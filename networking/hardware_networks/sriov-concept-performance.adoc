Learn about SR-IOV performance optimization techniques, including DPDK, hardware offloading, NUMA-aware scheduling, and interface tuning. Understand when each technique applies and what performance gains to expect.

[IMPORTANT]
====
These are advanced performance optimization techniques. Before exploring performance optimizations:

* Complete the xref:../../networking/hardware_networks/sriov-getting-started.adoc#sriov-getting-started[getting started guide]

* Identify specific performance bottlenecks in your workload
* Verify your hardware supports the optimization techniques
====

[id="sriov-concept-performance-overview_{context}"]
== SR-IOV performance capabilities

SR-IOV provides high-performance networking by:

* Bypassing the kernel network stack for reduced latency
* Providing direct hardware access to network adapters
* Eliminating virtualization overhead in network I/O
* Enabling hardware acceleration features

Additional optimizations can further improve performance for specific workloads.

[id="sriov-concept-performance-dpdk_{context}"]
== Data Plane Development Kit (DPDK)

DPDK is a framework for fast packet processing that:

* Moves packet processing from the kernel to user space
* Uses polling instead of interrupts to reduce latency
* Processes packets in batches for higher throughput
* Supports zero-copy packet handling

Use DPDK when:

* Your workload processes millions of packets per second
* You need single-digit microsecond latency
* You run network functions like routers, firewalls, or load balancers
* Your application can dedicate CPU cores to packet processing

Trade-offs:

* Requires dedicating CPU cores to polling (uses CPU even when idle)
* More complex application code
* Higher memory requirements

[id="sriov-concept-performance-hardware-offload_{context}"]
== Hardware offloading

Hardware offloading moves packet processing tasks from the CPU to the network adapter:

* Checksum calculation
* Segmentation and reassembly
* Encryption/decryption
* Packet filtering

Use hardware offloading when:

* Your workload is CPU-bound
* You want to improve throughput without using more CPU
* Your network adapter supports offloading features
* You process large amounts of network traffic

Trade-offs:

* Requires compatible network hardware
* May not work with all packet types or protocols
* Debugging can be more difficult

[id="sriov-concept-performance-numa_{context}"]
== NUMA-aware scheduling

Non-Uniform Memory Access (NUMA) scheduling places pods on the same NUMA node as the SR-IOV device:

* Reduces memory access latency
* Increases memory bandwidth
* Improves cache efficiency

Use NUMA-aware scheduling when:

* Your nodes have multiple NUMA nodes
* Your workload is memory-intensive
* You need consistent, predictable performance
* You already use CPU Manager and Topology Manager

Trade-offs:

* More complex pod scheduling
* May reduce cluster utilization
* Requires understanding your hardware topology

[id="sriov-concept-performance-interface-tuning_{context}"]
== Interface tuning

Interface-level tuning adjusts kernel parameters for the SR-IOV interface:

* Ring buffer sizes
* Interrupt coalescing
* TCP/UDP parameters
* Queue settings

Use interface tuning when:

* Default settings do not match your workload
* You experience packet drops under load
* You need to balance latency vs. throughput
* Other optimizations are not sufficient

Trade-offs:

* Requires deep networking knowledge
* Wrong settings can reduce performance
* Effects are workload-specific

[id="sriov-concept-performance-which-to-use_{context}"]
== Choosing the right optimizations

Start with baseline SR-IOV (no additional optimizations) and measure performance. Add optimizations only if measurements show they help your specific workload.

Typical optimization progression:

. Start with basic SR-IOV networking
. Add hardware offloading if CPU-bound
. Add NUMA scheduling if memory latency matters
. Add interface tuning if you see packet drops
. Add DPDK only if you need extreme performance and can dedicate CPU cores
