[role="_abstract"]
Learn how to achieve high availability with SR-IOV networks using interface bonding and Link Aggregation Control Protocol (LACP) monitoring. Understand the trade-offs and when to use each approach.

[IMPORTANT]
====
These are advanced high availability techniques. Before implementing HA for SR-IOV:

* Complete the xref:../../networking/hardware_networks/sriov-getting-started.adoc#sriov-getting-started[getting started guide]
* Understand your application's availability requirements
* Verify your network infrastructure supports bonding/LACP
* Consider whether pod-level or cluster-level HA is more appropriate
====

[id="sriov-concept-ha-overview_{context}"]
== High availability with SR-IOV

SR-IOV provides high-performance networking, but a single SR-IOV interface creates a single point of failure:

* If the physical network adapter fails, the pod loses connectivity
* If the network switch or cable fails, the pod loses connectivity
* Node maintenance requires pod eviction

High availability techniques address these risks by providing redundant network paths.

[id="sriov-concept-ha-bonding_{context}"]
== Pod-level bonding

Pod-level bonding combines multiple SR-IOV interfaces inside a single pod:

* Creates a bonded interface from two or more SR-IOV VFs
* The bond remains active if one interface fails
* Bonds operate in active-backup or other modes
* Provides network redundancy at the pod level

Use pod-level bonding when:

* Your pod requires continuous network connectivity
* You can tolerate brief interruptions during failover (seconds)
* You have multiple SR-IOV devices available per node
* Your application cannot handle reconnections

How it works:

. Your pod requests multiple SR-IOV VFs (for example, 2)
. A bonding CNI plugin creates a bonded interface inside the pod
. The bond monitors the interfaces and fails over if one goes down
. The application uses the bonded interface and sees a single network device

Trade-offs:

* Uses more VFs per pod (reduces total pod capacity)
* Requires bonding CNI configuration
* Failover takes seconds, not milliseconds
* More complex pod specifications

[id="sriov-concept-ha-lacp_{context}"]
== LACP monitoring

Link Aggregation Control Protocol (LACP) monitoring provides switch-level high availability:

* Detects switch or link failures using LACP protocol
* Works with network switches that support LACP
* Provides faster failover than basic bonding
* Enables active-active configurations for higher bandwidth

Use LACP monitoring when:

* Your network switches support LACP
* You need faster failover than basic bonding
* You want active-active load balancing across interfaces
* Your network team can configure LACP on switches

How it works:

. Configure your network switches for LACP
. Configure SR-IOV interfaces with LACP monitoring
. The SR-IOV interface and switch exchange LACP frames
. If LACP frames stop, the interface detects a failure immediately

Trade-offs:

* Requires LACP-capable network switches
* Requires coordination with network administrators
* More complex switch configuration
* LACP overhead on the network

[id="sriov-concept-ha-cluster-level_{context}"]
== Cluster-level HA versus pod-level HA

Not all workloads need pod-level HA. Consider cluster-level HA first:

Cluster-level HA (Kubernetes standard):

* Kubernetes restarts the pod on another node if a node fails
* Simpler configuration
* Better resource utilization
* Suitable for most stateless workloads

Pod-level HA (bonding):

* Pod survives single interface failure without restarting
* Required for stateful workloads that cannot restart
* Uses more resources
* More complex configuration

Use pod-level HA only when:

* Your application maintains long-lived connections that cannot reconnect
* Your application has slow startup or initialization
* You need to meet strict SLA requirements
* The cost of pod restart exceeds the cost of redundant VFs

[id="sriov-concept-ha-comparison_{context}"]
== Comparing HA approaches

[cols="1,3,3,3",options="header"]
|===
|Approach |Failover time |Complexity |When to use

|No HA
|Pod restart (minutes)
|Simple
|Stateless apps, dev/test, cost-sensitive deployments

|Basic bonding
|Seconds
|Medium
|Stateful apps, long-lived connections

|LACP bonding
|Sub-second
|High
|Mission-critical apps, strict SLA requirements

|Cluster-level HA
|Pod restart (minutes)
|Simple
|Most production workloads
|===
