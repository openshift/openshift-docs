[id="sriov-concept-architecture-overview_{context}"]
== SR-IOV overview

SR-IOV is a hardware specification that allows a single physical network adapter to be divided into multiple virtual network devices. Each virtual device can be assigned directly to a pod, bypassing the kernel networking stack for maximum performance.

[id="sriov-concept-architecture-pf-vf_{context}"]
== Physical functions and virtual functions

SR-IOV network devices expose two types of PCIe functions:

Physical Function (PF)::
The full-featured network adapter as seen by the host operating system. The PF manages the underlying hardware and creates Virtual Functions.

Virtual Function (VF)::
A lightweight instance of the network adapter that provides network I/O. Each VF operates as an independent network device that can be assigned to a pod.

For example, a physical network adapter might be recognized as device `ens1f0` on the host (the PF). You configure the SR-IOV Network Operator to create 8 VFs from this PF. OpenShift then allocates these VFs to pods as independent network interfaces.

[id="sriov-concept-architecture-drivers_{context}"]
== VF drivers and device types

The SR-IOV network device driver determines how the VF is exposed in the container:

`netdevice` driver::
Exposes the VF as a regular kernel network device in the container's network namespace. Use this for most workloads that need high-performance networking. The interface appears as `net1`, `net2`, etc., and works with standard Linux networking tools like `ip` and `ping`.

`vfio-pci` driver::
Exposes the VF as a character device mounted in the container. Use this for workloads that require direct hardware access without kernel involvement, such as Data Plane Development Kit (DPDK) applications.

You specify the driver type in the `SriovNetworkNodePolicy` custom resource when configuring SR-IOV devices.
