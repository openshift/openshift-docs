#### REQ-002: API Changes for GA Readiness
- **Source**: [ROX-30352](https://redhat.atlassian.net/browse/ROX-30352)
- **Summary**: The VM API requires significant cleanup before GA. This includes removing mutating endpoints (CREATE, UPDATE, DELETE), ensuring field consistency between storage and v2 protos, and standardizing naming conventions.
- **User impact**: API users will lose access to CREATE, UPDATE, DELETE operations for VMs. The API will be read-only for VM vulnerability data. Field names and data structures may change to ensure consistency.
- **Documentation action**:
  - [ ] Create `modules/vm-api-reference-411.adoc` (REFERENCE) - Document complete v2 API
  - [ ] Create `modules/vm-api-migration-guide.adoc` (PROCEDURE) - Guide for migrating from Tech Preview API
  - [ ] Update existing API documentation - Remove mutating operation references
- **Acceptance criteria**:
  - [ ] All API endpoints documented with request/response examples
  - [ ] Removed operations clearly documented with migration alternatives
  - [ ] Field mappings between storage and v2 protos explained
  - [ ] CVSSv3 conversion functions documented
  - [ ] Query and filtering capabilities documented
- **References**:

  - [ROX-27664](https://redhat.atlassian.net/browse/ROX-27664): Original API design and implementation (4.9)

#### REQ-003: VSOCK Networking Prerequisites
- **Source**: [ROX-32660](https://redhat.atlassian.net/browse/ROX-32660) - External blockers: CNV-68501, CNV-64172
- **Summary**: VM vulnerability scanning depends on VSOCK (Virtual Socket) networking support in OpenShift Virtualization. This is currently a developer/tech preview feature that must be GA for VM scanning to be GA.
- **User impact**: Users must enable VSOCK support in OpenShift Virtualization and configure VM resources with `autoattachVSOCK: true`. This is a critical prerequisite that may have namespace and network isolation implications.
- **Documentation action**:
  - [ ] Update `modules/vm-scanning-requirements.adoc` - Add detailed VSOCK prerequisites
  - [ ] Update `modules/configuring-vm-scanning.adoc` - Add troubleshooting for VSOCK issues
  - [ ] Create `modules/vsock-networking-for-vm-scanning.adoc` (CONCEPT) - Explain VSOCK architecture
  - [ ] Update `modules/troubleshooting-vm-scanning.adoc` - Add VSOCK connectivity troubleshooting
- **Acceptance criteria**:
  - [ ] VSOCK architecture and data flow explained
  - [ ] Prerequisites for enabling VSOCK documented
  - [ ] Namespace and networking requirements clarified
  - [ ] Troubleshooting steps for VSOCK connectivity issues
  - [ ] Version requirements for OpenShift Virtualization documented
- **References**:
  - [ROX-32660 Description](https://redhat.atlassian.net/browse/ROX-32660): References CNV-68501 (VSOCK networking namespaces)
  - [ROX-32660 Description](https://redhat.atlassian.net/browse/ROX-32660): References CNV-64172 (Enabling VSOCK)

### External Dependencies

The JIRA ticket explicitly lists external blockers for GA:
- **CNV-68501**: VSOCK and networking namespaces
- **CNV-64172**: Enabling VSOCK

These OpenShift Virtualization prerequisites must be GA before VM vulnerability scanning can fully transition to GA. Documentation must clearly state these dependencies.

### Feature Flag Status

Current documentation shows the feature requires `ROX_VIRTUAL_MACHINES=true` environment variable. It's unclear whether this will remain in GA or be removed. Documentation updates depend on final GA implementation.

### API Stability Questions

ROX-30352 lists many API TODO items. The ticket asks: "is it unclear to me if we want to try and share vulnerability and scan protobufs between images, nodes, and virtual machines. It seems like we should, but there may be subtle differences that make it difficult."

This architectural decision affects API documentation structure and whether VM, Image, and Node vulnerability APIs should be documented together or separately.
