== Logging
There are various methods to view your logs in {product-rosa} (ROSA). Use the following procedures to forward the logs to AWS CloudWatch, look at the logs directly through the pod using `oc logs`, and use Kibana (EFK Stack) to search the logs.

.Prerequisites

[IMPORTANT]
====
ROSA is not preconfigured with a logging solution.
====

* Set up a logging solution before viewing your logs. See the xref:../../observability/logging/cluster-logging-deploying.adoc#cluster-logging-deploying[Installing logging] documentation for more information.

The cluster logging components are based on Fluentd, Elasticsearch (ES), and Kibana. Fluentd is deployed to each node in the cluster. It collects application logs and either writes them to ES or forwards them to CloudWatch. Kibana is a centralized, web-based user interface (UI) where users and administrators can create visualizations and dashboards with the aggregated data.

.Additional resources
* xref:../../observability/logging/cluster-logging.adoc#cluster-logging[Cluster logging]

=== Installing the logging add-on service

Install the logging add-on service to forward the logs to AWS CloudWatch.

//These steps were adopted from the Managed OpenShift Black Belts xref:../../cloud_experts_tutorials/cloud-experts-rosa-cloudwatch-sts.adoc#cloud-experts-rosa-cloudwatch-sts[tutorial].

. Run the following script to configure your ROSA cluster to forward logs to CloudWatch:
+
[NOTE]
====
Configuring ROSA to send logs to CloudWatch goes beyond the scope of this tutorial. Integrating with AWS and enabling CloudWatch logging are important aspects of ROSA, so a script has been included to simplify the configuration process. The script will automatically set up AWS CloudWatch. You can examine the script to understand the steps involved.
====
+
[source,terminal]
----
curl https://raw.githubusercontent.com/openshift-cs/rosaworkshop/master/rosa-workshop/ostoy/resources/configure-cloudwatch.sh | bash
----
+
.Example output
+
[source,terminal]
----

clusterlogging.logging.openshift.io/instance created
----

=== Outputting the data to the streams and logs
. Output a message to `stdout`.
.. Click *Home* and then click the message box for *Log Message (stdout)*.
.. Write a message to output to the `stdout` stream.
.. Click *Send Message*.
+
image:cloud-experts-deploying-application-logging-ostoy-stdout.png[]
+
. Output a message to `stderr`.
.. Click the message box for *Log Message (stderr)* and write a message to output to the `stderr` stream.
.. Click *Send Message*.
+

=== Viewing the application logs using the `oc` command
. Enter the following command in the command line interface (CLI) to retrieve the name of your frontend pod:
+
[source,terminal]
----
$ oc get pods -o name
----
+
.Example output
+
[source,terminal]
----
pod/ostoy-frontend-679cb85695-5cn7x <1>
pod/ostoy-microservice-86b4c6f559-p594d
----
<1> The pod name is `ostoy-frontend-679cb85695-5cn7x`.
+  
. Run the following command to see both the `stdout` and `stderr` messages:
+
[source,terminal]
----
oc logs ostoy-frontend-679cb85695-5cn7x
----
+
.Example output
+
[source,terminal]
----
$ oc logs ostoy-frontend-679cb85695-5cn7x
[...]
ostoy-frontend-679cb85695-5cn7x: server starting on port 8080
Redirecting to /home
stdout: All is well!
stderr: Oh no! Error!
----

=== Viewing the logs with CloudWatch
. Navigate to *CloudWatch* on the AWS web console.
. In the left menu, click *Logs* and then *Log groups* to see the different groups of logs. You should see 2 groups. One for `<cluster-name>-XXXXX-application` and one for `<cluster-name>-XXXXX-infrastructure`.
+
image:cloud-experts-deploying-application-logging-cw.png[]
+
. Click `<cluster-name>-XXXXX.application`.
. Click the log stream for the frontend pod.
+
image:cloud-experts-deploying-application-logging-logstream2.png[]
+
. Filter for `stdout` and `stderr`.
. Expand the row to show the messages you entered earlier and other pertinent information.
+
image:cloud-experts-deploying-application-logging-stderr.png[]
+
. Enter "microservice" in the search bar to see other messages in your logs.
. Expand one of the entries to see the color the frontend pod received from microservice and which pod sent that color to the frontend pod.
+
image:cloud-experts-deploying-application-logging-messages.png[]

=== Viewing the logs with Kibana

[NOTE]
====
In order to use EFK, you must successfully complete the steps outlined in xref:../../observability/logging/cluster-logging-deploying.adoc#cluster-logging-deploying[Installing {logging}].
====

. Obtain the route for the Kibana console by running the following command:
+
[source,terminal]
----
oc get route -n openshift-logging
----
+
. Open a new browser tab and paste the URL in the browser.
. Define the index patterns. See xref:../../observability/logging/cluster-logging-deploying.adoc#cluster-logging-deploying[Installing {logging}] for further instructions.
+
image:cloud-experts-deploying-application-logging-kibana.png[]

=== Familiarization with the data
The main part of the console contains entries for the `stdout` and `stderr` messages. It also has information about each entry such as:

* namespace names
* pod names
* host ip addresses
* timestamps
* log levels
* messages

image:cloud-experts-deploying-application-logging-logoutput.png[]

There is data from multiple sources and multiple messages. These can be expanded for more details.

image:cloud-experts-deploying-application-logging-logdata.png[]

=== Filtering Results
Look for any errors in the app. You may need to filter to find errors.

Create a filter to find the error message for `stderr`.  

. Click *Add a filter*.
. Select or type "level" for *Fields...*.
. Select "is" for *Operators*.
. Type "err" in *Value...*.
. Click *Save*.
+
image:cloud-experts-deploying-application-logging-filtererr.png[]
+
The output should be a single row that contains the error message.
+
image:cloud-experts-deploying-application-logging-erronly.png[]
+
If too much time has elapsed since you output the messages to the `stdout` and `stderr` streams, nothing is returned. Set the time frame for the filter based on how much time has elapsed since the outputs. 
+
. In the Kibana console, click *Last 15 minutes* in the top right.
. Click *Quick*.
. Click *Last 1 hour*.
