== Scaling

You can manually scale your pods or you can automatically scale your pods by using the Horizontal Pod Autoscaler (HPA). You can also scale your cluster nodes.

=== Manual pod scaling

You can manually scale the pods and nodes of your applications:

* by changing your replicaset and deployment definition 
* by using the command line
* by using the web console
* by changing your *deployment* definition in the `ostoy-frontend-deployment.yaml`

This workshop starts by using only one pod for the microservice. By using one pod, the Kubernetes Replication Controller strives to keep one pod alive. You can also define pod autoscaling by using the link:https://docs.openshift.com/container-platform/latest/nodes/pods/nodes-pods-autoscaling.html[Horizontal Pod Autoscaler](HPA). The HPA is based on workload, and it expands past defined limits if needed.

.Prerequisites

* Create a cluster with the xref:../../cloud_experts_tutorials/cloud-experts-deploying-application/cloud-experts-deploying-application-lab-overview.adoc[OSToy microservice deployed].
* Download the link:https://www.rosaworkshop.io/ostoy/yaml/ostoy-microservice-deployment.yaml[ostoy-microservice-deployment.yaml].

.Procedure 

. In the OSToy app, click the *Networking* tab in the left menu.
. In the "Intra-cluster Communication" section, there is a box that randomly changes colors. Inside the box, you see the microservice's pod name. There is only one box shown because there is only one microservice pod. 
+
image::deploy-scale-network.png[HPA Menu]
+ 
Now, scale up the microservice pods and observe the number of boxes change.

. Run the following command to confirm that there is only one pod running on the microservice:
+
[source,terminal,highlight='4']
----
$ oc get pods
----
+
.Example output
[source,terminal]
----
NAME                                  READY     STATUS    RESTARTS   AGE
ostoy-frontend-679cb85695-5cn7x       1/1       Running   0          1h
ostoy-microservice-86b4c6f559-p594d   1/1       Running   0          1h
----

. Now, scale the application by using the CLI or using the Web UI:
+
** Run the following command increases the number of pods from `1` to `3`:
+
[source,terminal]
----
$ oc scale deployment ostoy-microservice --replicas=3
----
+
** In the OSToy application, from the left menu, click *Workloads > Deployments > ostoy-microservice*.  On the left side of the page, you see a blue circle with a "1 Pod" label in the middle. Selecting the arrows next to this scales the number of pods. Select the up arrow to `3`.
+
image::deploy-scale-uiscale.png[UI Scale]

.Verification

Check your pod counts with the following methods:

* In the web UI, select *Workloads > Deployments > ostoy-microservice*.
+
image::deploy-scale-verify-workload.png[Verify the workload pods]

* From the CLI, run the follow command to confirm that you are using three pods:
+
[source,terminal]
----
$ oc get pods
----
+
.Example output
[source,terminal]
----
NAME                                  READY     STATUS    RESTARTS   AGE
ostoy-frontend-679cb85695-5cn7x       1/1       Running   0          1h
ostoy-microservice-86b4c6f559-p594d   3/3       Running   0          1h
----

* You can also confirm that there are three pods being used by selecting **Networking** in the left menu of the OSToy app. There should be three colored boxes for the three pods.

=== Pod Autoscaling

{product-title} offers a link:https://docs.openshift.com/container-platform/latest/nodes/pods/nodes-pods-autoscaling.html[Horizontal Pod Autoscaler] (HPA). The horizontal pod autoscaler (HPA) uses metrics to decide if increase or decrease the scale your cluster's worker pods.

.Procedure

. In the web UI, select *Auto Scaling* in the left menu.
+
image::deploy-scale-hpa-menu.png[HPA Menu]

. Create the Horizontal Pod Autoscaler by running the following command:
+
[source,terminal]
----
$ oc autoscale deployment/ostoy-microservice --cpu-percent=80 --min=1 --max=10
----
+
This command creates an HPA that maintains between 1 and 10 replicas of the pods controlled by the *ostoy-microservice* deployment. HPA increases and decreases the number of replicas, through the deployment, to keep an average CPU use across all pods of 80%. There is an average CPU usage of 40 millicores since each pod requests 50 millicores.

. On the *Auto Scaling* page, select the *Increase the load* button. 
+
[IMPORTANT]
====
This action generates some CPU intensive calculations, the page can become unresponsive. This is an expected response. Click the *Increase the Load* button only once. For more information about the process, see the link:https://github.com/openshift-cs/ostoy/blob/master/microservice/app.js#L32[microservice's GitHub repository]. 
====
+
After a few minutes, the new pods display on the page, as represented by colored rectangles.
+
[NOTE]
====
The page can experience a bit of lag.
====

.Verification

Check your pod counts with the following methods:

* In the OSToy application's web UI, see the remote pods box:
+
image::deploy-scale-hpa-mainpage.png[HPA Main]
+
Since there is only one pod, increasing the workload should trigger an increase of nodes.
+
* In the CLI, run the following command:
+
[source,terminal]
----
oc get pods --field-selector=status.phase=Running | grep microservice
----
+
.Example output
+
[source,terminal]
----
ostoy-microservice-79894f6945-cdmbd   1/1     Running   0          3m14s
ostoy-microservice-79894f6945-mgwk7   1/1     Running   0          4h24m
ostoy-microservice-79894f6945-q925d   1/1     Running   0          3m14s
----

* You can also verify autoscaling from the {cluster-manager} 
+
. In the OpenShift web console left menu, click *Observe > Dashboards*.
. In the dashboard, select *Kubernetes / Compute Resources / Namespace (Pods)* and your namespace *ostoy*.
+
image::deploy-scale-hpa-metrics.png[Select metrics]
+
. A graph appears showing your resource usage across CPU and memory. The top graph will show recent CPU consumption per pod and the lower graph indicates memory usage. The following lists the call outs in the graph: 
.. As soon as the load started to increase (A), 
.. two new pods started to spin up (B, C). 
.. The thickness of each graph is its CPU consumption indicating which pods handled more load. 
.. You can also see that the load decreased (D), after which, the pods were spun back down.
+
image::deploy-scale-metrics.png[Select metrics]

=== Node Autoscaling

{product-title} allows you to use link:https://docs.openshift.com/rosa/rosa_cluster_admin/rosa_nodes/rosa-nodes-about-autoscaling-nodes.html[node autoscaling]. In this scenario, you will create a new project with a job that has a large workload that this cluster cannot handle. With autoscaling enabled, when your load is larger than you can currently handle, it should force the cluster to automatically create new nodes to handle the load.

.Prerequisites

* Autoscaling is enabled on your machine pools.

.Procedure

. Create a new project called `autoscale-ex` by running the following command:
+
[source,terminal]
----
$ oc new-project autoscale-ex
----

. Create the job with the following command:
+
[source,terminal]
----
$ oc create -f https://raw.githubusercontent.com/openshift-cs/rosaworkshop/master/rosa-workshop/ostoy/yaml/job-work-queue.yaml
----
+

. Allow the pods some time to create and then, run the following command to see the pods:
+
[source,terminal]
----
$ oc get pods
----
+
.Example output
+
[source,terminal]
----

work-queue-5x2nq-57zpt   0/1     Pending   0          10s
