= {oadp-short} 1.3 performance and use cases
include::_attributes/common-attributes.adoc[]
:context: oadp-performance-1-3

toc::[]

The objective of this performance testing was to run regression test cases for backup and restore performance of {oadp-first} 1.3 using the plugin types Kopia and CSI.

In addition to regular backup and restore flows, native data mover, and incremental backup were also tested.

Overall, the majority of CSI and Restic flows showed comparable performance to the previous {oadp-short} 1.2, including the Velero built-in DataMover.

[id="oadp-1-3-testing-summary_{context}"]
== {oadp-short} 1.3 performance testing summary

The main performance-related difference for Datamover in 1.2 vs 1.3 is concurrency limited to the number of workers for VBDM, which is a known limitation.

In terms of backup duration, Kopia performed slightly better than Restic in general.

Following the initial backup, Kopia showed reduced time for subsequent backups.

The biggest backup and restore of a 4TB PV system, of which 3TB was used, was done successfully using Kopia, Restic, and CSI.  

The maximum number of pods within a single namespace that have been tested for backup/restore is 5000 pods, which were successfully completed using Kopia, Restic, and Container Storage Interface (CSI).

The maximum number of namespaces tested in a single backup and single restore was 200. The maximum number of namespaces tested was 800 using the Kopia plugin.

[id="oadp-1-3-recommendations_{context}"]
== {oadp-short} 1.3 recommendations

The following recommendations are based on observations of performance seen during scale and performance testing. It is important to note that backup and restore resources can be impacted by the plugin type, the number of resources related to that backup or restore, and the respective data contained within persistent volumes (PVs) related to those resources.

.Adjust {oadp-first} Data Protection Applicaion API (DPA) pod resources & timeouts for the Container Storage Interface (CSI) and NodeAgent

The majority of average use cases showed acceptable performance when the following resources and changes were made to the DPA:

* Velero:
** CPU request changed to `200m`, and max limit to `1`
** Mem request changed to `256MiB`, max limit to `1024MiB`

* NodeAgent
** CPU request changed to `1000m`, max limit to `2`
** Mem request changed to `16GiB`, max limit to `32GiB`
** NodeAgent timeout value updated from default of `60m` to `900m`

[NOTE]
====
Please note that the NodeAgent (Restic / Kopia ) resource usage corresponds to the amount of data, and type of data, for example, many small files, or a large amount of data can cause Restic or Kopia to used large amounts of resources.
====
