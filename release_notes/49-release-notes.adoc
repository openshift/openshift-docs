//ROX-29160
[id="redhat-images-signed-release-key_{context}"]
=== New default system policy for Red Hat image signing

A new default policy, "Red Hat images must be signed by a Red Hat release key", is now available. This policy ensures that Red Hat images are signed by Red Hat's link:https://access.redhat.com/security/team/key[Release Key 3 product signing key]. In addition to ensuring supply chain provenance, this default policy serves as an example of using the "Image Signature" field and combining it with other criteria.

For more information, see xref:../operating/manage_security_policies/security-policy-reference.adoc#high-sev-security-policies_security-policy-reference[High severity security policies].

=== {product-title-short} integration with the ServiceNow Container Vulnerability Response Application is generally available (GA)

{product-title-short} integration with the ServiceNow Container Vulnerability Response Application is now GA in the  link:https://store.servicenow.com/store/app/edea7344476072502ec7c1c4f16d4343#summary[ServiceNow Marketplace]. {product-title-short} integration with ServiceNow populates rich container image vulnerability data from {product-title-short} in the ServiceNow Container Vulnerability Response Module. //ROX-29959
[id="declarative-config-m2m_{context}"]
=== Declarative configuration for machine-to-machine access configuration

You can now declaratively configure machine-to-machine (M2M) OIDC authentication. To configure M2M authentication resources, you create YAML files that contain configuration information. These files are used to create a config map or secret that is added to Central by using a mount point during installation of the {product-title-short} Central resource. 

For more information, see xref:../configuration/declarative-configuration-using.adoc#declarative-config-example-short-lived-token[Declarative configuration short-lived token example].

//ROX-20262
[id="certificate-rotation-operator-managed-clusters_{context}"]
=== Automatic certificate rotation for Operator-managed clusters

With this update, the automatic certificate renewal feature is introduced for Operator-managed Central and secured clusters. The Central CA certificate, which has a validity of 5 years, will now be automatically rotated. This enhancement simplifies the management of large cluster installations, as manually upgrading certificates can be a significant task.

[NOTE]
====
Automatic certificate rotation remains partially supported for Helm-installed secured clusters. These clusters can connect to Central with the rotated CA but their certificates remain signed by the older CA.
====

For more information, see xref:../configuration/reissue-internal-certificates.adoc#reissue-internal-certificates[Reissuing internal certificates].

//ROX-28029
[id="scannerv4-sbom-delegated-scans_{context}"]
=== SBOM creation is generally available and supports delegated scanning

With this release, using {product-title-short} to generate a Software Bill of Materials (SBOM) from scanned container images is generally available. SBOM generation includes images scanned with the {product-title-short} delegated scanning feature. These SBOMs provide a detailed overview of all software components, dependencies, and libraries within an application. The SBOMs created by {product-title-short} are of the link:https://www.cisa.gov/sites/default/files/2023-04/sbom-types-document-508c.pdf["Analyzed"] type and conform to the SPDX 2.3 specification.

For more information, see xref:../operating/manage-vulnerabilities/scanner-generate-sbom.adoc#scanner-generate-sbom[Generating SBOMs from scanned images].

//ROX-30654
[id="smtp-ehlo-helo-hostname_{context}"]
=== SMTP EHLO/HELO hostname field supported in the {product-title-short} email notifier

With this update, you can add the SMTP EHLO/HELO hostname in the {product-title-short} email notifier, which is often required in secure enterprise environments. This feature aligns with the strict SMTP policies enforced by mail relay servers, which often reject connections with generic or unauthorized EHLO/HELO values. 

For more information, see xref:../integration/integrate-with-email.adoc#integrate-with-email[Integrating with email].

//ROX-30087
[id="m2m-token-exchange-jwt"]
=== Support for machine-to-machine (M2M) token exchange for external JSON web token (JWT) issuers

With this update, {product-title-short} supports M2M token exchange with external JWT token issuers. {product-title-short} performs a token exchange between OpenID Connect (OIDC) tokens and {product-title-short} and uses role mapping to allow access to the {product-title-short} API. This is beneficial for third-party clients such as Prometheus server and entities such as the {ocp} console that do not support the full M2M token exchange flow. This feature allows custom Prometheus installations to scrape the custom metrics from the API endpoint by using Kubernetes service account tokens.

//ROX-28326
[id="prometheus-custom-metrics_{context}"]
=== {product-title-short} supports custom Prometheus metrics

With this update, you can now track statistics and trends over time for key metrics reported by {product-title-short}, such as the number of CVEs and violations. You can configure custom Prometheus metrics to be exposed on the API endpoint, enabling you to monitor vulnerability and violation trends over time more effectively. This data can be used in tools such as Perses to create dashboards for {ocp} or Kubernetes systems that provide information about {product-title-short}. 

For more information, see xref:../configuration/monitor-acs.adoc#prometheus-server-metrics_monitor-acs[Monitoring RHACS by using a Prometheus server].

1. API token authentication is deprecated. The corresponding cloud source integration now uses service accounts for authentication.

2. The current hierarchical implementation for defining Collections is deprecated and will be replaced by a more comprehensive search-based definition in a future release.

3. The Google Container Registry integration is deprecated in response to the deprecation of Container Registry. You can use the Artifact Registry as a registry replacement and Scanner V4 as a scanner replacement.
+
For more information, see link:https://cloud.google.com/artifact-registry/docs/transition/transition-from-gcr[Transition from Container Registry] (Google Cloud documentation).

4. Kernel support packages and driver download functionality are deprecated.

5. The manifest installation method is now deprecated and will be removed in the future. Manifest installation is currently done using the `roxctl {central,sensor,scanner} generate` command in the CLI, or by choosing the legacy installation method in the {product-title-short} portal. Use the Operator or Helm installation methods.

6. A feature flag controls this API object, and you can enable or disable this API object by using the `ROX_VULN_MGMT_LEGACY_SNOOZE` environment variable.

7. The format for specifying duration in JSON requests to `v1/nodecves/suppress`, `v1/clustercves/suppress`, and `v1/imagecves/suppress` has been changed to the ProtoJSON format.
Only a numeric value representing seconds with optional fractional seconds for nanosecond precision and followed by the `s` suffix is supported.
+
For example, `0.300s`, `-5400s`, or `9900s`. The previously valid time units of `ns`, `us`, `µs`, `ms`, `m`, and `h` are no longer supported.

8. The *Vulnerability Management* -> *Dashboard* view is deprecated and is planned to be removed in a future release. You can use the *User workload vulnerabilities*, *Exception management*, *Platform vulnerabilities*, and *Node CVEs* views as alternatives.
