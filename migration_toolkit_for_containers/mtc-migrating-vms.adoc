= Migrating virtual machine storage
include::_attributes/common-attributes.adoc[]
:context: mtc-migrating-vms

toc::[]

The migration can occur offline, when the VM is turned off. It can also occur online, when the VM is running.

For KubeVirt virtual machines (VMs), the primary use case for VM storage migration is migration from one storage class to another. For example, you might migrate VM storage if you are rebalancing between different storage providers or if there is new storage available that is better suited to the workload running inside the VM.

== Supported persistent volume actions

.Supported persistent volume actions
[width="100%",cols="31%,33%,36%",options="header",]
|===
|Action
|Supported
|User-initiated steps

|Copy
|Yes
a|
. Create a new persistent volume (PV) in the same namespace.
. Copy data from the source PV to the target PV, and modify the VM definition to point to the new PV.
** If the `liveMigrate` flag is set, the VM migrates live.
** If the `liveMigrate` flag is not set, the VM shuts down, the source PV contents are copied to the target PV, and the VM is restarted.

|Move
|No
|This action is not supported.
|===

[id="mtc-vm-storage-prerequisites-for-migration"]
== Prerequisites

Before migrating virtual machine storage, the following prerequisites must be met:

. Install xref:../virt/install/installing-virt.adoc#installing-virt-operator_installing-virt[{VirtProductName} Operator]. In order to support storage live migration, you need to deploy {VirtProductName} version 4.17. Earlier versions do not support live storage migration. You also need to configure KubeVirt to enable storage live migration according to the xref:../virt/live_migration/virt-configuring-live-migration.adoc#virt-configuring-live-migration-limits_virt-configuring-live-migration[Configuring live migration].
+
In {VirtProductName} 4.17.0, not all the required feature gates are enabled.
As a result, you have to enable the feature gate before being able to use the storage live migration feature.
+
To work around this, you can use a JSON patch:
+
[source,terminal]
----
$ oc annotate --overwrite -n openshift-cnv hco kubevirt-hyperconverged kubevirt.kubevirt.io/jsonpatch='[ {"op": "add", "path": "/spec/configuration/developerConfiguration/featureGates/-", "value": "VolumesUpdateStrategy"}, {"op": "add", "path": "/spec/configuration/developerConfiguration/featureGates/-", "value": "VolumeMigration"} ]'
----
+
[WARNING]
====
This should not be done in a production cluster, as it takes the  cluster out of support. Red{nbsp}Hat does not support clusters with this annotation.

When this annotation is created, a warning is given.
====

These deployments add the appropriate Custom Resource Definitions (CRDs) used by the migration controller to manipulate the VMs. For more details, see xref:../migrating_from_ocp_3_to_4/advanced-migration-options-3-4.adoc#migration-controller-options_advanced-migration-options-3-4[Migration controller options].

[NOTE]
====
If the `mig-controller` pod was started before {VirtProductName} was installed, the migration controller does not automatically see that the {VirtProductName} CustomResourceDefinition (CRD) is installed.

You have to restart the `mig-controller` pod in the `openshift-migration` namespace after installing OpenShift Virtualization.
====

[width="100%",cols="50%,50%",options="header",]
|===
|Resource |Purpose
|`MigCluster`
|Represents the cluster to use when doing the storage migration

|`StorageClass`
|The storage class, there should be at least two

|`VirtualMachine`
|A virtual machine definition, installed by KubeVirt

|`VirtualMachineInstance`
|A running virtual machine, installed by KubeVirt

|`DataVolume`
|A definition on how to populate a Persistent Volume with a virtual machine disk, installed by CDI
|===
