= Installing a two-node OpenShift cluster with fencing
include::_attributes/common-attributes.adoc[]
:context: install-tnf

toc::[]

You can deploy a two-node OpenShift cluster with fencing using either the installer-provisioned infrastructure (IPI) or the user-provisioned infrastructure (UPI) installation method. The following examples provide sample `install-config.yaml` configurations for both methods.

[id="sample-install-config-two-node-fencing-ipi_{context}"]
== Sample install-config.yaml for an IPI two-node cluster with fencing

You can use the following `install-config.yaml` configuration as a template for deploying a two-node OpenShift cluster with fencing using the installer-provisioned infrastructure (IPI) method:

.Sample `install-config.yaml` configuration for IPI two-node OpenShift cluster with fencing
[source,yaml]
----
apiVersion: v1
baseDomain: example.com
compute:
- name: worker
  replicas: 0 <1>
controlPlane:
  name: master
  replicas: 2 <2>
  fencing:
    credentials:
      - hostname: <control-0-hostname> <3>
        address: https://<redfish-api-url>
        username: <username>
        password: <password>
      - hostname: <control-1-hostname> <3>
        address: https://<redfish-api-url>
        username: <username>
        password: <password>
metadata:
  name: <cluster-name> <4>
featureSet: TechPreviewNoUpgrade <5>
platform:
  baremetal:
    apiVIPs:
      - <api_ip> <6>
    ingressVIPs:
      - <wildcard_ip> <6>
    hosts:
      - name: <control-0-hostname>
        role: master
        bmc:
          address: <bmc-address>
          username: <bmc-username>
          password: <bmc-password>
        bootMACAddress: <boot-mac>
      - name: <control-1-hostname>
        role: master
        bmc:
          address: <bmc-address>
          username: <bmc-username>
          password: <bmc-password>
        bootMACAddress: <boot-mac>
pullSecret: '<pull-secret>' <7>
sshKey: '<ssh-public-key>' <8>
----
<1> Set this field to `0` because a two-node fencing cluster does not include worker nodes.  
<2> Set this field to `2` for a two-node fencing deployment.  
<3> Provide BMC credentials for each control plane node. These credentials are required for node fencing and prevent split-brain scenarios.  
<4> The cluster name is used as a prefix for hostnames and DNS records.  
<5> Set this field to `TechPreviewNoUpgrade` to enable two-node OpenShift cluster deployments.  
<6> Virtual IPs for the API and Ingress endpoints. Ensure they are reachable by all nodes and external clients.  
<7> Contains credentials required to pull container images for the cluster components.  
<8> The SSH public key for accessing cluster nodes after installation.

== Sample install-config.yaml for an UPI two-node cluster with fencing

You can use the following `install-config.yaml` configuration as a template for deploying a two-node OpenShift cluster with fencing using the user-provisioned infrastructure (UPI) method:

.Sample `install-config.yaml` configuration for installing a UPI two-node OpenShift cluster with fencing
[source,yaml]
----
apiVersion: v1
baseDomain: example.com
compute:
- name: worker
  replicas: 0 <1>
controlPlane:
  name: master
  replicas: 2 <2>
  fencing:
    credentials:
      - hostname: <control-0-hostname> <3>
        address: https://<redfish-api-url>
        username: <username>
        password: <password>
      - hostname: <control-1-hostname> <3>
        address: https://<redfish-api-url>
        username: <username>
        password: <password>
metadata:
  name: <cluster-name> <4>
featureSet: TechPreviewNoUpgrade <5>
platform:
  none: {} <6>
pullSecret: '<pull-secret>' <7>
sshKey: '<ssh-public-key>' <8>
----
<1> Set this field to `0` because a two-node fencing cluster does not include worker nodes.  
<2> Set this field to `2` for a two-node fencing deployment.  
<3> Provide BMC credentials for each control plane node.  
<4> Cluster name is used as a prefix for hostnames and DNS records.  
<5> Enables two-node OpenShift cluster deployments.  
<6> Set the platform to `none` for UPI deployments. Bare metal hosts are pre-provisioned outside of the installer.  
<7> Contains credentials required to pull container images for the cluster components.  
<8> The SSH public key for accessing cluster nodes after installation.
