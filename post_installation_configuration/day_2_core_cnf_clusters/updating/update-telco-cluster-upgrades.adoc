= Upgrade Telco Core multi-node clusters

[role="_abstract"]
You can upgrade multi-node Telco Core clusters by using z-stream, y-stream, or Extended Update Support (EUS) to EUS approaches with GitOps and {cgu-operator-full} ({cgu-operator}).
Multi-node clusters require special considerations for control plane sequencing, worker node batching, and network connectivity during upgrades.

Multi-node Telco Core clusters typically include three or more control plane nodes and multiple worker nodes running network functions.
Understanding multi-node upgrade patterns ensures safe upgrades with minimal workload disruption.

== Multi-node cluster upgrade overview

Multi-node cluster upgrades differ from single-node cluster upgrades in several important ways:

* *Control plane rolling updates*: Control plane nodes upgrade one at a time in a rolling fashion.
The etcd cluster maintains quorum throughout the upgrade because you have three or more control plane nodes.
* *API server availability*: The API server might be briefly unavailable during individual control plane node reboots.
Most requests succeed, but some API calls might fail temporarily during control plane updates.
* *Worker node batching*: Worker nodes upgrade in batches controlled by the `maxUnavailable` setting in the MachineConfigPool.
For Telco workloads, set `maxUnavailable` to 1 to minimize disruption.
* *Network requirements*: Cluster network connectivity must remain available between nodes during rolling upgrades.
The CNI plugin must support version skew while control plane and worker nodes run different versions.

== Control plane upgrade sequencing

During multi-node cluster upgrades, the control plane updates according to this sequence:

. The Cluster Version Operator identifies the target version.
. Control plane nodes update one at a time in a rolling fashion.
. While one control plane node updates, the other control plane nodes continue serving API requests.
. The etcd cluster maintains quorum with the remaining healthy control plane nodes.
. After each control plane node completes its update, the next control plane node begins updating.

This rolling update pattern ensures the control plane remains available throughout the upgrade, although brief API disruptions might occur during individual node reboots.

== Worker node batching and pod disruption budgets

Worker nodes upgrade in batches determined by the `maxUnavailable` setting:

* *Default `maxUnavailable`*: 1 worker node or 33% of worker nodes, whichever is smaller
* *Recommended for Telco workloads*: `maxUnavailable: 1` to minimize disruption
* *Pod Disruption Budget interaction*: PDBs must allow at least one disruption, or node upgrades will be blocked

Configure Pod Disruption Budgets for all critical network functions:

[source,yaml]
----
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: cnf-workload-pdb
  namespace: telco-workload
spec:
  minAvailable: 2
  selector:
    matchLabels:
      app: cnf-workload
----

If you have 3 replicas, set `minAvailable: 2` to allow one pod disruption during node upgrades.
Do not set `minAvailable` equal to the total number of replicas, or the PDB will block node upgrades.

== Network connectivity during upgrades

Multi-node cluster upgrades require continuous network connectivity:

* *Cluster networking*: The CNI plugin (typically OVN-Kubernetes for Telco Core) must support version skew between nodes.
* *SR-IOV networks*: SR-IOV network attachments should remain available during worker node upgrades if workloads are spread across multiple nodes.
* *Control plane communication*: Worker nodes must maintain connectivity to control plane nodes throughout the upgrade.
* *Inter-node pod communication*: Pods on upgrading nodes are drained and rescheduled to other nodes.
Network policies and service connectivity must remain functional.

For Telco deployments, verify network latency requirements are met during rolling upgrades.

== Monitoring multi-node upgrades

Monitor these resources during multi-node cluster upgrades:

* *Cluster version*: `oc get clusterversion -w`
* *Cluster operators*: `oc get co`
* *Node status*: `oc get nodes -w`
* *Machine config pools*: `oc get mcp -w`
* *etcd quorum*: `oc get etcd`
* *{cgu-operator} CGU progress*: `oc get cgu <cgu_name> -w`

For multi-cluster upgrades, monitor the ClusterGroupUpgrade status to track progress across all target clusters.
