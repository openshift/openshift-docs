= Troubleshooting

:context: dist-tracing-tempo-troubleshooting

toc::[]

There are multiple ways on you can diagnistic and fix common problems with the TempoStack instance,

You can inspect the logs of the different components along with some useful metrics.

If the the TempoStack instance has no traces, there are two possible causes:
* TempoStack instance is not ingesting traces.
* There is an issue with the query path.

[id="getting-tempo-stack-logs_{context}"]
== Getting the TempoStack instance logs

You can get the logs from the Tempo component as follows.

.Procedure

. Localize the component you want to see the logs. You can do this by listing all the deployments on an specific namespace.

. Identify the pods belonging to that component.

. Watch for the logs using `oc logs` command or the web console to retrieve the logs.

[id="problems-ingesting-traces_{context}"]
== Problems ingesting traces

Use the following steps to troubleshoot when the TempoStack instance fails to ingest traces.

.Procedure

. Review the components in the ingestion path to determine which problem you are facing:
+
* The spans are not sent correctly.
* The spans are not sampled.

. Check for errors in the logs that point to unhealthy ingesters, which can result from failing due to out-of-memory (OOM) issues or scale-down events.

. Check the following metrics to verify that the TempoStack instance is receiving traces:
+
[source,yaml]
----
tempo_distributor_spans_received_total
tempo_ingester_traces_created_total
----
+
Both of these metrics must be greather than zero if the TempoStack instance is ingesting.
If one of them is wrong, review the following ?:

- If `tempo_distributor_spans_received_total` is `0`, the distributor is not reciving spans.
Troubleshoot by checking the network configuration, protocols, and ports of the applications that are attempting to send traces to the distributor.
- If `tempo_ingester_traces_created_total is `0`, there is a communcation problem between the ingester and the distributor. Troubleshoot by inspecting both logs to find errors.

[id="problems-quering-traces_{context}"]
== Problems quering traces

When data are successfully ingested into the TempoStack instance, it's time to look into potential issues with querying the data.


Check the logs of the `query-frontend`. The `query-frontend` pod runs with two containers (`query-frontend` and `querier`. If you see something similiar to the following errors, it means that there is an error in the query path.

[source,terminal]
----
level=info ts=XXXXXXX caller=frontend.go:63 method=GET traceID=XXXXXXXXX url=/api/traces/XXXXXXXXX duration=xxxx status=500
could not dial 10.X.X.X:3200 connection refused
----

The tempo-querier and tempo-queryfrontend were unable to communicate with each other. Review the logs of both services to determine if either has crashed. Additionally, check your network settings and policies to ensure proper communication between them.


Other useful metrics to review:

- `cortex_query_frontend_connected_clients` exposed by the `query-frontend`. If `cortex_query_frontend_connected_clients` is greater than `0`, it is an indication than the `queriers` are connected to the `query-frontend`.
