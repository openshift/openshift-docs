= Understanding the policy lifecycle
include::modules/common-attributes.adoc[]
:context: understanding-policy-lifecycle

toc::[]

[role="_abstract"]

{product-title-short} operates across the three lifecycle stages: the build stage, the deploy stage, and runtime. {product-title-short} security policies can be associated with multiple lifecycle stages and lifecycle related criteria.

Build phase policies::

These policies are constructed purely of image criteria, for example, CVEs found in images and Dockerfile instructions. These policies can be applied only during the build stage, by using the {product-title-short} `roxctl` CLI. When build stage policies are enforced, `roxctl` is designed to break the pipeline by returning a failure code.

Deploy phase policies::
Deploy stage policies can include image criteria such as the criteria from the build phase policies; workload criteria, such as a container running in privileged mode or that mounts the Docker daemon socket; and service account criteria such as RBAC attributes.

With a "shift left” approach, deploy stage policies are first used during the build stage pipeline, to ensure that deployments meet the desired requirements along with the images they use. The same policies are later used during the deployment creation attempt. After the deployment is active, the policies are evaluated during any update attempt.

Deploy stage policies can be enforced in the build pipeline in the same manner by using `roxctl`. When deploy stage policies are enforced in the cluster, they are designed to reject a deployment creation or update attempt by using the {product-title-short} admission controller.

By combining criteria from images and workload- or cluster-related criteria, policies can be designed to capture higher risk cases to behave differently. For example, if the image has a high severity CVE, and the deployment has an exposed port, then the risk may be higher. You can choose to enforce the policy for a particularly sensitive namespace, but, for example, if the port is not exposed, you can choose to only "inform".

Build and deploy::

This is a convenience mode to designate Build policies (which use only image related criteria) to be used for deployment validation too. Users may prefer this mode instead of “Build only” because the Build-type policies are typically also applicable for the Deploy stage. It allows you to run the same validations on the images associated with a deployment,  at the time of deployment creation or update. By using this mode users don’t have to duplicate the policy for that purpose.

Runtime::

Runtime policies can include any of the  build and deploy-stage criteria but they must include at least one Runtime related criterion (process activity, network flow, User-Pod interaction criteria). The runtime criteria serve as the primary factors when evaluating these policies. That is, a runtime policy will not trigger unless one of the runtime criteria was met.
While runtime policies may well be written using runtime criteria alone, combining build and deploy stage criteria allows users to craft specific policies for heightened risk. For example, the combined weaknesses of : High image vulnerability (Build), exposed port (Deploy), and a suspicious process execution (Run) can identify a dangerous situation that should be addressed with a high priority.


When runtime policies are enforced they result in the termination of the offending pod, followed by the natural respawning, by Kubernetes, of a fresh pod from the container imager.

Alternatively, a runtime policy may be constructed to analyze Kubernetes audit log events, to evaluate access to a set of sensitive cluster resources. For this special case, the image and deployment criteria are not applicable, and this policy cannot be enforced (since the events already happened).
