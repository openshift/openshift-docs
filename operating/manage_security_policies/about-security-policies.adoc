= About security policies
include::modules/common-attributes.adoc[]
:context: about-security-policies

toc::[]

[role="_abstract"]
{product-title-short} provides security for your Kubernetes environments by providing out-of-the-box security policies and giving you the ability to customize your own multifactor policies for your container environment. Configuring these policies enables you to automatically prevent high-risk service deployments in your environment and respond to runtime security incidents.

You can use policies to alert you or work with your system to stop it from performing an action, such as a deployment, when {product-title-short} scans your environment and discovers policy violations.

=== Policy details

The policy details include general information such as severity and MITRE ATT&CK tactic, and information that you create depending on your system, such as the policy name, rationale, and guidance to provide when a violation for the policy occurs.

[id="policy-severity-levels_{context}"]
==== Severity

The severity levels for policies in {product-title} are different from the severity levels that Red{nbsp}Hat Product Security assigns.

The {product-title} policy severity levels are Critical, High, Medium, and Low. Red{nbsp}Hat Product Security rates vulnerability severity levels as Critical, Important, Moderate, and Low.

While a policy's severity level and the Red{nbsp}Hat Product Security severity levels can interact, it is important to distinguish between them. For more information about the Red{nbsp}Hat Product Security severity levels, see link:https://access.redhat.com/security/updates/classification[Severity Ratings].

[id="policy-behavior-about_{context}"]
=== Policy behavior

When you configure policy behavior for a policy, you select the parts of your system to which a policy should or should not apply, and then configure actions that the system will take if the policy is violated.

[id="policy-scoping_{context}"]
==== Scope

When you configure policy behavior, you create scopes to restrict a policy to only a specific cluster, namespace, or deployment label, or exclude certain entities from your policy. You can only exclude policies in the deploy and runtime lifecycle stages.

For more information about scopes, see xref:../operating/manage_security_policies/understanding-policy-scopes.adoc#understanding-policy-scopes[Understanding policy scopes].

[id="share-security-policies_{context}"]
== Sharing security policies

Beginning from {product-title} version 3.0.44,  you can share your security policies between different Central instances, by exporting and importing policies.
Sharing policies helps you enforce the same standards for all your clusters.
To share policies, you export them as JSON files, and then import them back into another Central instance.

[NOTE]
====
Currently, you cannot export multiple security policies at once by using the {product-title-short} portal.
However, you can use the API for exporting multiple security policies.
In the {product-title-short} portal, go to *Help* -> *API reference* to see the API reference.
====
