= Creating a NodePool

[role="_abstract"]
Create a NodePool custom resource to define the compute capacity limits and instance requirements that {autonode} uses when provisioning nodes.

A NodePool defines the parameters that {autonode} uses to provision compute nodes for your workloads. It specifies allowed instance types, capacity types (on-demand or spot), availability zones, and resource limits. After you create a NodePool, {autonode} automatically provisions nodes that match your workload requirements within the limits you define.

.Prerequisites

* You have enabled {autonode} on your cluster.
* You have verified that the `ec2nodeclass` and `openshiftec2nodeclass` resources are present and ready.
* You are logged in to your cluster as a user with `cluster-admin` privileges.

.Procedure

. Create a file named `nodepool.yaml` with the following NodePool definition:
+
[source,yaml]
----
apiVersion: karpenter.sh/v1
kind: NodePool
metadata:
  name: default-np
spec:
  template:
    metadata:
      labels:
        autonode: "true"
    spec:
      requirements:
      - key: node.kubernetes.io/instance-type
        operator: In
        values:
        - c5.xlarge
      - key: karpenter.sh/capacity-type
        operator: In
        values: ["on-demand"]
      nodeClassRef:
        group: karpenter.k8s.aws
        kind: EC2NodeClass
        name: default
----
+
This NodePool configuration defines the following parameters:
+
--
* `autonode: "true"` label: Workloads can target this NodePool by using a node selector that matches this label.
* Instance type requirement: Allows only `c5.xlarge` instances. You can specify multiple instance types as a comma-separated list.
* Capacity type requirement: Restricts provisioning to on-demand instances. You can also specify `spot` for spot instances.
* Node class reference: References the default `EC2NodeClass` that was created when you enabled {autonode}.
--

. Create the NodePool in your cluster:
+
[source,terminal]
----
$ oc create -f nodepool.yaml
----

. Verify that the NodePool was created and is ready:
+
[source,terminal]
----
$ oc get nodepool
----
+
.Example output
[source,terminal]
----
NAME         NODECLASS   NODES   READY   AGE
default-np   default     0       True    3s
----
+
The `NODES` column shows `0` because no workloads are currently requesting capacity from this NodePool.

. Create a test deployment that uses the NodePool:
+
.. Create a file named `test-deployment.yaml` with the following content:
+
[source,yaml]
----
apiVersion: apps/v1
kind: Deployment
metadata:
  name: stress-load
  labels:
    app: stress-load
spec:
  replicas: 1
  selector:
    matchLabels:
      app: stress-load
  template:
    metadata:
      labels:
        app: stress-load
    spec:
      nodeSelector:
        autonode: "true"
      containers:
      - name: stress
        image: progrium/stress@sha256:e34d56d60f5caae79333cee395aae93b74791d50e3841986420d23c2ee4697bf
        imagePullPolicy: IfNotPresent
        command: ["stress"]
        args: ["--cpu", "2", "--timeout", "600"]
        resources:
          requests:
            cpu: "100m"
            memory: "64Mi"
          limits:
            cpu: "500m"
            memory: "128Mi"
----
+
The `nodeSelector: autonode: "true"` field ensures that these pods are scheduled only on nodes provisioned by the NodePool you created.

.. Deploy the test workload:
+
[source,terminal]
----
$ oc create -f test-deployment.yaml
----

. Verify that the pod is initially pending while {autonode} provisions a node:
+
[source,terminal]
----
$ oc get pods -n default
----
+
.Example output
[source,terminal]
----
NAME                           READY   STATUS    RESTARTS   AGE
stress-load-7b9464b566-f5sjz   0/1     Pending   0          7s
----
+
The pod remains in the `Pending` state until {autonode} creates a node that satisfies its requirements.

. Observe the NodePool and NodeClaim resources as {autonode} provisions capacity:
+
.. Check the NodePool to see the node count increase:
+
[source,terminal]
----
$ oc get nodepool
----
+
.Example output
[source,terminal]
----
NAME         NODECLASS   NODES   READY   AGE
default-np   default     1       True    95s
----

.. Check the NodeClaim to see the provisioning status:
+
[source,terminal]
----
$ oc get nodeclaim
----
+
.Example output (initial state)
[source,terminal]
----
NAME               TYPE        CAPACITY    ZONE         NODE   READY     AGE
default-np-ztsch   c5.xlarge   on-demand   us-east-1a          Unknown   26s
----
+
After approximately 3-5 minutes, the node registers with the cluster:
+
.Example output (ready state)
[source,terminal]
----
NAME               TYPE        CAPACITY    ZONE         NODE                          READY   AGE
default-np-ztsch   c5.xlarge   on-demand   us-east-1a   ip-10-0-141-45.ec2.internal   True    4m15s
----

. Verify that the pod is running on the {autonode}-provisioned node:
+
[source,terminal]
----
$ oc get pods -n default -o wide
----
+
.Example output
[source,terminal]
----
NAME                           READY   STATUS    RESTARTS   AGE    IP           NODE                          NOMINATED NODE   READINESS GATES
stress-load-7b9464b566-f5sjz   1/1     Running   0          7m8s   10.128.2.5   ip-10-0-141-45.ec2.internal   <none>           <none>
----

.Verification

* The NodePool shows `READY` as `True` and `NODES` greater than `0`.
* The NodeClaim shows `READY` as `True` and includes a node name.
* The test pod shows `STATUS` as `Running` and is scheduled on a node provisioned by {autonode}.

.Next steps

* Configure additional NodePools for different workload types or capacity requirements.
* Review the NodePool reference documentation to understand additional configuration options.
* Configure disruption settings to control when {autonode} can replace nodes.
