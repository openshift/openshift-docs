= DHCP

[role="_abstract"]
With DHCP, a DHCP server running on the physical network automatically assigns IP addresses when pods start.

== When to use DHCP

Use DHCP when:

* You already have a DHCP server on the physical network
* You are integrating with existing network infrastructure
* You need DHCP to provide DNS servers, NTP servers, or routing information
* Your network team manages the IP addressing infrastructure
* You want SR-IOV networking to work like traditional VM networking
* You need IP addresses that are consistent with other devices on the physical network

== How it works

The DHCP server runs on the physical network that your SR-IOV devices connect to:

1. Pod starts and the SR-IOV virtual function connects to the physical network
2. Pod sends a DHCP DISCOVER message through the virtual function
3. Physical DHCP server responds with a DHCP OFFER containing an IP address
4. Pod configures the interface with the assigned IP address

The DHCP server is external to {product-title} - it runs on your physical network infrastructure, just like DHCP for physical servers or virtual machines.

.Example SriovNetwork configuration with DHCP
[source,yaml]
----
apiVersion: sriovnetwork.openshift.io/v1
kind: SriovNetwork
metadata:
  name: sriov-dhcp-network
  namespace: openshift-sriov-network-operator
spec:
  resourceName: intelnics
  networkNamespace: default
  ipam: |
    {
      "type": "dhcp"
    }
----

== Advantages

* *Leverages existing infrastructure* - Uses DHCP servers you already have
* *Provides additional configuration* - Can provide DNS servers, default routes, NTP servers, and boot parameters
* *Familiar to network administrators* - Works exactly like DHCP for VMs and physical servers
* *Centralized management* - Network team manages IP addressing from DHCP server
* *Scales automatically* - Handles many pods without manual configuration
* *Consistent with physical network* - IP addresses come from the same pools as other devices

== Limitations

* *Requires DHCP server* - Must have a DHCP server on the SR-IOV network segment
* *External dependency* - DHCP server must be available when pods start
* *Limited visibility* - IP addresses are assigned outside {product-title}'s control
* *More complex troubleshooting* - Need to check DHCP server logs and lease databases
* *Network team coordination* - May require coordination with network administrators for IP pool management
* *Lease renewal* - Pods must renew DHCP leases, which could cause issues for long-running pods if DHCP server has problems

== DHCP server requirements

Your DHCP server must:

* Be reachable on the physical network connected to the SR-IOV devices
* Have IP address pools configured for the SR-IOV network
* Allow DHCP requests from the virtual function MAC addresses
* Be highly available if you need reliable pod networking

== Best practices

If you use DHCP:

* Ensure your DHCP server is highly available
* Configure appropriate lease times (longer leases for stable workloads, shorter for dynamic workloads)
* Monitor DHCP server capacity and available IP addresses
* Coordinate with your network team for DHCP server management
* Consider DHCP reservations for pods that need consistent IP addresses
* Test DHCP failover scenarios to ensure pods can obtain IP addresses even if the primary DHCP server fails
