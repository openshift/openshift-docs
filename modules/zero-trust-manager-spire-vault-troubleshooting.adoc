= Troubleshooting SPIRE Vault UpstreamAuthority plugin

[role="_abstract"]
To diagnose and resolve problems such as authentication failures, TLS certificate errors, connection issues, and certificate signing failures, use this troubleshooting guide. This guide describes common configuration and operational issues with the SPIRE Vault UpstreamAuthority plugin.

[id="spire-vault-troubleshooting-enable-debug_{context}"]
== Enabling debug logging

To troubleshoot SPIRE Server issues with the Vault plugin, increase the log level to capture detailed information.

. Edit the SPIRE Server configuration file to increase the log level by running the following command:
+
[source,hcl]
----
server {
  log_level = "DEBUG"
  # ... other server configuration
}
----

. Restart the SPIRE Server to apply the changes by running the following command:
+
[source,terminal]
----
$ systemctl restart spire-server
----

. View the detailed logs by running the following command:
+
[source,terminal]
----
$ journalctl -u spire-server -f
----

[id="spire-vault-troubleshooting-connection_{context}"]
== Connection failures

*Symptom:* SPIRE Server fails to connect to Vault, and logs show connection timeout or refused errors.

.Example error
[source,terminal]
----
time="2026-04-07T10:15:30Z" level=error msg="Failed to initialize UpstreamAuthority" error="Post \"https://vault.example.com:8200/v1/pki/root/sign-intermediate\": dial tcp: connect: connection refused"
----

*Common causes and solutions:*

Check network connectivity::
* Verify that SPIRE Server can reach the Vault server by running the following command:
+
[source,terminal]
----
$ curl -v https://vault.example.com:8200/v1/sys/health
----
+
If the connection fails, check firewall rules and network policies by running the following command.

Verify Vault address::
* Ensure the `vault_addr` parameter in the SPIRE configuration matches the actual Vault server URL by running the following command:
+
[source,hcl]
----
vault_addr = "https://vault.example.com:8200"
----

Check DNS resolution::
* Verify that the Vault hostname resolves correctly by running the following command:
+
[source,terminal]
----
$ nslookup vault.example.com
----

[id="spire-vault-troubleshooting-tls_{context}"]
== TLS certificate verification failures

*Symptom:* SPIRE Server cannot verify the Vault server's TLS certificate.

.Example error
[source,terminal]
----
time="2026-04-07T10:15:30Z" level=error msg="Failed to initialize UpstreamAuthority" error="x509: certificate signed by unknown authority"
----

*Solutions:*

Give CA certificate::
* Specify the path to the CA certificate that signed the Vault server's certificate by running the following command:
+
[source,hcl]
----
ca_cert_path = "/etc/spire/vault-ca.pem"
----

Verify CA certificate format:: 
* Ensure the CA certificate is in PEM format by running the following command:
+
[source,terminal]
----
$ openssl x509 -in /etc/spire/vault-ca.pem -text -noout
----
+
If the command fails, the file might not be a valid PEM certificate.

Check certificate permissions::
* Verify that the SPIRE Server process can read the CA certificate file by running the following command:
+
[source,terminal]
----
$ ls -l /etc/spire/vault-ca.pem
----

[NOTE]
====
Using `insecure_skip_verify = true` bypasses certificate verification and is not recommended for production environments.
====

[id="spire-vault-troubleshooting-auth-token_{context}"]
== Token authentication failures

*Symptom:* SPIRE Server fails to authenticate to Vault by using a token.

.Example error
[source,terminal]
----
time="2026-04-07T10:15:30Z" level=error msg="Failed to initialize UpstreamAuthority" error="Error making API request. Code: 403. Errors: * permission denied"
----

*Common causes and solutions:*

Verify token validity::

* Check that the Vault token is valid and not expired by running the following command:
+
[source,terminal]
----
$ VAULT_TOKEN="your-token" vault token lookup
----

Check token permissions:: 

* Ensure the token has the required policy permissions. The token must have `update` capability on `pki/root/sign-intermediate` by running the following command:
+
[source,terminal]
----
$ VAULT_TOKEN="your-token" vault token capabilities pki/root/sign-intermediate
----
+
.Expected output
[source,terminal]
----
update
----
+
If you see `deny` or a different capability, the token's policy is wrong.

Verify attached policies:: 

* List the policies attached to the token by running the following command:
+
[source,terminal]
----
$ VAULT_TOKEN="your-token" vault token lookup -format=json | jq -r '.data.policies'
----
+
Ensure `spire-server` (or your policy name) is in the list.

Update Vault policy::
* If the token lacks permissions, update the associated policy by running the following command:
+
[source,hcl]
----
path "pki/root/sign-intermediate" {
  capabilities = ["update"]
}
----
+
Then re-create the token or re-login with the authentication method to get a new token with updated permissions.

Verify environment variable::
* If using `${VAULT_TOKEN}`, confirm the environment variable is set by running the following command:
+
[source,terminal]
----
$ echo $VAULT_TOKEN
----

[id="spire-vault-troubleshooting-auth-approle_{context}"]
== AppRole authentication failures

*Symptom:* SPIRE Server fails to authenticate using `AppRole` credentials.

.Example error
[source,terminal]
----
time="2026-04-07T10:15:30Z" level=error msg="Failed to initialize UpstreamAuthority" error="failed to login to approle auth method: Error making API request. Code: 400. Errors: * invalid role ID or secret ID"
----

*Solutions:*

Verify AppRole credentials::

* Check that the `AppRole ID` and `Secret ID` are correct by running the following command:
+
[source,terminal]
----
$ vault write auth/approle/login role_id="$VAULT_APPROLE_ID" secret_id="$VAULT_APPROLE_SECRET_ID"
----

Check AppRole mount point::
* Ensure the `approle_auth_mount_point` matches the actual mount point in Vault by running the following command:
+
[source,terminal]
----
$ vault auth list
----

Verify AppRole policy::
* Confirm the `AppRole` has the necessary policy attached by running the following command:
+
[source,terminal]
----
$ vault read auth/approle/role/<role_name>
----

[id="spire-vault-troubleshooting-auth-kubernetes_{context}"]
== Kubernetes authentication failures

*Symptom:* SPIRE Server fails to authenticate using Kubernetes service account tokens.

.Example error
[source,terminal]
----
time="2026-04-07T10:15:30Z" level=error msg="Failed to initialize UpstreamAuthority" error="failed to login to kubernetes auth method: * service account name not authorized"
----

*Common causes and solutions:*

Verify service account token::
* Check that the token file exists and is readable by running the following command:
+
[source,terminal]
----
$ cat /var/run/secrets/kubernetes.io/serviceaccount/token
----

Check Vault role configuration::
* Verify the Kubernetes auth role is configured correctly in Vault by running the following command:
+
[source,terminal]
----
$ vault read auth/kubernetes/role/spire-server
----

Verify service account binding::
* Ensure the service account is bound to the Vault role by running the following command:
+
[source,terminal]
----
$ vault write auth/kubernetes/role/spire-server \
    bound_service_account_names=spire-server \
    bound_service_account_namespaces=spire \
    policies=spire-server \
    ttl=1h
----

[id="spire-vault-troubleshooting-auth-cert_{context}"]
== Client certificate authentication failures

*Symptom:* SPIRE Server fails to authenticate using client certificates.

.Example error
[source,terminal]
----
time="2026-04-07T10:15:30Z" level=error msg="Failed to initialize UpstreamAuthority" error="failed to login to cert auth method: Error making API request. Code: 400. Errors: * invalid certificate or no client certificate supplied"
----

*Solutions:*

Verify certificate and key files::
* Ensure both the client certificate and private key are in PEM format and readable by running the following command:
+
[source,terminal]
----
$ openssl x509 -in /etc/spire/client-cert.pem -text -noout
$ openssl rsa -in /etc/spire/client-key.pem -check
----

Check certificate registration::
* Verify the certificate is registered in Vault by running the following command:
+
[source,terminal]
----
$ vault list auth/cert/certs
----

Verify file permissions::
* Check that SPIRE Server can read the certificate and key files by running the following command:
+
[source,terminal]
----
$ ls -l /etc/spire/client-cert.pem /etc/spire/client-key.pem
----

[id="spire-vault-troubleshooting-signing_{context}"]
== Certificate signing failures

*Symptom:* Vault successfully authenticates but fails to sign the intermediate certificate.

.Example error
[source,terminal]
----
time="2026-04-07T10:15:30Z" level=error msg="Failed to sign intermediate CA" error="Error making API request. Code: 400. Errors: * unknown role"
----

*Common causes and solutions:*

Check PKI mount point::
* Verify the PKI secrets engine is enabled at the specified mount point by running the following command:
+
[source,terminal]
----
$ vault secrets list
----

Verify PKI configuration::
* Check that the PKI secrets engine is configured correctly by running the following command:
+
[source,terminal]
----
$ vault read pki/config/urls
----

Check root CA::
* Ensure the PKI secrets engine has a root CA configured by running the following command:
+
[source,terminal]
----
$ vault read pki/cert/ca
----
+
If no root CA exists, generate or import one by running the following command:
+
[source,terminal]
----
$ vault write pki/root/generate/internal common_name="Example Root CA" ttl=87600h
----

[id="spire-vault-troubleshooting-ttl_{context}"]
== TTL mismatch errors

*Symptom:* Certificate signing fails due to TTL configuration mismatches.

.Example error
[source,terminal]
----
time="2026-04-07T10:15:30Z" level=error msg="Failed to sign intermediate CA" error="requested TTL of 168h exceeds max_lease_ttl of 72h"
----

*Solutions:*

Adjust SPIRE Server ca_ttl::
* Reduce the `ca_ttl` value in the SPIRE Server configuration to be less than the Vault PKI maximum lease TTL by running the following command:
+
[source,hcl]
----
server {
  ca_ttl = "48h"
  # ... other configuration
}
----

Increase Vault PKI max_lease_ttl::
* Or, increase the maximum lease TTL in Vault by running the following command:
+
[source,terminal]
----
$ vault secrets tune -max-lease-ttl=8760h pki
----

Verify current TTL limits::
* Check the current PKI secrets engine configuration by running the following command:
+
[source,terminal]
----
$ vault read sys/mounts/pki/tune
----

[id="spire-vault-troubleshooting-namespace_{context}"]
== Vault Enterprise namespace issues

*Symptom:* When using Vault Enterprise, authentication or signing fails with namespace-related errors.

.Example error
[source,terminal]
----
time="2026-04-07T10:15:30Z" level=error msg="Failed to initialize UpstreamAuthority" error="Namespace not found"
----

*Solutions:*

Verify namespace exists::
* Check that the specified namespace exists in Vault by running the following command:
+
[source,terminal]
----
$ VAULT_NAMESPACE=<namespace> vault namespace list
----

Configure namespace parameter::
* Ensure the namespace is specified in the SPIRE configuration by running the following command:
+
[source,hcl]
----
namespace = "admin/team-a"
----

Use correct namespace path:: Verify the full namespace path is provided, including parent namespaces.

[id="spire-vault-troubleshooting-verification_{context}"]
== Verifying successful integration

After resolving issues, verify that the SPIRE Vault integration is working correctly.

. Check SPIRE Server health by running the following command:
+
[source,terminal]
----
$ /opt/spire/bin/spire-server healthcheck
----
+
.Expected output
[source,terminal]
----
Server is healthy.
----

. Verify the CA bundle includes a Vault-signed certificate by running the following command:
+
[source,terminal]
----
$ /opt/spire/bin/spire-server bundle show
----

. Examine SPIRE Server logs for successful Upstream Authority initialization by running the following command:
+
[source,terminal]
----
$ journalctl -u spire-server -n 100 | grep -i "upstream"
----
+
.Expected log entries
[source,terminal]
----
time="2026-04-07T10:15:30Z" level=info msg="Upstream authority loaded" subsystem_name=ca
time="2026-04-07T10:15:31Z" level=info msg="Server CA activated"
----

. Test SVID issuance to ensure the full chain is working by running the following command:
+
[source,terminal]
----
$ /opt/spire/bin/spire-server entry create \
    -spiffeID spiffe://example.org/test \
    -parentID spiffe://example.org/spire/agent/test \
    -selector unix:uid:1000
----
