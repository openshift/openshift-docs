= Creating a second `HostNetwork` endpoint publishing strategy

You can create a second `HostNetwork` endpoint publishing strategy and use the strategy to add an additional Ingress Controller to non-default node ports in your network. Consider creating this additional endpoint publishing strategy when you want to limit access to default ports and the default Ingress Controller.

Before you create a second endpoint publishing strategy, read the following considerations:

* You cannot run the additional Ingress Controller on nodes that use the default Ingress Controller because of node port conflicts.
* A HAProxy Virtual IP (VIP) or a MetalLB implementation can host the additional Ingress Controller.
* You must create a wildcard DNS record for the additional Ingress Controller.
* You must create a route service and specify the hostname of your additional Ingress Controller in this service.

.Prerequisites

* You installed the {oc-first}.
* Logged in as a user with `cluster-admin` privileges.
* You created a wildcard DNS record.
# https://docs.openshift.com/container-platform/4.16/networking/ingress-controller-dnsmgt.html (does not detail how to create the DNS)

.Procedure

. Create a custom resource (CR) file for the additional Ingress Controller:
+
.Example of a CR file that defines information for the additional Ingress Controller
[source,yaml]
----
apiVersion: v1
items:
- apiVersion: operator.openshift.io/v1
  kind: IngressController
  metadata:
    name: <ingress_controller_name> <1>
    namespace: openshift-ingress-operator
  spec:
    replicas: 1
    domain: <domain> <2>
    nodePlacement:
      nodeSelector:
        matchLabels:
          ingresscontroller: "<ingress_controller_custom_resource>" <3>
    namespaceSelector:
     matchLabels:
       ic: <ingress_controller_name> <4>
    endpointPublishingStrategy:
      type: HostNetwork
  status: {}
kind: List
metadata:
  resourceVersion: ""
  selfLink: ""
----
<1> Specify the `name` with a name for the `IngressController` object.
<2> Specify the `domain` for the wilcard DNS record.
<3> Specify the label for the `ingresscontroller` CR.
<4> Filter the namespace that is serviced by the Ingress Controller. 

. Add a label to a node by using the `oc label node` command:
+
[source,terminal]
----
$ oc label node <node_name> ingresscontroller=<ingress_controller_custom_resource>
----

. Update the `IngressController` CR to exclude the namespace for additional Ingress Controller from the default Ingress Controller:
+
[source,terminal]
----
$ oc patch --type=merge -n openshift-ingress-operator ingresscontroller/default --patch '{"spec":{"namespaceSelector":{"matchExpressions":[{"key":"ic","operator":"NotIn","values":["<ingress_controller_name>","custom-ic2","custom-ic3"]}]}}}'
----

. Create the `Pod` object:
+
[source,terminal]
----
$ oc create -f <pod_object_name>.yaml
----

. Use the `oc new-project` command to create a new project:
+
[source,terminal]
----
$ oc new-project <ingress_controller_name>
----

. Enter the following command to label the additional Ingress Controller:
+
[source,terminal]
----
$ oc label namespace <ingress_controller_name> ic=<ingress_controller_name>
----

. WHAT does this do?
+
[source,terminal]
----
$ oc new-app --image=quay.io/openshifttest/hello-openshift:multiarch
----

. Set the domain for the additional Ingress Controller hostname:
+
[source,terminal]
----
$ oc expose svc/hello-openshift --hostname=hello-openshift.<domain_name>
----

.Verification

. Verify that the wildcard DNS record is managed by the additioanl Ingress Controller: 
+
[source,terminal]
----
$ curl hello-openshift.<ingress_controller_name>.<domain_name>
----
