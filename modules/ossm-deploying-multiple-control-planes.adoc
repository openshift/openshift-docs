= Deploying multiple control planes
:context: ossm-deploying-multiple-control-planes

This procedure demonstrates how to configure a cluster to host two control planes in separate system namespaces. Each control plane will manage specific application workloads running in designated namespaces. The namespaces are associated with a control plane using revision and discovery selector configurations.

.Prerequisites
* The {SMProduct} operator is installed.
* An {istio} CNI resource is created.
* The `istioctl` binary is installed on your localhost.

.Procedure

. Create a system namespace called `usergroup-1`:
+
[source,terminal]
----
$ oc create namespace usergroup-1
----

. Add label to the namespace:
+
[source,terminal]
----
$ oc label namespace usergroup-1 usergroup=usergroup-1
----

. Create a YAML file named `istio-1.yaml`:
+
.Example configuration
[source,yaml,subs="attributes,verbatim"]
----
kind: Istio
apiVersion: sailoperator.io/v1alpha1
metadata:
  name: usergroup-1
spec:
  namespace: usergroup-1
  values:
    meshConfig:
      discoverySelectors:
        - matchLabels:
            usergroup: usergroup-1
  updateStrategy:
    type: InPlace
  version: v1.23.0
----

. Create `Istio` resource:
+
[source,terminal]
----
$ oc apply -f istio-1.yaml
----

. Create a second system namespace called `usergroup-2`:
+
[source,terminal]
----
$ oc create namespace usergroup-2
----

. Add label to the second namespace:
+
[source,terminal]
----
$ oc label namespace usergroup-2 usergroup=usergroup-2
----

. Create a YAML file named `istio-2.yaml`:
+
.Example configuration
[source,yaml,subs="attributes,verbatim"]
----
kind: Istio
apiVersion: sailoperator.io/v1alpha1
metadata:
  name: usergroup-2
spec:
  namespace: usergroup-2
  values:
    meshConfig:
      discoverySelectors:
        - matchLabels:
            usergroup: usergroup-2
  updateStrategy:
    type: InPlace
  version: v1.23.0
----

. . Create `Istio` resource:
+
[source,terminal]
----
$ oc apply -f istio-2.yaml
----

. Deploy a policy for workloads in the `usergroup-1` namespace to only accept mutual TLS traffic `peer-auth-1.yaml`:
+
[source,terminal]
----
$ oc apply -f peer-auth-1.yaml
----
+
.Example configuration
[source,yaml,subs="attributes,verbatim"]
----
apiVersion: security.istio.io/v1
kind: PeerAuthentication
metadata:
  name: "usergroup-1-peerauth"
  namespace: "usergroup-1"
spec:
  mtls:
    mode: STRICT
----

. Deploy a policy for workloads in the `usergroup-2` namespace to only accept mutual TLS traffic `peer-auth-2.yaml`:
+
[source,terminal]
----
$ oc apply -f peer-auth-2.yaml
----
+
.Example configuration
[source,yaml,subs="attributes,verbatim"]
----
apiVersion: security.istio.io/v1
kind: PeerAuthentication
metadata:
  name: "usergroup-2-peerauth"
  namespace: "usergroup-2"
spec:
  mtls:
    mode: STRICT
----

.Verfication

. Verify that both the control planes deploy and run:
+
.. Run the following command for `usergroup-1` verification:
+
[source,terminal]
----
$ oc get pods -n usergroup-1
----
+
.. Run the following command for `usergroup-2` verification:
+
[source,terminal]
----
$ oc get pods -n usergroup-2
----
