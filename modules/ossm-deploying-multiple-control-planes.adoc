= Deploying multiple control planes
:context: ossm-deploying-multiple-control-planes

This procedure explains how to configure a cluster to host two control planes, each managing independent service mesh data planes. The configuration involves creating separate `{istio}` resources with unique names in independent {istio} system namespaces. Each `{istio}` resource will use a unique `revision` name, which identifies the control planes, workloads, or namespaces associated with it. These `revision` names, applied using injection or `istio.io/rev` labels determine which control plane will inject the sidecar proxy into the application pods.

Each `{istio}` resource must also configure `discoverySelectors` to specify which namespaces the {istio} control plane observes. Only namespaces with labels that match the configured `discoverySelectors` can join the mesh. Additionally, discovery selectors determine which control plane creates the `istio-ca-root-cert` config map in each namespace, which is used to encrypt traffic between services with mutual TLS within each mesh.

When adding an additional {istio} control plane to a cluster with an existing control plane, ensure that the existing `{istio}` instance has `discoverySelectors` configured to avoid overlapping with the new control plane. Run the following command to check for existing `{istio}` instances.

This procedure may be extended to support more than two control planes. The primary limitation on the number of service meshes in a single cluster is the availability of cluster resources.

[NOTE]
====
There is only one `IstioCNI` resource shared by all control planes in the cluster, and it must be updated independently.
====

.Prerequisites
* The {SMProduct} operator is installed.
* An {istio} CNI resource is created.
* The `istioctl` binary is installed on your localhost.

.Procedure

. Create the namespace for the first {istio} control plane called `istio-system-1`:
+
[source,terminal]
----
$ oc create namespace istio-system-1
----

. Add the following label to the first namespace, which will be used with the Istio `discoverySelector`:
+
[source,terminal]
----
$ oc label namespace istio-system-1 istio-discovery=mesh-1
----

. Create a YAML file named `istio-1.yaml` with the name `mesh-1` and the discoverySelector `mesh-1`:
+
.Example configuration
[source,yaml,subs="attributes,verbatim"]
----
kind: Istio
apiVersion: sailoperator.io/v1alpha1
metadata:
  name: mesh-1
spec:
  namespace: istio-system-1
  values:
    meshConfig:
      discoverySelectors:
        - matchLabels:
            istio-discovery: mesh-1
----

. Create the first `Istio` resource:
+
[source,terminal]
----
$ oc apply -f istio-1.yaml
----

. Create a namespace for the second Istio control plane called `istio-system-2`:
+
[source,terminal]
----
$ oc create namespace istio-system-2
----

. Add the following label to the second namespace, which will be used with the Istio `discoverySelector`:
+
[source,terminal]
----
$ oc label namespace istio-system-2 istio-discovery=mesh-2
----

. Create a YAML file named `istio-2.yaml`:
+
.Example configuration
[source,yaml,subs="attributes,verbatim"]
----
kind: Istio
apiVersion: sailoperator.io/v1alpha1
metadata:
  name: mesh-2
spec:
  namespace: istio-system-2
  values:
    meshConfig:
      discoverySelectors:
        - matchLabels:
            istio-discovery: mesh-2
----

. . Create `Istio` resource:
+
[source,terminal]
----
$ oc apply -f istio-2.yaml
----

. To restrict workloads in `mesh-1` from communicating freely with unencrypted traffic between meshes, deploy a `PeerAuthentication` resource to enforce mutual TLS (mTLS) traffic within the `mesh-1` data plane. Apply the `PeerAuthentication` resource in the `istio-system-1` namespace by using a configuration file, such as `peer-auth-1.yaml`:
+
[source,terminal]
----
$ oc apply -f peer-auth-1.yaml
----
+
.Example configuration
[source,yaml,subs="attributes,verbatim"]
----
apiVersion: security.istio.io/v1
kind: PeerAuthentication
metadata:
  name: "mesh-1-peerauth"
  namespace: "istio-system-1"
spec:
  mtls:
    mode: STRICT
----

. Deploy a policy for workloads in the `istio-system-2` namespace to only accept mutual TLS traffic `peer-auth-2.yaml`:
+
[source,terminal]
----
$ oc apply -f peer-auth-2.yaml
----
+
.Example configuration
[source,yaml,subs="attributes,verbatim"]
----
apiVersion: security.istio.io/v1
kind: PeerAuthentication
metadata:
  name: "mesh-2-peerauth"
  namespace: "istio-system-2"
spec:
  mtls:
    mode: STRICT
----

.Verfication

. Verify that both of the {istio} control planes deploy and run. This can be done by validating that the `Istiod` pod is successfully running in each {istio} system namespace.
+
.. Run the following command for `istio-system-1` verification:
+
[source,terminal]
----
$ oc get pods -n istio-system-1
----
+
You should see output similar to the following example:
+
.Example output
[source,terminal]
----
NAME                            READY   STATUS    RESTARTS   AGE
istiod-mesh-1-b69646b6f-kxrwk   1/1     Running   0          4m14s
----
+
.. Run the following command for `istio-system-2` verification:
+
[source,terminal]
----
$ oc get pods -n istio-system-2
----
+
You should see output similar to the following example:
+
.Example output
[source,terminal]
----
NAME                            READY   STATUS    RESTARTS   AGE
istiod-mesh-2-8666fdfc6-mqp45   1/1     Running   0          118s
----
