= Understanding {autonode}

[role="_abstract"]
In {product-title}, {autonode} is a fully managed service with controllers from the open source Karpenter project that is hosted and managed along with other control plane components. This service eliminates not only the need to manually size cluster nodes using hundreds of Amazon EC2 instance types but also deploying Karpenter on your cluster worker nodes and managing them.

[id="rosa-nodes-autonode-how-it-works_{context}"]
== How {autonode} works

{autonode} builds on the open source `Karpenter` project to provide intelligent, workload-driven node provisioning. The `Karpenter` software performs the following work:

* Observes pods that the Kubernetes scheduler cannot place on existing capacity
* Evaluates resource, placement, and affinity requirements for those pods
* Creates nodes that match those requirements
* Removes or replaces nodes when they are no longer needed, based on disruption settings that you configure

You set specifications on how {autonode} can provision capacity through `NodePool` custom resources. Examples include acceptable instance families and market types such as on-demand or spot instances.

You express workload requirements through standard Kubernetes scheduling fields on your pods and controllers, such as resource requests, node selectors, affinity rules, and topology spread constraints. {autonode} works to place pending workloads within the limits you define.

[id="rosa-nodes-autonode-why-use_{context}"]
== Why use {autonode}

{autonode} addresses common challenges in managing Kubernetes cluster capacity:

Eliminates capacity planning guesswork::
Traditional machine pools require you to predict workload patterns and pre-provision capacity. {autonode} provisions nodes in response to actual demand, eliminating the need to estimate how many nodes you need.

Reduces infrastructure waste::
Karpenter always right-sizes Amazon EC2 instances based on the workload requirements, eliminating the rigid configuration based on single EC2 instance type that can leave underutilized nodes.

Handles diverse workload requirements::
Different workloads often have conflicting requirements - some need GPU instances, others need high memory, others need specific availability zones. Managing separate machine pools for each workload type creates complexity. {autonode} automatically selects the right instance type based on each pod's requirements.

Dynamic scaling::
Karpenter continuously looks out of reducing infrastructure costs even when there is no change in the workload. It can switch between market types or instance types based on availability and pricing instead of just waiting for a pending pod or an empty node to trigger scaling up or down, respectively.

Simplifies multi-zone deployments::
{autonode} can provision nodes across multiple availability zones based on pod topology requirements, without requiring separate machine pools per zone.

[id="rosa-nodes-autonode-when-to-use_{context}"]
== When to use {autonode}

Consider using {autonode} in the following scenarios:

Batch and data processing workloads::
Jobs that run periodically, require capacity for a few hours, then complete. Examples include data pipelines, training workloads, and nightly batch processes.

Development and testing environments::
Clusters where usage patterns vary widely throughout the day. {autonode} scales capacity up during working hours and down overnight.

Applications with unpredictable scaling::
Workloads where demand spikes are difficult to predict, such as event-driven applications or seasonal traffic patterns.

Mixed workload clusters::
Environments running many different application types with different resource requirements. {autonode} can provision the right instance types for each workload.

Cost optimization priorities::
When reducing infrastructure costs is a primary concern, {autonode} eliminates paying for idle capacity during low-demand periods.

[id="rosa-nodes-autonode-operational-considerations_{context}"]
== Operational considerations

Using {autonode} introduces some operational changes:

Node lifecycle is dynamic::
Nodes are created and removed automatically based on workload demand. Applications must tolerate node disruptions and pod rescheduling.

Capacity limits are important::
You define maximum capacity through `NodePool` limits. If limits are too restrictive, pods may remain unschedulable even when demand exists.

IAM permissions are required::
{autonode} needs Amazon Web Services (AWS) Identity and Access Management (IAM) permissions to create and terminate Amazon Elastic Compute Cloud (Amazon EC2) instances, manage security groups, and create instance profiles.

Resource tagging is required::
{autonode} uses AWS resource tags to discover which subnets and security groups to use. Incorrectly tagged resources will prevent node provisioning.

Monitoring and observability::
You monitor {autonode} through `NodePool` and `NodeClaim` resources, not machine pools. Metrics and logs are accessed differently than traditional machine pool autoscaling.
