= Manage virtual machines with the KubeVirt Redfish API

[role="_abstract"]
After KubeVirt Redfish is running, you can list managed systems, read their power state, and request power and reset operations through HTTPS with the same Redfish request patterns you use in other environments.
The following steps walk through discovery and power control with `curl`.

.Prerequisites

* The deployment from xref:../../virt/post_installation_configuration/virt-kubevirt-redfish.adoc#proc_virt-installing-kubevirt-redfish_{context}[Installing KubeVirt Redfish] is complete and the Route is reachable from your client.
* You have the **endpoint URL** (https://host/...) and credentials configured in the Secret.
* `curl` is available on the management host.
* Optional: read link:https://www.dmtf.org/standards/redfish[Redfish standard] for general protocol terms such as *ComputerSystem* and *Reset*.

.Procedure

. Set environment variables to avoid typing secrets on the command line in shell history, replacing placeholders with the host name or IP, user name, and password from your deployment.
+
[source,terminal]
----
$ export RFHOST="<redfish_host>"
$ export RFUSER="<user>"
$ export RFPASS="<password>"
----
+
[NOTE]
====
Storing credentials in environment variables is convenient for testing; in production, use a secrets manager and your organization's hardening requirements.
====

. Query the service root to confirm the endpoint responds and the TLS configuration matches your expectations, adding `-k` or CA flags per your test policy.
+
[source,terminal]
----
$ curl -sk -u "${RFUSER}:${RFPASS}" https://${RFHOST}/redfish/v1/
----

. List the `Systems` collection to discover the managed VMs exposed as Redfish `ComputerSystem` entries.
+
[source,terminal]
----
$ curl -sk -u "${RFUSER}:${RFPASS}" https://${RFHOST}/redfish/v1/Systems
----
+
From the response, take the value of the `@odata.id` (or the member URI) of the system you need; this identifier encodes the namespace, VM name, or other mapping defined by your `chassis` settings.

. Optional: `GET` the individual system to read `PowerState` and `Status` fields before you change power, replacing `<system_odata_id>` with the path from the `Members` list (for example, `/redfish/v1/Systems/namespace__vmname`).
+
[source,terminal]
----
$ curl -sk -u "${RFUSER}:${RFPASS}" \
  "https://${RFHOST}<system_odata_id>"
----

. Power on a VM that is off by `POST`ing a `Reset` action with the `On` `ResetType`, matching the body used in the reference examples.
+
[source,terminal]
----
$ curl -sk -u "${RFUSER}:${RFPASS}" \
  -H "Content-Type: application/json" \
  -d '{"ResetType":"On"}' \
  -X POST \
  "https://${RFHOST}<system_odata_id>/Actions/ComputerSystem.Reset"
----

. Initiate a graceful shutdown, if supported, to request a guest-aware stop (behavior depends on the VM and the implementation).
+
[source,terminal]
----
$ curl -sk -u "${RFUSER}:${RFPASS}" \
  -H "Content-Type: application/json" \
  -d '{"ResetType":"GracefulShutdown"}' \
  -X POST \
  "https://${RFHOST}<system_odata_id>/Actions/ComputerSystem.Reset"
----

. When you need a hard stop (for example, for a hung guest), use a `ResetType` your product documents for a forced off or reset, and verify data integrity with your runbooks.
+
[IMPORTANT]
====
**Boot media** behavior (for example, virtual CD-ROM) and the effect of a forced **shutdown** on file systems are common pain points.
If results differ from a physical host, use xref:../../virt/post_installation_configuration/virt-kubevirt-redfish.adoc#proc_virt-troubleshooting-kubevirt-redfish_{context}[Troubleshoot KubeVirt Redfish] and confirm expectations with the SME.
====

. Optional: if your product exposes `VirtualMedia` or related actions, follow the product chart and SME guidance; do not assume full hardware-style media ejection on every cluster.

.Verification

* After each `POST` to `.../Actions/ComputerSystem.Reset`, re-run the `GET` of the `ComputerSystem` and confirm `PowerState` and `Status` report the state you expect.
* If the API returns an error body, copy the request identifier, HTTP status, and response and follow xref:../../virt/post_installation_configuration/virt-kubevirt-redfish.adoc#proc_virt-troubleshooting-kubevirt-redfish_{context}[Troubleshoot KubeVirt Redfish].
