= Configuring service discovery for OpenAPI services

[role="_abstract"]
You can configure service discovery to automatically resolve Kubernetes resource endpoints in your workflow configurations. This eliminates the need to hard code service URLs and enables workflows to adapt to different deployment environments.

.Prerequisites

* You have {ServerlessLogicOperatorName} installed on your cluster.
* You have access to a {ServerlessLogicProductName} project with the appropriate roles and permissions to create applications and other workloads in {ocp-product-title}.
* You have created your {ServerlessLogicProductName} project.
* The target Kubernetes resources exist in your cluster.
* Your workflow service account has permissions to query the Kubernetes API for the required resource types.

.Procedure

. Open the `application.properties` file in your {ServerlessLogicProductName} project directory.

. Add the REST client configuration using a service discovery URI pattern enclosed in `${}`:
+
You can see the following example for a Kubernetes Service:
+
[source,properties]
----
$ quarkus.rest-client.my_service_yaml.url=${kubernetes:services.v1//my-backend-service}
----
+
In this example:
+
* `my_service_yaml` is the REST client configuration key.
* The service discovery URI references a Kubernetes Service named `my-backend-service` in the current namespace.

. For Knative services, use either the full format or the shorthand notation:
+
You can see the following example by using the Knative shorthand:
+
[source,properties]
----
$ quarkus.rest-client.financial_service_yaml.url=${knative:finance/financial-service}
----
+
You can see the following example by using the full Knative format:
+
[source,properties]
----
$ quarkus.rest-client.financial_service_yaml.url=${kubernetes:services.v1.serving.knative.dev/finance/financial-service}
----

. To filter resources by labels, add label query parameters:
+
You can see the following example with label filters:
+
[source,properties]
----
quarkus.rest-client.backend_yaml.url=${kubernetes:services.v1//backend-service?app=backend&env=production}
----

. To specify a custom port when the service exposes multiple ports, use the port query parameter:
+
You can see the following example with custom port:
+
[source,properties]
----
$ quarkus.rest-client.api_service_yaml.url=${kubernetes:services.v1//api-service?port=https}
----

. For OpenShift Routes, use the `openshift` scheme:
+
You can see the following example for OpenShift Route:
+
[source,properties]
----
$ quarkus.rest-client.web_service_yaml.url=${openshift:routes.v1.route.openshift.io/web/frontend-route}
----

. Save the `application.properties` file.

. Deploy your workflow to the cluster. The operator automatically resolves the service discovery URIs during deployment.

.Verification

. After deployment, check the managed properties `ConfigMap` created by the operator:
+
[source,terminal]
----
$ oc get configmap <workflow_name> -o yaml
----

. Verify that the service discovery URIs are resolved to actual network endpoints:
+
[source,yaml]
----
data:
  application.properties: |
    quarkus.rest-client.my_service_yaml.url=http://my-backend-service.default.svc.cluster.local
----

. Test your workflow to ensure it successfully connects to the resolved service endpoints.
