= Understanding RBAC for Perses dashboards

[role="_abstract"]
Perses dashboards use the standard Kubernetes role-based access control (RBAC) model to control who can view, create, edit, and delete dashboards and datasources.

The Cluster Observability Operator creates the following `ClusterRole` resources:

[cols="1,2",options="header"]
|===
|ClusterRole | Permissions
|`persesdashboard-editor-role` | Create, read, update, and delete `PersesDashboard` resources. View dashboards in the console.
|`persesdashboard-viewer-role` | Read `PersesDashboard` resources. View dashboards in the console.
|`persesdatasource-editor-role` | Create, read, update, and delete `PersesDatasource` resources.
|`persesdatasource-viewer-role` | Read `PersesDatasource` resources.
|`persesglobaldatasource-editor-role` | Create, read, update, and delete `PersesGlobalDatasource` resources.
|`persesglobaldatasource-viewer-role` | Read `PersesGlobalDatasource` resources.
|===

The scope of access depends on how you bind the role:

`RoleBinding`::
Grants access to dashboards and datasources in a single namespace. Use this binding for team members who should only access a specific project.

`ClusterRoleBinding`::
Grants access to dashboards and datasources in all namespaces. Use this binding for cluster administrators or shared service accounts.

The common access patterns include:

Developer::
Grant `persesdashboard-editor-role` and `persesdatasource-viewer-role` in their project namespace using a `RoleBinding`. Developers can create and edit dashboards but cannot modify datasources.

SRE (site reliability engineer)::
Grant `persesdashboard-editor-role` and `persesdatasource-editor-role` in their project namespaces using `RoleBinding` resources.

Cluster administrator::
Grant `persesdashboard-editor-role`, `persesdatasource-editor-role`, and `persesglobaldatasource-editor-role` using `ClusterRoleBinding` resources.
