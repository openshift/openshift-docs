= Developer metrics and namespace filtering

[role="_abstract"]
The Metrics tab in the Developer perspective automatically filters queries to the selected project. This namespace-scoped filtering ensures developers query only the metrics they have permission to access.

When you run a `PromQL` query in the Developer perspective Metrics tab, the namespace selector automatically adds a namespace filter to your query. This behavior differs from the Administrator perspective, where you can query metrics across all namespaces.

== How namespace filtering works

The Developer perspective Metrics tab restricts metric queries to a single project at a time. When you select a project from the Project dropdown and click the Metrics tab, all queries are automatically scoped to that project's namespace.

For example, if you select the `my-application` project and run the query `up`, the system automatically filters it to `up{namespace="my-application"}`. You cannot query metrics from other projects in the Developer perspective.

== Contrast with Administrator perspective

The Administrator perspective Metrics view (menu:Observe[Metrics]) provides cluster-wide query capabilities:

* **Administrator perspective**: Can query metrics across all namespaces. You can write queries that total data from many projects or filter by specific namespaces.
* **Developer perspective**: Can query metrics for only one project at a time. The namespace filter is automatically applied and you cannot remove it.

This scoping aligns with the Developer perspective's focus on namespace-level operations and ensures developers access only the monitoring data they have permission to view.

== Prerequisites for using the Metrics tab

Before you can query metrics in the Developer perspective, you must meet the following prerequisites:

* A cluster administrator enables user-defined monitoring at the cluster level.
+
[NOTE]
====
To verify your cluster administrator has enabled user-defined monitoring, check if the Metrics tab displays data.
If the tab is empty or shows an error message "User-defined monitoring is not enabled", contact your cluster administrator.
====
* `ServiceMonitor` or `PodMonitor` custom resources exist in your project to expose application metrics endpoints.
* Your application exposes Prometheus-compatible metrics on an HTTP endpoint.

If `ServiceMonitor` or `PodMonitor` resources do not exist in your project, the Metrics tab displays predefined Kubernetes resource metrics for CPU, memory, and network bandwidth. Application-specific metrics do not display.

== `PromQL` autocomplete features

The Expression field in the Metrics tab provides autocomplete support to help you write `PromQL` queries:

* Metric names appear as you type
* Label names appear after you open a brace `{`
* Label values appear after you select a label name
* Functions and operators appear based on query context

The system filters autocomplete suggestions to show only metrics available in the selected project namespace.
