= Optimizing Clone Performance at Scale in {rh-storage}

When you use {rh-storage}, the storage profile configures the default link:https://docs.redhat.com/en/documentation/openshift_container_platform/4.18/html/virtualization/storage#virt-customizing-storage-profile-default-cloning-strategy_virt-configuring-storage-profile[cloning strategy] as `csi-clone`. However, this method has link:https://docs.redhat.com/en/documentation/red_hat_openshift_data_foundation/4.18/html/managing_and_allocating_storage_resources/volume-cloning_rhodf#creating-a-clone_rhodf[limitations]. After a certain number of clones are created from a persistent volume claim (PVC), a background flattening process begins. This process can significantly reduce clone creation performance at scale.

To improve performance when creating hundreds of clones from a single source PVC, use the link:https://docs.redhat.com/en/documentation/openshift_container_platform/4.18/html/storage/using-container-storage-interface-csi#persistent-storage-csi-snapshots-overview_persistent-storage-csi-snapshots[`VolumeSnapshot`] cloning method instead of the default `csi-clone` strategy.


To use this method, first create a `VolumeSnapshot` of the source image.
[source,yaml]
----
apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshot
metadata: 
  name: golden-volumesnapshot
  namespace: golden-ns
spec:
  volumeSnapshotClassName: ocs-storagecluster-rbdplugin-snapclass
  source:
    persistentVolumeClaimName: golden-snap-source

----

Then, reference the `VolumeSnapshot` as the source for the `DataVolume clone`.


[source,yaml]
----
spec:
  source:
    snapshot:
      namespace: golden-ns
      name: golden-volumesnapshot
----
