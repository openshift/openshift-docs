= Enable Gateway API for the Ingress Operator

[role="_abstract"]
To configure Gateway API for use on your cluster, you must create a `GatewayClass` resource. During the creation of the `GatewayClass` resource, the Ingress Operator installs a lightweight Istio control plane, based on {SMProductName}, in the `openshift-ingress` namespace.

.Prerequisites

* You have access to the cluster as a user with the `cluster-admin` role.
* You have installed the {oc-first}.

.Procedure

. Create a `GatewayClass` object:
+
.. Create a YAML file, `openshift-default.yaml`, that contains the following information:
+
.Example `GatewayClass` CR
[source,yaml]
----
apiVersion: gateway.networking.k8s.io/v1
kind: GatewayClass
metadata:
  name: openshift-default
spec:
  controllerName: openshift.io/gateway-controller/v1
----
+
* `metadata.name`: The name of your `GatewayClass` object. The name must consist of a maximum of 63 lowercase alphanumeric characters or hyphens (`-`). The name must also start and end with an alphanumeric character.
+
[IMPORTANT]
====
The `controllerName` value must be exactly as shown for the Ingress Operator to manage it. If you set this field to anything else, the Ingress Operator ignores the `GatewayClass` object and all associated `Gateway`, `GRPCRoute`, and `HTTPRoute` objects. The controller name is tied to the implementation of Gateway API in {product-title}, and `openshift.io/gateway-controller/v1` is the only controller name allowed.
====
+
.. Run the following command to create the `GatewayClass` resource:
+
[source,terminal]
----
$ oc create -f openshift-default.yaml
----
+
.Example output
[source,terminal]
----
gatewayclass.gateway.networking.k8s.io/openshift-default created
----

.Verification

* Verify that the `GatewayClass` CR has been accepted and the controller is successfully installed by running the following command:
+
[source,terminal]
----
$ oc get gatewayclass openshift-default -o yaml
----
+
.Example output
[source,yaml]
----
# ...
status:
  conditions:
  - lastTransitionTime: "2026-05-15T10:00:00Z"
    message: The GatewayClass has been accepted
    reason: Accepted
    status: "True"
    type: Accepted
  - lastTransitionTime: "2026-05-15T10:00:00Z"
    message: Istio installed successfully
    reason: Installed
    status: "True"
    type: ControllerInstalled
  - lastTransitionTime: "2026-05-15T10:00:00Z"
    message: Istio CRDs are being managed by cluster-ingress-operator
    reason: ManagedByCIO
    status: "True"
    type: CRDsReady
# ...
----
+
Inspect the `status.conditions` block in the output. A healthy `GatewayClass` CR reports a status of `True` for the `Accepted`, `ControllerInstalled`, and `CRDsReady` conditions.

.Troubleshooting

If you encounter issues with your Gateway API configuration, you can troubleshoot by checking logs, verifying status conditions, and monitoring metrics.

. Verify the Gateway API infrastructure is running correctly:

.. Check that the Istio deployment is ready:
+
[source,terminal]
----
$ oc get deployment -n openshift-ingress istiod-openshift-gateway
----
+
.Example output
[source,terminal]
----
NAME                       READY   UP-TO-DATE   AVAILABLE   AGE
istiod-openshift-gateway   1/1     1            1           5h
----

.. Verify that your gateway deployment is available:
+
[source,terminal]
----
$ oc get deployment -n openshift-ingress <gateway_name>-openshift-default
----
+
Replace `<gateway_name>` with the name of your `Gateway` object.

. Check the status conditions of your `Gateway`:
+
[source,terminal]
----
$ oc describe gateway <gateway_name> -n openshift-ingress
----
+
Review the `Status` section for any error messages or warnings. The `Gateway` should show `Programmed` and `Accepted` conditions as `True`.

. Inspect the `HTTPRoute` or `GRPCRoute` status:
+
[source,terminal]
----
$ oc describe httproute <route_name> -n <namespace>
----
+
Verify that the route shows `Accepted` and `ResolvedRefs` conditions as `True`, and that the `parentRef` correctly references your `Gateway`.

. Review the access logs for traffic analysis:

.. Access logs are available through the gateway pod logs. List the gateway pods:
+
[source,terminal]
----
$ oc get pods -n openshift-ingress -l gateway.networking.k8s.io/gateway-name=<gateway_name>
----

.. View the access logs from a gateway pod:
+
[source,terminal]
----
$ oc logs -n openshift-ingress <gateway_pod_name> -c istio-proxy
----
+
Access logs show each HTTP request with details including:
+
* Request time and duration
* HTTP method and path
* Response status code
* Upstream service
* Request and response sizes
* User agent and client IP
+
For information about interpreting Istio access log fields, see link:https://istio.io/latest/docs/tasks/observability/logs/access-log/[Istio Access Logging documentation^].

. Check for DNS resolution issues:

.. Verify that the `DNSRecord` is created and has a `Published` status:
+
[source,terminal]
----
$ oc get dnsrecord -n openshift-ingress -l gateway.networking.k8s.io/gateway-name=<gateway_name>
----

.. Inspect the `DNSRecord` status:
+
[source,terminal]
----
$ oc describe dnsrecord -n openshift-ingress <dnsrecord_name>
----
+
The `zones` status should show `Published: True` with the message "The DNS provider succeeded in ensuring the record".

. Troubleshoot load balancing issues:

.. Verify that the backend service endpoints are available:
+
[source,terminal]
----
$ oc get endpoints -n <namespace> <service_name>
----
+
Ensure that the service has ready endpoints listed.

.. Check if requests are reaching the backend pods:
+
[source,terminal]
----
$ oc logs -n <namespace> <backend_pod_name>
----

.. Review the `HTTPRoute` `backendRefs` configuration:
+
[source,terminal]
----
$ oc get httproute <route_name> -n <namespace> -o yaml
----
+
Verify that:
+
* The service name matches your backend service
* The port number is correct
* The namespace is correct if using cross-namespace routing

. Monitor Gateway API metrics:

.. Access the Istio proxy metrics from a gateway pod:
+
[source,terminal]
----
$ oc exec -n openshift-ingress <gateway_pod_name> -c istio-proxy -- curl localhost:15020/stats/prometheus
----
+
Key metrics include:
+
* `istio_requests_total`: Total number of requests
* `istio_request_duration_milliseconds`: Request latency
* `istio_request_bytes`: Request payload size
* `istio_response_bytes`: Response payload size
+
For detailed information about Istio metrics, see link:https://istio.io/latest/docs/reference/config/metrics/[Istio Standard Metrics^].

.. View metrics in the {product-title} web console:
+
... Navigate to *Observe* -> *Metrics*.
... Query Istio metrics using PromQL, for example:
+
[source,terminal]
----
rate(istio_requests_total{destination_service_namespace="<namespace>"}[5m])
----

. Diagnose performance issues:

.. Check the resource usage of gateway pods:
+
[source,terminal]
----
$ oc adm top pods -n openshift-ingress -l gateway.networking.k8s.io/gateway-name=<gateway_name>
----

.. Scale the `Gateway` deployment horizontally for increased capacity:
+
[source,terminal]
----
$ oc scale deployment -n openshift-ingress <gateway_name>-openshift-default --replicas=<number>
----
+
Replace `<number>` with the desired number of replicas.
