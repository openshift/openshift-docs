= Updating Istio control plane with RevisionBased strategy and IstioRevisionTag

When updating {istio} using the `RevisionBased` strategy, you can create an `IstioRevisionTag` resource to tag a specific `IstioRevision` resource. You can use the `IstioRevisionTag` resource to attach workloads to a specific `IstioRevision` resource without modifying the `istio.io/rev` label on namespaces or pods.

.Prerequisites

* You are logged in to {ocp-product-title} as a user with the `cluster-admin` role.
* You have installed the {SMProductName} Operator 3, and deployed {istio} with the `RevisionBased` strategy.
* You have created an `IstioRevisionTag` resource and the `targetRef` field is referencing the desired `{istio}` resource.
* You have installed the {istio} CNI plugin with the desired version.
* You have application workloads running in the cluster.
* You have labeled the bookinfo namespace to enable sidecar injection.
* You have updated the `{istio}` resource to the desired version.
* You have confirmed that both revisions of the control plane are running and ready.
* You have set the field `InUse` in the `IstioRevisionTag` resource to true.
* You have installed `istioctl` on your local machine.

.Procedure

When the `IstioRevisionTag` resource references an `{istio}` resource, it automatically tracks the active `IstioRevisionTag` resource associated with that `{istio}` resource instance. When you update the control plane to a new revision, the {SMProductName} Operator automatically updates the tag to reference the new revision, ensuring that namespaces and workloads referencing the tag stay current without requiring manual updates.

.  Confirm that the proxy sidecar version is the same by running the following command:
+
[source,terminal]
----
$ istioctl proxy-status
----
+
The `VERSION` column should match the old control plane version.

. Restart the application workloads so that the new version of the sidecar gets injected by running the following command:
+
[source,terminal]
----
$ oc rollout restart deployment <deployment_name>
----

.Verification

. Verify that the application workloads are running and ready by running the following command:
+
[source,terminal]
----
$ oc get pods -n <deployment_name>
----

. Verify that the new version of the sidecar is running by entering the following command:
+
[source,terminal]
----
$ istioctl proxy-status
----
+
The `VERSION` column should match the new control plane version.

. Verify the deletion of the old control plane by running the following command:
+
[source,terminal]
----
$ oc get istiorevisions
----
+
.Example Output
+
[source,yaml]
----
NAME              TYPE    READY   STATUS    IN USE   VERSION   AGE
default-v1-24-3   Local   True    Healthy   True     v1.24.3   5m31s
----

The Operator deletes the old `IstioRevision` resource and the associated control plane after the grace period defined in the `spec.updateStrategy.inactiveRevisionDeletionGracePeriodSeconds` field expires. The default grace period is 30 seconds.

You can increase the grace period to allow sufficient time to test the new control plane before removing the previous revision. Set a higher value during canary upgrades to ensure workload stability before fully transitioning.
