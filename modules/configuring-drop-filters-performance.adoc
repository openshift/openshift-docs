= Configuring drop filters for performance

[role="_abstract"]
Drop filters discard log records before forwarding, reducing network bandwidth, storage costs, and collector CPU usage. The collector still reads and evaluates all logs before it applies filters. Use drop filters when high log volume degrades performance or when certain logs provide no operational value.

.Prerequisites

* You have installed the {clo}.
* You have administrator permissions.
* You have created a `ClusterLogForwarder` custom resource.

.Procedure

. Choose the filtering scenario that matches your environment:
+
** *Scenario 1*: Drop debug logs from a specific namespace
** *Scenario 2*: Drop health check requests cluster-wide
** *Scenario 3*: Drop logs by many criteria with complex filtering logic

. Edit your `ClusterLogForwarder`:
+
[source,terminal]
----
$ oc edit clusterlogforwarder instance -n openshift-logging
----

. Add the appropriate drop filter configuration:
+
*Scenario 1: Drop debug logs from a specific namespace*
+
This filter drops all debug-level logs from the `test-app` namespace while keeping all other logs.
+
[source,yaml]
----
apiVersion: observability.openshift.io/v1
kind: ClusterLogForwarder
metadata:
  name: instance
  namespace: openshift-logging
spec:
  filters:
  - name: drop-test-debug-logs
    type: drop
    drop:
    - test:
      - field: .kubernetes.namespace_name
        matches: test-app
      - field: .level
        matches: debug

  pipelines:
  - name: application-logs
    inputRefs:
    - application
    filterRefs:
    - drop-test-debug-logs
    outputRefs:
    - default-lokistack
----
+
This filter drops records only when both conditions are true: the namespace is `test-app` and the level is `debug`.
+
*Scenario 2: Drop health check requests cluster-wide*
+
This filter drops noisy HTTP health check requests from all applications.
+
[source,yaml]
----
apiVersion: observability.openshift.io/v1
kind: ClusterLogForwarder
metadata:
  name: instance
  namespace: openshift-logging
spec:
  filters:
  - name: drop-health-checks
    type: drop
    drop:
    - test:
      - field: .message
        matches: 'GET /(health|readiness|liveness)'

  pipelines:
  - name: application-logs
    inputRefs:
    - application
    filterRefs:
    - drop-health-checks
    outputRefs:
    - default-lokistack
----
+
The regular expression pattern matches any GET request to health check endpoints.
+
*Scenario 3: Drop logs by many criteria with complex filtering*
+
This filter demonstrates advanced drop logic. It drops all logs from `dev-*` namespaces that are either debug level or contain "connection established" messages.
+
[source,yaml]
----
apiVersion: observability.openshift.io/v1
kind: ClusterLogForwarder
metadata:
  name: instance
  namespace: openshift-logging
spec:
  filters:
  - name: drop-dev-noise
    type: drop
    drop:
    - test:
      - field: .kubernetes.namespace_name
        matches: ^dev-
      - field: .level
        matches: debug
    - test:
      - field: .kubernetes.namespace_name
        matches: ^dev-
      - field: .message
        matches: connection established

  pipelines:
  - name: application-logs
    inputRefs:
    - application
    filterRefs:
    - drop-dev-noise
    outputRefs:
    - default-lokistack
----
+
The first test group drops records if namespace starts with `dev-` and level is `debug`. The second test group drops records if namespace starts with `dev-` and message contains "connection established". The filter drops records that match either test group. Within a test group, all conditions must be true.

.Verification

. Check the collector pod logs for drop filter metrics:
+
[source,terminal]
----
$ oc logs -n openshift-logging -l app.kubernetes.io/component=collector --tail=100 | grep -i drop
----
+
.Example output
[source,terminal]
----
2026-06-23T14:32:15.123Z INFO VectorSink: Records dropped by filter 'drop-health-checks': 1250
2026-06-23T14:32:20.456Z INFO VectorSink: Records dropped by filter 'drop-test-debug-logs': 340
----

. Query your log store to confirm the filtered logs are not present:
+
.. In the {product-title} web console, go to *Observe* > *Logs*.
.. Run a query for the pattern you expect to be dropped:
+
[source,terminal]
----
{kubernetes_namespace_name="test-app"} |~ "debug"
----
+
If the filter works correctly, no results appear.

. Monitor collector resource usage before and after applying filters:
+
[source,terminal]
----
$ oc adm top pods -n openshift-logging -l app.kubernetes.io/component=collector
----
+
You should see reduced CPU and memory usage after dropping high-volume logs.
