= Release notes for the {otel} 3.10.0

[role="_abstract"]
The {otel} 3.10.0 is available through the link:https://catalog.redhat.com/software/containers/rhosdt/opentelemetry-operator-bundle/615618406feffc5384e84400/history[{OTelOperator} 0.152.0] and is based on the open source link:https://opentelemetry.io/docs/collector/[OpenTelemetry] release 0.152.0.

[NOTE]
====
You can access some linked Jira tickets only with Red{nbsp}Hat credentials.
====

The {otel} 3.10.0 release adds these features and enhancements:

General availability of Collector components::
These components, available as a Technology Preview before this update, are fully supported from version 3.10.0:

* Kafka Receiver
* Kubernetes Objects Receiver
* Probabilistic Sampling Processor
* Tail Sampling Processor
* Prometheus Remote Write Exporter

Cluster TLS profile adherence::
This update introduces support for cluster TLS profile adherence. The Operator uses the TLS configuration from the `APIServer` custom resource in all TLS communication in the Operator and its operands. As a result, you can configure the TLS cluster profile by using environment variables.
+

The {otel} 3.10.0 release has this known issue:

Automatic RBAC of the Operator fails::
The automatic role-based access control (RBAC) feature of the {OTelOperator} fails due to a regression. As a consequence, the Kubernetes Cluster Receiver fails.
+
To work around this problem, add these two entries to the `ClusterRole` object named `otelcontribcol`:
+
[source,yaml]
----
# ...
  resources:
# ...
  - persistentvolumeclaims
  - persistentvolumes
# ...
----
+
link:https://redhat.atlassian.net/browse/TRACING-6381[TRACING-6381]

The {otel} 3.10.0 release fixes these issues:

Prometheus Receiver `api_server` port is exposed on Collector service and network policy::
Before this update, the `api_server` port of the Prometheus Receiver was not exposed on the Collector service and network policy. As a consequence, the Prometheus Receiver API endpoint was inaccessible from outside the Collector pod. With this update, the `api_server` port is correctly exposed. As a result, the Prometheus Receiver API endpoint is accessible from outside the Collector pod.
+
link:https://redhat.atlassian.net/browse/TRACING-5823[TRACING-5823]

Scrape class `tlsConfig` configuration is preserved in Target Allocator::
Before this update, `scrapeClass` entries with a `tlsConfig` block in the `prometheusCR` configuration were silently dropped when passed to the Target Allocator. As a consequence, TLS configuration for scrape classes was not applied, causing scrape failures. With this update, `tlsConfig` fields in scrape classes are correctly preserved. As a result, TLS configuration is correctly applied during scraping.
+
link:https://redhat.atlassian.net/browse/TRACING-6032[TRACING-6032]

The filesystem scraper of the Host Metrics Receiver collects filesystem usage metrics::
Before this update, a regression in the filesystem scraper of the Host Metrics Receiver prevented it from collecting the `system.filesystem.inodes.usage` and `system.filesystem.usage` metrics. As a consequence, these metrics were unavailable. With this update, the filesystem scraper correctly collects these metrics. As a result, the `system.filesystem.inodes.usage` and `system.filesystem.usage` metrics are available.
+
link:https://redhat.atlassian.net/browse/TRACING-5963[TRACING-5963]
