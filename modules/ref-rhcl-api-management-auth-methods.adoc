= API management authentication methods

[role="_abstract"]
You can use one of two authentication methods for protecting APIs with the {prodname} {ocp} web console plugin. Configure your chosen method at the platform level through `AuthPolicy` custom resources (CRs), which the {prodname} controller automatically discovers.

[id="rhcl-api-key-authentication_{context}"]
== API key authentication

API key authentication uses Kubernetes secrets to store credentials. This method is suitable for internal APIs, development environments, or scenarios that require fine-grained control over API access.

The following list is an example workflow:

. An API consumer creates an `APIKey` resource requesting access to an `APIProduct` resource.
. Depending on the `APIProduct` resource `approvalMode` field, one of the following events happens:

* *Automatic*: The controller immediately approves the request.
* *Manual*: The request enters `Pending` state, awaiting API owner approval.

. For manual approval, the API owner or administrator clicks *Approve* in the web console.
. After approval, credentials become available to the API consumer.
. The API consumer retrieves the key from the console and uses it in API requests.
. The `AuthPolicy` CR validates incoming requests against the credentials.

[id="rhcl-oidc-jwt-authentication_{context}"]
== OIDC and JWT authentication

OpenID Connect (OIDC) authentication delegates credential management to an external identity provider. This method is suitable for APIs that integrate with existing identity providers, require stronger authentication, or need integration with enterprise single sign-on (SSO) systems. No `APIKey` resources are created. Access control happens at the identity provider level.

There is no request or approval workflow in the {ocp} web console for this authentication method, but certain parts of the authentication process are visible. The following list is an example workflow:

. A cluster administrator configures an `AuthPolicy` resource with JWT validation pointing to an OIDC issuer.
. The {prodname} controller discovers the JSON web token (JWT) authentication scheme and performs OIDC discovery to find the token endpoint.
. Discovered authentication details are surfaced in the `APIProduct` resource status.
. An API consumer views the identity provider URL and token endpoint in the web console.
. The API consumer obtains a JWT token directly from the identity provider. For example, by using client credentials flow or any other available flow.
. The consumer uses the JWT token in API requests.
. The `AuthPolicy` CR validates the token signature and claims against the OIDC issuer.

[IMPORTANT]
====
You can create an `OIDCPolicy` object in the web console, but this policy is experimental and not supported. If you want to use OIDC authentication in your {prodname} deployment, use it with an `AuthPolicy` object.
====
