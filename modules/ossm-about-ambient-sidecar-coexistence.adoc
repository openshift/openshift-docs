= Overview of sidecar and ambient interoperability

[role="_abstract"]
Coexisting sidecar and ambient workloads operate under a single {istio} control plane that uses the ambient profile. With this configuration, you can enforce policies and route traffic across both deployment types. Review the following capabilities, limitations, and operational guidelines for managing this architecture.

[id="ossm-about-ambient-sidecar-coexistence-supported-features_{context}"]
== Supported features

[NOTE]
====
Ambient and sidecar coexistence requires {SMProductName} version 3.2 or later.
====

The following features work when sidecar and ambient workloads coexist.

[options="header"]
[cols="1,1,2"]
|===
|Category |Functionality |Description

|Connectivity
|East-west traffic
|Sidecar pods and ambient pods communicate bidirectionally across namespace boundaries.

|Security
|mTLS
|Sidecar proxies automatically detect HBONE (HTTP-Based Overlay Network Environment) endpoints and use HBONE for outbound connections to ambient pods. Ztunnel uses HBONE to reach sidecar proxies. Traffic between both modes is encrypted with mutual TLS.

|Security
|L4 `AuthorizationPolicy`
|The mesh enforces authorization policies based on principals, source IPs, and ports in both directions across modes.

|Security
|`PeerAuthentication` with `STRICT` mTLS
|A `PeerAuthentication` policy set to `STRICT` mode accepts traffic from both sidecar and ambient workloads.

|Gateways
|Egress gateways
|Ambient pods interoperate with egress gateways.

|Gateways
|Ingress gateways
|Ingress gateways deployed in non-ambient namespaces expose services in both sidecar and ambient namespaces.

|Observability
|L4 TCP metrics
|Basic Layer 4 TCP metrics and telemetry are available for traffic between sidecar and ambient workloads.
|===

[id="ossm-about-ambient-sidecar-coexistence-limitations_{context}"]
== Limitations

* **Unsupported `VirtualService` features:** Ambient mode does not support a gateway API required by the following features. Workloads that rely on these features must remain in sidecar mode:
+
** `HTTPFaultInjection`
** JWT claim-based routing
** Delegate rules
** Rate limiting
** `ignoreCase` for path matching

* **One-Way L7 policy bypass (sidecar to ambient):** Traffic from a sidecar pod to an ambient pod bypasses the waypoint proxy of the destination.
+
As a result, the mesh enforces only L4 authorization policies on this path; L7 policies (such as HTTP routing or rules) do not apply.
+
[NOTE]
====
Traffic in the opposite direction (ambient to sidecar) works without this limitation. The destination sidecar proxy enforces L7 policies normally.
====

* **Ingress gateway waypoint bypass:** By default, traffic from an ingress gateway to an ambient service bypasses the waypoint proxy. To force traffic through the waypoint for L7 enforcement, you must:
+
.. Set the environment variable `ENABLE_INGRESS_WAYPOINT_ROUTING` to `true` in the {istio} CR.
.. Label the target service or namespace with `istio.io/ingress-use-waypoint: "true"`.

[id="ossm-about-ambient-sidecar-coexistence-best-practices_{context}"]
== Best practices

* Keep ambient and sidecar workloads in separate namespaces to make the data plane mode of each workload explicit and to avoid label conflicts.

* Do not label a namespace or pod with both `istio-injection: enabled` and `istio.io/dataplane-mode: ambient`. If both labels are present, sidecar injection takes precedence, but the resulting configuration is ambiguous and difficult to maintain.
