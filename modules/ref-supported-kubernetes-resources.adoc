= Supported Kubernetes resources for service discovery

[role="_abstract"]
Service discovery in {ServerlessLogicProductName} supports automatic resolution of network endpoints for various Kubernetes resource types. Each resource type has a specific Group, Version, and Kind (GVK) identifier that you use in service discovery URI patterns.

== Supported resource types

The following table lists the Kubernetes resource types supported by service discovery:

[cols="2,3,2",options="header"]
|===
|Resource Type
|GVK (Group/Version/Kind)
|URI Pattern Example

|Services
|`v1` / `Service`
|`kubernetes:services.v1//my-service`

|Knative Services
|`serving.knative.dev/v1` / `Service`
|`kubernetes:services.v1.serving.knative.dev/default/my-knative-service` or `knative:default/my-knative-service`

|Pods
|`v1` / `Pod`
|`kubernetes:pods.v1//my-pod`

|Deployments
|`apps/v1` / `Deployment`
|`kubernetes:deployments.v1.apps//my-deployment`

|DeploymentConfigs (OpenShift)
|`apps.openshift.io/v1` / `DeploymentConfig`
|`openshift:deploymentconfigs.v1.apps.openshift.io//my-dc`

|StatefulSets
|`apps/v1` / `StatefulSet`
|`kubernetes:statefulsets.v1.apps//my-statefulset`

|Routes (OpenShift)
|`route.openshift.io/v1` / `Route`
|`openshift:routes.v1.route.openshift.io//my-route`

|Ingresses
|`networking.k8s.io/v1` / `Ingress`
|`kubernetes:ingresses.v1.networking.k8s.io//my-ingress`
|===

== Resource-specific considerations

Services::
The most common resource type for service discovery. Services provide stable network endpoints and built-in load balancing across pods.

Knative Services::
Support both the full GVK format and a shorthand notation (`knative:<namespace>/<serviceName>`). Knative services automatically scale based on-demand and can scale to zero when not in use.

Pods::
Directly reference individual pod endpoints. This approach is less common because pods are ephemeral and their IP addresses change when pods are recreated.

Deployments and StatefulSets::
Reference the underlying service or pods managed by these controllers. For most use cases, reference the Service resource instead of the Deployment or StatefulSet directly.

OpenShift Routes::
Provide external access to services in OpenShift clusters. Routes expose services through a hostname that external clients can access.

Ingresses::
Similar to OpenShift Routes but use the standard Kubernetes Ingress resource. Ingresses provide HTTP and HTTPS routing to services based on rules.

== Using resource types in workflows

When you reference a resource in your workflow configuration, ensure that:

* The resource exists in the specified namespace or the current namespace if no namespace is specified.
* Your workflow service account has the necessary permissions to query the Kubernetes API for the resource type.
* The resource exposes the network endpoint that your workflow requires.
