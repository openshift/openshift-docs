= Deploying Applications that Use Generic Devices

After the Generic Device Plugin is configured and enabled in {microshift-short}, you can deploy Kubernetes workloads (such as pods, deployments, or StatefulSets) that request access to the host devices that you have exposed. The device is made available inside the container without requiring the pod to run with elevated privileges.

.Prerequisites
* The Generic Device Plugin must be enabled and configured on your MicroShift node, as described in the "Configuring the Generic Device Plugin" module.
* You have `oc` configured to interact with your MicroShift cluster.

.Procedure
. Define the device request in your Pod specification*:
+
In your Kubernetes workload manifest (for example, for a pod, deployment, or StatefulSet), add a `resources.limits` entry to the container that needs device access. The resource name should follow the pattern `device.microshift.io/<device-name>`, where `<device-name>` matches the `name` tath you specified in your `config.yaml` file. The value indicates the number of instances of that device type the container requires.
+
[source,yaml]
----
apiVersion: v1
kind: Pod
metadata:
  name: my-device-app
spec:
  containers:
  - name: my-container
    image: <your-application-image> # Replace with your container image
    command: ["/path/to/your/app"] # Replace with your application's command
    args: ["--device-path=/dev/video0"] # Example: how your app might use the device
    resources:
      limits:
        device.microshift.io/video: 1 # Request one instance of the 'video' device
----
+
. Deploy the Kubernetes workload
+
Apply your manifest to the {microshift-short} cluster:
+
[source,bash]
----
oc apply -f <your-workload-manifest.yaml>
----
+
. Verify device access from within the container
+
After the pod is running, the specified host device is available at its original path (or `mountPath` if specified) inside the container. Your application can then interact with it as if it were a local device.
+
For example, if you requested `device.microshift.io/serial` (which maps to `/dev/ttyUSB*`), your application might find the device at `/dev/ttyUSB0` or similar inside the container.
+
You can verify by executing a command inside the running pod:
+
[source,bash]
----
kubectl exec -it <pod-name> -- ls -l /dev/video0
----
+
This command should show the device file inside the container, confirming access.
```
