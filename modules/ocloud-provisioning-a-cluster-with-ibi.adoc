= Provision a single-node cluster with image-based installation

[role="_abstract"]
You can provision a single-node OpenShift cluster using image-based installation (IBI) through the O-Cloud Manager. IBI uses a pre-built seed image and a factory-provisioned live ISO to significantly reduce installation time. Servers arrive pre-provisioned with the operating system and base cluster content, so provisioning skips the full OS installation stage.

This procedure documents the differences from the standard provisioning flow. The overall process of creating a `ProvisioningRequest` and monitoring its status is the same.

.Prerequisites

* A validated `ClusterTemplate` exists that references IBI-specific `clusterinstance-defaults` ConfigMap.
* Image-Based Install Operator is enabled in the `MultiClusterEngine` resource on the hub cluster.
* Servers are factory-provisioned with the seed image and live ISO.
* Bare-metal servers are onboarded with an IBI-identifying resource selector label.

.Procedure

. Create a `ClusterTemplate` that uses IBI-specific cluster instance defaults:
+
[source,yaml]
----
apiVersion: clcm.openshift.io/v1alpha1
kind: ClusterTemplate
metadata:
  name: sno-ran-du-ibi.v4-Y-Z-1
  namespace: <template_namespace>
spec:
  name: sno-ran-du-ibi
  version: v4-Y-Z-1
  release: 4.Y.Z
  templateDefaults:
    hwMgmtDefaults:
      nodeGroupData:
        - name: controller
          role: master
          resourceSelector:
            resourceselector.clcm.openshift.io/server-type: XR8620t
            resourceselector.clcm.openshift.io/ibi-ready: "true"
    clusterInstanceDefaults: ibi-clusterinstance-defaults-v1
    policyTemplateDefaults: policytemplates-defaults-v1
  templateParameterSchema:
    properties:
      nodeClusterName:
        type: string
      oCloudSiteId:
        type: string
      policyTemplateParameters:
        properties:
          cpu-isolated:
            type: string
        type: object
      clusterInstanceParameters:
        properties:
          clusterName:
            type: string
          baseDomain:
            type: string
          nodes:
            items:
              properties:
                hostName:
                  type: string
              required:
                - hostName
              type: object
            type: array
        required:
          - clusterName
          - nodes
        type: object
    required:
      - nodeClusterName
      - oCloudSiteId
      - policyTemplateParameters
      - clusterInstanceParameters
    type: object
----
+
The key difference from a standard `ClusterTemplate` is the `clusterInstanceDefaults` reference. The IBI-specific ConfigMap references `ibi-cluster-templates-v1` and `ibi-node-templates-v1` SiteConfig Operator templates, which instruct the installation to use the pre-provisioned seed image instead of performing a full OS installation.

. Create the IBI-specific `clusterinstance-defaults` ConfigMap:
+
[source,yaml]
----
apiVersion: v1
kind: ConfigMap
metadata:
  name: ibi-clusterinstance-defaults-v1
  namespace: <template_namespace>
data:
  clusterinstance-defaults: |
    clusterImageSetNameRef: "4.Y.Z"
    pullSecretRef:
      name: pull-secret
    networkType: OVNKubernetes
    templateRefs:
      - name: ibi-cluster-templates-v1
        namespace: open-cluster-management
    nodes:
      - role: master
        bootMode: UEFI
        ironicInspect: "disabled"
        automatedCleaningMode: disabled
        nodeNetwork:
          interfaces:
            - name: ens3f0
              label: boot-interface
        templateRefs:
          - name: ibi-node-templates-v1
            namespace: open-cluster-management
----
+
* `templateRefs` uses `ibi-cluster-templates-v1` and `ibi-node-templates-v1` instead of the standard `ai-cluster-templates-v1` and `ai-node-templates-v1`.
* IBI templates are provided by the Image-Based Install Operator through {rh-rhacm-first}.

. Create a `ProvisioningRequest` referencing the IBI `ClusterTemplate`:
+
[source,yaml]
----
apiVersion: clcm.openshift.io/v1alpha1
kind: ProvisioningRequest
metadata:
  name: <valid_uuid>
spec:
  name: sno1-ibi-request
  description: "IBI-based SNO cluster"
  templateName: sno-ran-du-ibi
  templateVersion: v4-Y-Z-1
  templateParameters:
    nodeClusterName: sno1-ibi
    oCloudSiteId: site-rack-a
    clusterInstanceParameters:
      clusterName: sno1-ibi
      baseDomain: example.com
      nodes:
        - hostName: sno1-ibi.example.com
    policyTemplateParameters:
      cpu-isolated: "2-63"
----
+
The `ProvisioningRequest` structure is identical to the standard flow. The O-Cloud Manager determines the installation method based on the `ClusterTemplate` and its referenced defaults.

. Apply the `ProvisioningRequest`:
+
[source,terminal]
----
$ oc apply -f <provisioning_request_filename>.yaml
----
+
Because servers are pre-provisioned with the operating system, the cluster installation stage completes significantly faster than standard provisioning. The server boots from the pre-installed seed image and applies cluster-specific configuration without requiring a full OS installation.
+
[NOTE]
====
When you delete a `ProvisioningRequest` for an IBI-provisioned cluster, the server returns to the `Available` state. You can re-use deprovisioned IBI servers in two ways:

* Re-provision with another IBI `ClusterTemplate` to re-use the existing seed image.
* Convert to standard provisioning by onboarding the server without the IBI resource selector label and using a standard `ClusterTemplate`.
====

.Verification

* Verify the `ProvisioningRequest` reaches the `fulfilled` phase:
+
[source,terminal]
----
$ oc get provisioningrequests.clcm.openshift.io <uuid>
----
