= Onboard bare-metal servers to resource pools

[role="_abstract"]
You can register bare-metal servers with the O-Cloud Manager by defining an infrastructure hierarchy and creating `BareMetalHost` resources with resource selector labels. Onboarded servers become available for allocation when a `ProvisioningRequest` references a `ClusterTemplate` with matching resource selectors.

.Prerequisites

* The O-Cloud Manager Operator is installed on the hub cluster.
* You have BMC (Baseboard Management Controller) credentials for each server.
* The Bare Metal Operator is available on the hub cluster.

.Procedure

. Create the infrastructure hierarchy by defining `Location`, `OCloudSite`, and `ResourcePool` custom resources:
+
[source,yaml]
----
apiVersion: ocloud.openshift.io/v1alpha1
kind: Location
metadata:
  name: us-east-dc1
  namespace: <server_namespace>
spec:
  description: "US East Data Center 1"
  address: "123 Data Center Blvd, Virginia, US"
---
apiVersion: ocloud.openshift.io/v1alpha1
kind: OCloudSite
metadata:
  name: site-rack-a
  namespace: <server_namespace>
spec:
  description: "Rack A at US East DC1"
  globalLocationName: us-east-dc1
---
apiVersion: ocloud.openshift.io/v1alpha1
kind: ResourcePool
metadata:
  name: pool-dell-xr8620t
  namespace: <server_namespace>
spec:
  description: "Dell XR8620t server pool"
  oCloudSiteName: site-rack-a
----
+
* A `Location` represents a geographic place where infrastructure is deployed. You must specify at least one of `address`, `coordinate`, or `civicAddress`.
* An `OCloudSite` represents a logical site at a given location. The `globalLocationName` field references the parent `Location` by its `metadata.name`.
* A `ResourcePool` represents a group of bare-metal resources at a site. The `oCloudSiteName` field references the parent `OCloudSite` by its `metadata.name`.

. Create a `Secret` containing the BMC credentials for the server:
+
[source,yaml]
----
apiVersion: v1
kind: Secret
metadata:
  name: bmc-credentials-server1
  namespace: <server_namespace>
type: Opaque
data:
  username: <base64_encoded_username>
  password: <base64_encoded_password>
----

. Create a `BareMetalHost` resource with resource selector labels that associate the server with a resource pool and identify its characteristics:
+
[source,yaml]
----
apiVersion: metal3.io/v1alpha1
kind: BareMetalHost
metadata:
  name: server1
  namespace: <server_namespace>
  labels:
    resourceselector.clcm.openshift.io/resource-pool: pool-dell-xr8620t
    resourceselector.clcm.openshift.io/server-type: XR8620t
    resourceselector.clcm.openshift.io/site: site-rack-a
  annotations:
    bmac.agent-install.openshift.io/hostname: server1.example.com
spec:
  online: false
  bootMACAddress: "AA:BB:CC:DD:EE:FF"
  automatedCleaningMode: disabled
  bmc:
    address: idrac-virtualmedia+https://192.168.1.100/redfish/v1/Systems/System.Embedded.1
    credentialsName: bmc-credentials-server1
    disableCertificateVerification: true
----
+
* Labels with the `resourceselector.clcm.openshift.io/` prefix are used by the O-Cloud Manager to select servers during provisioning. These labels must match the `resourceSelector` criteria defined in the `ClusterTemplate`.
* `spec.online: false` prevents the server from being powered on before provisioning.
* `spec.bmc.address` specifies the BMC endpoint using the Redfish virtual media protocol.

. Apply the resources to the hub cluster:
+
[source,terminal]
----
$ oc apply -f <infrastructure_hierarchy_filename>.yaml
$ oc apply -f <bmc_secret_filename>.yaml
$ oc apply -f <baremetalhost_filename>.yaml
----
+
After the `BareMetalHost` is created, the Bare Metal Operator connects to the BMC, inspects the physical hardware, and populates a `HardwareData` resource with CPU, memory, NIC, and storage details. This data is used for hardware data filtering during resource selection.

.Verification

. Verify that the infrastructure hierarchy resources are created:
+
[source,terminal]
----
$ oc get locations.ocloud.openshift.io
$ oc get ocloudsites.ocloud.openshift.io
$ oc get resourcepools.ocloud.openshift.io
----

. Verify that the `BareMetalHost` is registered and inspection is complete:
+
[source,terminal]
----
$ oc get bmh -A
----
+
The server should show a provisioning state of `available` after inspection completes, indicating it is ready for allocation.
