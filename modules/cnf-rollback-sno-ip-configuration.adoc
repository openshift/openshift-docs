= Roll back a network reconfiguration

[role="_abstract"]
If you encounter issues after a network reconfiguration, you can roll back to the previous network configuration. The rollback reboots the node into the previous stateroot with the original network settings.

.Prerequisites

* You have completed a network reconfiguration that has not been finalized.
* The `status.validNextStages` field in the `IPConfig` CR includes `Rollback`.
* You have cluster administrator privileges.

.Procedure

. Verify that rollback is available by checking the `IPConfig` CR:
+
[source,terminal]
----
$ oc get ipc ipconfig -o jsonpath='{.status.validNextStages}'
----
+
Verify that the output includes `Rollback`.

. Check the rollback availability expiration timestamp:
+
[source,terminal]
----
$ oc get ipc ipconfig -o jsonpath='{.status.rollbackAvailabilityExpiration}'
----
+
[NOTE]
====
The `rollbackAvailabilityExpiration` field indicates when rolling back might require manual recovery due to expired control plane or kubelet certificates in the rollback stateroot. Plan your rollback before this timestamp if possible.
====

. Trigger the rollback by setting the stage to `Rollback`:
+
[source,terminal]
----
$ oc patch ipc ipconfig --type merge -p '{"spec":{"stage":"Rollback"}}'
----
+
The {lcao} reboots the node into the previous stateroot.

. After the node reboots, monitor the rollback progress:
+
[source,terminal]
----
$ oc get ipc ipconfig -o yaml
----
+
Wait for the `RollbackCompleted` condition to be set.

. Verify the node is using the original network configuration:
+
[source,terminal]
----
$ oc get nodes -o wide
----

. Finalize the rollback by setting the stage to `Idle`:
+
[source,terminal]
----
$ oc patch ipc ipconfig --type merge -p '{"spec":{"stage":"Idle"}}'
----
