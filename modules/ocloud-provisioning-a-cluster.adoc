= Provision a cluster using a ProvisioningRequest

[role="_abstract"]
You can provision a cluster by creating a `ProvisioningRequest` custom resource that references a validated `ClusterTemplate` and supplies site-specific parameters. The O-Cloud Manager validates the request, allocates hardware from resource pools, and orchestrates the end-to-end provisioning flow.

.Prerequisites

* A validated `ClusterTemplate` exists on the hub cluster (`ClusterTemplateValidated` condition is `True`).
* Bare-metal servers are onboarded and in the `Available` provisioning state with resource selector labels matching the `ClusterTemplate` requirements.
* The ACM policy templates are synchronized to the hub cluster through ArgoCD.

.Procedure

. Create a `ProvisioningRequest` custom resource that references the `ClusterTemplate` and supplies site-specific parameters:
+
[source,yaml]
----
apiVersion: clcm.openshift.io/v1alpha1
kind: ProvisioningRequest
metadata:
  name: 123e4567-e89b-12d3-a456-426614174000 <1>
spec:
  name: sno1-request
  description: "SNO cluster for site rack-a"
  templateName: sno-ran-du
  templateVersion: v4-Y-Z-1
  templateParameters:
    nodeClusterName: sno1
    oCloudSiteId: site-rack-a
    clusterInstanceParameters:
      clusterName: sno1
      baseDomain: example.com
      nodes:
        - hostName: sno1.example.com
    policyTemplateParameters:
      cpu-isolated: "2-63"
      cpu-reserved: "0-1"
      hugepages-default: "1G"
      hugepages-size: "1G"
      hugepages-count: "32"
----
<1> The `metadata.name` must be a valid UUID. Any attempt to create a `ProvisioningRequest` with an invalid name is rejected by the webhook.
+
* `spec.templateName` and `spec.templateVersion` identify the `ClusterTemplate` to use.
* `spec.templateParameters.clusterInstanceParameters` provides site-specific values that are merged with the defaults from the `clusterinstance-defaults` ConfigMap. The merged data is used to render the `ClusterInstance` resource.
* `spec.templateParameters.policyTemplateParameters` provides site-specific values for Day-2 configuration policies. These are merged with the `policytemplates-defaults` ConfigMap values.

. Apply the `ProvisioningRequest`:
+
[source,terminal]
----
$ oc apply -f <provisioning_request_filename>.yaml
----
+
The O-Cloud Manager processes the request through the following stages:
+
.. Validates the parameters against the `ClusterTemplate` schema.
.. Renders and validates the `ClusterInstance` resource via dry-run.
.. Creates a `NodeAllocationRequest` to allocate hardware from resource pools.
.. Applies firmware and BIOS settings if a `HardwareProfile` is specified.
.. Creates the `ClusterInstance` resource, triggering cluster installation through the SiteConfig Operator.
.. Monitors ACM policy compliance until all configuration is applied.
+
[NOTE]
====
Each provisioning stage has a default timeout: 90 minutes for hardware provisioning, 90 minutes for cluster installation, and 30 minutes for cluster configuration. You can override these timeouts in the `ClusterTemplate` by setting `hardwareProvisioningTimeout` in `hwMgmtDefaults`, `clusterInstallationTimeout` in the `clusterinstance-defaults` ConfigMap, or `clusterConfigurationTimeout` in the `policytemplates-defaults` ConfigMap.
====

.Verification

* Verify that provisioning completes successfully:
+
[source,terminal]
----
$ oc get provisioningrequests.clcm.openshift.io 123e4567-e89b-12d3-a456-426614174000
----
+
.Example output
[source,terminal]
----
NAME                                   DISPLAYNAME    AGE   PROVISIONPHASE   PROVISIONDETAILS
123e4567-e89b-12d3-a456-426614174000   sno1-request   71m   fulfilled        The cluster provisioning has completed successfully
----
+
A `provisioningPhase` of `fulfilled` indicates that hardware allocation, cluster installation, and configuration are all complete.
