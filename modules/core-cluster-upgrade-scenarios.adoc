[id="core-cluster-upgrade-scenarios_{context}"]
= Bare-metal core cluster update scenarios

[role="_abstract"]
You can perform z-stream, y-stream, and EUS-to-EUS updates on multi-node bare-metal core clusters.
Each update scenario has different risk profiles, release cadences, and use cases that align with your operational requirements and maintenance windows.

Bare-metal core clusters require careful update planning to maintain service availability for network functions.
Understanding the differences between update scenarios helps you select the right approach for your deployment.

[id="z-stream-upgrades_{context}"]
== Z-stream updates

Z-stream updates apply patch releases within a minor version, such as updating from 4.20.0 to 4.20.1.
These updates address security vulnerabilities and bug fixes without introducing new features.

Z-stream releases follow a weekly cadence and should be applied as soon as they become available.
The risk profile for z-stream updates is low because they do not include API changes or feature modifications.

The following list describes use cases for z-stream updates:

* Applying critical security patches (CVEs)
* Resolving production bugs identified in the current minor version
* Maintaining compliance with security policies that require timely patching
* Regular maintenance updates during normal operating windows

When configured with proper pod disruption budgets, z-stream updates cause minimal workload disruption.
Control plane nodes update in a rolling fashion, and worker nodes update one at a time or in small batches.

[id="y-stream-upgrades_{context}"]
== Y-stream updates

Y-stream updates move between minor versions, such as updating from 4.20.x to 4.21.0.
These updates introduce new features, performance improvements, and might include API deprecations.

Y-stream releases follow a quarterly cadence aligned with the {product-title} release schedule.
You must update through consecutive y-stream versions and cannot skip versions.
For example, to update from 4.20 to 4.22, you must update from 4.20 to 4.21, and then from 4.21 to 4.22.

The risk profile for y-stream updates is medium because they might include deprecated APIs and configuration changes.
Review release notes carefully before updating to identify breaking changes that affect your workloads.

The following list describes use cases for y-stream updates:

* Accessing new {product-title} features and capabilities
* Maintaining support lifecycle compliance (older versions eventually lose support)
* Meeting requirements for updated operator versions
* Adopting performance improvements and optimizations

Y-stream updates require a planned maintenance window.
The control plane update typically completes in 60 to 120 minutes, and worker node updates take an additional 30 to 60 minutes.
You can minimize workload disruption by pausing worker nodes and updating the control plane first.

[id="eus-to-eus-upgrades_{context}"]
== EUS to EUS updates

Extended Update Support (EUS) releases receive 18 months of support and include even-numbered minor versions such as 4.18, 4.20, and 4.22.
With EUS-to-EUS updates, you can jump between EUS versions while skipping odd-numbered releases.

With control-plane-only updates, you can update the control plane to a new EUS version while leaving worker nodes at the previous version.
The control plane and workers can be up to two minor versions apart during this staged approach.

The risk profile for EUS-to-EUS updates is medium to high because they involve larger version jumps.
Thorough testing and validation are critical before performing EUS-to-EUS updates in production.

The following list describes use cases for EUS-to-EUS updates:

* Staging control plane updates separately from worker node updates
* Minimizing workload disruption by deferring worker updates to a later maintenance window
* Aligning updates with long-term support cycles (18-month support windows)
* Managing capacity-constrained environments where full cluster downtime is not feasible

Control-plane-only EUS updates affect only control plane components.
Workloads continue running on worker nodes at the previous version.
Worker nodes can be updated days or weeks after the control plane, depending on your maintenance schedule.

[id="selecting-an-upgrade-scenario_{context}"]
== Selecting an update scenario

The following decision matrix helps you determine which update scenario to use:

.Decision matrix for update scenarios
[cols="1,2,2",options="header"]
|===
|Scenario
|When to use
|Key considerations

|Z-stream
|Security patches or bug fixes needed
|Apply weekly; minimal risk; minimal downtime

|Y-stream
|New features required or approaching end of support
|Quarterly cadence; review release notes; plan maintenance window

|EUS-to-EUS
|Long-term planning with staged rollouts
|18-month support; control plane and workers can update separately; larger version jumps require more validation
|===

[id="application-and-cluster-configuration-policy-changes_{context}"]
== Application and cluster configuration policy changes

In addition to {product-title} and OLM operator update policies, review and prepare policies for application changes and cluster configuration changes that might be required as part of the update.
This can include changes to user policies, security context constraints (SCCs), or version-specific configuration for your cluster.

For further policy configuration, see the {rh-rhacm-first} documentation.

[id="multi-node-cluster-considerations_{context}"]
== Multi-node cluster considerations

Multi-node bare-metal core clusters have additional considerations for each update scenario.

The following list describes the key considerations:

* Control plane quorum: During updates, etcd quorum must be maintained.
With three or more control plane nodes, one control plane node updates at a time while the others maintain quorum.
* Worker node batching: Worker nodes update in batches controlled by the `maxUnavailable` setting in the `MachineConfigPool` resource.
The appropriate value for `maxUnavailable` depends on your application configuration and how your workloads tolerate disruption.
* Pod topology spread constraints: You can use pod topology spread constraints to improve update resilience by spreading workloads across fault-isolated zones.
* Network requirements: Cluster network connectivity must remain available between nodes during rolling updates.
The Container Network Interface (CNI) plugin must support version skew during control plane and worker node updates.
The default OVN-Kubernetes CNI plugin supports version skew.
If you are using a third-party CNI plugin, such as Calico, Cilium, or a vendor-specific CNI, consult the CNI vendor documentation to verify version skew support before performing control-plane-only updates.
* Pod disruption budgets: Configure `PodDisruptionBudget` resources for all critical network functions to ensure minimum replica availability during node updates.

[id="cgu-operator-orchestration-for-upgrades_{context}"]
== {cgu-operator} orchestration for updates

{cgu-operator-full} ({cgu-operator}) automates policy-based updates across multiple bare-metal core clusters.
{cgu-operator} uses `ClusterGroupUpgrade` custom resources to orchestrate update rollouts in batches.

{cgu-operator} supports all three update scenarios (z-stream, y-stream, and EUS-to-EUS) and integrates with {rh-rhacm} policies.
You define the required cluster version in a GitOps policy, and {cgu-operator} applies that policy to target clusters according to your batching and concurrency settings.
