= Understanding the PromQL expression for alerts
////
:FeatureName: Network observability alerts
include::snippets/technology-preview.adoc[]
////
[role="_abstract"]
Learn about the base query for Prometheus Query Language (`PromQL`), and how to customize it so you can configure network observability alerts for your specific needs.

The alerting API in the network observability `FlowCollector` custom resource (CR) is a mapping to the Prometheus Operator API, generating a `PrometheusRule`. You can see the `PrometheusRule` in the default `netobserv` namespace by running the following command:

[source,terminal]
----
$ oc get prometheusrules -n netobserv -oyaml
----

[id="example-base-query-alert-for-surge-in-incoming-traffic_{context}"]
== Example of a base query for an alert in a surge in incoming traffic

This example gives you the formula of a base `PromQL` query pattern for an alert about a surge in incoming traffic:

[source,text]
----
sum(rate(netobserv_workload_ingress_bytes_total{SrcK8S_Namespace="openshift-ingress"}[30m])) by (DstK8S_Namespace)
----

This is the bytes rate coming from an `openshift-ingress` namespace to any of your workload's namespaces over the last 30 minutes.

Appending `> 1000` to this query keeps only the rates observed that are greater than `1 KB/s`, which eliminates noise from low-bandwidth consumers:

[source,text]
----
(sum(rate(netobserv_workload_ingress_bytes_total{SrcK8S_Namespace="openshift-ingress"}[30m])) by (DstK8S_Namespace) > 1000)
----

The bytes rate is relative to the sampling ratio defined in the `FlowCollector` custom resource (CR)configuration. If you have a sampling ratio of `1:100`, the actual traffic might be approximately 100 times higher than what is reported by the metrics.

You can run the same query for a particular period of time. For example, `offset 1d` runs the query for one day earlier, and `offset 5h` runs the query for five hours ago:

[source,text]
----
sum(rate(netobserv_workload_ingress_bytes_total{SrcK8S_Namespace="openshift-ingress"}[30m] offset 1d)) by (DstK8S_Namespace))
----

This gives you the formula `100 * (<query now> - <query yesterday>) / <query yesterday>`, which is the percentage of increase compared to the previous day. This value can be negative if the bytes rate today is lower than yesterday.

You can also eliminate increases that are lower than you want so you won't get alerted for them. For example, `> 100` eliminates increases that are lower than 100%.

Together, the expression looks something like this:

[source,text]
----
...
      expr: |-
        (100 *
          (
            (sum(rate(netobserv_workload_ingress_bytes_total{SrcK8S_Namespace="openshift-ingress"}[30m])) by (DstK8S_Namespace) > 1000)
            - sum(rate(netobserv_workload_ingress_bytes_total{SrcK8S_Namespace="openshift-ingress"}[30m] offset 1d)) by (DstK8S_Namespace)
          )
          / sum(rate(netobserv_workload_ingress_bytes_total{SrcK8S_Namespace="openshift-ingress"}[30m] offset 1d)) by (DstK8S_Namespace))
        > 100
----

You can configure `PromQL` queries to fit your specific needs.

== Alert Metadata for Prometheus and Network Health

The following standard metadata is required for Prometheus and the Network Observability Operator `AlertManager` object:

[source,yaml]
----
      annotations:
        message: |-
          NetObserv is detecting a surge of incoming traffic: current traffic to {{ $labels.DstK8S_Namespace }} has increased by more than 100% since yesterday.
        summary: "Surge in incoming traffic"
      labels:
        severity: warning
----

You can leverage the output labels from the defined `PromQL` expression in the `message` annotation. For example, since results are grouped per `DstK8S_Namespace`, the expression `{{ $labels.DstK8S_Namespace }}` can be used in the message text.

[IMPORTANT]
====
The `severity` label must be set to `critical`, `warning`, or `info`.
====

To ensure the alert is recognized and displayed in the *Network Health* dashboard, the following metadata is required:

[source,yaml]
----
      annotations:
        netobserv_io_network_health: '{"namespaceLabels":["DstK8S_Namespace"],"threshold":"100","unit":"%","upperBound":"500"}'
      labels:
        netobserv: "true"
----

[IMPORTANT]
====
The label `netobserv: "true"` is required for the dashboard to detect the alert.
====

The annotation `netobserv_io_network_health` is optional and provides control over how the alert is rendered on the *Network Health* page. It is a JSON string that consists of the following fields:

.Fields for `netobserv_io_network_health` annotation
[cols="2,2,6",options="header"]
|===
| Field
| Type
| Description

| `namespaceLabels`
| List of strings
| One or more labels that hold namespaces. When provided, the alert appears under the **`*Namespaces*`** tab.

| `nodeLabels`
| List of strings
| One or more labels that hold node names. When provided, the alert appears under the **`*Nodes*`** tab.

| `threshold`
| String
| The alert threshold, expected to match the threshold defined in the `PromQL` expression.

| `unit`
| String
| The data unit, used only for display purposes.

| `upperBound`
| String
| An upper bound value used to compute the score on a closed scale. Metric values exceeding this bound are clamped.

| `links`
| List of objects
| A list of links to display contextually with the alert. Each link requires a `name` (display name) and `url`.

| `trafficLinkFilter`
| String
| An additional filter to inject into the URL for the **`*Network Traffic*`** page.
|===

The `namespaceLabels` and `nodeLabels` are mutually exclusive. If neither is provided, the alert appears under the *Global* tab.
