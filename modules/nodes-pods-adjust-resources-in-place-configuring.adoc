= Configuring in-place pod resizing

In-place pod resizing requires that you add a resize policy to a pod specification. If you are editing an existing pod specification by using the {oc-first}, you must use the `--subresource resize` flag in the edit command, as shown in the following procedure. Because you need to use the flag, you cannot edit the pods in the {product-title} web console.

You cannot add or modify a resize policy to an existing pod, but you can add or edit the policy in the pod's owner object, such as a deployment, if the pod has an owner object. 

.Procedure

. Create a pod spec with a resize policy or add a resize policy to the owner object of an existing pod:

.. Create a YAML file similar to the following example:
+
[source,yaml]
----
apiVersion: v1
kind: Pod
metadata:
  name: resize-pod
spec:
# ...
  containers:
  - name: pause
    resizePolicy: <1>
    - resourceName: cpu
      restartPolicy: NotRequired
    - resourceName: memory
      restartPolicy: RestartContainer
# ...
----
<1> Specifies a resize policy. For CPU and/or memory resources specify one of the following values:
+
* `NotRequired`: Apply any resource changes without restarting the pod. This is the default.
* `RestartContainer`: Apply any resource changes and restart the pod.

.. Create the `Pod` object by running the following command:
+
[source,terminal]
----
$ oc create -f <file_name>.yaml
----

. Optional: After the pod is created, add or modify the CPU or memory resource and limits for the pod by using a command similar to the following. You must include the `--subresource resize` flag with the command.
+
[source,terminal]
----
$ oc edit pod <pod_name> --subresource resize
----
+
[source,yaml]
----
apiVersion: v1
kind: Pod
metadata:
  name: resize-pod
spec:
# ...
  containers:
  - name: pause
    resizePolicy:
    - resourceName: cpu
      restartPolicy: NotRequired
    - resourceName: memory
      restartPolicy: RestartContainer
    resources: <1>
      limits:
        cpu: "700m"
        memory: "200Mi"
      requests:
        cpu: "700m"
        memory: "200Mi"
# ...
----
<1> Specifies CPU and/or memory resources and limits. For more information, see "Understanding resource requests and limits". In-place pod resizing is available for only CPU and memory resources. 
+
While the change is being applied, you can check the progress of the resize in the pod specification by using a command similar to the following:
+
[source,terminal]
----
$ oc edit pod <pod_name>  --subresource resize
----
+
.Example output:
[source,yaml]
----
apiVersion: v1
kind: Pod
metadata:
  name: resize-pod
spec:
# ...
status:
  conditions:
  - lastProbeTime: "2025-09-09T20:50:31Z"
    lastTransitionTime: "2025-09-09T20:50:31Z"
    status: "True"
    type: PodResizeInProgress
# ...
----
+
After the change is applied, the pod in this example is updated without a restart for changes to CPU because the resize policy for CPU is `NotRequired`.
+
[source,terminal]
----
$oc get pods
----
+
.Example output
[source,terminal]
----
NAME                          READY   STATUS    RESTARTS     AGE
resize-pod                    1/1     Running   0            9m33s
----
+
In this example, if you change the memory request or limits, the pod is restarted because the resize policy for memory is `RestartContainer`.
+
[source,terminal]
----
$oc get pods
----
+
.Example output
[source,terminal]
----
NAME                         READY   STATUS    RESTARTS    AGE
resize-pod                   1/1     Running   1 (5s ago)  5s
----

.Verification

* Check that the modified resources are applied to the container in the pod by using a command similar to the following:
+
[source,terminal]
----
$oc describe pod <pod_name> -o yaml
----
+
[source,yaml]
----
apiVersion: v1
kind: Pod
metadata:
  name: resize-pod
# ...
spec:
  containers:
# ...
    resources: <1>
      limits:
        cpu: 800m
        memory: 300Mi
      requests:
        cpu: 800m
        memory: 300Mi
# ...
status:
# ...
 containerStatuses:
    resources: <2>
      limits:
        cpu: 800m
        memory: 300Mi
      requests:
        cpu: 800m
        memory: 300Mi
# ...
----
<1> Shows the resources that were requested by the changes to the pod spec. 
<2> Shows the actual resources allocated to the container on the pod.
