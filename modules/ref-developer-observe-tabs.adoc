= Developer perspective Observe tabs

[role="_abstract"]
The Observe view in the Developer perspective provides seven tabs for monitoring your project. Each tab offers specific capabilities for tracking application health, performance, and alerts.

The following table describes the Developer perspective Observe tabs and their capabilities:

.Developer perspective Observe tabs
[cols="20%,40%,40%",options="header"]
|===
|Tab name |Purpose |Key capabilities

|Events
|View Kubernetes events for project resources
|* Filter events by resource type
* Filter events by event type
* Search events by name or message
* View event timestamps and reasons

|Alerting Rules
|View alerting rules defined for the project
|* View rules created via PrometheusRule custom resources
* Filter rules by state or severity
* View rule expressions and labels
* View which alerts each rule triggers

|Alerts
|View firing and pending alerts
|* Filter alerts by state and severity
* View alert details and metrics
* Create silences for specific alerts
* View alert history

|Dashboard
|View pre-built Kubernetes compute resources dashboards
|* View CPU, memory, bandwidth metrics
* View network packet rates
* Select different dashboard views
* Set custom time ranges
* Export or inspect specific graph details

|Logs
|View application logs for project pods
|* View logs for specific pods or deployments
* Filter logs by resource type
* Stream real-time logs
* View historical logs

|Metrics
|Query and visualize project metrics using PromQL
|* Select predefined metric queries
* Create custom PromQL queries
* View results in graph or table format
* Set time range and refresh interval
* PromQL autocomplete support

|Silences
|Create and manage alert silences
|* Create silences using label matchers
* View active and expired silences
* Edit or expire active silences
* Set silence duration and comments
|===

[NOTE]
====
A regression in {product-title} versions 4.20.12 through 4.20.19 caused five tabs (Alerting Rules, Alerts, Dashboard, Metrics, and Silences) to disappear.
Version 4.20.20 fixed this issue and backported the fix to 4.19.z releases.
If you are running an affected version, upgrade to 4.20.20, 4.19.z, or later to restore all seven tabs.
Contact your cluster administrator to verify your cluster version.
All seven tabs are available in versions 4.20.20 and later, 4.19.z, 4.21, 4.22, and 4.23.
====

== Role-based access control requirements

The following table shows the required RBAC roles for each Developer Observe tab:

.Required roles for Developer Observe tabs
[cols="30%,70%",options="header"]
|===
|Tab name |Required roles

|Events
|View permissions for the project

|Alerting Rules
|`monitoring-rules-view` or `monitoring-rules-edit` role for the namespace

|Alerts
|`monitoring-rules-view` or `monitoring-rules-edit` role for the namespace

|Dashboard
|View permissions for the project

|Logs
|View permissions for the project

|Metrics
|`monitoring-rules-view` or `monitoring-rules-edit` role for the namespace

|Silences
|`cluster-monitoring-view` cluster role (cluster-wide scope) and `monitoring-alertmanager-edit` role for your namespace (project-scoped)
|===

[NOTE]
====
A cluster administrator must enable user-defined monitoring at the cluster level for the Alerting Rules, Alerts, Metrics, and Silences tabs to display data.
====

== Administrator perspective comparison

The Developer perspective Observe view differs from the Administrator perspective monitoring UI:

* **Developer perspective**: Provides integrated tabs within a single Observe view. All monitoring capabilities are namespace-scoped to the selected project.
* **Administrator perspective**: Provides separate pages under menu:Observe[Alerting]. Monitoring capabilities are cluster-wide and can query across all namespaces.

Developers can only query metrics and view alerts for one project at a time.
Namespace filtering is automatically applied to all queries in the Developer perspective.

For cluster-wide monitoring capabilities, see xref:../observability/monitoring/managing-alerts.adoc#managing-alerts[Managing alerts].
