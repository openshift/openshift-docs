[id="rosa-nodes-autonode-nodepool-reference_{context}"]
= NodePool configuration reference

[role="_abstract"]
Reference documentation for NodePool custom resource fields and configuration options.

A NodePool defines the constraints and requirements that {autonode} uses when provisioning compute nodes for your workloads. This reference describes the available configuration fields and their usage.

[id="rosa-nodes-autonode-nodepool-structure_{context}"]
== NodePool structure

The NodePool custom resource has the following high-level structure:

[source,yaml]
----
apiVersion: karpenter.sh/v1
kind: NodePool
metadata:
  name: <nodepool-name>
spec:
  template:
    metadata:
      labels: <key-value-pairs>
      annotations: <key-value-pairs>
    spec:
      requirements: <instance-requirements>
      nodeClassRef: <node-class-reference>
      taints: <node-taints>
      startupTaints: <startup-taints>
      expireAfter: <expiration-duration>
  limits: <resource-limits>
  disruption: <disruption-settings>
  weight: <priority-weight>
----

[id="rosa-nodes-autonode-nodepool-metadata_{context}"]
== Metadata fields

`metadata.name`::
The NodePool name. Must be unique within the cluster. This name appears in node labels and is used to identify which NodePool provisioned each node.

`metadata.annotations`::
Optional annotations to apply to the NodePool resource for documentation or tooling purposes. These annotations are not propagated to nodes.

[id="rosa-nodes-autonode-nodepool-template_{context}"]
== Template fields

The `spec.template` section defines the configuration template for nodes provisioned by this NodePool.

`spec.template.metadata.labels`::
Key-value pairs to apply as labels to all nodes provisioned by this NodePool. Workloads can use node selectors to target these labels. Common use cases include environment labels (`env: production`), workload type labels (`workload: batch`), or cost center labels.
+
.Example
[source,yaml]
----
spec:
  template:
    metadata:
      labels:
        environment: production
        team: platform
        cost-center: eng-123
----

`spec.template.metadata.annotations`::
Key-value pairs to apply as annotations to all nodes provisioned by this NodePool. Unlike labels, annotations are not used for selection but can store additional metadata.

[id="rosa-nodes-autonode-nodepool-requirements_{context}"]
== Requirements

The `spec.template.spec.requirements` field defines constraints that {autonode} uses to select Amazon Elastic Compute Cloud (Amazon EC2) instance types for this NodePool. Each requirement is a key-operator-values tuple.

Available operators::
* `In`: The node must have one of the specified values for this key
* `NotIn`: The node must not have any of the specified values for this key
* `Exists`: The key must be present (values are ignored)
* `DoesNotExist`: The key must not be present (values are ignored)
* `Gt`: The value must be greater than the specified value (numeric comparison)
* `Lt`: The value must be less than the specified value (numeric comparison)

[id="rosa-nodes-autonode-nodepool-requirements-keys_{context}"]
=== Common requirement keys

`node.kubernetes.io/instance-type`::
Restricts provisioning to specific Amazon EC2 instance types. Use this when workloads require specific instance characteristics.
+
.Example
[source,yaml]
----
requirements:
  - key: node.kubernetes.io/instance-type
    operator: In
    values: ["m5.large", "m5.xlarge", "m5.2xlarge"]
----

`karpenter.sh/capacity-type`::
Specifies the capacity type for instances. Valid values are `on-demand` or `spot`. On-demand instances provide guaranteed capacity, while spot instances offer significant cost savings but can be interrupted.
+
.Example (on-demand only)
[source,yaml]
----
requirements:
  - key: karpenter.sh/capacity-type
    operator: In
    values: ["on-demand"]
----
+
.Example (allow both)
[source,yaml]
----
requirements:
  - key: karpenter.sh/capacity-type
    operator: In
    values: ["on-demand", "spot"]
----

`karpenter.k8s.aws/instance-category`::
Restricts provisioning to instance categories. Common values include `c` (compute-optimized), `m` (general-purpose), `r` (memory-optimized), `t` (burstable), and `g` (GPU instances).
+
.Example
[source,yaml]
----
requirements:
  - key: karpenter.k8s.aws/instance-category
    operator: In
    values: ["c", "m"]
----

`karpenter.k8s.aws/instance-generation`::
Restricts provisioning to instance generations. Use the `Gt` operator to require instances newer than a specific generation.
+
.Example (generation 3 or newer)
[source,yaml]
----
requirements:
  - key: karpenter.k8s.aws/instance-generation
    operator: Gt
    values: ["2"]
----

`kubernetes.io/arch`::
Specifies the CPU architecture. Valid values are `amd64` and `arm64`.
+
.Example
[source,yaml]
----
requirements:
  - key: kubernetes.io/arch
    operator: In
    values: ["amd64"]
----

`kubernetes.io/os`::
Specifies the operating system. For {product-title}, the value is always `linux`.

`topology.kubernetes.io/zone`::
Restricts provisioning to specific availability zones.
+
.Example
[source,yaml]
----
requirements:
  - key: topology.kubernetes.io/zone
    operator: In
    values: ["us-east-1a", "us-east-1b"]
----

[id="rosa-nodes-autonode-nodepool-nodeclass_{context}"]
== Node class reference

`spec.template.spec.nodeClassRef`::
References the EC2NodeClass that defines the Amazon EC2 configuration for nodes provisioned by this NodePool. The reference includes the API group, kind, and name.
+
.Example
[source,yaml]
----
spec:
  template:
    spec:
      nodeClassRef:
        group: karpenter.k8s.aws
        kind: EC2NodeClass
        name: default
----

[id="rosa-nodes-autonode-nodepool-taints_{context}"]
== Taints

`spec.template.spec.taints`::
Taints to apply to all nodes provisioned by this NodePool. Taints prevent pods from scheduling on nodes unless the pods have matching tolerations. Use taints to dedicate nodes to specific workloads.
+
.Example
[source,yaml]
----
spec:
  template:
    spec:
      taints:
        - key: workload
          value: database
          effect: NoSchedule
----
+
Valid effects: `NoSchedule`, `PreferNoSchedule`, `NoExecute`

`spec.template.spec.startupTaints`::
Taints applied only during node startup. These taints are removed automatically once the node is ready. Use startup taints to prevent pods from scheduling on nodes until initialization completes, such as when using custom user data that installs software.
+
.Example
[source,yaml]
----
spec:
  template:
    spec:
      startupTaints:
        - key: node.kubernetes.io/not-ready
          effect: NoSchedule
----

[id="rosa-nodes-autonode-nodepool-expiration_{context}"]
== Node expiration

`spec.template.spec.expireAfter`::
Duration after which nodes are automatically deleted and replaced, even if they are still running workloads. This forces regular node rotation for security updates or configuration changes. The value is a duration string such as `720h` for 30 days.
+
.Example
[source,yaml]
----
spec:
  template:
    spec:
      expireAfter: 720h
----
+
[NOTE]
====
Expiring nodes does not respect disruption budgets. Pods are evicted when nodes expire.
====

[id="rosa-nodes-autonode-nodepool-limits_{context}"]
== Resource limits

The `spec.limits` field caps the total resources that {autonode} can provision for this NodePool. When limits are reached, {autonode} stops provisioning additional nodes for this NodePool, even if workloads remain pending.

`spec.limits.cpu`::
Maximum total CPU cores that can be provisioned. This counts virtual CPUs (vCPUs) for the instance types.
+
.Example
[source,yaml]
----
spec:
  limits:
    cpu: "100"
----

`spec.limits.memory`::
Maximum total memory that can be provisioned. The value is a quantity such as `256Gi`.
+
.Example
[source,yaml]
----
spec:
  limits:
    memory: 256Gi
----

[NOTE]
====
Limits are soft caps. {autonode} provisions nodes to satisfy pending pods until adding another node would exceed the limit. The actual resource usage might slightly exceed limits.
====

[id="rosa-nodes-autonode-nodepool-disruption_{context}"]
== Disruption settings

The `spec.disruption` field controls when and how {autonode} can replace or remove nodes to optimize cost and resource utilization.

`spec.disruption.consolidationPolicy`::
Controls whether {autonode} can consolidate workloads onto fewer nodes to reduce costs. Valid values:
+
--
* `WhenEmptyOrUnderutilized` (default): {autonode} can delete empty nodes immediately and can consolidate workloads from underutilized nodes onto fewer nodes
* `WhenEmpty`: {autonode} can only delete nodes that have no pods (other than DaemonSets)
--
+
.Example
[source,yaml]
----
spec:
  disruption:
    consolidationPolicy: WhenEmpty
----

`spec.disruption.consolidateAfter`::
How long {autonode} waits after a node becomes empty or underutilized before consolidating it. The default is `0s` for empty nodes. Setting a longer duration prevents rapid consolidation and un-consolidation cycles.
+
.Example
[source,yaml]
----
spec:
  disruption:
    consolidateAfter: 5m
----

`spec.disruption.budgets`::
Limits the rate of disruption by specifying the maximum number or percentage of nodes that can be disrupted during a time window. Use budgets to prevent too many nodes from being replaced simultaneously.
+
.Example
[source,yaml]
----
spec:
  disruption:
    budgets:
      - nodes: "10%"
        schedule: "0 9 * * *"
        duration: 8h
----
+
This example allows disruption of up to 10% of nodes daily between 9:00 AM and 5:00 PM.

[id="rosa-nodes-autonode-nodepool-weight_{context}"]
== Priority weight

`spec.weight`::
Relative priority for this NodePool when multiple NodePools can satisfy a workload's requirements. Higher weight values indicate higher priority. {autonode} prefers provisioning from higher-weighted NodePools. The default weight is `0`.
+
.Example
[source,yaml]
----
spec:
  weight: 10
----
+
Use weights to prefer on-demand instances over spot instances, or to prefer newer instance generations.

[id="rosa-nodes-autonode-nodepool-examples_{context}"]
== Complete examples

.General-purpose NodePool with limits
[source,yaml]
----
apiVersion: karpenter.sh/v1
kind: NodePool
metadata:
  name: general-purpose
spec:
  template:
    metadata:
      labels:
        workload-type: general
    spec:
      requirements:
        - key: karpenter.sh/capacity-type
          operator: In
          values: ["on-demand"]
        - key: karpenter.k8s.aws/instance-category
          operator: In
          values: ["c", "m", "r"]
        - key: karpenter.k8s.aws/instance-generation
          operator: Gt
          values: ["2"]
      nodeClassRef:
        group: karpenter.k8s.aws
        kind: EC2NodeClass
        name: default
  limits:
    cpu: "100"
    memory: 256Gi
  disruption:
    consolidationPolicy: WhenEmptyOrUnderutilized
----

.Spot instance NodePool for batch workloads
[source,yaml]
----
apiVersion: karpenter.sh/v1
kind: NodePool
metadata:
  name: spot-batch
spec:
  template:
    metadata:
      labels:
        workload-type: batch
    spec:
      requirements:
        - key: karpenter.sh/capacity-type
          operator: In
          values: ["spot"]
        - key: karpenter.k8s.aws/instance-category
          operator: In
          values: ["c", "m"]
      taints:
        - key: workload
          value: batch
          effect: NoSchedule
      nodeClassRef:
        group: karpenter.k8s.aws
        kind: EC2NodeClass
        name: default
  disruption:
    consolidationPolicy: WhenEmpty
  weight: 5
----
