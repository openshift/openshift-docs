= Service discovery URI pattern reference

[role="_abstract"]
Service discovery URI patterns follow a structured format that identifies Kubernetes resources and specifies how to resolve their network endpoints. Understanding the URI pattern components enables you to reference any supported Kubernetes resource in your workflow configurations.

== URI pattern structure

The service discovery URI pattern uses the following format:

[source,text]
----
<scheme>:<kind>.<version>.<group>/<namespace>/<resourceName>?<attributeName>=<attributeValue>
----

== URI pattern components

The following table describes each component of the service discovery URI pattern:

[cols="1,3",options="header"]
|===
|Component
|Description

|`scheme`
|The protocol identifier. Supported values are `kubernetes`, `openshift`, or `knative`.

|`kind`
|The Kubernetes resource kind, such as `services`, `deployments`, or `pods`.

|`version`
|The API version of the resource, such as `v1` or `v1beta1`.

|`group`
|The API group of the resource. For core resources such as Services and Pods, this can be empty. For other resources, specify the group such as `apps` or `serving.knative.dev`.

|`namespace`
|The Kubernetes namespace where the resource is located. If empty, the operator uses the current namespace context.

|`resourceName`
|The name of the specific resource to resolve.

|`attributeName=attributeValue`
|Optional query parameters for filtering or specifying additional attributes. Multiple parameters are separated by `&`.
|===

== URI pattern examples

The following examples displays different URI pattern formats:

For Kubernetes Service in the same namespace:

[source,text]
----
$ kubernetes:services.v1//my-service
----

For Knative Service in a specific namespace:
[source,text]
----
$ kubernetes:services.v1.serving.knative.dev/finance/financial-service
----

For Deployment with custom label filter:
[source,text]
----
$ kubernetes:deployments.v1.apps/myapp/backend-deployment?app=backend&env=prod
----

For OpenShift Route:
[source,text]
----
openshift:routes.v1.route.openshift.io/web/frontend-route
----

== Knative service shorthand notation

For Knative services, you can use a simplified URI format:

[source,text]
----
$ knative:<namespace>/<serviceName>
----

This shorthand is equal to the full Knative service URI pattern:

[source,text]
----
$ kubernetes:services.v1.serving.knative.dev/<namespace>/<serviceName>
----

== Query parameters

Service discovery supports the following query parameters:

Custom labels::
Filter resources by label selectors when multiple resources share similar characteristics.
+
[source,text]
----
$ label-name1=label-value1&label-name2=label-value2
----

Custom port::
Specify a target port when the service exposes multiple ports.
+
[source,text]
----
$ port=<PORT_NAME>
----
+
You can combine label and port parameters:
+
[source,text]
----
$ app=myapp&port=http
----
