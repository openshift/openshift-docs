= Whereabouts

[role="_abstract"]
Whereabouts is an IPAM CNI plugin that runs inside {product-title} and dynamically assigns IP addresses from a pool you define in the network configuration.

== When to use Whereabouts

Use Whereabouts when:

* You need dynamic IP assignment but do not have a DHCP server
* You want {product-title} to manage all IP assignments
* You want IP address pools defined in Kubernetes resources
* You are running many pods that need SR-IOV networks
* You prefer infrastructure-as-code over external DHCP infrastructure
* You want all networking configuration visible in {product-title} resources

== How it works

Whereabouts runs as a CNI plugin inside {product-title}:

1. You define an IP address pool in the `SriovNetwork` resource
2. When a pod starts, the Whereabouts CNI plugin allocates the next available IP address from the pool
3. Whereabouts stores the allocation in Kubernetes as a CustomResource
4. When the pod terminates, the IP address returns to the available pool for reuse

All IP address management happens within {product-title} - no external server required.

.Example SriovNetwork configuration with Whereabouts
[source,yaml]
----
apiVersion: sriovnetwork.openshift.io/v1
kind: SriovNetwork
metadata:
  name: sriov-whereabouts-network
  namespace: openshift-sriov-network-operator
spec:
  resourceName: intelnics
  networkNamespace: default
  ipam: |
    {
      "type": "whereabouts",
      "range": "192.168.1.0/24",
      "range_start": "192.168.1.10",
      "range_end": "192.168.1.100",
      "gateway": "192.168.1.1"
    }
----

== Advantages

* *No external infrastructure* - No DHCP server required
* *All configuration in {product-title}* - IP pools defined in network resources
* *Visible state* - Allocations visible as Kubernetes resources
* *Prevents conflicts* - Automatically prevents IP conflicts within the defined pool
* *Simple infrastructure* - No DHCP server to install, configure, or maintain
* *GitOps-friendly* - Network configuration can be managed as code
* *Kubernetes-native* - Familiar to platform teams managing {product-title}

== Limitations

* *Only provides IP addresses* - Does not provide DNS servers, routes, or other DHCP options
* *Requires planning* - Must define IP pools in advance
* *State in Kubernetes* - IP allocation state stored in etcd; potential issues if etcd has problems
* *Less familiar to network teams* - Network administrators may prefer traditional DHCP
* *No integration with IP address management systems* - Cannot integrate with existing IPAM tools
* *Per-network configuration* - Must configure IP pools for each network, cannot share DHCP infrastructure

== IP pool planning

When using Whereabouts, you must plan your IP address pools:

* Define non-overlapping ranges for different networks
* Ensure pools are large enough for your expected number of pods
* Avoid using IP addresses that conflict with other systems on the physical network
* Reserve IP addresses for network infrastructure (gateways, DNS servers, etc.)

== Whereabouts state management

Whereabouts stores IP allocation state as Kubernetes CustomResources:

* Each IP assignment creates an `IPPool` or `OverlappingRangeIPReservation` resource
* These resources track which IP addresses are allocated to which pods
* When a pod is deleted, its IP allocation is removed and the address returns to the pool

== Best practices

If you use Whereabouts:

* Plan IP address ranges carefully to avoid conflicts
* Use non-overlapping ranges for different SR-IOV networks
* Monitor IP pool utilization to ensure you have enough addresses
* Document your IP pool assignments
* Consider network segmentation - use different pools for different purposes
* Ensure etcd is properly backed up, as it stores IP allocation state
* Test pod creation and deletion to verify IP addresses are properly allocated and reclaimed
