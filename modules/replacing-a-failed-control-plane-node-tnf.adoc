= Replacing a failed control plane node in two-node OpenShift cluster with fencing

[role="_abstract"]
Follow this procedure to safely replace a failed control plane node and restore your two-node OpenShift cluster with fencing (TNF) cluster to full health. This process leverages built-in Cluster etcd Operator (CEO) automation and Pacemaker integration to handle the heavy lifting, keeping your manual recovery efforts to an absolute minimum.

.Prerequisites

* SSH access to the surviving control plane node is configured.
* Replacement node hardware meets the CPU, memory, and disk requirements.
* BMC credentials for the fencing device are available. Only Redfish is supported.
* Cluster backups are up to date.
* The cluster uses the bare-metal installer-provisioned infrastructure platform, which requires the Ironic provisioning service.
* For VM-based deployments, hypervisor access is available for VM operations.
* The failed node is powered off or inaccessible.

.Procedure

. Verify that the system completed fencing successfully by checking the pacemaker status on the surviving node by running the following command:
+
[source,terminal]
----
$ sudo pcs status
----

. Record the failed node name, IP address, and MAC address from the `BareMetalHost` resource.

. Back up the node configuration.

. For VM-based deployments, export the VM XML definition by running the following command:
+
[source,terminal]
----
$ virsh dumpxml <vm_name> > vm-backup.xml
----

. Back up the `BareMetalHost` resource by running the following command:
+
[source,terminal]
----
$ oc get bmh -n openshift-machine-api <bmh_name> -o yaml > bmh.yaml
----

. Back up the `Machine` resource by running the following command:
+
[source,terminal]
----
$ oc get machine -n openshift-machine-api <machine_name> -o yaml > machine.yaml
----

. Back up the BMC secret if you need to re-create it by running the following command:
+
[source,terminal]
----
$ oc get secret -n openshift-machine-api <bmc_secret_name> -o yaml > bmc-secret.yaml
----
+
[NOTE]
====
etcd TLS secrets are automatically generated by the CEO and do not need to be backed up.
====

. Delete the Kubernetes node object by running the following command:
+
[source,terminal]
----
$ oc delete node <failed_node_name>
----

. Delete the `BareMetalHost` resource and wait for the process to complete by running the following command:
+
[NOTE]
====
During this step, Ironic automatically cleans and resets the host hardware in the background.
====
+
[source,terminal]
----
$ oc delete bmh -n openshift-machine-api <bmh_name>
----

. Delete the `Machine` resource by running the following command:
+
[source,terminal]
----
$ oc delete machine -n openshift-machine-api <machine-name>
----

. Delete stale installer pods in the static-pod operator namespaces by running the following command:
+
[NOTE]
====
This step is critical when replacing a node with the same name, because existing completed installer pods block new pod installations:
====
+
[source,terminal]
----
$ oc delete pod -n openshift-etcd -l app=installer \
  --field-selector spec.nodeName=<failed_node_name>
$ oc delete pod -n openshift-kube-apiserver -l app=installer \
  --field-selector spec.nodeName=<failed_node_name>
$ oc delete pod -n openshift-kube-controller-manager -l app=installer \
  --field-selector spec.nodeName=<failed_node_name>
$ oc delete pod -n openshift-kube-scheduler -l app=installer \
  --field-selector spec.nodeName=<failed_node_name>
----
+
The operators schedule fresh installer pods when the replacement node becomes `Ready`.

. Provision the replacement node hardware:
+
* For VM-based deployments: Restore the VM from backup or provision a fresh one using `virsh define`. Ensure the MAC address matches the original.
* For physical hardware: Prepare hardware matching the original specifications.

** Ensure the MAC address matches the original node. This is required for `BareMetalHost` binding.

** Power on the node.

** Create a new `BareMetalHost` resource without the detached annotation. Replace all placeholder values before applying:
+
[source,yaml]
----
apiVersion: metal3.io/v1alpha1
kind: BareMetalHost
metadata:
  name: <bmh-name>               
  namespace: openshift-machine-api
  labels:
    installer.openshift.io/role: control-plane
  finalizers:
  - baremetalhost.metal3.io
spec:
  online: true
  bootMACAddress: <mac-address> 
  bmc:
    address: redfish+https://<bmc-ip>:<port>/redfish/v1/Systems/<uuid>
    credentialsName: <bmc-secret-name>
    disableCertificateVerification: true
  bootMode: UEFI
  automatedCleaningMode: metadata
  customDeploy:
    method: install_coreos
  userData:
    name: master-user-data-managed
    namespace: openshift-machine-api
  rootDeviceHints:
    deviceName: /dev/sda
----
+
where:
+
** `metadata.name`: Use the original BMH name to reuse the same node identity.
+
** `spec.bootMACAddress`: Must match the hardware MAC address exactly.
** `bmc.rootDeviceHints.deviceName`:  Adjust the device path to match your hardware configuration.

. Apply the resource and wait for provisioning to complete by running the following commands:
+
[source,terminal]
----
$ oc create -f bmh.yaml
$ oc get bmh -n openshift-machine-api <bmh_name> -w
----

. Wait until `status.provisioning.state` shows `provisioned`.
+
. After provisioning is complete, apply the detached annotation to prevent cluster-api from managing the `BareMetalHost` during replacement by running the following command:
+
[source,terminal]
----
$ oc annotate bmh -n openshift-machine-api <bmh_name> \
  baremetalhost.metal3.io/detached=
----

. Create a new `Machine` resource. Replace all placeholder values before applying:
+
[source,yaml]
----
apiVersion: machine.openshift.io/v1beta1
kind: Machine
metadata:
  name: <machine_name>                
  namespace: openshift-machine-api
  labels:
    machine.openshift.io/cluster-api-cluster: ostest-<cluster_hash>
    machine.openshift.io/cluster-api-machine-role: master
    machine.openshift.io/cluster-api-machine-type: master
  annotations:
    metal3.io/BareMetalHost: openshift-machine-api/<bmh_name>
  finalizers:
  - machine.machine.openshift.io
spec:
  authoritativeAPI: MachineAPI
  providerSpec:
    value:
      apiVersion: baremetal.cluster.k8s.io/v1alpha1
      kind: BareMetalMachineProviderSpec
      customDeploy:
        method: install_coreos
      userData:
        name: master-user-data-managed
----

. To find the cluster hash, run the following command against the surviving node's `Machine` resource:
+
[source,terminal]
----
$ oc get machine -n openshift-machine-api <survivor_machine_name> \
  -o jsonpath='{.metadata.labels.machine\.openshift\.io/cluster-api-cluster}'
----

. Apply the resource by running the following command:
+
[source,terminal]
----
$ oc create -f machine.yaml
----

. Monitor the node until it becomes `Ready` by running the following command:
+
[source,terminal]
----
$ oc get node <node-name> -w
----

. Verify that node certificates are approved by running the following command. The `cluster-machine-approver` should approve CSRs automatically:
+
[source,terminal]
----
$ oc get csr | grep <node-name>
----
+
[NOTE]
====
Manual CSR approval is only needed if the node registered from a backup image before Ironic reprovisioned it.
====

. Monitor cluster operator status by running the following command. Deleting the stale installer pods in Phase 3 should trigger new installer pod scheduling automatically:
+
[source,terminal]
----
$ oc get co kube-apiserver kube-controller-manager kube-scheduler etcd
----

. If installer pods do not trigger automatically, force a static pod revision bump as a fallback by running the following command:
+
[source,terminal]
----
$ oc patch kubeapiserver cluster --type=merge -p '{"spec":{"logLevel":"Trace"}}'
----
+
Wait for the rollout, then revert by running the following command:
+
[source,terminal]
----
$ oc patch kubeapiserver cluster --type=merge -p '{"spec":{"logLevel":"Normal"}}'
----
+
. If required, repeat this process for `kube-controller-manager` and `kube-scheduler` 

. Check east-west connectivity by verifying `PodNetworkConnectivityCheck` success for survivor-to-replacement paths by running the following command:
+
[source,terminal]
----
$ oc get podnetworkconnectivitycheck
----

. Monitor the CEO-managed jobs that configure the replacement node by running the following command:
+
[source,terminal]
----
$ oc get jobs -n openshift-etcd
----
+
The following jobs run automatically after the node becomes `Ready`:
+
* `tnf-auth-<nodename>`: Configures authentication for the replacement node.
* `tnf-update-setup-<survivor-name>`: Updates pacemaker on the surviving node for the new peer.
* `tnf-update-setup-<nodename>`: Configures pacemaker on the replacement node.
