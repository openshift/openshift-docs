= Troubleshooting X.509 authentication

[role="_abstract"]
If requests are either bypassing your X.509 authentication, or not accessing your backend as expected, you can diagnose the cause by taking steps that are based on the feedback being displayed.

.Prerequisites

* You are logged into {ocp} as a cluster administrator.
* You installed {prodname}.
* You created your CAs and X.509-formatted certificates.
* You created your `Gateway` object.
* You created your route.

.Procedure

. If the TLS handshake succeeds without client certificate, the Envoy filter might not be properly applied. Troubleshoot this situation by taking the following steps:

.. Verify that the `EnvoyFilter` exists by running the following command:
+
[source,terminal,subs="+quotes"]
----
$ oc get envoyfilter mtls-validation -n _<gateway_system>_
----
+
Replace `_<gateway_system>_` with the namespace where your `Gateway` object is applied.

.. Check that the `targetRef` in the Envoy filter points to the `Gateway` object by running the following command:
+
[source,terminal,subs="+quotes"]
----
$ oc get envoyfilter mtls-validation -n _<gateway_system>_ -o yaml | grep -A 3 targetRefs
----
+
Replace `_<gateway_system>_` with the namespace where your `Gateway` object is applied.

.. Check that the Envoy configuration has the correct configuration by running the following command:
+
[source,terminal,subs="+quotes"]
----
$ oc exec -n _<gateway_system>_ \
  $(oc get pod -n _<gateway_system>_ -l gateway.networking.k8s.io/gateway-name=mtls-gateway -o jsonpath='{.items[0].metadata.name}') \
  -- curl -s localhost:15000/config_dump?include_eds | grep -A 20 validation_context
----
+
Replace `_<gateway_system>_` with the namespace where your `Gateway` object is applied.

. If the Gateway object logs show `failed to load trusted CA`, the certificate authority (CA) might not be mounted. Use the following steps to check the CA:

.. Verify that the ConfigMap exists by running the following command:
+
[source,terminal,subs="+quotes"]
----
$ oc get configmap gateway-client-ca -n _<gateway_system>_
----
+
Replace `_<gateway_system>_` with the namespace where your `Gateway` object is applied.

.. Check that the volume mount in the `Gateway` pod by running the following command:
+
[source,terminal,subs="+quotes"]
----
$ oc describe pod -n _<gateway_system>_ -l gateway.networking.k8s.io/gateway-name=mtls-gateway | grep -A 5 Mounts
----
+
Replace `_<gateway_system>_` with the namespace where your `Gateway` object is applied.

.. Verify that the CA file exists in the `Gateway` pod by running the following command:
+
[source,terminal,subs="+quotes"]
----
$ oc exec -n _<gateway_system>_ \
  $(oc get pod -n _<gateway_system>_ -l gateway.networking.k8s.io/gateway-name=mtls-gateway -o jsonpath='{.items[0].metadata.name}') \
  -- cat /etc/certs/ca.crt
----
+
Replace `_<gateway_system>_` with the namespace where your `Gateway` object is applied.

. If Authorino rejects requests with a `certificate not found` error, the XFCC header might not be set correctly. Check the configuration and logs to help resolve this issue as follows:

.. Verify the `forward_client_cert_details` configuration by running the following command:
+
[source,terminal,subs="+quotes"]
----
$ oc exec -n _<gateway_system>_ \
  $(oc get pod -n _<gateway_system>_ -l gateway.networking.k8s.io/gateway-name=mtls-gateway -o jsonpath='{.items[0].metadata.name}') \
  -- curl -s localhost:15000/config_dump?include_eds | grep -i forward_client_cert
----
+
Replace `_<gateway_system>_` with the namespace where your `Gateway` object is applied.

.. Check the Envoy access logs for the XFCC header by running the following command:
+
[source,terminal,subs="+quotes"]
----
$ oc logs -n _<gateway_system>_ -l gateway.networking.k8s.io/gateway-name=mtls-gateway | grep XFCC
----
+
Replace `_<gateway_system>_` with the namespace where your `Gateway` object is applied.
