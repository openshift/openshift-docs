[id="rhacs-compliance-tailored-profile-troubleshooting_{context}"]
= Troubleshooting tailored profile issues

[role="_abstract"]
This reference provides solutions for common problems with tailored profiles in {product-title} ({product-title-short}).
Use these troubleshooting steps to diagnose and resolve issues with profile scheduling, result aggregation, and scan execution.

[id="troubleshooting-tailored-profile-not-appearing_{context}"]
== Tailored profile does not appear in {product-title-short}

*Symptom*:: The expected tailored profile does not appear in the {product-title-short} profile list.

*Cause*:: The `TailoredProfile` CR does not exist on the target cluster, is not ready, or {product-title-short} cannot discover it.

*Resolution*::

. Verify that you have selected the correct cluster in the {product-title-short} portal.

. Check whether the `TailoredProfile` CR exists on the cluster:
+
[source,terminal]
----
$ oc get tailoredprofiles -n openshift-compliance
----

. If the `TailoredProfile` resource is missing, create it by using the {ocp} console or CLI. See the Compliance Operator documentation for information about creating tailored profiles.

. If the `TailoredProfile` resource exists, check its status:
+
[source,terminal]
----
$ oc get tailoredprofile <profile_name> -n openshift-compliance -o yaml
----
+
Look for the `status.state` field.
A ready profile shows `status.state: READY`.
If the status shows `ERROR` or another state, see the troubleshooting section for tailored profile creation errors.

. If the `TailoredProfile` resource exists and shows `READY` status, verify that {product-title-short} can connect to the cluster:
+
[source,terminal]
----
$ oc get pods -n stackrox
----
+
. Ensure that the Sensor pods are running and healthy on the target cluster. Check the Compliance Operator integration status in {product-title-short}:
.. Navigate to *Platform Configuration* -> *Clusters*.
.. Verify that you enabled compliance scanning for the cluster.
.. Check the cluster health status for any connection issues.

*Prevention*:: Always create `TailoredProfile` CRs on target clusters before attempting to schedule scans from {product-title-short}.
Verify that tailored profiles show `status.state: READY` before scheduling scans.

[id="troubleshooting-tailored-profile-creation-error_{context}"]
== Tailored profile shows ERROR status

*Symptom*:: A `TailoredProfile` CR exists, but shows `status.state: ERROR` or does not show `status.state: READY`.

*Cause*:: The `TailoredProfile` resource has configuration errors that prevent the Compliance Operator from processing it.

*Resolution*::

. Check the tailored profile status for error messages:
+
[source,terminal]
----
$ oc get tailoredprofile <profile_name> -n openshift-compliance -o yaml
----

. Review the `status.errorMessage` field for details about the configuration problem. Common configuration errors and their resolutions include:
+
--
* **Profile not found**: The profile name specified in `extends` does not exist.
Verify that the profile exists by running `oc get profiles.compliance -n openshift-compliance` and check that the name matches exactly.
* **Rule not found**: A rule name in `disableRules`, `enableRules`, or `manualRules` does not exist in the extended profile.
List available rules by running `oc get rules.compliance -n openshift-compliance` and verify that your rule names match.
* **Invalid setValues**: The `setValues` field references a variable that does not exist in the profile.
Check the base profile definition for valid variable names.
* **API version not supported**: The `TailoredProfile` API version in your CR does not match the version supported by your Compliance Operator.
Check the Compliance Operator documentation for the correct API version for your operator version.
--

. After correcting the configuration, update the `TailoredProfile` resource:
+
[source,terminal]
----
$ oc apply -f <tailored_profile_file>.yaml
----

. Verify that the status changes to `READY`:
+
[source,terminal]
----
$ oc get tailoredprofile <profile_name> -n openshift-compliance -o jsonpath='{.status.state}'
----
+
The output should show `READY`.

*Prevention*:: Before creating a `TailoredProfile` CR, verify that the profile you want to extend exists on the cluster and that all rule names you reference are valid for that profile.

[id="troubleshooting-cannot-schedule-tailored-profile_{context}"]
== Cannot schedule tailored profile on multiple clusters

*Symptom*:: A tailored profile is displayed for some clusters but not others, or scheduling fails for specific clusters.

*Cause 1: Profile name differs across clusters*

*Resolution*::
Standardize the profile name across all target clusters:

. Identify clusters where the profile is missing or has a different name:
+
[source,terminal]
----
$ oc get tailoredprofiles -n openshift-compliance
----

. Delete the incorrectly named `TailoredProfile` resources:
+
[source,terminal]
----
$ oc delete tailoredprofile <profile_name> -n openshift-compliance
----

. Re-create the `TailoredProfile` resources with a consistent name on all clusters.

*Cause 2: Profile content differs across clusters*

*Resolution*::
Compare and update `TailoredProfile` specifications to match exactly:

. Export the `TailoredProfile` resource from each cluster:
+
[source,terminal]
----
$ oc get tailoredprofile <profile_name> -n openshift-compliance -o yaml > profile-cluster1.yaml
----

. Compare the `spec` sections across all exported YAML files.
Look for differences in the `extends`, `disableRules`, `enableRules`, and `setValues` fields.

. Update `TailoredProfile` resources on all clusters to match the required specification.

*Cause 3: Profile does not exist on all target clusters*

*Resolution*::
Create the `TailoredProfile` CR on missing clusters by using the same specification.

[id="troubleshooting-results-not-aggregating_{context}"]
== Results not aggregating across clusters

*Symptom*:: The same tailored profile shows separate result sets for different clusters instead of an aggregated view.

*Cause*:: The profile name or content differs between clusters.

*Resolution*::

. Verify that profile names are identical and case-sensitive:
+
[source,terminal]
----
$ oc get tailoredprofiles -n openshift-compliance
----

. Export and compare `TailoredProfile` specifications from each cluster:
+
[source,terminal]
----
$ oc get tailoredprofile <profile_name> -n openshift-compliance -o yaml
----

. Check for differences in the following fields:
+
--
* `spec.extends`
* `spec.disableRules`
* `spec.enableRules`
* `spec.manualRules`
* `spec.setValues`
* `spec.description`
* `spec.title`
--
+
[NOTE]
====
{product-title-short} compares the `spec` section of the `TailoredProfile` CR to assess content matching.
Differences in metadata such as labels or annotations, or in the `status` section, do not affect aggregation if the `spec` fields are identical.
====

. Update `TailoredProfile` resources to match exactly on all clusters where you want aggregated results.

*Diagnostic*:: Look for {product-title-short} informational messages in the *Tailored Profiles* tab that indicate duplicate profile names with different content.

[id="troubleshooting-profile-flagged-as-duplicate_{context}"]
== Profile flagged as duplicate with different name

*Symptom*:: {product-title-short} displays a message indicating that profiles with identical content but different names exist.

*Cause*:: The same `TailoredProfile` specification has different names on different clusters.

*Resolution*::

. Review the flagged profiles in the {product-title-short} *Tailored Profiles* tab.

. Choose a standardized name for the profile that reflects its purpose.

. Delete the `TailoredProfile` resources with inconsistent names:
+
[source,terminal]
----
$ oc delete tailoredprofile <old_profile_name> -n openshift-compliance
----

. Re-create the `TailoredProfile` resources with the standardized name on all target clusters.

. Reschedule scans in {product-title-short} by using the updated profile name.

[NOTE]
====
This informational flag helps you identify naming inconsistencies but does not prevent scanning.
Scans continue to run successfully even after flagging profiles. 
====

[id="troubleshooting-scan-fails-profile-not-found_{context}"]
== Scan fails with profile not found error

*Symptom*:: A scheduled scan fails immediately with an error message indicating that the profile is missing.

*Cause*:: You deleted the `TailoredProfile` CR after scheduling the scan in {product-title-short}.

*Resolution*::

. Verify that the `TailoredProfile` resource exists on the cluster:
+
[source,terminal]
----
$ oc get tailoredprofiles -n openshift-compliance
----

. If the resource is missing, re-create it by using the original `TailoredProfile` specification.

. Reschedule the scan in {product-title-short}.

*Prevention*:: Do not delete `TailoredProfile` CRs that are actively used in {product-title-short} scan schedules.
If you need to change a profile, update the existing resource rather than deleting and recreating it.

[id="troubleshooting-custom-rule-incorrect-description_{context}"]
== Custom rule results show incorrect description

*Symptom*:: Custom rules display unexpected, missing, or incomplete descriptions in the scan results.

*Cause*:: The `CustomRule` CR is not properly defined or not synchronized with the TailoredProfile.

*Resolution*::

. Verify that the `CustomRule` resource exists on the cluster:
+
[source,terminal]
----
$ oc get customrules -n openshift-compliance
----

. Check the `CustomRule` specification:
+
[source,terminal]
----
$ oc get customrule <rule_name> -n openshift-compliance -o yaml
----
+
Verify that the custom rule resource contains the expected check definition.
The specific fields available depend on your Compliance Operator version.
Consult the Compliance Operator `CustomRule` documentation for the exact schema.

. Update the `CustomRule` resource as needed based on your Compliance Operator version.

[NOTE]
====
Custom rule description rendering in {product-title-short} depends on the Compliance Operator rule definition format.
If custom rule descriptions appear incomplete or missing, verify that your Compliance Operator version supports the fields you are using in the `CustomRule` resource.
====

[id="troubleshooting-setvalues-not-reflecting_{context}"]
== Modified check descriptions not reflecting setValues overrides

*Symptom*:: Compliance check descriptions do not display customized thresholds or parameters specified in the `setValues` field.

*Cause*:: The check description update depends on the Compliance Operator rule definition.
Some rules might not support dynamic description updates.

*Resolution*::

. Verify that the `spec.setValues` field in the `TailoredProfile` CR is correct:
+
[source,terminal]
----
$ oc get tailoredprofile <profile_name> -n openshift-compliance -o yaml
----

. Check whether the modified rule supports description updates in the Compliance Operator documentation.
Not all rules expose modified values in their descriptions.

. Use the `rationale` field in the `TailoredProfile` specification to document why you changed values.
The rationale is displayed in the check details even if the description does not reflect the modified value.

*Workaround*:: Refer to the `TailoredProfile` CR YAML for the authoritative `setValues` configuration.
The `TailoredProfile` resource contains the exact values used for compliance checks.

[id="troubleshooting-scan-stuck-pending_{context}"]
== Scan stuck in pending or launching status

*Symptom*:: A compliance scan remains in `PENDING` or `LAUNCHING` phase for more than 10 minutes and does not progress to `RUNNING`.

*Cause*:: The Compliance Operator cannot create or schedule the scan pods, typically due to resource constraints, node scheduling issues, or Compliance Operator problems.

*Resolution*::

. Check the `ComplianceScan` resource for detailed status information:
+
[source,terminal]
----
$ oc describe compliancescan <scan_name> -n openshift-compliance
----
+
Look for error messages in the Events section that indicate why the scan cannot start.

. Check recent events in the openshift-compliance namespace:
+
[source,terminal]
----
$ oc get events -n openshift-compliance --sort-by='.lastTimestamp' | tail -20
----
+
Look for events related to pod creation failures, scheduling errors, or resource constraints.

. Verify that the Compliance Operator pods are running:
+
[source,terminal]
----
$ oc get pods -n openshift-compliance
----
+
If the Compliance Operator pod is not running or is restarting, check its logs for errors.

. Check for resource constraints that might prevent scan pod scheduling:
+
[source,terminal]
----
$ oc describe nodes | grep -A 5 "Allocated resources"
----
+
If nodes are under resource pressure, the scan pods might not be able to schedule.

. If the scan remains stuck after checking these issues, delete the scan and reschedule it:
+
[source,terminal]
----
$ oc delete compliancescan <scan_name> -n openshift-compliance
----
+
Then reschedule the scan from the {product-title-short} portal.

[id="troubleshooting-scan-result-error_{context}"]
== Scan completes with ERROR result

*Symptom*:: A compliance scan reaches `PHASE: DONE` but shows `RESULT: ERROR` instead of `COMPLIANT` or `NONCOMPLIANT`.

*Cause*:: The scan encountered an error during execution, such as pod failures, timeout issues, or problems accessing cluster resources.

*Resolution*::

. Check the `ComplianceScan` resource for error details:
+
[source,terminal]
----
$ oc get compliancescan <scan_name> -n openshift-compliance -o yaml
----
+
Look for the `status.errorMessage` field for details about what failed.

. Check the scan pod logs for errors:
+
[source,terminal]
----
$ oc logs -n openshift-compliance -l compliance.openshift.io/scan-name=<scan_name>
----
+
Look for error messages that indicate why the scan failed.

. Common causes and resolutions:
+
--
* **Pod timeout**: The scan took longer than the configured timeout. Increase the scan timeout in the `ScanSetting` resource.
* **Permission denied**: The scan pod cannot access required cluster resources. Verify that the Compliance Operator has the necessary RBAC permissions.
* **Node failure**: A node where a scan pod was running became unavailable. Reschedule the scan.
* **Profile content error**: The tailored profile references rules that cannot the system cannot evaluate. Verify the `TailoredProfile` configuration.
--

. After resolving the underlying issue, delete the failed scan and reschedule:
+
[source,terminal]
----
$ oc delete compliancescan <scan_name> -n openshift-compliance
----
+
Then reschedule the scan from the {product-title-short} portal.
