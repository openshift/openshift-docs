= Verify recovered file integrity

[role="_abstract"]
Verify file integrity by using checksums to confirm recovered data is uncorrupted and matches the backup source.
Checksum verification is essential for compliance requirements and validating critical file recovery.

.Prerequisites

* You downloaded files from a VM backup using web browser or SSH access.
* You have access to the original VM or a baseline checksum from before the backup was created.

.Procedure

. If the VM is still running and accessible, generate a baseline checksum from the running VM:
+
[source,terminal]
----
$ virtctl ssh <vm_name> -n <vm_namespace> -- md5sum /path/to/file
----
+
Replace `<vm_name>` with the name of the running VM and `<vm_namespace>` with the VM's namespace.
+
Save the checksum output as a baseline for comparison with recovered files.
+
If the VM is not accessible (disaster recovery scenario), skip to step 3 to compare checksums across backup versions instead.

. After recovering the file from a backup, generate a checksum for the recovered file:
+
[source,terminal]
----
$ md5sum /local/path/to/recovered/file
----

. Compare the checksums to verify data integrity:
+
* If the checksums match, the file was recovered successfully without corruption.
* If the checksums differ, verify that you selected the correct backup timestamp. Compare checksums across multiple backup versions to identify when the file changed. If a file should be identical to the running VM but checksums differ, this might indicate backup corruption or incomplete backup. Review Velero backup logs to verify backup completion.

. For more secure verification, use SHA256 instead of MD5:
+
[source,terminal]
----
$ sha256sum /local/path/to/recovered/file
----
+
SHA256 provides better protection against hash collisions and is recommended for security-sensitive files.

. To verify multiple files in a directory, create a checksum manifest:
+
[source,terminal]
----
$ find /local/path/to/recovered/directory -type f -exec sha256sum {} \; > checksums.txt
----

. Compare checksums across multiple backup versions to identify when a file changed. First, retrieve the SSH address and date directory, then use them in the checksum command:
+
[source,bash]
----
#!/bin/bash
SSH_KEY="/tmp/vmfr-key"
SSH_ADDRESS=$(oc get vmfr <vmfr_name> -n openshift-adp -o jsonpath='{.status.fileServingInfo.ssh.address}')
USERNAME="oadp"
VM_NAME="<vm_name>"
REMOTE_FILE="/etc/myapp/config.yaml"
BACKUPS=("daily-backup-20260630" "daily-backup-20260629" "daily-backup-20260628")

# Retrieve the date directory from SSH server
DATE_DIR=$(ssh -p 2222 -i "$SSH_KEY" "${USERNAME}@${SSH_ADDRESS}" "ls -1 /restores/ | head -1")

for backup in "${BACKUPS[@]}"; do
    echo "Checksum for $backup:"
    ssh -p 2222 -i "$SSH_KEY" "${USERNAME}@${SSH_ADDRESS}" \
        "md5sum /restores/${DATE_DIR}/${backup}/${VM_NAME}${REMOTE_FILE}"
done
----
+
Replace `<vmfr_name>` with the name of your VMFR CR and `<vm_name>` with the virtual machine name. Different checksums indicate the file changed between backup versions.

.Verification

For ongoing operations, generate and store checksum manifests alongside backup metadata during normal operations to enable future integrity validation:

. Document checksums for critical files in a secure location for future comparison.

. For compliance requirements, store checksum manifests alongside backup metadata to prove data integrity.

For recovery operations without baselines, compare checksums across multiple backup versions to identify corruption.
Different checksums for the same file across adjacent backups might indicate corruption in one version.
