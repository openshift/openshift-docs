= Network Observability Operator 1.10 known issues

[role="_abstract"]
Review the following known issues and their recommended workarounds (where available) that might affect your use of the Network Observability Operator 1.10 release.

[id="loki-connectivity-lost-with-networkpolicy_{context}"]
== Loki loses object storage connectivity after upgrade with network policies

After upgrading the Operator to version 1.10, if the `FlowCollector` custom resource has `networkPolicy` enabled, Loki cannot connect to object storage (such as AWS S3 or GCS) when the Loki workloads reside in the same namespace as the `FlowCollector` resource.

This limitation prevents Loki from accessing required cloud object storage targets.

To work around this problem, you can either:

. Move your Loki workloads to a separate namespace
. Create your own network policy that includes additional connective rules for Loki. For example:
+
[source,yaml]
----
kind: NetworkPolicy
apiVersion: networking.k8s.io/v1
metadata:
  name: policy-loki
  namespace: netobserv
spec:
  podSelector:
    matchLabels:
      app.kubernetes.io/name: lokistack
  egress:
    - to:
        - ipBlock:
            cidr: <IP ranges for their S3>
  policyTypes:
    - Egress
----

[id="eBPF-agent-compatibility-with-older-open-shift-versions_{context}"]
== eBPF agent compatibility with older {product-title} versions
The eBPF agent used in the Network Observability Command Line Interface (CLI) packet capture feature is incompatible with {product-title} (OCP) versions 4.16 and older.

This limitation prevents the eBPF-based PCA from functioning correctly on those older clusters.

To work around this problem, you must manually configure PCA to use an older, compatible eBPF agent container image.

link:https://issues.redhat.com/browse/NETOBSERV-2358[NETOBSERV-2358]
