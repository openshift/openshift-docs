[id="serverless-event-transformation-deployment-patterns_{context}"]
= Deployment patterns for event transformation

You can use `EventTransform` at different points in your event flow depending on your architecture needs. The following patterns are supported:

[id="serverless-event-transformation-source-to-broker_{context}"]
== Source to Broker event transformation

In this pattern, events emitted by a Source are first routed to an `EventTransform` resource, which applies the transformation logic, and are then forwarded to a Broker. This ensures that only normalized or enriched events enter the Broker for further routing and consumption. 

You can configure an `ApiServerSource` to send events to an `EventTransform`, which then transforms and routes them to the default Broker, as shown in the following example: 

[source,yaml]
----
apiVersion: sources.knative.dev/v1
kind: ApiServerSource
metadata:
  name: k8s-events
spec:
  serviceAccountName: event-watcher
  resources:
    - apiVersion: v1
      kind: Event
  sink:
    ref:
      apiVersion: eventing.knative.dev/v1alpha1
      kind: EventTransform
      name: event-transformer
---
apiVersion: eventing.knative.dev/v1alpha1
kind: EventTransform
metadata:
  name: event-transformer
spec:
  sink:
    ref:
      apiVersion: eventing.knative.dev/v1
      kind: Broker
      name: default
  jsonata:
    expression: |
      # transformation expression
----

[id="serverless-event-transformation-trigger-to-service_{context}"]
== Trigger to Service event transformation

In this pattern, events flow through the sequence Broker → Trigger → EventTransform → Service or Sink. The Broker receives all events, the Trigger filters them based on defined attributes, and the filtered events are then routed to an `EventTransform` resource. The `EventTransform` reshapes or enriches the events before finally forwarding them to the target Service or Sink. 

This flow makes it possible to tailor events for specific consumers without modifying the original event producer or affecting other subscribers. By applying transformations after filtering, you ensure that only relevant events are reshaped, which improves efficiency and reduces unnecessary processing.

The following configuration filters events of type `original.event.type`, routes them to an `EventTransform`, and then delivers the transformed events to a Service:

[source,yaml]
----
apiVersion: eventing.knative.dev/v1
kind: Trigger
metadata:
  name: transform-trigger
spec:
  broker: default
  filter:
    attributes:
      type: original.event.type
  subscriber:
    ref:
      apiVersion: eventing.knative.dev/v1alpha1
      kind: EventTransform
      name: event-transformer
---
apiVersion: eventing.knative.dev/v1alpha1
kind: EventTransform
metadata:
  name: event-transformer
spec:
  sink:
    ref:
      apiVersion: serving.knative.dev/v1
      kind: Service
      name: destination-service
  jsonata:
    expression: |
      # transformation expression
----

[id="serverless-event-transformation-broker-reply_{context}"]
== Broker reply event transformation

You can configure `EventTransform` without a sink so that it republishes transformed events back to the Broker. This pattern is useful when you want transformed events to be routed to additional Triggers or consumers.

[NOTE]
====
When using the Broker reply feature, ensure the transformed events do not match the same Trigger that invoked the `EventTransform`. Otherwise, you risk creating an infinite event loop.
====

You can transform events of type `original.event.type` and republish them back to the Broker as shown in the following example:

[source,yaml]
----
apiVersion: eventing.knative.dev/v1
kind: Trigger
metadata:
  name: transform-trigger
spec:
  broker: default
  filter:
    attributes:
      type: original.event.type
  subscriber:
    ref:
      apiVersion: eventing.knative.dev/v1alpha1
      kind: EventTransform
      name: event-transformer
---
apiVersion: eventing.knative.dev/v1alpha1
kind: EventTransform
metadata:
  name: event-transformer
spec:
  # No sink specified - reply to Broker
  jsonata:
    expression: |
      {
        "specversion": "1.0",
        "id": id,
        "time": time,
        "type": "transformed.event.type",
        "source": "transform.event-transformer",
        "data": $
      }
----
