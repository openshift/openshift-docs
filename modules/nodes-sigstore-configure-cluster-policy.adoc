= About Creating a cluster image policy


.Prerequisites

// Taken from https://issues.redhat.com/browse/OCPSTRAT-918

The following example shows general guidelines on how to configure a `ClusterImagePolicy` object. For more details on the parameters, see _Configuring sigstore support_. 

Customers have a supported PKI and private keys or are willing to use OIDC to generate a certificate for the signature
Customers have a signing process in place to sign their images (will be supported in Quay later on)
Customers have a registry that supports cosign signatures (Quay as of 3.6, most other registries are already supported)

* You enabled the required Technology Preview features for your cluster by editing the `FeatureGate` CR named `cluster`:
+
[source,terminal]
----
$ oc edit featuregate cluster
----
+
.Example `FeatureGate` CR
[source,yaml]
----
apiVersion: config.openshift.io/v1
kind: FeatureGate
metadata:
  name: cluster
spec:
  featureSet: TechPreviewNoUpgrade <1>
----
<1> Enables the required `UserNamespacesSupport` and `ProcMountType` features.
+
[WARNING]
====
Enabling the `TechPreviewNoUpgrade` feature set on your cluster cannot be undone and prevents minor version updates. This feature set allows you to enable these Technology Preview features on test clusters, where you can fully test them. Do not enable this feature set on production clusters.
====
+
After you save the changes, new machine configs are created, the machine config pools are updated, and scheduling on each node is disabled while the change is being applied.
	
.Procedure

.Example cluster image policy object with a public key policy 
[source,yaml]
----
apiVersion: config.openshift.io/v1alpha1
kind: ClusterImagePolicy 
metadata:
  name: p1
spec:
  scopes: <1>
    - example.com/global
    - quay.io/openshift-release-dev/ocp-release
    - quay.io/openshift-release-dev/ocp-v4.0-art-dev
  policy: <2>
    rootOfTrust: <3>
      policyType: PublicKey <4>
      publicKey:
        keyData: Zm9vIGJhcg== <5>
        rekorKeyData: Zm9vIGJhcg== <6>
    signedIdentity: <7>
      matchPolicy: MatchRepoDigestOrExact
----
<1> Defines a list of image identities assigned to this policy. Each item refers to a scope in a registry.
<2> Specifies the parameters that define how the scopes are to be verified.
<3> Defines a root of trust for the policy.
<4> Specifies the policy types that define the root of trust, either a public key or a Fulcio certification. You can optionally include a Rekor verification.
<5> For a public key policy, specifies a base64-encoded public key in the PEM format. The maximum length is 8192 characters.
<6> Optional: Specifies a base64-encoded Rekor public key in the PEM format. The maximum length is 8192 characters.
<7> Specifies one of the following processes to verify the identity in the signature and the actual image identity:
* `matchRepoDigestOrExact` 
* `matchRepository`.
* `exactRepository`
* `remapIdentity`

.Example cluster image policy object with a Fulcio certificate policy and the `remapIdentity` match policy
[source,yaml]
----
apiVersion: config.openshift.io/v1alpha1
kind: ClusterImagePolicy 
metadata:
  name: p1
spec:
  scopes: <1>
    - example.com/global
    - quay.io/openshift-release-dev/ocp-release
    - quay.io/openshift-release-dev/ocp-v4.0-art-dev
  policy: <2>
    rootOfTrust: <3>
      policyType: FulcioCAWithReko <4>
      publicKey: <5>
        fulcioCAData:
        fulcioSubject:
          oidcIssuer:
          signedEmail:
        rekorKeyData: Zm9vIGJhcg== <6>
    signedIdentity:
      matchPolicy: remapIdentity <7>
        prefix: <8>
        signedPrefix: <9>
        
----
<1> Defines a list of image identities assigned to this policy. Each item refers to a scope in a registry.
<2> Specifies the parameters that define how the scopes are to be verified.
<3> Defines a root of trust for the policy.
<4> Specifies the policy types that define the root of trust, either a public key or a Fulcio certification. You can optionally include a Rekor verification.
<5> For a Fulcio certificate policy, the following parameters are required:
* `fulcioCAData`: Specifies a base64-encoded Fulcio CA in the PEM format. The maximum length is 8192 characters.
* `fulcioSubject`: Specifies OpenID Connect (OIDC) issuer and the email of the Fulcio authentication configuration.
<6> Optional: specifies a base64-encoded Rekor public key in the PEM format. The maximum length is 8192 characters.
<7> Specifies one of the following processes to verify the identity in the signature and the actual image identity:
* `matchRepoDigestOrExact` 
* `matchRepository`.
* `exactRepository`
* `remapIdentity`
<8> For the `remapIdentity` match policy, specifies the repository that contains the image identity and signature.
<9> For the `remapIdentity` match policy, specifies the prefix of the image identity to be matched in the signature. The format is the same as "prefix".
