= Creating a cluster image policy CR

A `ClusterImagePolicy` custom resource (CR) enables a cluster administrator to configure sigstore support for the entire cluster. When enabled, the Machine Config Operator (MCO) watches the `ClusterImagePolicy` object and updates the `/etc/containers/policy.json` file in the `99-<pool>-generated-registries` machine config. 

The following example shows general guidelines on how to configure a `ClusterImagePolicy` object. For more details on the parameters, see _Configuring sigstore support_. 

.Prerequisites
// Taken from https://issues.redhat.com/browse/OCPSTRAT-918
* You have a supported public key infrastructure (PKI) and private keys or are willing to use OpenID Connect (OIDC) to generate a certificate for the signature.
* You have a signing process in place to sign your images.
* You have access to a registry that supports cosign signatures.
* You enabled the required Technology Preview features for your cluster by editing the `FeatureGate` CR named `cluster`:
+
[source,terminal]
----
$ oc edit featuregate cluster
----
+
.Example `FeatureGate` CR
[source,yaml]
----
apiVersion: config.openshift.io/v1
kind: FeatureGate
metadata:
  name: cluster
spec:
  featureSet: TechPreviewNoUpgrade <1>
----
<1> Enables the required `UserNamespacesSupport` and `ProcMountType` features.
+
[WARNING]
====
Enabling the `TechPreviewNoUpgrade` feature set on your cluster cannot be undone and prevents minor version updates. This feature set allows you to enable these Technology Preview features on test clusters, where you can fully test them. Do not enable this feature set on production clusters.
====
+
After you save the changes, new machine configs are created, the machine config pools are updated, and scheduling on each node is disabled while the change is being applied.
	
.Procedure

. Create a cluster image policy object similar to the following examples:
+
--
.Example cluster image policy object with a public key policy 
[source,yaml]
----
apiVersion: config.openshift.io/v1alpha1
kind: ClusterImagePolicy <1>
metadata:
  name: p1
spec:
  scopes: <2>
    - example.com
    - quay.io/openshift-release-dev/ocp-release
    - quay.io/openshift-release-dev/ocp-v4.0-art-dev
  policy: <3>
    rootOfTrust: <4>
      policyType: PublicKey <5>
      publicKey:
        keyData: a2V5RGF0YQ== <6>
        rekorKeyData: cmVrb3JLZXlEYXRh <7>
    signedIdentity: <8>
      matchPolicy: MatchRepoDigestOrExact
----
<1> Creates a `ClusterImagePolicy` object.
<2> Defines a list of image identities assigned to this policy. In a cluster image policy, make sure that the policy allows deployment of the {product-title} images in the `quay.io/openshift-release-dev/ocp-release` and `quay.io/openshift-release-dev/ocp-v4.0-art-dev` repositories that are required for cluster operation.
<3> Specifies the parameters that define how the image identities are verified.
<4> Defines a root of trust for the policy.
<5> Specifies the policy types that define the root of trust, either a public key or a Fulcio certification. You can optionally include a Rekor verification.
<6> For a public key policy, specifies a base64-encoded public key in the PEM format. The maximum length is 8192 characters.
<7> Optional: Specifies a base64-encoded Rekor public key in the PEM format. The maximum length is 8192 characters.
<8> Specifies one of the following processes to verify the identity in the signature and the actual image identity:
* `matchRepoDigestOrExact` 
* `matchRepository`.
* `exactRepository`
* `remapIdentity`
--
+
--
.Example cluster image policy object with a Fulcio certificate policy and the `remapIdentity` match policy
[source,yaml]
----
apiVersion: config.openshift.io/v1alpha1
kind: ClusterImagePolicy <1>
metadata:
  name: p1
spec:
  scopes: <2>
    - example.com
    - quay.io/openshift-release-dev/ocp-release
    - quay.io/openshift-release-dev/ocp-v4.0-art-dev
  policy: <3>
    rootOfTrust: <4>
      policyType: FulcioCAWithReko <5>
      publicKey: <6>
        fulcioCAData: nMIICGjCCAaGgAwIBAgIUALnViVfnU
        fulcioSubject:
          oidcIssuer: https://OIDC.example.com
          signedEmail: username@example.com
        rekorKeyData: cmVrb3JLZXlEYXRh <7>
    signedIdentity:
      matchPolicy: remapIdentity <8>
        prefix: test-remap-prefix <9>
        signedPrefix: test-remap-signed-prefix <10>
----
<1> Creates a `ClusterImagePolicy` object.
<2> Defines a list of image identities assigned to this policy. In a cluster image policy, make sure that the policy allows deployment of the {product-title} images in the `quay.io/openshift-release-dev/ocp-release` and `quay.io/openshift-release-dev/ocp-v4.0-art-dev` repositories that are required for cluster operation.
<3> Specifies the parameters that define how the image identities are verified.
<4> Defines a root of trust for the policy.
<5> Specifies the policy types that define the root of trust, either a public key or a Fulcio certification. You can optionally include a Rekor verification.
<6> For a Fulcio certificate policy, the following parameters are required:
* `fulcioCAData`: Specifies a base64-encoded Fulcio CA in the PEM format. The maximum length is 8192 characters.
* `fulcioSubject`: Specifies the OIDC issuer and the email of the Fulcio authentication configuration.
<7> Optional: specifies a base64-encoded Rekor public key in the PEM format. The maximum length is 8192 characters.
<8> Specifies one of the following processes to verify the identity in the signature and the actual image identity:
* `matchRepoDigestOrExact` 
* `matchRepository`.
* `exactRepository`
* `remapIdentity`
<9> For the `remapIdentity` match policy, specifies the repository that contains the image identity and signature.
<10> For the `remapIdentity` match policy, specifies the prefix of the image identity to be matched in the signature. The format is the same as "prefix".
--

. Create the cluster image policy object:
+
[source,terminal]
----
$ oc create -f <file-name>.yaml
----
+
Scheduling is disabled on each node as it is updated.

.Verification

. After the nodes in your cluster are updated, you can verify that the cluster image policy has been configured:

.. Start a debug pod for the node by running the following command:
+
[source,terminal]
----
$ oc debug node/<node_name>
----

.. Set `/host` as the root directory within the debug shell by running the following command:
+
[source,terminal]
----
sh-5.1# chroot /host/
----

. Examine the `policy.json` file  by running the following command: 
+
[source,terminal]
----
sh-5.1# cat /etc/containers/policy.json
----
+
.Example output
[source,json]
----
{
  "default": [
    {
      "type": "insecureAcceptAnything"
    }
  ],
  "transports": {
    "atomic": {
      "example.com": [
        {
          "type": "sigstoreSigned",
          "keyData": "MFkwEwYHKoZIzj0CAQYIKoZIzj0DAQcDQgAEvJgd5dGEwolhKl6GeH1FT7S9HryF+V3shL3tFzAhxP06Wih1obsmZ7vs1cWGAQcEMUeYJyYjzrtWydWdPVZVbw==",
          "signedIdentity": {
            "type": "matchRepoDigestOrExact"
          }
        }
      ],
# ...
----

. Examine the `sigstore-registries.yaml` file  by running the following command: 
+
[source,terminal]
----
sh-5.1# cat /etc/containers/registries.d/sigstore-registries.yaml
----
+
.Example output
[source,yaml]
----
docker:
  docker.io:
    use-sigstore-attachments: true
  quay.io/openshift-release-dev/ocp-release:
    use-sigstore-attachments: true
  registry.build03.ci.openshift.org:
    use-sigstore-attachments: true
  registry.ci.openshift.org:
    use-sigstore-attachments: true
----
