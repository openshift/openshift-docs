= Set up {cgu-operator} for cluster upgrades

[role="_abstract"]
You can set up {cgu-operator-full} ({cgu-operator}) on your {rh-rhacm} hub cluster to enable policy-based upgrades for your bare metal Core clusters.
This setup includes installing {cgu-operator}, configuring {rh-rhacm} for policy management, and organizing your Git repository for upgrade policies.

.Prerequisites

* {rh-rhacm} hub cluster deployed and managing target bare metal Core clusters.
* Git repository for storing upgrade policies with appropriate access controls.
* Cluster-admin privileges on the {rh-rhacm} hub cluster.
* The `oc` CLI tool is installed and configured.

.Procedure

. Verify {rh-rhacm} is installed and managing spoke clusters:
+
Verify {rh-rhacm} installation:
+
[source,terminal]
----
$ oc get csv -n open-cluster-management | grep advanced-cluster-management
----
+
.Example output
[source,terminal]
----
advanced-cluster-management.v2.12.0   Red Hat Advanced Cluster Management   2.12.0   Succeeded
----
+
Verify spoke clusters are managed:
+
[source,terminal]
----
$ oc get managedclusters
----
+
.Example output
[source,terminal]
----
NAME     HUB ACCEPTED   MANAGED CLUSTER URLS                       JOINED   AVAILABLE   AGE
spoke1   true           https://spoke1.example.com:6443            True     True        30d
spoke2   true           https://spoke2.example.com:6443            True     True        30d
spoke3   true           https://spoke3.example.com:6443            True     True        30d
----
+
Verify spoke cluster status is Available:
+
[source,terminal]
----
$ oc get managedclusters -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.status.conditions[?(@.type=="ManagedClusterConditionAvailable")].status}{"\n"}{end}'
----
+
.Example output
[source,terminal]
----
spoke1	True
spoke2	True
spoke3	True
----
+
All managed clusters must show `True` for Available status before proceeding.

. Install {cgu-operator} from OperatorHub by using the web console:
.. In the {rh-rhacm} hub cluster web console, navigate to *Operators* -> *OperatorHub*.
.. Search for "{cgu-operator-full}" and select the operator.
.. Click *Install* and select the *openshift-operators* namespace.
.. Select the update channel and approval strategy according to your requirements.
.. Click *Install* to create the operator subscription.

. Alternatively, install {cgu-operator} by using the CLI. Create a subscription:
+
[source,yaml]
----
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: openshift-cluster-group-upgrades
  namespace: openshift-operators
spec:
  channel: stable
  name: cluster-group-upgrades-operator
  source: redhat-operators
  sourceNamespace: openshift-marketplace
  installPlanApproval: Automatic
----
+
Apply the subscription:
+
[source,terminal]
----
$ oc apply -f talm-subscription.yaml
----

. Verify the {cgu-operator} installation:
+
[source,terminal]
----
$ oc get csv -n openshift-operators -o json | \
  jq -r '.items[] | select(.spec.displayName | contains("Topology Aware Lifecycle Manager")) | .metadata.name, .status.phase'
----
+
.Example output
[source,terminal]
----
cluster-group-upgrades-operator.v4.20.0
Succeeded
----
+
The output shows the CSV name and phase.
The phase must be `Succeeded`.

. Create a namespace for upgrade policies:
+
[source,terminal]
----
$ oc create namespace openshift-upgrade-policies
----

. Configure RBAC for policy management by creating a ClusterRole:
+
[source,yaml]
----
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: upgrade-policy-manager
rules:
- apiGroups: ["policy.open-cluster-management.io"]
  resources: ["policies", "placementbindings", "placementrules"]
  verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
- apiGroups: ["apps.open-cluster-management.io"]
  resources: ["placementrules"]
  verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
- apiGroups: ["cluster.open-cluster-management.io"]
  resources: ["managedclusters"]
  verbs: ["get", "list", "watch"]
----
+
Apply the ClusterRole:
+
[source,terminal]
----
$ oc apply -f upgrade-policy-clusterrole.yaml
----

. Create a ClusterRoleBinding to grant permissions to your GitOps service account:
+
[source,yaml]
----
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: upgrade-policy-manager-binding
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: upgrade-policy-manager
subjects:
- kind: ServiceAccount
  name: gitops-service-account
  namespace: openshift-gitops
----
+
Apply the ClusterRoleBinding:
+
[source,terminal]
----
$ oc apply -f upgrade-policy-binding.yaml
----

. Prepare your Git repository structure for upgrade policies:
+
Create the following directory structure in your Git repository:
+
[source,text]
----
upgrade-policies/
├── policies/
│   ├── zstream/
│   │   └── upgrade-4.20.1-policy.yaml
│   ├── ystream/
│   │   └── upgrade-4.20-policy.yaml
│   └── eus/
│       └── upgrade-4.20-eus-policy.yaml
└── cgu/
    ├── core-zstream-upgrade.yaml
    ├── core-ystream-upgrade.yaml
    └── core-eus-upgrade.yaml
----
+
NOTE: If you are using Kustomize for policy management with ArgoCD or Flux, you can add a `kustomization.yaml` file to manage policy overlays.
This file is optional if you apply policies directly with `oc apply`.

. Create a `Placement` rule so that upgrade policies can be bound to clusters through labels.
+
The following example uses the label `upgrade-16to17=""` to bind a policy to a cluster:
+
[source,yaml]
----
apiVersion: cluster.open-cluster-management.io/v1beta1
kind: Placement
metadata:
  name: openshift-upgrade-placement
  namespace: policies
  annotations:
    policy.open-cluster-management.io/description: |
      Placement rule for OpenShift upgrade policies. Targets clusters with
      the label "upgrade-16to17=true" for 4.16 to 4.17 upgrades.
spec:
  predicates:
    - requiredClusterSelector:
        labelSelector:
          matchLabels:
            upgrade-16to17: ""
----

. Create a `PlacementBinding` to connect the policies to the placement rule:
+
[source,yaml]
----
apiVersion: policy.open-cluster-management.io/v1
kind: PlacementBinding
metadata:
  name: openshift-upgrade-placement-binding
  namespace: policies
  annotations:
    policy.open-cluster-management.io/description: |
      Binding that connects OpenShift upgrade policies to their placement rule.
      This binding ensures that both the main upgrade policy and pre-upgrade
      health check policy are applied to the same set of target clusters.
placementRef:
  name: openshift-upgrade-placement
  kind: Placement
  apiGroup: cluster.open-cluster-management.io
subjects:
  - name: openshift-upgrade-4.16-to-4.17-policy
    kind: Policy
    apiGroup: policy.open-cluster-management.io
  - name: pre-upgrade-health-check-policy
    kind: Policy
    apiGroup: policy.open-cluster-management.io
----
+
A single policy, placement rule, and placement binding set can be bound to multiple clusters by adding the matching label to each cluster.

. Create cluster labels for upgrade batching.
+
After creating the policies and placement bindings, bind them to clusters through cluster labels. You can manage labels through the CLI or the {rh-rhacm} UI.
+
To view the current labels on a managed cluster by using the CLI:
+
[source,terminal]
----
$ oc get mcl <cluster-name> -o json | jq .metadata.labels
----
+
.Example output
[source,terminal]
----
{
  "app.kubernetes.io/instance": "clusters",
  "cluster.open-cluster-management.io/clusterset": "default",
  "name": "test1",
  "ocp-version": "4.18",
  "openshiftVersion": "4.18.15",
  "upgrade-version-to-4-19": "",
  "upgrade-version-to-4-20": "",
  "unpause-worker-mcp": ""
}
----
+
To add an upgrade label to a cluster:
+
[source,terminal]
----
$ oc label mcl <cluster-name> upgrade-version-to-4-19=""
----
+
To remove a label from a cluster:
+
[source,terminal]
----
$ oc label mcl <cluster-name> upgrade-version-to-4-19-
----
+
In the {rh-rhacm} UI, navigate to *Infrastructure* -> *Clusters*, click the cluster, and scroll to *Labels*. Click the pencil icon to edit labels in the window.
+
Label clusters by site location for topology awareness:
+
[source,terminal]
----
$ oc label managedcluster spoke1 site=north
$ oc label managedcluster spoke2 site=south
$ oc label managedcluster spoke3 site=east
----

. Create a ManagedClusterSet for upgrade targets:
+
[source,yaml]
----
apiVersion: cluster.open-cluster-management.io/v1beta2
kind: ManagedClusterSet
metadata:
  name: core-bm-clusters
spec:
  clusterSelector:
    selectorType: LabelSelector
    labelSelector:
      matchLabels:
        cluster-type: bare-metal-core
----
+
Apply the ManagedClusterSet:
+
[source,terminal]
----
$ oc apply -f managedclusterset.yaml
----

. Configure resource limits for {cgu-operator}:
+
[source,terminal]
----
$ oc patch deployment cluster-group-upgrades-controller-manager \
  -n openshift-operators \
  --type merge \
  -p '{"spec":{"template":{"spec":{"containers":[{"name":"manager","resources":{"limits":{"cpu":"200m","memory":"256Mi"},"requests":{"cpu":"100m","memory":"128Mi"}}}]}}}}'
----
+
Verify the resource limits were applied:
+
[source,terminal]
----
$ oc get deployment cluster-group-upgrades-controller-manager \
  -n openshift-operators \
  -o jsonpath='{.spec.template.spec.containers[0].resources}' | jq
----
+
.Example output
[source,terminal]
----
{
  "limits": {
    "cpu": "200m",
    "memory": "256Mi"
  },
  "requests": {
    "cpu": "100m",
    "memory": "128Mi"
  }
}
----
+
Verify the pods restarted with the new resource limits:
+
[source,terminal]
----
$ oc get pods -n openshift-operators | grep cluster-group-upgrades
----
+
.Example output
[source,terminal]
----
cluster-group-upgrades-controller-manager-5d7b9c8f7d-xyz99   2/2     Running   0          1m
----

. Test {cgu-operator} connectivity and policy propagation:
+
Create a test policy:
+
[source,yaml]
----
apiVersion: policy.open-cluster-management.io/v1
kind: Policy
metadata:
  name: test-policy
  namespace: default
spec:
  remediationAction: inform
  disabled: false
  policy-templates:
  - objectDefinition:
      apiVersion: policy.open-cluster-management.io/v1
      kind: ConfigurationPolicy
      metadata:
        name: test-config
      spec:
        severity: low
        object-templates:
        - complianceType: musthave
          objectDefinition:
            apiVersion: v1
            kind: ConfigMap
            metadata:
              name: test-configmap
              namespace: default
            data:
              message: "TALM is working"
----
+
Apply the test policy:
+
[source,terminal]
----
$ oc apply -f test-policy.yaml
----

. Verify policy compliance in {rh-rhacm}:
+
[source,terminal]
----
$ oc get policy test-policy -n default
----
+
.Example output
[source,terminal]
----
NAME          REMEDIATION ACTION   COMPLIANCE STATE   AGE
test-policy   inform               Compliant          1m
----

.Verification

. Verify {cgu-operator} is running:
+
[source,terminal]
----
$ oc get pods -n openshift-operators | grep cluster-group-upgrades
----
+
.Example output
[source,terminal]
----
cluster-group-upgrades-controller-manager-5d7b9c8f7d-abc12   2/2     Running   0          5m
----

. Verify cluster labels are applied:
+
[source,terminal]
----
$ oc get managedclusters --show-labels
----

. Verify the ManagedClusterSet exists:
+
[source,terminal]
----
$ oc get managedclusterset core-bm-clusters
----
