= Automate file recovery with rsync

[role="_abstract"]
Use `rsync` to automate bulk file recovery operations and directory synchronization from VM backups.
This enables scripted recovery workflows and integration with existing backup automation.

.Prerequisites

* You created a `VirtualMachineFileRestore` CR with SSH access configured.
* You retrieved the SSH service address and private key from the `VirtualMachineFileRestore` status.
* `rsync` is installed on your local system.

.Procedure

. Retrieve the SSH connection details from the `VirtualMachineFileRestore` status:
+
[source,terminal]
----
$ SSH_ADDRESS=$(oc get vmfr <vmfr_name> -n openshift-adp -o jsonpath='{.status.fileServingInfo.ssh.address}')
----
+
Replace `<vmfr_name>` with the name of your `VirtualMachineFileRestore` CR. This sets the `SSH_ADDRESS` variable for use in subsequent commands.

. Save the private key from the autogenerate secret to a local file:
+
[source,terminal]
----
$ oc get secret <private_key_secret> -n openshift-adp \
    -o jsonpath='{.data.privateKey}' | base64 -d > /tmp/vmfr-key
----
+
Replace `<private_key_secret>` with the secret name from `status.fileServingInfo.ssh.credentialsSecretRef.name`.

. Set appropriate permissions on the private key file:
+
[source,terminal]
----
$ chmod 600 /tmp/vmfr-key
----

. Perform a dry run to preview the synchronization operation:
+
[source,terminal]
----
$ rsync -av --dry-run \
    -e 'ssh -p 2222 -i /tmp/vmfr-key' \
    <username>@<address>:/restores/<date>/<backup_name>/<vm_name>/<remote_path> \
    <local_path>
----
+
where:
+
`-a`:: Archive mode preserves permissions, timestamps, and symbolic links.
`-v`:: Verbose output shows files being transferred.
`--dry-run`:: Previews the operation without transferring files.
`-e`:: Specifies the SSH command with port and key location.
`<username>`:: SSH username from `status.fileServingInfo.ssh.username` or `oadp` if autogenerate.
`<address>`:: SSH service address from the previous step.
`<date>`:: Backup creation date in the format `YYYYMMDD-HHMMSS`.
`<backup_name>`:: Velero backup name.
`<vm_name>`:: Virtual machine name.
`<remote_path>`:: Path to the directory or file in the backup.
`<local_path>`:: Local destination directory.

. Review the dry run output to verify the correct files will be transferred.

. Run the synchronization operation by removing the `--dry-run` flag:
+
[source,terminal]
----
$ rsync -av \
    -e 'ssh -p 2222 -i /tmp/vmfr-key' \
    <username>@<address>:/restores/<date>/<backup_name>/<vm_name>/<remote_path> \
    <local_path>
----

. For incremental recovery across multiple backups, use `rsync` to compare and retrieve only changed files:
+
[source,terminal]
----
$ rsync -av --update \
    -e 'ssh -p 2222 -i /tmp/vmfr-key' \
    <username>@<address>:/restores/<date>/<backup_name>/<vm_name>/<remote_path> \
    <local_path>
----
+
The `--update` flag skips files that are newer in the destination than the source.

. For large transfers, enable compression and bandwidth limiting:
+
[source,terminal]
----
$ rsync -avz --bwlimit=10000 --progress \
    -e 'ssh -p 2222 -i /tmp/vmfr-key' \
    <username>@<address>:/restores/<date>/<backup_name>/<vm_name>/<remote_path> \
    <local_path>
----
+
where:
+
`-z`:: Enables compression during transfer.
`--bwlimit=10000`:: Limits bandwidth to 10000 KB per second.
Set this value based on your network capacity and recovery time requirements.
For a 1 Gbps network with 50% allocated to backup traffic, use `--bwlimit=62500` (500 Mbps / 8).
For a 100 Mbps network, use `--bwlimit=12500` (100 Mbps / 8).
For shared networks, limit to 10-20% of capacity to avoid impacting production traffic.
To calculate the appropriate value, divide your available bandwidth in Mbps by 8 to get kilobytes per second.
`--progress`:: Displays transfer progress.

. To compare files across multiple backup versions, create a script that iterates through backups:
+
[source,bash]
----
#!/bin/bash
BACKUPS=("daily-backup-20260630" "daily-backup-20260629" "daily-backup-20260628")
SSH_KEY="/tmp/vmfr-key"
SSH_ADDRESS=$(oc get vmfr <vmfr_name> -n openshift-adp -o jsonpath='{.status.fileServingInfo.ssh.address}')
USERNAME="oadp"
VM_NAME="<vm_name>"
REMOTE_FILE="/etc/myapp/config.yaml"
LOCAL_DIR="/tmp/backup-comparison"

mkdir -p "$LOCAL_DIR"

# Retrieve the date directory from SSH server
DATE_DIR=$(ssh -p 2222 -i "$SSH_KEY" "${USERNAME}@${SSH_ADDRESS}" "ls -1 /restores/ | head -1")

for backup in "${BACKUPS[@]}"; do
    rsync -av \
        -e "ssh -p 2222 -i $SSH_KEY" \
        "${USERNAME}@${SSH_ADDRESS}:/restores/${DATE_DIR}/${backup}/${VM_NAME}${REMOTE_FILE}" \
        "${LOCAL_DIR}/${backup}-config.yaml"
done

echo "Files downloaded to $LOCAL_DIR for comparison"
----
+
Replace `<vmfr_name>` with the name of your VMFR CR and `<vm_name>` with the virtual machine name.

.Verification

. Verify the transferred files match the source using checksums. First retrieve the date directory, then generate checksums:
+
[source,bash]
----
$ DATE_DIR=$(ssh -p 2222 -i /tmp/vmfr-key ${USERNAME}@${SSH_ADDRESS} "ls -1 /restores/ | head -1")
$ ssh -p 2222 -i /tmp/vmfr-key ${USERNAME}@${SSH_ADDRESS} \
    "md5sum /restores/${DATE_DIR}/<backup_name>/<vm_name>/<remote_path>"
----
+
[source,terminal]
----
$ md5sum <local_path>
----
+
Compare the checksums to confirm successful transfer.

. Clean up the private key file after recovery completes:
+
[source,terminal]
----
$ rm -f /tmp/vmfr-key
----
