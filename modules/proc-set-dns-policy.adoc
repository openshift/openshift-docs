= Set the DNS policy

[role="_abstract"]
Control multicluster ingress by using DNS to bring traffic to your gateways with `DNSPolicy` custom resources (CRs). Setting a DNS policy automates the link between your `Gateway` IP address and a human-readable hostname.

Create and apply a default `DNSPolicy` custom resource (CR) to ensure consistency and cleanup. You can standardized naming by creating a pattern for developers who need URLs, manage time-to-live values, and set up automatic cleanup of unused DNS records.

.Prerequisites

* You installed {prodname} on at least one {ocp} cluster.
* You installed the {oc-first}.
* You have write access to the {ocp} namespaces you need to work with.
* You have access to external or on-premise DNS.
* You created a DNS-provider `Secret` object.
* You created a `Gateway` object.

.Procedure

. Create a `DNSPolicy` custom resource (CR), for example, `{KUADRANT_GATEWAY_NAME}-dnspolicy.yaml`, that includes the following information:
+
[source,yaml,subs="+quotes"]
----
apiVersion: kuadrant.io/v1
kind: DNSPolicy
metadata:
  name: _<${KUADRANT_GATEWAY_NAME}-dnspolicy>_
  namespace: _<${KUADRANT_GATEWAY_NS}>_
spec:
  healthCheck:
    failureThreshold: 3
    interval: 1m
    path: /health
  loadBalancing:
    defaultGeo: true
    geo: GEO-NA
    weight: 120
  targetRef:
    name: _<${KUADRANT_GATEWAY_NAME}>_
    group: gateway.networking.k8s.io
    kind: Gateway
  providerRefs:
  - name: _<dns_provider_credentials>_
----
+
* Replace `_<{KUADRANT_GATEWAY_NAME}-dnspolicy.yaml>_` with the environment variable you defined.
* Replace `_<{KUADRANT_GATEWAY_NS}>_` with the environment variable you defined.
* `spec.loadBalancing.geo`: Defines a geographically relevant load balancer. In this example, `GEO-NA` is used. Change this to match your requirements.
* `spec.providerRefs`: Replace `_<dns_provider_credentials>_` with a reference to the {ocp} secret containing the credentials for your DNS provider.

. Apply the `DNSPolicy` CR by running the following command:
+
[source,terminal,subs="+quotes"]
----
$ oc apply -f _<{KUADRANT_GATEWAY_NAME}_dnspolicy.yaml>_ -n _<gateway_namespace>_
----
+
* Replace `_<{KUADRANT_GATEWAY_NAME}_dnspolicy.yaml>_` with the filename you used.
* Replace `_<gateway_namespace>_` with name of the {ocp} namespace where the `Gateway` object is applied.

.Verification

. Verify that your `DNSPolicy` object has a status of `Accepted` and `Enforced` by running the following command:
+
[source,terminal,subs="+quotes"]
----
$ oc get dnspolicy _<${KUADRANT_GATEWAY_NAME}-dnspolicy>_ -n _<${KUADRANT_GATEWAY_NS}>_ -o=jsonpath='{.status.conditions[?(@.type=="SubResourcesHealthy")].message}'
----
+
* Replace `_<{KUADRANT_GATEWAY_NAME}_dnspolicy.yaml>_` with the filename you used.
* Replace `_<{KUADRANT_GATEWAY_NS}>_` with the environment variable you used.
* This process might take a few minutes.

. Check the status of the DNS health checks that are enabled on your DNS policy by running the following command:
+
[source,terminal,subs="+quotes"]
----
$ oc get dnspolicy _<${KUADRANT_GATEWAY_NAME}_dnspolicy>_ -n _<${KUADRANT_GATEWAY_NS}>_ -o=jsonpath='{.status.conditions[?(@.type=="SubResourcesHealthy")].message}'
----
+
* Replace `_<{KUADRANT_GATEWAY_NAME}_dnspolicy.yaml>_` with the filename you used.
* Replace `_<{KUADRANT_GATEWAY_NS}>_` with the environment variable you used.
* These health checks flag a published endpoint as healthy or unhealthy based on the defined configuration. When unhealthy, an endpoint is not published if it has not already been published to the DNS provider. An endpoint is only unpublished if it is part of an `A` record that has more than one value. You can see the endpoint status for all cases in the `DNSPolicy` object status.
