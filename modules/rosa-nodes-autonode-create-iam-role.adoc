= Creating the {autonode} IAM role

[role="_abstract"]
Create an Amazon Web Services (AWS) Identity and Access Management (IAM) role with a trust policy that allows the Karpenter service account to assume the role by using web identity federation, and attach the {autonode} IAM policy to it.

.Prerequisites

* You have created the OpenID Connect (OIDC) configuration for your cluster.
* You have created the {autonode} IAM policy.
* You have the `OIDC_CONFIG_ID` and `POLICY_ARN` environment variables set from previous procedures.
* You have the AWS command-line interface (CLI) installed and configured.
* You have your AWS account ID available.

.Procedure

. Get the OIDC provider URL from your OIDC configuration:
+
[source,terminal]
----
$ OIDC_PROVIDER_URL=$(rosa list oidcconfig | \
  grep $OIDC_CONFIG_ID | \
  awk '{ gsub(/^https:\/\//, "", $3); print $3 }')
----
+
This command extracts the OIDC provider URL and removes the `https://` prefix, which is required for the trust policy.

. Verify the OIDC provider URL:
+
[source,terminal]
----
$ echo $OIDC_PROVIDER_URL
----
+
The output should display a URL without the `https://` prefix, similar to: `rh-oidc.s3.us-east-1.amazonaws.com/<unique-id>`

. Set your AWS account ID:
+
[source,terminal]
----
$ export AWS_ACCOUNT_ID=<your_aws_account_id>
----
+
Replace `<your_aws_account_id>` with your 12-digit AWS account ID.

. Create a trust policy template file named `trust-policy-template.json`:
+
[source,terminal]
----
$ cat > trust-policy-template.json <<'EOF'
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Principal": {
                "Federated": "arn:aws:iam::{AWS_ACCOUNT_ID}:oidc-provider/{OIDC_PROVIDER_URL}"
            },
            "Action": "sts:AssumeRoleWithWebIdentity",
            "Condition": {
                "StringEquals": {
                    "{OIDC_PROVIDER_URL}:sub": "system:serviceaccount:kube-system:karpenter"
                }
            }
        }
    ]
}
EOF
----
+
This template defines a trust relationship that allows the Karpenter service account in the `kube-system` namespace to assume the IAM role by using the OIDC provider.

. Substitute the placeholder values in the template to create the final trust policy:
+
[source,terminal]
----
$ sed -e "s|{AWS_ACCOUNT_ID}|$AWS_ACCOUNT_ID|g" \
      -e "s|{OIDC_PROVIDER_URL}|$OIDC_PROVIDER_URL|g" \
      trust-policy-template.json > ${PREFIX}-trust-policy.json
----

. Verify the trust policy was created correctly:
+
[source,terminal]
----
$ cat ${PREFIX}-trust-policy.json
----
+
Ensure that the `{AWS_ACCOUNT_ID}` and `{OIDC_PROVIDER_URL}` placeholders have been replaced with actual values.

. Create the IAM role with the trust policy:
+
[source,terminal]
----
$ ROLE_ARN=$(aws iam create-role \
  --role-name ${PREFIX}-autonode-operator-role \
  --assume-role-policy-document file://${PREFIX}-trust-policy.json \
  --query 'Role.Arn' \
  --output text)
----
+
This command creates the IAM role and captures its ARN in the `ROLE_ARN` variable.

. Attach the {autonode} policy to the role:
+
[source,terminal]
----
$ aws iam attach-role-policy \
  --role-name ${PREFIX}-autonode-operator-role \
  --policy-arn $POLICY_ARN
----

. Verify the role was created and the policy was attached:
+
[source,terminal]
----
$ echo $ROLE_ARN
$ aws iam get-role --role-name ${PREFIX}-autonode-operator-role
$ aws iam list-attached-role-policies --role-name ${PREFIX}-autonode-operator-role
----

[IMPORTANT]
====
Save the `ROLE_ARN` value. You will need it when enabling {autonode} on your {product-title} cluster.
====

.Verification

* The `aws iam get-role` command displays the role details, including the trust policy.
* The `aws iam list-attached-role-policies` command displays the {autonode} policy attached to the role.
* The role Amazon Resource Name (ARN) follows the format: `arn:aws:iam::<account_id>:role/<PREFIX>-autonode-operator-role`
