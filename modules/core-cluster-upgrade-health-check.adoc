= Perform health checks before bare metal Core cluster upgrades

[role="_abstract"]
You can perform comprehensive health checks before upgrading bare metal Core clusters to ensure cluster components are ready for the upgrade.
These checks validate cluster operators, etcd health, storage systems, network connectivity, and custom resources specific to your deployment.

Complete these health checks before starting any cluster upgrade to minimize the risk of upgrade failures.

.Prerequisites

* Access to target clusters with cluster-admin privileges.
* The `oc` CLI tool is installed and configured for the target cluster.

.Procedure

. Verify control plane node readiness:
+
[source,terminal]
----
$ oc get nodes -l node-role.kubernetes.io/master=
----
+
.Example output
[source,terminal]
----
NAME                   STATUS   ROLES                  AGE   VERSION
master-0.example.com   Ready    control-plane,master   30d   v1.30.0
master-1.example.com   Ready    control-plane,master   30d   v1.30.0
master-2.example.com   Ready    control-plane,master   30d   v1.30.0
----
+
All control plane nodes must show `Ready` status with no `SchedulingDisabled` notation.

. Verify worker node readiness:
+
[source,terminal]
----
$ oc get nodes -l node-role.kubernetes.io/worker=
----
+
All worker nodes must show `Ready` status.
If any nodes show `NotReady` or `SchedulingDisabled`, investigate and resolve before upgrading.

. Check cluster operator status:
+
[source,terminal]
----
$ oc get co
----
+
.Example output
[source,terminal]
----
NAME                                       VERSION   AVAILABLE   PROGRESSING   DEGRADED   SINCE   MESSAGE
authentication                             4.20.0    True        False         False      30d
cloud-controller-manager                   4.20.0    True        False         False      30d
cluster-autoscaler                         4.20.0    True        False         False      30d
----
+
All cluster operators must show:
+
--
* `AVAILABLE`: `True`
* `PROGRESSING`: `False`
* `DEGRADED`: `False`
--
+
If any operator is degraded or progressing, resolve the issue before proceeding.

. Verify etcd cluster health:
+
[source,terminal]
----
$ oc get etcd -o=jsonpath='{range .items[0].status.conditions[?(@.type=="EtcdMembersAvailable")]}{.message}{"\n"}{end}'
----
+
.Example output
[source,terminal]
----
3 members are available
----
+
All etcd members must be available.
If the output indicates unhealthy members, investigate etcd pod logs before upgrading.

. Check for recent etcd backup:
+
Verify that an etcd backup exists from within the past 24 hours.
Check your backup storage location for recent backup timestamps.
+
If no recent backup exists, create an etcd backup before proceeding.
+
Identify a control plane node to use for the backup:
+
[source,terminal]
----
$ oc get nodes -l node-role.kubernetes.io/master= -o jsonpath='{.items[0].metadata.name}'
----
+
.Example output
[source,terminal]
----
master-0.example.com
----
+
Create the etcd backup on the selected control plane node:
+
[source,terminal]
----
$ oc debug node/master-0.example.com -- chroot /host /usr/local/bin/cluster-backup.sh /home/core/assets/backup
----
+
.Example output
[source,terminal]
----
Creating etcd snapshot...
Snapshot saved to /home/core/assets/backup/snapshot_2026-03-31_120000.db
Backing up static pod resources...
Backup saved to /home/core/assets/backup/static_kuberesources_2026-03-31_120000.tar.gz
----
+
Verify the backup files were created:
+
[source,terminal]
----
$ oc debug node/master-0.example.com -- chroot /host ls -lh /home/core/assets/backup
----
+
.Example output
[source,terminal]
----
-rw-------. 1 root root  98M Mar 31 12:00 snapshot_2026-03-31_120000.db
-rw-------. 1 root root 2.1M Mar 31 12:00 static_kuberesources_2026-03-31_120000.tar.gz
----
+
Copy the backup files from the node to secure storage for safekeeping.
The snapshot file should be several megabytes in size.
If the file size is unexpectedly small, investigate before proceeding with the upgrade.

. Verify machine config pool status:
+
[source,terminal]
----
$ oc get mcp
----
+
.Example output
[source,terminal]
----
NAME     CONFIG                                             UPDATED   UPDATING   DEGRADED   MACHINECOUNT   READYMACHINECOUNT   UPDATEDMACHINECOUNT   DEGRADEDMACHINECOUNT   AGE
master   rendered-master-abc123                             True      False      False      3              3                   3                     0                      30d
worker   rendered-worker-def456                             True      False      False      3              3                   3                     0                      30d
----
+
All machine config pools must show:
+
--
* `UPDATED`: `True`
* `UPDATING`: `False`
* `DEGRADED`: `False`
* `MACHINECOUNT` must equal `READYMACHINECOUNT` and `UPDATEDMACHINECOUNT`
--

. Check persistent volume status:
+
[source,terminal]
----
$ oc get pv
----
+
All persistent volumes must show `Bound` or `Available` status.
No persistent volumes should show `Failed` status.

. Verify storage class configuration:
+
[source,terminal]
----
$ oc get storageclass
----
+
Verify that a default storage class exists for your cluster.

. Validate network connectivity for workloads:
+
If your cluster uses SR-IOV, verify network node policies:
+
[source,terminal]
----
$ oc get sriovnetworknodepolicy -n openshift-sriov-network-operator
----
+
All SR-IOV network node policies must show `Succeeded` or `InProgress` state.

. Verify OLM operator health:
+
[source,terminal]
----
$ oc get csv -A
----
+
All ClusterServiceVersions must show `Succeeded` phase.
If any operators show `Failed` or `Installing` phase, investigate before upgrading.

. Verify pod status in all namespaces:
+
[source,terminal]
----
$ oc get pod -A | grep -E -i -v 'complete|running'
----
+
This command must produce no output.
If any pods are listed, review their status before upgrading.

. Check Pod Disruption Budget configuration:
+
[source,terminal]
----
$ oc get pdb -A
----
+
Verify that PDBs are configured for all critical CNF workloads.
+
For example:
+
[source,terminal]
----
$ oc get pdb -n cnf-workloads
----
+
.Example output
[source,terminal]
----
NAME                    MIN AVAILABLE   MAX UNAVAILABLE   ALLOWED DISRUPTIONS   AGE
cnf-workload-pdb        2               N/A               1                     30d
----
+
Ensure PDBs allow at least one disruption (do not set `minAvailable` to 100% of replicas).

. Validate custom resource status:
+
Check Performance Profile status:
+
[source,terminal]
----
$ oc get performanceprofile
----
+
.Example output
[source,terminal]
----
NAME               AGE
core-performance   30d
----
+
Verify the PerformanceProfile shows `Applied` status:
+
[source,terminal]
----
$ oc get performanceprofile core-performance -o jsonpath='{.status.conditions[?(@.type=="Applied")].status}'
----
+
The output should be `True`.

. Verify API server responsiveness:
+
[source,terminal]
----
$ oc version
----
+
The command should complete within a few seconds.
If the API server is slow to respond, investigate before upgrading.

. Check certificate expiration:
+
[source,terminal]
----
$ oc get certificates -A
----
+
Verify that no certificates are near expiration (within 30 days).

. Verify node resource utilization:
+
Check CPU and memory utilization on nodes:
+
[source,terminal]
----
$ oc adm top nodes
----
+
.Example output
[source,terminal]
----
NAME                   CPU(cores)   CPU%   MEMORY(bytes)   MEMORY%
master-0.example.com   500m         25%    8Gi             50%
worker-0.example.com   1000m        50%    16Gi            60%
----
+
Nodes should have sufficient headroom for rolling updates.
CPU usage should be below 80%, and memory usage should be below 85%.

.Verification

. Review the results of all health checks.

. If all checks pass, the cluster is ready for upgrade.

. If any checks fail, resolve the issues before proceeding with the upgrade.

.Troubleshooting

* If cluster operators are degraded, check operator logs:
+
[source,terminal]
----
$ oc logs -n openshift-<operator-namespace> <operator-pod-name>
----

* If etcd members are unhealthy, investigate etcd pod logs:
+
[source,terminal]
----
$ oc logs -n openshift-etcd <etcd-pod-name>
----

* If PDBs block upgrades, review PDB configuration to ensure they allow sufficient disruptions.
