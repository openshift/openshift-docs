= Understanding web browser file access

[role="_abstract"]
Web browser file access provides an interactive interface for browsing and downloading files from VM backups without command-line tools.
This method is appropriate for interactive recovery scenarios where visual file inspection and preview capabilities improve recovery workflows.

The FileBrowser web interface provides HTTPS access to files from VM backups with authentication enforced through Kubernetes secrets.
Files are mounted read-only from backup storage and served through a dedicated pod in a temporary namespace.
The interface supports directory navigation, file preview for common formats, and archive download for multiple files.

== Web browser capabilities

You can browse backup file systems interactively with directory tree navigation across multiple backup versions.
File previews are available for text files, images, and other common formats to validate content before download.
Individual files can be downloaded directly or multiple files and directories can be downloaded as compressed archives.

The file path format `/restores/<date>/<backup_name>/<vm_name>/<path>` organizes files by backup creation date and backup name.
This structure enables comparison of file versions across multiple backups simultaneously.

== Authentication and access control

Authentication credentials are stored in a Kubernetes secret referenced in the `VirtualMachineFileRestore` CR.
The secret contains a username and password generated automatically or provided by the user.
Credentials are retrieved from the secret and used to access the web interface.

Service exposure can be configured as cluster-internal or external:

Cluster-internal:: The file browser is accessible only from within the cluster by using a ClusterIP service.
This provides additional security by limiting access to cluster users.

External route:: An OpenShift route exposes the file browser to users outside the cluster.
This enables file recovery from external networks but requires careful consideration of security policies and network access controls.

== When to use web browser access

Web browser access is appropriate for the following scenarios:

Interactive recovery:: When you need to browse backup contents visually to locate files without knowing exact file paths.

File validation:: When you need to preview file contents before downloading to confirm the correct version or validate file integrity visually.

Non-technical users:: When recovery operations are performed by users who prefer graphical interfaces over command-line tools.

Ad hoc restoration:: When recovering small numbers of files where scripted automation is not required.

== Security considerations

External route exposure introduces potential security risks:

* Credentials are transmitted over HTTPS but the route may be accessible to unauthorized users if network policies are not configured properly.
* Consider using cluster-internal access unless external access is specifically required.
* Review network policies and firewall rules before exposing routes externally.
* Rotate credentials after recovery operations complete.
* Delete the `VirtualMachineFileRestore` CR promptly after file recovery to remove the exposed service.

Read-only access is enforced at the file system level using libguestfs mounts with the `--ro` flag.
This prevents modification of backup data through the web interface.
