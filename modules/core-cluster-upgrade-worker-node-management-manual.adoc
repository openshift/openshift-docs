= Pausing and unpausing worker nodes by using machine config pools

[role="_abstract"]
You can pause and unpause worker nodes directly by patching the `MachineConfigPool` resource on the target cluster.
Use this method when you are managing a single cluster or when you do not have {cgu-operator-full} ({cgu-operator}) configured.

.Prerequisites

* Access to the target cluster with cluster-admin privileges.
* The `oc` CLI tool is installed and configured.

.Procedure

. Pause the worker `MachineConfigPool` resource by running the following command:
+
[source,terminal]
----
$ oc patch mcp worker --type merge -p '{"spec":{"paused":true}}'
----

. Verify the worker `MachineConfigPool` resource is paused by running the following command:
+
[source,terminal]
----
$ oc get mcp worker -o jsonpath='{.spec.paused}'
----
+
.Example output
[source,terminal]
----
true
----

. Monitor the paused worker status during the control plane update by running the following command:
+
[source,terminal]
----
$ oc get mcp worker
----
+
The worker `MachineConfigPool` resource should show `paused=true` and workers should remain at the previous version.

. Verify that workloads continue running on worker nodes during the control plane update by running the following command:
+
[source,terminal]
----
$ oc get pods -A -o wide | grep worker
----

. After the control plane update completes and you are ready to update worker nodes, unpause the worker `MachineConfigPool` resource by running the following command:
+
[source,terminal]
----
$ oc patch mcp worker --type merge -p '{"spec":{"paused":false}}'
----

. Monitor worker node updates after unpausing by running the following command:
+
[source,terminal]
----
$ oc get mcp worker -w
----
+
Nodes update in a rolling fashion based on the `maxUnavailable` setting.
The default `maxUnavailable` is 1 or 33% of workers, whichever is smaller.

. Watch individual worker node status by running the following command:
+
[source,terminal]
----
$ oc get nodes -l node-role.kubernetes.io/worker= -w
----
+
Nodes progress through the following states:
+
--
* `Ready,SchedulingDisabled`: Node is being drained
* `NotReady,SchedulingDisabled`: Node is rebooting with new configuration
* `Ready`: Node update complete
--

.Verification

. Verify all worker nodes are at the target version by running the following command:
+
[source,terminal]
----
$ oc get nodes
----

. Verify the worker `MachineConfigPool` resource is fully updated by running the following command:
+
[source,terminal]
----
$ oc get mcp worker
----
+
Verify `MACHINECOUNT` = `READYMACHINECOUNT` = `UPDATEDMACHINECOUNT`.

. Verify workload health after worker updates by running the following command:
+
[source,terminal]
----
$ oc get pods -A | grep -v Running | grep -v Completed
----
