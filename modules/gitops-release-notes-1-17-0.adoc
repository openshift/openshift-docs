== New features

[id="gitops-1-17-0-extended-json-logging-support-for-additional-argo-cd-components_{context}"]
=== Extended JSON logging support to additional Argo CD components

* With this update, the `logFormat: json` setting in the Argo CD custom resource (CR) is supported by the `ApplicationSet` controller and `Notifications` controller. Previously, JSON logging was limited to components such as the API server, repo server, and application controller. link:https://issues.redhat.com/browse/GITOPS-6668[GITOPS-6668]
+
[NOTE]
====
Redis and Dex do not support configurable JSON logging and are unaffected by this change.
====

[id="fixed-issues-1-17-0_{context}"]
== Fixed issues

[id="gitops-1-17-0-fixed-rolebinding-errors-by-long-SourceNamespace-project-names_{context}"]
=== Fixed `RoleBinding` errors caused by long `SourceNamespace` project names

* Before this update, the Argo CD Operator failed to create `RoleBindings` when `sourceNamespaces` contained long namespace names. This issue was caused by Kubernetes label validation errors, as namespace names exceeding `63` characters exceeded the label value length limit. The Argo CD Operator attempted to use the full namespace name as a label value, resulting in reconciliation failures with errors such as `must be no more than 63 characters`. With this update, the issue has been fixed by introducing a `truncateWithHash` function that:

** Truncates names to comply with the 63-character limit
** Appends a deterministic hash suffix to maintain uniqueness
** Ensures consistent and unique label values for long namespace names
+
As a result, `RoleBindings` are now created successfully for long `sourceNamespaces`. link:https://issues.redhat.com/browse/GITOPS-7018[GITOPS-7018]

[id="gitops-1-17-0-fixed-replica-count-for-redis-ha-haproxy-to-ensure-ha_{context}"]
=== Fixed replica count for `redis-ha-haproxy` to ensure high availability

* Before this update, a misconfiguration caused the `redis-ha-haproxy` deployment to use only one replica, resulting in intermittent access issues with the Redis cache. With this update, the number of `redis-ha-haproxy` replicas now matches the High Availability (HA) configuration. Two replicas are created by default, improving Redis cache availability and resilience. link:https://issues.redhat.com/browse/GITOPS-6806[GITOPS-6806]

[id="gitops-1-17-0-gitops-plugin-pods-comply-with-pod-security-policy_{context}"]
=== `gitops-plugin` Pods comply with Pod Security policy

* Before this update, `gitops-plugin` Pods failed to start due to missing security context properties, resulting in non-compliance with the Pod security policy. With this update, `gitops-plugin` includes the required security context settings, ensuring compliance with the Pod security policy. link:https://issues.redhat.com/browse/GITOPS-6777[GITOPS-6777]

[id="gitops-1-17-0-fixed-invalid-route-hostnames-for-argo-cd-webhook-server_{context}"]
=== Fixed invalid Route hostnames for Argo CD webhook server

* Before this update, enabling `applicationSet.webhookServer.route` caused an invalid Route error due to the default hostname exceeding the Kubernetes label value limit. This issue resulted in application deployment failures. With this update, the default hostname is now generated using the Argo CD instance name and namespace, ensuring it stays within the valid length limits. This fix resolves the error and allows successful creation of the webhook server Route. link:https://issues.redhat.com/browse/GITOPS-6707[GITOPS-6707]

[id="gitops-1-17-0-support-added-for-repeated-flags-in-extraCommandArgs_{context}"]
=== Support added for repeated flags in `extraCommandArgs`

* Before this update, the `extraCommandArgs` field in the Argo CD custom resource did not support using the same flag multiple times with different values. With this update, repeated flags are now correctly handled. Multiple instances of flags can be specified, and all values are included in the controller Pod configuration. link:https://issues.redhat.com/browse/GITOPS-6703[GITOPS-6703]

[id="gitops-1-17-0-fixed-regular-for-repeated-flags-in-extraCommandArgs_{context}"]
=== Fixed regular expression handling in `sourceNamespaces`

* Before this update, incorrect regular expression processing in the `sourceNamespaces` field caused improper labeling of namespaces. As a result, the `argocd.argoproj.io/managed-by-cluster-argocd` label was not applied, affecting namespace management. With this update, Argo CD correctly processes regular expressions in `sourceNamespaces`. The {gitops-title} Operator correctly applies labels based on regex patterns, enabling management of multiple namespaces using a single expression. link:https://issues.redhat.com/browse/GITOPS-6675[GITOPS-6675]

[id="gitops-1-17-0-fixed-UI-error-for-http-based-repository-credentials_{context}"]
=== Fixed UI error when editing HTTP-based repository credentials

* Before this update, the Argo CD Web UI failed to handle repositories hosted over HTTP due to URL validation errors. This prevented users from editing credentials for HTTP-based repositories through the Web UI. With this update, URL validation logic has been enhanced, and users can edit HTTP repository credentials directly in the Web UI. link:https://issues.redhat.com/browse/GITOPS-5969[GITOPS-5969]

[id="deprecated-features-1-17-0_{context}"]
== Deprecated and removed features

[id="deprecation-of-some-fields-in-argo-cd_{context}"]
=== Deprecation of Keycloak support in OpenShift GitOps

* In {gitops-title} 1.17, the support for Keycloak-based authentication is deprecated. When Keycloak is configured for single sign-on (SSO), deprecation warnings appear in the logs. Dex is the default and recommended authentication provider. It integrates with the OpenShift OAuth server and supports authentication using OpenShift users and groups. Users are required to migrate to Dex-based authentication for continued support. To continue using Keycloak, you must deploy and manage their own link:https://docs.redhat.com/en/documentation/red_hat_build_of_keycloak/26.0/html/operator_guide/installation-[Red Hat Build of Keycloak (RHBK)] instance and manually configure the integration in the Argo CD custom resource. link:https://issues.redhat.com/browse/GITOPS-7070[GITOPS-7070]

[id="removed-support-for-deprecated-initialRepositories-repositoryCredentials-fields_{context}"]
=== Removed support for deprecated `initialRepositories` and `repositoryCredentials` fields

* In {gitops-title} 1.17, support for the deprecated `initialRepositories` and `repositoryCredentials` fields has been removed. If these fields are used, Argo CD logs a warning indicating their removal. Users can add repositories and repository credentials manually using the Argo CD CLI or the Argo CD dashboard. link:https://issues.redhat.com/browse/GITOPS-5961[GITOPS-5961]

[id="breaking-changes-1-17-0_{context}"]
== Breaking changes

[id="upgrade-to-argo-cd-3-0_{context}"]
=== Upgrade to Argo CD 3.0

This release introduces an upgrade to the Argo CD component, transitioning from v2.0 to v3.0. This is a minor upgrade that contains only minor breaking changes. The following section includes a list of minor breaking changes affecting Argo CD v3.0 and some additional information to remediate the changes. For more information, see link:https://argo-cd.readthedocs.io/en/stable/operator-manual/upgrading/2.14-3.0/[Upgrading Argo CD from v2.0 to v3.0].

[id="rbac-behavior-changes-with-argo-cd-dex-sso_{context}"]
=== RBAC behavior changes with Argo CD 3.0 and Dex SSO

* Argo CD 3.0 introduces a change in RBAC handling when using Dex for single sign-on (SSO). The RBAC system uses the `federated_claims.user_id` field instead of the previously encoded sub claim. This change affects how user identities are evaluated and matched in RBAC policies.
+
To address this, the Argo CD Operator documentation has been updated with:

** Detailed migration guidance for transitioning from sub to federated_claims.user_id
** Step-by-step remediation procedures
** Practical RBAC policy examples
** Best practices for managing RBAC with the new hands-off approach
+
Users must review and update their RBAC policies to ensure compatibility with Argo CD 3.0 when using the {gitops-title} Operator. For more information, see link:https://argocd-operator.readthedocs.io/en/latest/upgrading/#rbac-with-dex-sso-authentication[Configuring RBAC with Dex SSO Authentication].
+

link:https://issues.redhat.com/browse/GITOPS-6887[GITOPS-6887]

[id="added-guidance-for-logs-rbac-enforcement-in-argocd_{context}"]
=== Added guidance for logs RBAC enforcement in Argo CD 3.0

* Before this update, the Argo CD Operator upstream documentation did not explain the changes to logs RBAC introduced in Argo CD 3.0. In contrast to Argo CD 2.x, where users with access to Applications could also view logs, Argo CD 3.0 treats logs as a distinct RBAC resource requiring explicit permissions. With this update, the Argo CD Operator upstream documentation includes:

** A detailed explanation of logs-specific RBAC enforcement
** Migration guides and remediation steps
** Examples for updating policies to include logs permissions
** Best practices for RBAC under the new hands-off management model
+

[id="gitops-1-17-0-fine-grained-rbac-inheritance-enabled-by-default_{context}"]
=== Fine-grained RBAC inheritance enabled by default

* Starting with Argo CD v3.0, the `update` and `delete` actions apply only to the application resource. To perform these actions on managed resources, you must configure policies to allow `update/\*` or `delete/\*`. To maintain the Argo CD v2.0 behavior, the `server.rbac.disableApplicationFineGrainedRBACInheritance` parameter is set to `false` by default in the `argocd-cm` config map. You can enable the new fine-grained RBAC functionality by setting the following configuration in the Argo CD custom resource:
+
[source,YAML]
----
apiVersion: argoproj.io/v1beta1
kind: ArgoCD
metadata:
  name: example
spec:
  extraConfig:
    server.rbac.disableApplicationFineGrainedRBACInheritance: true # <1>
----
<1> Set `server.rbac.disableApplicationFineGrainedRBACInheritance` to `true` to maintain Argo CD v3.0 behavior. 
+
