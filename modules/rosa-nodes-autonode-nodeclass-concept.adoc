= Understanding node classes

[role="_abstract"]
{autonode} uses node classes to define the configuration for Amazon Elastic Compute Cloud (Amazon EC2) instances that it provisions as worker nodes in your cluster.

A node class defines the technical parameters for the Amazon EC2 instances that {autonode} creates, including the Amazon Machine Image (AMI), security groups, subnets, instance profile, and user data. Node classes work with NodePools to control how {autonode} provisions compute capacity for your workloads.

[id="rosa-nodes-autonode-nodeclass-types_{context}"]
== Node class types

{autonode} uses two types of node class custom resources:

EC2NodeClass::
The upstream Karpenter custom resource definition (CRD) that defines Amazon EC2 instance configuration parameters. NodePools reference `EC2NodeClass` resources to determine how to configure provisioned instances.

OpenShiftEC2NodeClass::
A {product-title}-specific CRD that wraps and manages `EC2NodeClass` resources. The `OpenShiftEC2NodeClass` resource automatically creates and maintains a corresponding `EC2NodeClass` resource with the appropriate configuration for {product-title} clusters.

When you enable {autonode} on your cluster, default instances of both resources are created automatically. The default `OpenShiftEC2NodeClass` manages the default `EC2NodeClass`, ensuring it has the correct configuration for your cluster.

[id="rosa-nodes-autonode-nodeclass-relationship_{context}"]
== Relationship between node class types

The two node class types work together as follows:

. You create or modify an `OpenShiftEC2NodeClass` resource.
. The {autonode} operator automatically creates or updates the corresponding `EC2NodeClass` resource with the correct configuration for your {product-title} cluster.
. NodePools reference the `EC2NodeClass` by name to provision instances with that configuration.

This two-tier architecture ensures that Amazon EC2 instances provisioned by {autonode} have the correct configuration for {product-title} clusters, including the appropriate AMI, security groups, instance profiles, and user data.

[NOTE]
====
Always create or modify `OpenShiftEC2NodeClass` resources, not `EC2NodeClass` resources directly. The {autonode} operator manages the `EC2NodeClass` resources and might overwrite manual changes.
====

[id="rosa-nodes-autonode-nodeclass-default_{context}"]
== Default node classes

When you enable {autonode}, the following default node classes are created:

Default OpenShiftEC2NodeClass::
Manages the default Amazon EC2 node configuration for your cluster. This resource automatically configures the corresponding `EC2NodeClass` with the correct AMI, security groups, subnets, and instance profile for your cluster.

Default EC2NodeClass::
Created and maintained automatically by the default `OpenShiftEC2NodeClass`. Provides the actual configuration that NodePools reference when provisioning instances.

The default node classes are configured to work with the subnets and security groups you tagged during cluster setup. NodePools that reference the default `EC2NodeClass` provision instances in the subnets and security groups tagged with `karpenter.sh/discovery=<cluster-id>`.

[id="rosa-nodes-autonode-nodeclass-custom_{context}"]
== When to create custom node classes

Consider creating custom `OpenShiftEC2NodeClass` resources in the following scenarios:

Multiple subnet sets::
When you want different NodePools to provision instances in different sets of subnets, such as separating production and development workloads across different network segments.

Custom AMIs::
When you need to use a custom AMI that includes additional software, configuration, or hardening beyond the default {product-title} AMI.

Different security configurations::
When different workloads require different security group configurations or network access policies.

Instance profiles with additional permissions::
When specific workloads need Amazon EC2 instances with additional Amazon Web Services (AWS) Identity and Access Management (IAM) permissions beyond the default worker node permissions.

Custom user data::
When you need to run custom initialization scripts on instances at launch time.

[id="rosa-nodes-autonode-nodeclass-configuration_{context}"]
== Node class configuration options

The `OpenShiftEC2NodeClass` resource supports the following configuration options:

AMI selection::
Specify a custom AMI to use for provisioned instances. By default, {autonode} uses the cluster's AMI.

Subnet selector::
Define which subnets to use for instance provisioning. By default, {autonode} uses subnets tagged with `karpenter.sh/discovery=<cluster-id>`.

Security group selector::
Define which security groups to attach to provisioned instances. By default, {autonode} uses security groups tagged with `karpenter.sh/discovery=<cluster-id>`.

Instance profile::
Specify a custom IAM instance profile for provisioned instances. By default, {autonode} uses the worker node instance profile.

User data::
Provide custom initialization scripts that run when instances launch.

Tags::
Define additional AWS resource tags to apply to provisioned instances and volumes.

[IMPORTANT]
====
Custom node class configurations must be compatible with {product-title} cluster requirements. Using incompatible AMIs, security groups, or instance profiles can prevent nodes from joining the cluster.
====

[id="rosa-nodes-autonode-nodeclass-verification_{context}"]
== Verifying node class status

You can verify that node classes are configured correctly by checking their ready status:

[source,terminal]
----
$ oc get openshiftec2nodeclass
----

.Example output
[source,terminal]
----
NAME      READY   AGE
default   True    1h
----

[source,terminal]
----
$ oc get ec2nodeclass
----

.Example output
[source,terminal]
----
NAME      READY   AGE
default   True    1h
----

Both resources should show `READY` as `True`. If a node class shows `READY` as `False`, check its status conditions for error messages:

[source,terminal]
----
$ oc describe openshiftec2nodeclass <name>
$ oc describe ec2nodeclass <name>
----
