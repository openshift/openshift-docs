= Upgrade with EUS-to-EUS control-plane-only updates

[role="_abstract"]
You can upgrade Telco Core clusters from one Extended Update Support (EUS) release to another using control-plane-only updates.
This approach upgrades the control plane while leaving worker nodes at the previous EUS version, enabling staged rollouts that minimize workload disruption.

EUS releases provide 18 months of support and include even-numbered minor versions such as 4.18, 4.20, and 4.22.
Control plane and worker nodes can be up to two minor versions apart during EUS-to-EUS upgrades.

WARNING: Control plane and worker version skew is limited to N-2 (two minor versions).
For example, if the control plane is at 4.20, workers must be upgraded before the control plane reaches 4.22, or they will be in an unsupported configuration.
Plan worker upgrades accordingly to avoid exceeding the version skew policy.
For more information about version skew policies, see the Kubernetes version and version skew support policy.

.Prerequisites

* Pre-upgrade health check completed successfully.
* TALM operator is installed and configured on the RHACM hub cluster.
* Current cluster is at the latest z-stream of a source EUS version (for example, 4.18.latest).
* Target EUS version is available (for example, 4.20).
* EUS-to-EUS upgrade path is supported (verify in release notes).

.Procedure

. Verify the current cluster version is at the latest z-stream of the source EUS:
+
[source,terminal]
----
$ oc --context=spoke1 get clusterversion
----
+
.Example output
[source,terminal]
----
NAME      VERSION   AVAILABLE   PROGRESSING   SINCE   STATUS
version   4.18.35   True        False         30d     Cluster version is 4.18.35
----
+
If not at the latest z-stream, upgrade to the latest patch release before proceeding.

. Review the EUS-to-EUS release notes:
+
Check the {product-title} documentation for:
+
--
* Supported EUS-to-EUS upgrade paths
* API deprecations between EUS versions
* Breaking changes affecting Telco workloads
* Control plane and worker version skew policies
--

. Validate that CNF workloads tolerate control plane and worker version skew:
+
Verify that your network functions can operate with the control plane at 4.20 while workers remain at 4.18.
+
Consult with application teams if needed.

. Plan the worker node upgrade schedule:
+
Determine when to upgrade worker nodes after the control plane upgrade completes.
+
Worker upgrades can be scheduled days or weeks after the control plane, depending on your maintenance windows.

. Create a GitOps policy to pause worker nodes:
+
[source,yaml]
----
apiVersion: policy.open-cluster-management.io/v1
kind: Policy
metadata:
  name: telco-pause-worker-nodes
  namespace: openshift-upgrade-policies
spec:
  remediationAction: enforce
  disabled: false
  policy-templates:
  - objectDefinition:
      apiVersion: policy.open-cluster-management.io/v1
      kind: ConfigurationPolicy
      metadata:
        name: pause-worker-mcp
      spec:
        severity: high
        object-templates:
        - complianceType: musthave
          objectDefinition:
            apiVersion: machineconfiguration.openshift.io/v1
            kind: MachineConfigPool
            metadata:
              name: worker
            spec:
              paused: true
----

. Verify the current cluster channel before creating the upgrade policy:
+
[source,terminal]
----
$ oc --context=spoke1 get clusterversion -o jsonpath='{.items[0].spec.channel}'
----
+
.Example output
[source,terminal]
----
eus-4.18
----
+
If the cluster is on a non-EUS channel (such as `stable-4.18`), changing to an EUS channel and setting the desired version simultaneously can cause unexpected upgrade behavior.

. Create a GitOps policy for the EUS upgrade:
+
[source,yaml]
----
apiVersion: policy.open-cluster-management.io/v1
kind: Policy
metadata:
  name: telco-eus-upgrade-4.20-policy
  namespace: openshift-upgrade-policies
spec:
  remediationAction: enforce
  disabled: false
  policy-templates:
  - objectDefinition:
      apiVersion: policy.open-cluster-management.io/v1
      kind: ConfigurationPolicy
      metadata:
        name: upgrade-cluster-version
      spec:
        severity: high
        object-templates:
        - complianceType: musthave
          objectDefinition:
            apiVersion: config.openshift.io/v1
            kind: ClusterVersion
            metadata:
              name: version
            spec:
              channel: eus-4.20
              desiredUpdate:
                version: 4.20.0
----
+
Note the use of the `eus-4.20` channel for EUS releases.
+
IMPORTANT: If the cluster is currently on a different channel (for example, `stable-4.18`), consider changing the channel first, verifying the change took effect, and then applying the desired version in a separate policy update.
This ensures a deterministic upgrade path.

. Apply both policies to RHACM:
+
[source,terminal]
----
$ oc apply -f telco-pause-worker-policy.yaml
$ oc apply -f telco-eus-upgrade-4.20-policy.yaml
----

. Create a ClusterGroupUpgrade CR for the EUS upgrade:
+
[source,yaml]
----
apiVersion: ran.openshift.io/v1alpha1
kind: ClusterGroupUpgrade
metadata:
  name: telco-eus-upgrade-4.18-to-4.20
  namespace: openshift-upgrade-policies
spec:
  clusters:
  - spoke1
  managedPolicies:
  - telco-pause-worker-nodes
  - telco-eus-upgrade-4.20-policy
  enable: false
  remediationStrategy:
    maxConcurrency: 1
    timeout: 10800
----
+
EUS upgrades may take longer than single y-stream upgrades, so the timeout is set to 10800 seconds (3 hours).

. Apply and enable the ClusterGroupUpgrade CR:
+
[source,terminal]
----
$ oc apply -f telco-eus-upgrade-cgu.yaml
$ oc patch cgu telco-eus-upgrade-4.18-to-4.20 \
  -n openshift-upgrade-policies \
  --type merge \
  -p '{"spec":{"enable":true}}'
----

. Monitor the EUS control plane upgrade progress:
+
[source,terminal]
----
$ oc get cgu telco-eus-upgrade-4.18-to-4.20 -n openshift-upgrade-policies -w
----

. On the target cluster, monitor cluster version:
+
[source,terminal]
----
$ oc --context=spoke1 get clusterversion -w
----
+
The cluster version progresses through intermediate versions during the EUS upgrade.
For example, upgrading from 4.18 to 4.20 goes through 4.19 internally.

. Monitor cluster operator status:
+
[source,terminal]
----
$ oc --context=spoke1 get co
----
+
Cluster operators may take longer to stabilize during EUS upgrades because of the larger version jump.

. After the control plane upgrade completes, verify control plane health:
+
[source,terminal]
----
$ oc --context=spoke1 get nodes -l node-role.kubernetes.io/master=
----
+
.Example output
[source,terminal]
----
NAME                   STATUS   ROLES                  AGE   VERSION
master-0.example.com   Ready    control-plane,master   60d   v1.30.1
master-1.example.com   Ready    control-plane,master   60d   v1.30.1
master-2.example.com   Ready    control-plane,master   60d   v1.30.1
----
+
Control plane nodes should show the new kubelet version (4.20).

. Verify cluster operators are healthy:
+
[source,terminal]
----
$ oc --context=spoke1 get co | grep -v "True.*False.*False"
----
+
All operators must be healthy before upgrading worker nodes.

. Verify worker nodes remain at the previous EUS version:
+
[source,terminal]
----
$ oc --context=spoke1 get nodes -l node-role.kubernetes.io/worker=
----
+
.Example output
[source,terminal]
----
NAME                   STATUS   ROLES    AGE   VERSION
worker-0.example.com   Ready    worker   60d   v1.27.10
worker-1.example.com   Ready    worker   60d   v1.27.10
worker-2.example.com   Ready    worker   60d   v1.27.10
----
+
Worker nodes should still show the previous kubelet version (4.18).

. Update OLM operators for the new EUS version:
+
Operators may need multiple version jumps for EUS-to-EUS upgrades.
+
[source,terminal]
----
$ oc --context=spoke1 get csv -A | grep -v Succeeded
----
+
Update operator subscriptions to EUS-compatible channels:
+
[source,terminal]
----
$ oc --context=spoke1 patch subscription sriov-network-operator-subscription \
  -n openshift-sriov-network-operator \
  --type merge \
  -p '{"spec":{"channel":"stable-4.20"}}'
----

. Schedule worker node upgrades:
+
Determine the timing for worker node upgrades based on your maintenance schedule.
+
Workers can remain at the previous EUS version for an extended period if needed.

. When ready to upgrade worker nodes, create and apply an unpause policy:
+
Follow the procedure in xref:telco-cluster-upgrade-worker-node-management_{context}[Manage worker nodes during upgrades] to unpause and upgrade workers.

. Monitor worker node upgrades to completion:
+
[source,terminal]
----
$ oc --context=spoke1 get mcp worker -w
----

.Verification

. Verify all nodes are at the target EUS version:
+
[source,terminal]
----
$ oc --context=spoke1 get nodes
----

. Verify all cluster operators are healthy:
+
[source,terminal]
----
$ oc --context=spoke1 get co
----

. Validate Telco-specific custom resources:
+
[source,terminal]
----
$ oc --context=spoke1 get performanceprofile
$ oc --context=spoke1 get ptpconfig -n openshift-ptp
$ oc --context=spoke1 get sriovnetworknodepolicy -n openshift-sriov-network-operator
----

. Run comprehensive smoke tests for all CNF workloads.

.Troubleshooting

* If the EUS upgrade path is not available, verify you are at the latest z-stream of the source EUS version.
* If the control plane upgrade gets stuck, check for operator compatibility issues with the target EUS version.
* If worker nodes fail to upgrade after unpause, check MachineConfigPool status and node logs.
