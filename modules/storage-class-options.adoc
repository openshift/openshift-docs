= Customizing the StorageClass for LVMS Device Classes


You can customize the StorageClass that LVM Storage (LVMS) creates for each device class by using the optional `storageClassOptions` field in the `LVMCluster` custom resource (CR).

Before this feature, LVMS automatically created a StorageClass for each device class and did not allow customization. Any manual edits to the StorageClass were overwritten by the operator during reconciliation.

The `storageClassOptions` field lets you control four properties of the generated StorageClass: reclaim policy, volume binding mode, additional parameters, and additional labels. If you omit `storageClassOptions`, LVMS creates the StorageClass with the same defaults as in previous versions. Existing `LVMCluster` configurations are fully backward compatible.

[NOTE]
====
No user action is required after upgrading. The `storageClassOptions` field is optional, and default values match the behavior before this feature was introduced.
====

= StorageClass options

You can configure the following fields under `storageClassOptions` in the device class specification.

`reclaimPolicy`::
Controls what happens to the PersistentVolume (PV) and its underlying logical volume when the PersistentVolumeClaim (PVC) is deleted.
+
--
Allowed values:: `Delete` (default), `Retain`
Immutable:: Yes. You cannot change this field after creating the `LVMCluster`.

When set to `Retain`, deleting a PVC does not delete the PV or the underlying logical volume on disk. Data is preserved, which is useful for data protection scenarios where accidental PVC deletion must not cause data loss. However, you must perform manual cleanup before you can delete the `LVMCluster`. See xref:deleting-an-lvmcluster[Deleting an LVMCluster] for the required steps.

When set to `Delete`, both the PV and the on-disk logical volume are removed when the PVC is deleted. This is the default behavior.

.Example
[source,yaml]
----
storageClassOptions:
  reclaimPolicy: Retain
----
--

`volumeBindingMode`::
Controls when volume binding and dynamic provisioning occur.
+
--
Allowed values:: `WaitForFirstConsumer` (default), `Immediate`
Immutable:: Yes. You cannot change this field after creating the `LVMCluster`.

`WaitForFirstConsumer` delays PV provisioning until a pod that uses the PVC is scheduled. This enables topology-aware scheduling, where LVMS creates the PV on the node where the pod will run.

`Immediate` provisions and binds the PV as soon as the PVC is created, without waiting for a consumer pod. This mode is useful for pre-provisioning workflows. On multi-node clusters, PVs might be provisioned on nodes where the consuming pod cannot run. Use `Immediate` only on single-node clusters or when node affinity is managed externally.

.Example
[source,yaml]
----
storageClassOptions:
  volumeBindingMode: Immediate
----
--

`additionalParameters`::
Adds custom key-value pairs to the StorageClass `.parameters` map.
+
--
Type:: `map[string]string`
Default:: `{}` (empty)
Maximum entries:: 16
Immutable:: Yes. You cannot change this field after creating the `LVMCluster`.

StorageClass parameters are passed to the CSI driver (TopoLVM) during volume provisioning. TopoLVM only reads two parameters that it recognizes: `topolvm.io/device-class` and `csi.storage.k8s.io/fstype`. Any other parameter is stored on the StorageClass object but is not acted upon by TopoLVM. Use `additionalParameters` for forward-compatibility when TopoLVM or a future CSI sidecar adds support for new StorageClass parameters, or for parameters consumed by other Kubernetes components such as volume snapshot controllers.

The following parameter keys are managed by LVMS and are rejected at admission if you set them through `additionalParameters`:

* `topolvm.io/device-class` â€” automatically set to the device class name
* `csi.storage.k8s.io/fstype` â€” automatically set from the `fstype` field on the device class

[IMPORTANT]
====
To change the filesystem type, use the `fstype` field on the device class directly. Do not use `additionalParameters`.
====

.Example
[source,yaml]
----
storageClassOptions:
  additionalParameters:
    custom-param-key: custom-param-value
----
--

`additionalLabels`::
Adds custom labels to the StorageClass metadata.
+
--
Type:: `map[string]string`
Default:: none
Maximum entries:: 16
Immutable:: No. This is the only mutable field. You can add, change, or remove labels after creation.

Use additional labels for organizational tagging, integration with cluster policies, monitoring, or other labeling use cases. When you remove a label from `additionalLabels`, the operator automatically removes it from the StorageClass during the next reconciliation. Labels added directly to the StorageClass by other tools are not affected by label pruning; only labels managed through `additionalLabels` are pruned.

Labels must follow Kubernetes label naming rules. Keys must be valid qualified names, and values must be valid label values.

The following label keys are reserved by the operator and cannot be set through `additionalLabels`:

* `app.kubernetes.io/managed-by`
* `app.kubernetes.io/part-of`
* `app.kubernetes.io/name`
* `app.kubernetes.io/component`
* Any key with the prefix `owned-by.topolvm.io/`

.Example
[source,yaml]
----
storageClassOptions:
  additionalLabels:
    environment: production
    team: storage
----


= Example configurations

The following examples show common `LVMCluster` configurations using `storageClassOptions`.

.Default StorageClass behavior (no options)
[source,yaml]
----
apiVersion: lvm.topolvm.io/v1alpha1
kind: LVMCluster
metadata:
  name: my-lvmcluster
  namespace: openshift-lvm-storage
spec:
  storage:
    deviceClasses:
    - name: vg1
      default: true
      thinPoolConfig:
        name: thin-pool-1
        sizePercent: 90
        overprovisionRatio: 10
----

This produces a StorageClass with `reclaimPolicy: Delete` and `volumeBindingMode: WaitForFirstConsumer`, which is the same as the behavior before this feature.

.Retain policy for data protection
[source,yaml]
----
apiVersion: lvm.topolvm.io/v1alpha1
kind: LVMCluster
metadata:
  name: my-lvmcluster
  namespace: openshift-lvm-storage
spec:
  storage:
    deviceClasses:
    - name: vg1
      default: true
      thinPoolConfig:
        name: thin-pool-1
        sizePercent: 90
        overprovisionRatio: 10
      storageClassOptions:
        reclaimPolicy: Retain
----

.Immediate binding for pre-provisioning
[source,yaml]
----
apiVersion: lvm.topolvm.io/v1alpha1
kind: LVMCluster
metadata:
  name: my-lvmcluster
  namespace: openshift-lvm-storage
spec:
  storage:
    deviceClasses:
    - name: vg1
      default: true
      thinPoolConfig:
        name: thin-pool-1
        sizePercent: 90
        overprovisionRatio: 10
      storageClassOptions:
        volumeBindingMode: Immediate
----

.All options configured together
[source,yaml]
----
apiVersion: lvm.topolvm.io/v1alpha1
kind: LVMCluster
metadata:
  name: my-lvmcluster
  namespace: openshift-lvm-storage
spec:
  storage:
    deviceClasses:
    - name: vg1
      default: true
      thinPoolConfig:
        name: thin-pool-1
        sizePercent: 90
        overprovisionRatio: 10
      storageClassOptions:
        reclaimPolicy: Retain
        volumeBindingMode: WaitForFirstConsumer
        additionalParameters:
          custom-key: custom-value
        additionalLabels:
          environment: production
          team: storage
----

.Multiple device classes with different options
[source,yaml]
----
apiVersion: lvm.topolvm.io/v1alpha1
kind: LVMCluster
metadata:
  name: my-lvmcluster
  namespace: openshift-lvm-storage
spec:
  storage:
    deviceClasses:
    - name: vg-fast
      default: true
      thinPoolConfig:
        name: thin-pool-1
        sizePercent: 90
        overprovisionRatio: 10
      deviceSelector:
        paths:
        - /dev/nvme0n1
      storageClassOptions:
        reclaimPolicy: Delete
        volumeBindingMode: WaitForFirstConsumer
        additionalLabels:
          tier: fast
    - name: vg-archive
      thinPoolConfig:
        name: thin-pool-1
        sizePercent: 90
        overprovisionRatio: 10
      deviceSelector:
        paths:
        - /dev/sda
      storageClassOptions:
        reclaimPolicy: Retain
        volumeBindingMode: WaitForFirstConsumer
        additionalLabels:
          tier: archive
----

= Resulting StorageClass

For a device class named `vg1` with the full configuration, LVMS generates a StorageClass named `lvms-vg1` with the following structure:

[source,yaml]
----
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: lvms-vg1
  annotations:
    description: "Provides RWO and RWOP Filesystem & Block volumes"
    storageclass.kubernetes.io/is-default-class: "true"
  labels:
    environment: production
    team: storage
provisioner: topolvm.io
reclaimPolicy: Retain
volumeBindingMode: WaitForFirstConsumer
allowVolumeExpansion: true
parameters:
  custom-key: custom-value
  topolvm.io/device-class: vg1
  csi.storage.k8s.io/fstype: xfs
----

The StorageClass name always follows the convention `lvms-<device-class-name>`.

= Updating labels

Labels are the only field you can update after the `LVMCluster` is created.

.Procedure

. Patch the `LVMCluster` resource to update `additionalLabels`:
+
[source,terminal]
----
$ oc -n openshift-lvm-storage patch lvmcluster <name> --type=merge \
  -p '{"spec":{"storage":{"deviceClasses":[{"name":"vg1","storageClassOptions":{"additionalLabels":{"environment":"staging"}}}]}}}'
----

. To remove a specific label, update `additionalLabels` without the label you want to remove. The operator removes the label from the StorageClass during the next reconciliation.

. To remove all custom labels, set `additionalLabels` to an empty map `{}`.

= Changing an immutable field

Three of the four `storageClassOptions` fields â€” `reclaimPolicy`, `volumeBindingMode`, and `additionalParameters` â€” are immutable after you create the `LVMCluster`. This mirrors the behavior of Kubernetes StorageClasses, which do not allow changes to these fields after creation.

If you attempt to modify an immutable field, the API server rejects the request:

[source,terminal]
----
Invalid value: "object": reclaimPolicy is immutable once set
----

There is no way to patch or update immutable fields in place. To change an immutable field, you must delete the `LVMCluster` and recreate it with the new values.

[NOTE]
====
The deletion gates require all PVCs and, for the `Retain` policy, all PVs to be removed before the `LVMCluster` can be deleted. After you recreate the `LVMCluster` with the new values, new PVCs use the updated StorageClass configuration.
====
