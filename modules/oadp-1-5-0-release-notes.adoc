= OADP 1.5.0 release notes

The {oadp-first} 1.5.0 release notes lists resolved issues and known issues.

[id="new-features-1-5-0_{context}"]
== New features

.Collecting logs with the must-gather tool is now possible 

You can collect logs, and information about {oadp-first} custom resources by using the `must-gather` tool. The `must-gather` data must be attached to all customer cases. 
This tool generates a Markdown output file with the collected information, which is is located in a cluster directory.
link:https://issues.redhat.com/browse/OADP-5384[OADP-5384]

[id="resolved-issues-1-5-0_{context}"]
== Resolved issues

.Containers now use `FallbackToLogsOnError` for `terminationMessagePolicy

With this release, for all the {oadp-first} Operator containers such as `operator-manager`, `velero`, `node-agent`, and `non-admin-controller`, the `terminationMessagePolicy` field can now set to the `FallbackToLogsOnError` value. This change ensures that if a container exits with an error and the termination message file is empty, {OCP-short} will use the last portion of the container's log output as the termination message.
link:https://issues.redhat.com/browse/OADP-5183[OADP-5183]

.Namespace admin can now access the application after restore

Previously, the Namespace admin could not execute an application after the restore operation with the errors such as `exec operation is not allowed because the pod's security context exceeds your permissions`, `unable to validate against any security context constraint`, or `not usable by user or serviceaccount, provider restricted-v2`.
With this update, this issue is now resolved and the namespace admin can access the application successfully after restore.
link:https://issues.redhat.com/browse/OADP-5611[OADP-5611]

.Specifying status restoration at the individual resource instance level using the annotation is now possible

Previously, status restoration was only configured at the resource type using the `restoreStatus` field in the Restore CR. With this release, you can now specify the status restoration at the individual resource instance level using the following annotation:
[source,terminal]
----
metadata:
  annotations:
    velero.io/restore-status: "true"
----
link:https://issues.redhat.com/browse/OADP-5968[OADP-5968]


.Restore is now successful with `excludedClusterScopedResources`

Previously, on performing the backup of an application with the `excludedClusterScopedResources` field set to `"storageclasses", "Namespace"` parameter, the backup was successful but the restore partially failed. With this update, even the restore is successful.
link:https://issues.redhat.com/browse/OADP-5239[OADP-5239]

.The `oc get dpa` command now displays Status

With this release, the `oc get dpa` command now displays Status to improve user experience.
link:https://issues.redhat.com/browse/OADP-1338[OADP-1338]

.Backup is completed even if it gets restarted during the `waitingForPluginOperations` phase

Previously, a backup was marked as failed with the `failureReason` as,  "Found a backup with status "InProgress" during the server starting, mark it as "Failed" during the `waitingForPluginsOperation` phase. With this update, backup is completed if it gets restarted during the `waitingForPluginOperations` phase.
link:https://issues.redhat.com/browse/OADP-2941[OADP-2941]

.OADP logs are now available in the JSON format

With this release, the logs are now available in the JSON format. It helps to have pre-parsed data in their elastic logs management system.
link:https://issues.redhat.com/browse/OADP-3391[OADP-3391]

[id="known-issues-1-5-0_{context}"]
== Known issues

.Kopia does not delete the artifacts after backup expiration

Even after deleting a backup, Kopia does not delete the volume artifacts from the ${bucket_name}/kopia/${namespace} on the S3 location after backup expired. To ensure that {oadp-first} functions properly, the data is not deleted and it exists in the `/kopia/` directory, for example:

* `kopia.repository`: Main repository format information such as encryption, version, and other details. 
* `kopia.blobcfg`: Configuration for how data blobs are named. 
* `kopia.maintenance`: Tracks maintenance owner, schedule, and last successful build. 
* `log*`: Log blobs. 
link:https://issues.redhat.com/browse/OADP-5131[OADP-5131]

[id="deprecated-features-1-5-0_{context}"]
== Deprecated features

.The `configuration.restic.podConfig.resourceAllocations` specification field has been deprecated

With {oadp-first} 1.5.0, the `configuration.restic.podConfig.resourceAllocations` specification field has been deprecated. Use the `nodeAgent` section with the `uploaderType` field set to `Kopia` instead of `Restic`.
link:https://issues.redhat.com/browse/OADP-5158[OADP-5158]


[id="technoloy-preview-1-5-0_{context}"]
== Technology Preview

.Support for HyperShift hosted OpenShift clusters is available as a Technology Preview

OADP can support and facilitate application migrations within HyperShift hosted {OCP-short} clusters as a Technology Preview. It ensures a seamless backup and restore operation for applications in HyperShift hosted clusters.
link:https://issues.redhat.com/browse/OADP-3930[OADP-3930]

[id="upgrade-notes-1-5-0_{context}"]
== Upgrade notes

[NOTE]
====
Always upgrade to the next minor version. *Do not* skip versions. To update to a later version, upgrade only one channel at a time. For example, to upgrade from {oadp-first} 1.1 to 1.3, upgrade first to 1.2, and then to 1.3.
====

[id="changes-oadp-1-4-to-1-4_{context}"]
=== Changes from OADP 1.4 to 1.5

The Velero server has been updated from version 1.14 to 1.16. Note that there are no changes in the Data Protection Application (DPA).

This changes the following:

Version Support changes::
OADP will implement a streamlined version support policy. Red Hat will support only one version of {oadp-first} on one {OCP-short} version to ensure better stability and maintainability. {oadp-first} 1.5.0 will be supported only on {OCP-short} 4.19 version.

OADP Self-Service::
{oadp-full} 1.5.0 introduces a new feature named {oadp-first} Self-Service, enabling namespace admin users to back up and restore applications on the {product-title}.
In the earlier versions of {oadp-full}, you needed the cluster-admin role to perform {oadp-first} operations such as backing up and restoring an application, creating a backup storage location, and so on.
+
From {oadp-first} 1.5.0 onward, you do not need the cluster-admin role to perform the backup and restore operations. You can use {oadp-first} with the namespace admin role. The namespace admin role has administrator access only to the namespace the user is assigned to.
You can use the Self-Service feature only after the cluster administrator installs the {oadp-first} Operator and provides the necessary permissions.

`backupPVC` and `restorePVC` configurations::
A `backupPVC` is an intermediate `PersistentVolumeClaim` (PVC) to access data during the data movement backup operation. You create a `readonly` backup PVC by using the `nodeAgent.backupPVC` section of the `DataProtectionApplication` (DPA).
+
A `restorePVC` is an intermediate PVC that is used to write data during the Data Mover restore operation.
+
You can configure the restorePVC in the DataProtectionApplication (DPA) by using the `ignoreDelayBinding` field.
