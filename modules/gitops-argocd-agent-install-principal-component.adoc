= Installing the Principal component

You can deploy the Principal component on the hub cluster to manage communication with connected Agent components. Create the required secrets and then deploy the Principal component on the hub cluster.

.Procedure

. Create Required Secrets.
The principal component uses a Public Key Infrastructure (PKI) to secure all communications.
+
[NOTE]
====
The CLI-generated PKI is intended for development and testing purposes only. For production environments, use certificates issued by your organization's PKI or a trusted certificate authority.
====

.. Initialize the PKI

... To initialize the Certificate Authority (CA) that signs other certificates, run the following command:
+
[source,terminal]
----
$ argocd-agentctl pki init \
     --principal-namespace "<PRINCIPAL_NAMESPACE>" \
     --principal-context "<PRINCIPAL_CONTEXT>"
----
+
where:

`<-f>`:: Overwrites the existing CA if one already exists.
`<principal-namespace>`:: Specifies the namespace where the Principal component is installed.
`<principal-context>`:: Specifies the context of the Principal component.
+
This command generates a self-signed CA certificate and private key, and creates the `argocd-agent-ca` secret that contains the CA credentials. The CA is valid for signing certificates and includes a default expiration period.

... To generate the server certificate for the Principal's gRPC service, run the following command:
+
[source,terminal]
----
$ argocd-agentctl pki issue principal
--principal-namespace "<PRINCIPAL_NAMESPACE>"
--principal-context "<PRINCIPAL_CONTEXT>"
--dns "<PRINCIPAL_DNS_NAME>"
----
+
where:

`<ip>`:: Specifies the comma-separated list of IP addresses where the Principal component is accessible.
`<dns>`:: Specifies the comma-separated list of DNS names for the Principal service.
`<u>`, `<upsert>`:: Updates the existing certificate if one already exists.
`<principal-namespace>`:: Specifies the namespace where the Principal component is installed.
`<principal-context>`:: Specifies the context of the Principal component.
+
This command issues a TLS certificate for the Principal to enable secure gRPC communication between the Principal and Agent components. It also creates the `argocd-agent-principal-tls` secret that contains the generated certificate and key.

... The resource proxy enables Argo CD to access live resources on remote clusters through the Principal component. It requires its own certificate for mutual TLS (mTLS) authentication. To generate the server certificate for the resource proxy service, run the following command:
+
[source,terminal]
----
$ argocd-agentctl pki issue resource-proxy
--principal-namespace "<PRINCIPAL_NAMESPACE>"
--principal-context "<PRINCIPAL_CONTEXT>"
--dns "<RESOURCE_PROXY_DNS>"
----
+
where:

`<ip>`:: Specifies the comma-separated list of IP addresses for the resource proxy that are reachable by the Argo CD API server.
`<dns>`:: Specifies the DNS name for the Principal service.
`<u>`, `<upsert>`:: Updates the existing certificate if one already exists.
`<principal-namespace>`:: Specifies the namespace where the Principal component is installed.
`<principal-context>`:: Specifies the context of the Principal component.
+
This command creates the `argocd-agent-resource-proxy-tls` secret that contains the TLS certificate for the resource proxy component. The certificate includes the required Subject Alternative Names (SANs) for the resource proxy service and is signed by the same Certificate Authority (CA) that was created during PKI initialization.

... You can create the JSON Web Token (JWT) signing key that the Principal component uses to sign authentication tokens. To generate the RSA private key, run the following command:
+
[source,terminal]
----
$ argocd-agentctl jwt create-key
--principal-namespace "<PRINCIPAL_NAMESPACE>"
--principal-context "<PRINCIPAL_CONTEXT>"
----
+
where:

`<u>`, `<upsert>`:: Updates the existing JWT key if one already exists.
`<principal-namespace>`:: Specifies the namespace where the Principal component is installed.
`<principal-context>`:: Specifies the context of the Principal component.
+
This command generates a 4096-bit RSA private key in the `PKCS#8 PEM` format and stores it in the `argocd-agent-jwt` secret.

... After you generate all required certificates and keys, verify that the corresponding secrets have been created in the Principal namespace by running the following command:
+
[source,terminal]
----
$ oc get secrets -n "<PRINCIPAL_NAMESPACE>" --context "<PRINCIPAL_CONTEXT>" | grep agent
----

... Run the following command to verify that all four required secrets are created before deploying the Principal component. Example output:
+
[source,terminal]
----
NAME TYPE
argocd-agent-ca kubernetes.io/tls
argocd-agent-principal-tls kubernetes.io/tls
argocd-agent-resource-proxy-tls kubernetes.io/tls
argocd-agent-jwt Opaque
----

. The {gitops-title} Operator supports the deployment of the Principal component by using the Argo CD custom resource (CR). After creating all required secrets, you can deploy the Argo CD instance with the Principal component enabled.

.. To deploy the Argo CD instance with the Principal component, create an Argo CD resource similar to the following example:
+
[source,yaml]
----
apiVersion: argoproj.io/v1beta1
kind: ArgoCD
metadata:
name: <ARGOCD_NAME>
spec:
controller:
enabled: false
argoCDAgent:
principal:
enabled: true
server:
auth: "mtls:CN=([^,]+)"
logLevel: "info"
namespace:
allowedNamespaces:
- "*"
tls:
insecureGenerate: false
jwt:
insecureGenerate: false
sourceNamespaces:
- "<APPLICATION_NAMESPACE>"
----

.. Apply the Argo CD custom resource to the hub cluster by running the following command:
+
[source,terminal]
----
$ oc apply -f argocd.yaml -n "<PRINCIPAL_NAMESPACE>" --context "<PRINCIPAL_CONTEXT>"
----

.. Verify that the Argo CD instance has deployed the Principal pod by running the following command:
+
[source,terminal]
----
$ oc get pod -n "<PRINCIPAL_NAMESPACE>" --context "<PRINCIPAL_CONTEXT>"
----

... As a result, the following output is displayed:
+
[source,terminal]
----
NAME READY STATUS
openshift-gitops-agent-principal-d4b64667b-xqzfr 1/1 Running
openshift-gitops-redis-64f6c486b6-qvttr 1/1 Running
openshift-gitops-repo-server-5467664495-z4m5g 1/1 Running
openshift-gitops-server-7bfdccc64b-k6qxk 1/1 Running
----

.. The {gitops-title} Operator also creates several services for the Principal component by running the following command:
+
[source,terminal]
----
$ oc get svc -n "<PRINCIPAL_NAMESPACE>" --context "<PRINCIPAL_CONTEXT>"
----

... As a result, the following output is displayed:
+
[source,terminal]
----
NAME TYPE PORT(S)
openshift-gitops-agent-principal ClusterIP 443/TCP
openshift-gitops-agent-principal-healthz ClusterIP 8030/TCP
openshift-gitops-agent-principal-metrics ClusterIP 9000/TCP
openshift-gitops-agent-principal-redisproxy ClusterIP 6379/TCP
openshift-gitops-agent-principal-resource-proxy ClusterIP 9090/TCP
----

.. The {gitops-title} Operator also creates a route for the Principal component, which is used to access the Principal services by running the following command:
+
[source,terminal]
----
$ oc get route -n "<PRINCIPAL_NAMESPACE>" --context "<PRINCIPAL_CONTEXT>"
----

... As a result, the following output is displayed:
+
[source,terminal]
----
NAME HOST/PORT
openshift-gitops-agent-principal principal-route.com
----

.Verification

Ensure that all Principal pods show a `Running` status and that the corresponding services and route are created. The Principal component is now ready to communicate securely with the Agent components on workload clusters.
