= Additional considerations

The following behaviors are related to `storageClassOptions` but are not controlled by it.

`allowVolumeExpansion` is always enabled::
LVMS always sets `allowVolumeExpansion: true` on generated StorageClasses. There is no `storageClassOptions` field to control this. All LVMS volumes support online expansion.

VolumeSnapshotClass is not affected::
`storageClassOptions` only affects StorageClasses. The VolumeSnapshotClass generated for each device class when thin provisioning is configured always uses a hardcoded `deletionPolicy: Delete`, regardless of the `reclaimPolicy` set in `storageClassOptions`. The `additionalParameters` and `additionalLabels` fields are also not applied to VolumeSnapshotClasses. If you need to retain snapshot data, manage it separately from the StorageClass reclaim policy.

Default StorageClass annotation behavior::
The `default` field on a device class controls the `storageclass.kubernetes.io/is-default-class` annotation on the generated StorageClass. Setting `default: true` does not guarantee the LVMS StorageClass becomes the cluster default. If another default StorageClass already exists on the cluster, for example `gp3-csi` on AWS-based {product-title} clusters, the LVMS StorageClass annotation is set to `"false"` to avoid creating a second cluster-wide default. The operator actively manages this annotation, so manual out-of-band changes are reverted during the next reconciliation.
