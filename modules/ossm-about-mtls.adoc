= About mutual Transport Layer Security (mTLS)


In {SMProduct} 2, the `ServiceMeshControlPlane` resource controlled mTLS settings. Enabling `STRICT` mTLS mode required setting `spec.security.dataPlane.mtls` to `true`. The minimum and maximum TLS protocol versions were defined using `spec.security.controlPlane.tls.minProtocolVersion` and `spec.security.controlPlane.tls.maxProtocolVersion`.


In {SMProduct} 3, the `Istio` resource replaces `ServiceMeshControlPlane` and does not include these settings. Strict mTLS mode is configured using PeerAuthentication and DestinationRule resources. TLS protocol versions are set through Istio Workload Minimum TLS Version Configuration.

Configuring mTLS settings can be complex and often leads to misconfiguration. Understanding the following `Istio` resources and concepts is essential for proper configuration:


`PeerAuthentication`:: defines the type of mTLS traffic a sidecar accepts. In `PERMISSIVE` mode, both plaintext and mTLS traffic are accepted. In `STRICT` mode, only mTLS traffic is allowed.


`DestinationRule`:: configures the type of TLS traffic a sidecar sends. In `DISABLE` mode, the sidecar sends plaintext. In `SIMPLE`, `MUTUAL`, and `ISTIO_MUTUAL` modes, the sidecar establishes a TLS connection.


`Auto mTLS`:: ensures that all inter-mesh traffic is encrypted with mTLS by default. It is controlled by the global mesh configuration field `enableAutoMtls`, which is enabled by default in {SMProduct} 2 and 3.

`Auto mTLS` is enabled by default, ensuring that all traffic through a sidecar is mTLS encrypted, regardless of the configured `PeerAuthentication` mode. mTLS operates entirely between sidecar proxies, requiring no changes to application or service code.

By default, `PeerAuthentication` is set to `PERMISSIVE` mode, allowing sidecars in the {SMProductShortName} to accept both plain-text and mTLS-encrypted traffic.
