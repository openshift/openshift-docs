= Deploy application workloads per user group
:context: ossm-deploy-application-workloads-per-user-group

This procedure demonstrates how to deploy application workloads in separate namespaces based on user groups.

.Procedure

. Create an application namespace called `app-ns-1`:
+
[source,terminal]
----
$ oc create namespace app-ns-1
----

. Label the namespace with the respective control plane:
+
[source,terminal]
----
$ oc label namespace app-ns-1 usergroup=usergroup-1 istio.io/rev=usergroup-1
----

. Deploy the `sleep` and `httpbin` applications:
+
[source,terminal]
----
$ oc apply -f https://raw.githubusercontent.com/istio/istio/release-1.23/samples/sleep/sleep.yaml -n app-ns-1
$ oc apply -f https://raw.githubusercontent.com/istio/istio/master/samples/httpbin/httpbin.yaml -n app-ns-1
----

. Wait for the `httpbin` and `sleep` pods to run with sidecars injected:
+
[source,terminal]
----
$ oc get pods -n app-ns-1
----
+
You should see output similar to the following example:
+
.Example output
[source,terminal]
----
NAME                       READY   STATUS    RESTARTS   AGE
httpbin-7f56dc944b-kpw2x   2/2     Running   0          2m26s
sleep-5577c64d7c-b5wd2     2/2     Running   0          91m
----

. Repeat the above steps for two more namespaces: `app-ns-2` and `app-ns-3`.

[NOTE]
====
You can use the following command to wait for deployment to be ready:
[source,terminal]
----
$ oc wait deployment sleep -n app-ns-1
----
====

.Verification

After deploying the applications, use the `istioctl ps` command to verify that each application workload is managed by its assigned control plane. For example, `app-ns-1` is managed by `usergroup-1`, and `app-ns-2` and `app-ns-3` are managed by `usergroup-2`.

Use this procedure to verify that application connectivity is restricted to workloads within their respective user groups:

. Send a request from the `sleep` pod in `app-ns-1` in `usergroup-1` to the `httpbin` service in `app-ns-2` in `usergroup-2`. The communication should fail:
+
[source,terminal]
----
$ oc -n app-ns-1 exec "$(oc -n app-ns-1 get pod -l app=sleep -o jsonpath={.items..metadata.name})" -c sleep -- curl -sIL http://httpbin.app-ns-2.svc.cluster.local:8000
----
+
You should see output similar to the following example:
+
.Example output
[source,terminal]
----
HTTP/1.1 503 Service Unavailable
content-length: 95
content-type: text/plain
date: Wed, 16 Oct 2024 12:05:37 GMT
server: envoy
----

. Send a request from the `sleep` pod in `app-ns-2` in `usergroup-2` to the `httpbin` service in `app-ns-3` in `usergroup-2`. The communication should work:
+
[source,terminal]
----
oc -n app-ns-2 exec "$(oc -n app-ns-2 get pod -l app=sleep -o jsonpath={.items..metadata.name})" -c sleep -- curl -sIL http://httpbin.app-ns-3.svc.cluster.local:8000
----
+
You should see output similar to the following example:
+
.Example output
[source,terminal]
----
HTTP/1.1 200 OK
access-control-allow-credentials: true
access-control-allow-origin: *
content-security-policy: default-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' camo.githubusercontent.com
content-type: text/html; charset=utf-8
date: Wed, 16 Oct 2024 12:06:30 GMT
x-envoy-upstream-service-time: 8
server: envoy
transfer-encoding: chunked
----
