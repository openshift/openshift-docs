= Ingress Controller router timeout variables

[role="_abstract"]
Reference for router timeout environment variables on the Ingress Controller deployment. These variables control how long the router waits for client, server, and tunnel connections during normal traffic.

A `TimeUnits` value uses a number and a unit: `us` (microseconds), `ms` (milliseconds, default), `s` (seconds), `m` (minutes), `h` (hours), `d` (days).

The regular expression is: [1-9][0-9]*(`us`|`ms`|`s`|`m`|`h`|`d`).

[cols="2,1,2a", options="header"]
|===
|Variable | Default | Description
| `ROUTER_BACKEND_CHECK_INTERVAL` | `5000ms` | Length of time from one liveness check to the next on back ends.
| `ROUTER_CLIENT_FIN_TIMEOUT` | `1s` | Controls the TCP finish (FIN) timeout period for the client connecting to the route. If the FIN sent to close the connection does not answer within the given time, HAProxy closes the connection. Setting a low value is harmless and uses fewer resources on the router.
| `ROUTER_DEFAULT_CLIENT_TIMEOUT` | `30s` | Length of time that a client has to acknowledge or send data.
| `ROUTER_DEFAULT_CONNECT_TIMEOUT` | `5s` | The maximum connection time.
| `ROUTER_DEFAULT_SERVER_FIN_TIMEOUT` | `1s` | Controls the TCP finish (FIN) timeout from the router to the pod backing the route.
| `ROUTER_DEFAULT_SERVER_TIMEOUT` | `30s` | Length of time that a server has to acknowledge or send data.
| `ROUTER_DEFAULT_TUNNEL_TIMEOUT` | `1h` | Maximum time that _idle_ TCP or WebSocket tunnel connections can remain open. Router reloads start a new HAProxy process for new connections while the prior process drains existing ones.
| `ROUTER_SLOWLORIS_HTTP_KEEPALIVE` | `300s` | Set the maximum time to wait for a new HTTP request to appear. If you set this value too low, it can cause problems with browsers and applications that do not expect a small `keepalive` value. When a timeout expires, HAProxy closes the connection with a TCP finish (FIN) to the client and the server; most clients open a new connection if they still need one, but some clients or applications do not handle proxy-initiated closure well.
| `ROUTER_SLOWLORIS_TIMEOUT` | `10s` | Length of time the transmission of an HTTP request can take.
| `RELOAD_INTERVAL` | `5s` | Minimum frequency for the router to reload and apply configuration changes that require a reload.
| `ROUTER_METRICS_HAPROXY_TIMEOUT` | `5s` | Timeout for the gathering of HAProxy metrics.

|===

[NOTE]
====
Several timeout settings can affect the same connection, but they do not always combine as a simple sum. On SSL passthrough routes, `tcp-request inspect-delay` is the maximum time a new connection waits for the payload HAProxy needs to pick a back end. The default is short, and HAProxy closes the connection when the wait exceeds that limit. That behavior is separate from HTTP keep-alive timers controlled by `ROUTER_SLOWLORIS_HTTP_KEEPALIVE` (`timeout http-keep-alive`).
====

[id="ingress-controller-router-timeout-example_{context}"]
.Example of a route setting custom timeout
[source,yaml]
----
apiVersion: route.openshift.io/v1
kind: Route
metadata:
  annotations:
    haproxy.router.openshift.io/timeout: 5500ms
...
----

[NOTE]
====
Setting a server-side timeout value for passthrough routes too low can cause WebSocket connections to time out often on that route.
====
