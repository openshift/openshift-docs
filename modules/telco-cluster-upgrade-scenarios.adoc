= Telco Core cluster upgrade scenarios

[role="_abstract"]
You can upgrade Telco Core clusters by using z-stream, y-stream, or Extended Update Support (EUS) to EUS approaches.
Each upgrade scenario has different risk profiles, release cadences, and use cases that align with your operational requirements and maintenance windows.

Telco Core clusters require careful upgrade planning to maintain service availability for network functions.
Understanding the differences between upgrade scenarios helps you select the right approach for your deployment.

== Z-stream upgrades

Z-stream upgrades apply patch releases within a minor version, such as upgrading from 4.20.0 to 4.20.1.
These upgrades address security vulnerabilities and bug fixes without introducing new features.

Z-stream releases follow a weekly cadence and should be applied as soon as they become available.
The risk profile for z-stream upgrades is low because they do not include API changes or feature modifications.

*Use cases for z-stream upgrades:*

* Applying critical security patches (CVEs)
* Resolving production bugs identified in the current minor version
* Maintaining compliance with security policies that require timely patching
* Regular maintenance updates during normal operating windows

*Downtime expectations:*

When configured with proper Pod Disruption Budgets (PDBs), z-stream upgrades cause minimal workload disruption.
Control plane nodes update in a rolling fashion, and worker nodes update one at a time or in small batches.

== Y-stream upgrades

Y-stream upgrades move between minor versions, such as upgrading from 4.19.x to 4.20.0.
These upgrades introduce new features, performance improvements, and might include API deprecations.

Y-stream releases follow a quarterly cadence aligned with the {product-title} release schedule.
You must upgrade through contiguous y-stream versions and cannot skip versions.
For example, to upgrade from 4.18 to 4.20, you must upgrade from 4.18 to 4.19, and then from 4.19 to 4.20.

The risk profile for y-stream upgrades is medium because they might include deprecated APIs and configuration changes.
Review release notes carefully before upgrading to identify breaking changes that affect your workloads.

*Use cases for y-stream upgrades:*

* Accessing new {product-title} features and capabilities
* Maintaining support lifecycle compliance (older versions eventually lose support)
* Meeting requirements for updated operator versions
* Adopting performance improvements and optimizations

*Downtime expectations:*

Y-stream upgrades require a planned maintenance window.
The control plane upgrade typically completes in 60 to 120 minutes, and worker node upgrades take an additional 30 to 60 minutes.
You can minimize workload disruption by pausing worker nodes and upgrading the control plane first.

== EUS to EUS upgrades

Extended Update Support (EUS) releases receive 18 months of support and include even-numbered minor versions such as 4.18, 4.20, and 4.22.
EUS-to-EUS upgrades allow you to jump between EUS versions while skipping odd-numbered releases.

Control-plane-only upgrades enable you to update the control plane to a new EUS version while leaving worker nodes at the previous version.
The control plane and workers can be up to two minor versions apart during this staged approach.

The risk profile for EUS-to-EUS upgrades is medium to high because they involve larger version jumps.
Thorough testing and validation are critical before performing EUS-to-EUS upgrades in production.

*Use cases for EUS-to-EUS upgrades:*

* Staging control plane upgrades separately from worker node upgrades
* Minimizing workload disruption by deferring worker upgrades to a later maintenance window
* Aligning upgrades with long-term support cycles (18-month support windows)
* Managing capacity-constrained environments where full cluster downtime is not feasible

*Downtime expectations:*

Control-plane-only EUS upgrades affect only control plane components.
Workloads continue running on worker nodes at the previous version.
Worker nodes can be upgraded days or weeks after the control plane, depending on your maintenance schedule.

== Selecting an upgrade scenario

The following decision matrix helps you determine which upgrade scenario to use:

.Decision matrix for upgrade scenarios
[cols="1,2,2",options="header"]
|===
|Scenario
|When to use
|Key considerations

|Z-stream
|Security patches or bug fixes needed
|Apply weekly; minimal risk; minimal downtime

|Y-stream
|New features required or approaching end of support
|Quarterly cadence; review release notes; plan maintenance window

|EUS-to-EUS
|Long-term planning with staged rollouts
|18-month support; control plane and workers can upgrade separately; larger version jumps require more validation
|===

== Multi-node cluster considerations

Multi-node Telco Core clusters have additional considerations for each upgrade scenario:

* *Control plane quorum*: During upgrades, etcd quorum must be maintained.
With three or more control plane nodes, one control plane node upgrades at a time while the others maintain quorum.
* *Worker node batching*: Worker nodes upgrade in batches controlled by the `maxUnavailable` setting in the MachineConfigPool.
For Telco workloads, set `maxUnavailable` to 1 to minimize disruption.
* *Network requirements*: Cluster network connectivity must remain available between nodes during rolling upgrades.
The CNI plugin must support version skew during control plane and worker node updates.
The default OpenShift SDN and OVN-Kubernetes CNI plugins support version skew.
If you are using a third-party CNI plugin (such as Calico, Cilium, or a vendor-specific CNI), consult the CNI vendor documentation to verify version skew support before performing control-plane-only upgrades.
* *Pod Disruption Budgets*: Configure PDBs for all critical network functions to ensure minimum replica availability during node upgrades.

== TALM orchestration for upgrades

Topology Aware Lifecycle Manager (TALM) automates policy-based upgrades across multiple Telco Core clusters.
TALM uses ClusterGroupUpgrade custom resources to orchestrate upgrade rollouts in batches with automated pre-caching for disconnected environments.

TALM supports all three upgrade scenarios (z-stream, y-stream, and EUS-to-EUS) and integrates with Red Hat Advanced Cluster Management (RHACM) policies.
You define the desired cluster version in a GitOps policy, and TALM applies that policy to target clusters according to your batching and concurrency settings.

For detailed information about configuring TALM for cluster upgrades, see "GitOps and TALM overview for Telco upgrades".
