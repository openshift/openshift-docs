[id="zero-trust-manager-cert-troubleshooting_{context}"]
= Troubleshooting the cert-manager upstream authority plugin

[role="_abstract"]
To diagnose and resolve problems such as permission errors, authentication failures, and certificate request issues, use this troubleshooting reference. This reference describes common configuration and operational issues with the cert-manager upstream authority plugin.

[id="cert-manager-plugin-permission-denied_{context}"]
== Permission denied when creating CertificateRequest

*Problem*

SPIRE Server logs show permission errors when attempting to create `CertificateRequest` resources:

[source,terminal]
----
ERROR: failed to create CertificateRequest: certificaterequests.cert-manager.io is forbidden: User "system:serviceaccount:sandbox:spire-server" cannot create resource "certificaterequests"
----

*Solution*

Verify that the service account has the required RBAC permissions by running the following command:

. Check if the role exists and has the correct permissions:
+
[source,terminal]
----
$ oc get role spire-server-cert-manager -n sandbox -o yaml
----

. Verify that the role binding exists and references the correct service account:
+
[source,terminal]
----
$ oc get rolebinding spire-server-cert-manager -n sandbox -o yaml
----


[id="cert-manager-plugin-issuer-not-found_{context}"]
== Issuer not found

*Problem*

SPIRE Server logs show that the specified issuer does not exist:

[source,terminal]
----
ERROR: failed to create CertificateRequest: issuer.cert-manager.io "spire-ca" not found
----

*Solution*

Verify that the issuer exists and is in the correct namespace:

. List all issuers in the namespace by running the following command:
+
[source,terminal]
----
$ oc get issuer -n sandbox
----

. If using a `ClusterIssuer`, verify it exists by running the following command:
+
[source,terminal]
----
$ oc get clusterissuer
----
+
Ensure the `issuer_name`, `issuer_kind`, and `namespace` parameters in the SPIRE Server configuration match the actual issuer resource.

[id="cert-manager-plugin-issuer-not-ready_{context}"]
== Issuer not ready

*Problem*

The `CertificateRequest` remains in a pending state, and the issuer shows as not ready:

[source,terminal]
----
$ oc get issuer spire-ca -n sandbox
NAME       READY   AGE
spire-ca   False   10m
----

*Solution*

Check the issuer status and events:

. Describe the issuer to view detailed status and events by running the following command:
+
[source,terminal]
----
$ oc describe issuer spire-ca -n sandbox
----

. Check the cert-manager logs for errors by running the following command:
+
[source,terminal]
----
$ oc logs -n cert-manager deployment/cert-manager
----
+
Resolve any configuration issues with the issuer, such as missing secrets or invalid credentials, based on the error messages.

[id="cert-manager-plugin-certificate-request-pending_{context}"]
== CertificateRequest stuck in pending state

*Problem*

The `CertificateRequest` is created but remains in a pending state and is never signed:

[source,terminal]
----
$ oc get certificaterequest -n sandbox
NAME                           READY   AGE
spire-intermediate-ca-xxxxx    False   5m
----

*Solution*

Check the `CertificateRequest` status and approval state:

. Describe the `CertificateRequest` to view its status by running the following command:
+
[source,terminal]
----
$ oc describe certificaterequest spire-intermediate-ca-xxxxx -n sandbox
----

. Check if the request requires manual approval by running the following command:
+
[source,terminal]
----
$ oc get certificaterequest spire-intermediate-ca-xxxxx -n sandbox -o jsonpath='{.status.conditions}'
----
+
If using a custom approver, verify that the approver is running and configured correctly.

. If manual approval is required, approve the request by running the following command:
+
[source,terminal]
----
$ oc annotate certificaterequest spire-intermediate-ca-xxxxx -n sandbox cert-manager.io/approval="approved"
----

[id="cert-manager-plugin-authentication-failure_{context}"]
== Authentication failure to Kubernetes API

*Problem*

SPIRE Server logs show authentication errors when connecting to the Kubernetes API:

[source,terminal]
----
ERROR: failed to connect to Kubernetes: Unauthorized
----

*Solution*

Verify the kubeconfig or in-cluster authentication:

. If using a kubeconfig file, verify the file exists and has correct permissions by running the following command:
+
[source,terminal]
----
$ ls -l /etc/kubernetes/kubeconfig
----

. Test the kubeconfig authentication by running the following command:
+
[source,terminal]
----
$ oc --kubeconfig=/etc/kubernetes/kubeconfig get pods
----

. If using in-cluster configuration, verify that the service account token is mounted correctly by running the following command:
+
[source,terminal]
----
$ oc describe pod <spire-server-pod> -n sandbox | grep -A 5 "Mounts:"
----

. Regenerate the service account token if it has expired by running the following command:
+
[source,terminal]
----
$ oc create token spire-server -n sandbox --duration=8760h > /etc/kubernetes/kubeconfig
----

[id="cert-manager-plugin-namespace-not-found_{context}"]
== Namespace not found

*Problem*

SPIRE Server logs show that the configured namespace does not exist:

[source,terminal]
----
ERROR: failed to create CertificateRequest: namespace "sandbox" not found
----

*Solution*

Create the namespace or update the configuration:

. Verify the namespace exists by running the following command:
+
[source,terminal]
----
$ oc get namespace sandbox
----

. If the namespace does not exist, create it by running the following command:
+
[source,terminal]
----
$ oc create namespace sandbox
----
+
Verify that the `namespace` parameter in the SPIRE Server configuration matches the actual namespace name.

[id="cert-manager-plugin-connection-timeout_{context}"]
== Connection timeout to Kubernetes API

*Problem*

SPIRE Server logs show connection timeout errors:

[source,terminal]
----
ERROR: failed to connect to Kubernetes: dial tcp 10.0.0.1:6443: i/o timeout
----

*Solution*

Verify network connectivity and API server accessibility:

. Check if the Kubernetes API server is accessible by running the following command:
+
[source,terminal]
----
$ oc cluster-info
----
+
If SPIRE Server runs outside the cluster, verify firewall rules allow connections to the API server port (typically 6443).

. Verify DNS resolution for the API server hostname if using a hostname in the kubeconfig by running the following command:
+
[source,terminal]
----
$ nslookup api.cluster.example.com
----

. Check the kubeconfig file for the correct API server URL by running the following command:
+
[source,terminal]
----
$ grep server /etc/kubernetes/kubeconfig
----

[id="cert-manager-plugin-invalid-configuration_{context}"]
== Invalid plugin configuration

*Problem*

SPIRE Server fails to start with configuration errors:

[source,terminal]
----
ERROR: failed to load plugin cert-manager: invalid configuration: issuer_name is required
----

*Solution*

Verify the plugin configuration syntax and required parameters:

* Check that all required parameters are specified:

** `issuer_name` - The name of the cert-manager issuer

** `namespace` - The namespace for CertificateRequests

. Verify the HCL syntax in the configuration file by running the following command:
+
[source,terminal]
----
$ cat /etc/spire/server/server.conf
----

. Check the SPIRE Server logs for specific configuration errors by running the following command:
+
[source,terminal]
----
$ journalctl -u spire-server -n 50
----
+
Or, if running in O{productg-title} run the following command:
+
[source,terminal]
----
$ oc logs -n sandbox deployment/spire-server --tail=50
----
+
Correct the configuration and restart SPIRE Server.

[id="cert-manager-plugin-certificate-request-denied_{context}"]
== CertificateRequest denied by approver

*Problem*

The `CertificateRequest` is denied by a custom approver:

[source,terminal]
----
$ oc get certificaterequest spire-intermediate-ca-xxxxx -n sandbox
NAME                           READY   DENIED   AGE
spire-intermediate-ca-xxxxx    False   True     2m
----

*Solution*

Check the denial reason and adjust the request or approver policy:

. View the denial reason by running the following command:
+
[source,terminal]
----
$ oc describe certificaterequest spire-intermediate-ca-xxxxx -n sandbox
----
+
* Common denial reasons include:

** Insufficient permissions for the requesting service account

** Request does not meet policy requirements (e.g., key size, validity period)

** UserInfo in the request does not match approver expectations
+
Update the RBAC permissions, approver policy, or service account configuration as needed.

. Delete the denied `CertificateRequest` to allow SPIRE to create a new one by running the following command:
+
[source,terminal]
----
$ oc delete certificaterequest spire-intermediate-ca-xxxxx -n sandbox
----

. Restart SPIRE Server to retry the request by running the following command:
+
[source,terminal]
----
$ oc rollout restart deployment/spire-server -n sandbox
----
