= Configuring ClusterGroupUpgrade custom resources for cluster updates

[role="_abstract"]
You can configure `ClusterGroupUpgrade` custom resources to orchestrate cluster updates with {cgu-operator-full} ({cgu-operator}).
A `ClusterGroupUpgrade` resource defines which clusters to update, which policies to apply, and how to manage batching and concurrency.

.Prerequisites

* {cgu-operator} is installed and configured on the {rh-rhacm-first} hub cluster.
* Update policies are defined and applied to {rh-rhacm}.
* Target clusters are labeled and grouped according to your update strategy.

.Procedure

. Create a `ClusterGroupUpgrade` custom resource for your update:
+
[source,yaml]
----
apiVersion: ran.openshift.io/v1alpha1
kind: ClusterGroupUpgrade
metadata:
  name: <cgu_name>
  namespace: <namespace>
spec:
  clusters:
  - <cluster_name_1>
  - <cluster_name_2>
  managedPolicies:
  - <policy_name>
  enable: false
  remediationStrategy:
    maxConcurrency: <max_concurrency>
    timeout: <timeout_seconds>
----
+
Replace the following values:
+
--
* `<cgu_name>`: Name for your `ClusterGroupUpgrade` resource, for example `core-zstream-upgrade-4.20.1`.
* `<namespace>`: Namespace for the resource, for example `openshift-upgrade-policies`.
* `<cluster_name_1>`, `<cluster_name_2>`: Names of your target clusters.
* `<policy_name>`: Name of your update policy.
* `enable`: Set to `false` to create the resource without starting the update. Set to `true` to start the update immediately.
* `<max_concurrency>`: Number of clusters to update simultaneously. Use `1` for canary testing, `2-5` for production rollouts, or higher for large fleets with proven procedures.
* `<timeout_seconds>`: Maximum time in seconds to wait for each cluster update. See the timeout guidelines that follow.
--
+
Calculate appropriate timeout values based on your environment:
+
--
* Base timeout: 3600 seconds (60 minutes) for z-stream updates, 7200 seconds (120 minutes) for y-stream updates
* Add 1800 seconds (30 minutes) for each additional worker node beyond 3
* Add 3600 seconds (60 minutes) for disconnected environments
* Add 1800 seconds (30 minutes) for clusters with large image registries
--

. Optional: Configure a canary cluster in the `ClusterGroupUpgrade` resource to test the update before rolling it out to the full fleet:
+
[source,yaml]
----
apiVersion: ran.openshift.io/v1alpha1
kind: ClusterGroupUpgrade
metadata:
  name: <cgu_name>
  namespace: <namespace>
spec:
  clusterSelector:
  - <label_selector>
  managedPolicies:
  - <policy_name>
  enable: false
  remediationStrategy:
    canaries:
    - <canary_cluster_name>
    maxConcurrency: 1
    timeout: <timeout_seconds>
----
+
Replace the following values:
+
--
* `<label_selector>`: Label selector for the target clusters, for example `upgrade-wave=1`.
* `<canary_cluster_name>`: Name of the canary cluster to update first. {cgu-operator} updates the canary cluster first, waits for success, and then proceeds with other clusters.
--

. Optional: Add blocking custom resources for pre-update checks:
+
[source,yaml]
----
apiVersion: ran.openshift.io/v1alpha1
kind: ClusterGroupUpgrade
metadata:
  name: <cgu_name>
  namespace: <namespace>
spec:
  clusters:
  - <cluster_name>
  managedPolicies:
  - <policy_name>
  enable: false
  remediationStrategy:
    maxConcurrency: 1
    timeout: <timeout_seconds>
  beforeEnable:
    addClusterLabels:
      pre-upgrade-check: pending
  blockingCRs:
  - name: <blocking_cr_name>
    namespace: <blocking_cr_namespace>
    kind: ConfigMap
    conditions:
    - type: <condition_type>
      status: "True"
----
+
Replace the following values:
+
--
* `<blocking_cr_name>`: Name of the blocking custom resource, for example `etcd-backup`.
* `<condition_type>`: Condition that must be met before the update can proceed, for example `BackupComplete`.
--

. Optional: Configure post-update actions to update cluster labels after the update completes:
+
[source,yaml]
----
apiVersion: ran.openshift.io/v1alpha1
kind: ClusterGroupUpgrade
metadata:
  name: <cgu_name>
  namespace: <namespace>
spec:
  clusters:
  - <cluster_name>
  managedPolicies:
  - <policy_name>
  enable: false
  remediationStrategy:
    maxConcurrency: 1
    timeout: <timeout_seconds>
  afterCompletion:
    addClusterLabels:
      upgraded-to: "<target_version>"
      upgrade-status: complete
    removeClusterLabels:
    - pre-upgrade-check
----
+
Replace `<target_version>` with the target version, for example `4.20`.

. For Extended Update Support (EUS) to EUS updates, create chained `ClusterGroupUpgrade` resources that use `blockingCRs` and `actions` to orchestrate a multi-step update:
+
[source,yaml]
----
---
apiVersion: ran.openshift.io/v1alpha1
kind: ClusterGroupUpgrade
metadata:
  name: upgrade-20to21
  namespace: upgrade
spec:
  actions:
    afterCompletion:
      deleteClusterLabels:
        upgrade-version-to-4-21: ""
      deleteObjects: true
      addClusterLabels:
        unpause-workers-4-22: ""
  backup: false
  clusters:
  - <cluster_name>
  enable: true
  managedPolicies:
    - prep-20to22
    - upgrade-ocp-20to21
    - upgrade-olm-20to21
    - upgrade-validate-21
  preCaching: false
  preCachingConfigRef: {}
  remediationStrategy:
    maxConcurrency: 1
    timeout: 480
---
apiVersion: ran.openshift.io/v1alpha1
kind: ClusterGroupUpgrade
metadata:
  name: upgrade-21to22
  namespace: upgrade
spec:
  actions:
    afterCompletion:
      deleteClusterLabels:
        upgrade-version-4-22: ""
        unpause-workers-4-22: ""
      deleteObjects: true
  backup: false
  clusters:
  - <cluster_name>
  enable: true
  managedPolicies:
    - upgrade-ocp-21to22
    - upgrade-olm-21to22
    - upgrade-validate-22
    - upgrade-worker-1
    - upgrade-worker-2
  preCaching: false
  preCachingConfigRef: {}
  remediationStrategy:
    maxConcurrency: 1
    timeout: 480
  blockingCRs:
  - name: upgrade-20to21
    namespace: upgrade
----
+
Replace `<cluster_name>` with the name of the target cluster.
+
The chained `ClusterGroupUpgrade` configuration uses:
+
--
* The `actions` section to change cluster labels as each step completes. This removes temporary update labels and unbinds completed update policies from the cluster.
* The `blockingCRs` section to require the first `ClusterGroupUpgrade` resource to complete before the second `ClusterGroupUpgrade` resource begins enforcing its policies.
* Separate unpause policies (`upgrade-worker-1`, `upgrade-worker-2`) for each `MachineConfigPool` resource. Your cluster might have more machine config pools, requiring additional policies.
--

. Apply the `ClusterGroupUpgrade` resource by running the following command:
+
[source,terminal]
----
$ oc apply -f <cgu_file>.yaml
----

. Enable the `ClusterGroupUpgrade` resource to start the update by running the following command:
+
[source,terminal]
----
$ oc patch cgu <cgu_name> \
  -n <namespace> \
  --type merge \
  -p '{"spec":{"enable":true}}'
----

. Monitor the `ClusterGroupUpgrade` status by running the following command:
+
[source,terminal]
----
$ oc get cgu <cgu_name> -n <namespace> -o yaml
----
+
Check the `status` section for update progress:
+
[source,yaml]
----
status:
  conditions:
  - message: Upgrade is progressing
    reason: InProgress
    status: "True"
    type: Progressing
  managedPoliciesForUpgrade:
  - name: core-upgrade-4.20.1-policy
    namespace: openshift-upgrade-policies
  remediationPlan:
  - - spoke1
    - spoke2
  status:
    spoke1: complete
    spoke2: inprogress
----

. Watch the `ClusterGroupUpgrade` progress in real time by running the following command:
+
[source,terminal]
----
$ oc get cgu <cgu_name> -n <namespace> -w
----

.Verification

. Verify the `ClusterGroupUpgrade` resource entered the expected state by running the following command:
+
[source,terminal]
----
$ oc get cgu -n <namespace>
----
+
.Example output
[source,terminal]
----
NAME                             AGE   STATE        DETAILS
core-zstream-upgrade-4.20.1     5m    InProgress   Remediating non-compliant policies
----

. Check the detailed status for each cluster by running the following command:
+
[source,terminal]
----
$ oc get cgu <cgu_name> -n <namespace> -o jsonpath='{.status.clusters}'
----

. When the cluster is fully updated and the state of all `ClusterGroupUpgrade` resources shows `Completed`, verify that all policies show `Compliant` by running the following command:
+
[source,terminal]
----
$ oc get policies -n <cluster_name>
----
+
.Example output
[source,terminal]
----
NAME                        REMEDIATION ACTION   COMPLIANCE STATE   AGE
upgrade.upgrade-ocp-21to22  inform               Compliant          1h
upgrade.upgrade-olm-21to22  inform               Compliant          10m
----

. Delete the completed `ClusterGroupUpgrade` resources from the update namespace by running the following command:
+
[source,terminal]
----
$ oc delete cgu -n <namespace> <cgu_name>
----

.Troubleshooting

* If the `ClusterGroupUpgrade` resource is stuck in `Preparing` state, verify that all managed policies exist and are bound to target clusters.
* If the `ClusterGroupUpgrade` resource fails with a timeout error, increase the `timeout` value in the `remediationStrategy` section.
