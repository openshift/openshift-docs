= Best practices for Loki query performance

The following are the best practices for Loki query performance:

* Make sure you are running the latest version of the operator.

* Make sure you have migrated your schema to v13.

* Make sure you are using reliable and fast object storage. Loki is a database capable of doing a tremendous amount of work on object storage.
+
To better understand Loki utilization of object storage you can use the following query:
+
[source]
----
sum by(status, container, operation) (label_replace(rate(loki_s3_request_duration_seconds_count{namespace="openshift-logging"}[5m]), "status", "${1}xx", "status_code", "([0-9]).."))
----
+
The section on “Query performance” also shows a way to understand if `objstore` is the bottleneck.

* Make sure you are using automatic stream sharding. For information see "Enabling automatic stream sharding"

* If you’re already using the new OTLP data model make sure you read the section on “The importance of labels” and configure the streamLabels in the LokiStack if necessary.

* Different types of queries will have different performance characteristics. Loki prefers simply filter queries over regular expressions.
