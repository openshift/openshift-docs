= Configuring revision response start timeout

By setting the revision response start timeout, you can specify the maximum duration in seconds that Serving will wait for a revision to start sending network traffic after a request has been routed to it. The revision response start timeout must not exceed the revision timeout. The default is 300 seconds.

.Prerequisites

* You have installed the {ServerlessOperatorName} and Knative Serving.
* You have cluster administrator permissions on {ocp-product-title}, or cluster or dedicated administrator permissions on {rosa-product-title} or {dedicated-product-title}.

.Procedure

* Choose the appropriate method to configure the revision response start timeout:
** To configure the timeout globally, set the `revision-response-start-timeout-seconds` field in your `KnativeServing` custom resource (CR). If your required response start timeout exceeds the revision timeout, also adjust `revision-timeout-seconds` accordingly:
+
.Example of revision response start timeout globally set to 600 seconds (10 minutes)
[source,yaml]
----
apiVersion: operator.knative.dev/v1beta1
kind: KnativeServing
metadata:
  name: knative-serving
  namespace: knative-serving
spec:
  config:
    defaults:
      revision-timeout-seconds: "900"
      revision-response-start-timeout-seconds: "600"
----
+
** To configure the timeout per revision, set the `responseStartTimeoutSeconds` field in your service definition. If your required response start timeout exceeds the revision timeout, also adjust `timeoutSeconds` accordingly:
+
.Example of a service definition with revision response start timeout set to 600 seconds (10 minutes)
[source,yaml]
----
apiVersion: serving.knative.dev/v1
kind: Service
metadata:
  namespace: my-ns
spec:
  template:
    spec:
      timeoutSeconds: 900
      responseStartTimeoutSeconds: 600
      containers:
# ...
----
