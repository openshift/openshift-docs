= Resource requirements for VM scanning

When using {product-title-short} to scan virtual machines for vulnerabilities (TODO LINK TO scanning-virtual-machines), certain components may need adjusted CPU and memory resource allocations to match the scale of your environment.

The guidelines below were developed by running tests that deployed large numbers of virtual machines, and identifying which resources determine the overall system throughput.

The tests identified the number of VMs and the frequency of their scans as the primary driver of resource utilization, and showed that Scanner V4 is the component whose resources can be modified to achieve higher throughput figures.

These are the specific settings that impact throughput, when it comes to allocating resources at {ocp} level:

- Scanner V4 Matcher replicas: controlled by the Scanner V4 Matcher Horizontal Pod Autoscaler (HPA).
- Scanner V4 Matcher CPU limit
- Scanner V4 DB CPU limit

When calculating infrastructure requirements, use the standard guidance for the rest of the system, and adjust Scanner V4's resources according to the following recommendations.

|===
|Number of VMs | Scanner V4 Matcher HPA max replicas | Scanner V4 Matcher CPU limit | Scanner V4 DB CPU limit

|< 4500  | 3 | 1 core  | 4 cores
|< 14000 | 3 | 1 core  | 8 cores
|< 25000 | 3 | 1 core  | 16 cores
|< 40000 | 3 | 2 cores | 32 cores
|< 50000 | 6 | 2 cores | 32 cores
|===

== Factors affecting throughput

While Scanner V4's resources constrain throughput, other factors are relevant when determining the number of VMs that a specific ACS deployment supports:

- VM scan interval: system throughput is actually measured in VM scans per unit of time, and therefore scan interval (a configuration parameter of roxagent, see TODO LINK TO ROXAGENT INTERVAL) directly impacts the number of VMs ACS can handle. The figures above were calculated for a scan interval of 4h (the default setting). A scan interval of 2h would result in halved numbers of VMs, and an interval of 8h in doubled numbers of VMs.

- Number of packages installed in each VM: Increased numbers of packages result in increased scan times, and therefore reduces throughput.

- Vulnerabilities in installed packages: Greater numbers of overal vulnerabilities found in VMs result in longer scan times, reducing throughput.

- Other workloads: while VMs are being processed, ACS components keep processing the rest of the usual workloads. This includes Scanner V4, and therefore such workloads reduce the VM scanning throughput.

To account for these factors and their variablity, the suggested capacity for each deployment flavor includes a headroom of at least 100%: even if the processing time doubles due to e.g. increased number of packages/vulnerabilities, ACS will be able to handle the specified number of VMs.
