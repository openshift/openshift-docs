= Updating bare-metal core clusters through y-stream releases

[role="_abstract"]
You can update bare-metal core clusters through minor version releases (y-stream) by using GitOps policies and {cgu-operator-full} ({cgu-operator}).
Y-stream updates introduce new features and might include API deprecations, requiring careful planning and validation.

Y-stream updates must be performed through consecutive versions.
You cannot skip versions during y-stream updates.

.Prerequisites

* Pre-update health check completed successfully.
* {cgu-operator} is installed and configured on the {rh-rhacm-first} hub cluster.
* Y-stream release is available (for example, 4.21 when updating from 4.20).
* Y-stream release notes reviewed for API deprecations and breaking changes.
* Maintenance window scheduled (y-stream updates take 1-2 hours).

.Procedure

. Review the y-stream release notes for deprecated APIs:
+
Check the {product-title} documentation for your target version to identify deprecated or removed APIs.

. Identify deprecated APIs in use on your cluster by running the following command:
+
[source,terminal]
----
$ oc get apirequestcounts -o jsonpath='{range .items[?(@.status.removedInRelease!="")]}{.status.removedInRelease}{"\t"}{.metadata.name}{"\n"}{end}' | sort -k1
----
+
This command shows APIs that will be removed in future releases.
Update any resources that use deprecated APIs before updating.

. Validate OLM operator compatibility with the target y-stream version by running the following command:
+
[source,terminal]
----
$ oc get csv -A
----
+
Review operator channels to ensure they support the target {product-title} version.
Update operator subscriptions if needed before the cluster update.

. Create a GitOps policy to pause worker nodes:
+
[source,yaml]
----
apiVersion: policy.open-cluster-management.io/v1
kind: Policy
metadata:
  name: core-pause-worker-nodes
  namespace: openshift-upgrade-policies
spec:
  remediationAction: inform
  disabled: false
  policy-templates:
  - objectDefinition:
      apiVersion: policy.open-cluster-management.io/v1
      kind: ConfigurationPolicy
      metadata:
        name: pause-worker-mcp
      spec:
        remediationAction: inform
        severity: high
        object-templates:
        - complianceType: musthave
          objectDefinition:
            apiVersion: machineconfiguration.openshift.io/v1
            kind: MachineConfigPool
            metadata:
              name: worker
            spec:
              paused: true
----

. Create a GitOps policy for the y-stream update:
+
[source,yaml]
----
apiVersion: policy.open-cluster-management.io/v1
kind: Policy
metadata:
  name: core-upgrade-4.20-policy
  namespace: openshift-upgrade-policies
spec:
  remediationAction: inform
  disabled: false
  policy-templates:
  - objectDefinition:
      apiVersion: policy.open-cluster-management.io/v1
      kind: ConfigurationPolicy
      metadata:
        name: upgrade-cluster-version
      spec:
        remediationAction: inform
        severity: high
        object-templates:
        - complianceType: musthave
          objectDefinition:
            apiVersion: config.openshift.io/v1
            kind: ClusterVersion
            metadata:
              name: version
            spec:
              channel: stable-4.20
              desiredUpdate:
                version: 4.20.0
----
+
Replace `4.20.0` with your target y-stream version.
Both policies are set to `inform` so they do not immediately push changes.
The `ClusterGroupUpgrade` CR enforces these policies at the scheduled update time.

. Apply the worker pause policy to {rh-rhacm} by running the following command:
+
[source,terminal]
----
$ oc apply -f core-pause-worker-policy.yaml
----

. Apply the y-stream update policy by running the following command:
+
[source,terminal]
----
$ oc apply -f core-upgrade-4.20-policy.yaml
----

. Verify the worker pause policy was created by running the following command:
+
[source,terminal]
----
$ oc get policy core-pause-worker-nodes -n openshift-upgrade-policies
----

. Verify the y-stream update policy was created by running the following command:
+
[source,terminal]
----
$ oc get policy core-upgrade-4.20-policy -n openshift-upgrade-policies
----

. Verify the placement binding exists by running the following command:
+
[source,terminal]
----
$ oc get placementbinding -n openshift-upgrade-policies
----

. Verify the placement rule exists by running the following command:
+
[source,terminal]
----
$ oc get placementrule -n openshift-upgrade-policies
----
+
If no `PlacementBinding` or `PlacementRule` exists, the policies will not apply to any clusters.
Create appropriate placement resources for your target clusters.

. Create a `ClusterGroupUpgrade` CR for the y-stream update:
+
[source,yaml]
----
apiVersion: ran.openshift.io/v1alpha1
kind: ClusterGroupUpgrade
metadata:
  name: core-ystream-upgrade-4.20
  namespace: openshift-upgrade-policies
spec:
  clusters:
  - spoke1
  managedPolicies:
  - core-pause-worker-nodes
  - core-upgrade-4.20-policy
  enable: false
  remediationStrategy:
    maxConcurrency: 1
    timeout: 7200
  actions:
    beforeEnable:
      addClusterLabels:
        upgrade-in-progress: "true"
    afterCompletion:
      addClusterLabels:
        upgraded-to: "4.20"
        control-plane-upgraded: "true"
----
+
For canary deployments, use a single cluster first before updating additional clusters.

. Apply the `ClusterGroupUpgrade` CR by running the following command:
+
[source,terminal]
----
$ oc apply -f core-ystream-cgu.yaml
----

. Enable the `ClusterGroupUpgrade` resource to start the control plane update by running the following command:
+
[source,terminal]
----
$ oc patch cgu core-ystream-upgrade-4.20 \
  -n openshift-upgrade-policies \
  --type merge \
  -p '{"spec":{"enable":true}}'
----

. Monitor the control plane update progress by running the following command:
+
[source,terminal]
----
$ oc get cgu core-ystream-upgrade-4.20 -n openshift-upgrade-policies -w
----

. On the target cluster, monitor cluster version by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get clusterversion -w
----
+
Cluster Operators update sequentially during the y-stream update.
The control plane update typically takes 60-120 minutes.

. Monitor cluster Operator status by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get co
----

. After the control plane update completes, verify control plane node health by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get nodes -l node-role.kubernetes.io/master=
----

. Verify control plane Operator health by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get co | grep -v "True.*False.*False"
----
+
All control plane nodes and Operators must be healthy before proceeding.

. Create an OLM Operator update policy that updates Operator subscription channels to versions compatible with the new y-stream:
+
[source,yaml]
----
apiVersion: policy.open-cluster-management.io/v1
kind: Policy
metadata:
  name: core-upgrade-olm-4.20
  namespace: openshift-upgrade-policies
spec:
  remediationAction: inform
  disabled: false
  policy-templates:
  - objectDefinition:
      apiVersion: policy.open-cluster-management.io/v1
      kind: ConfigurationPolicy
      metadata:
        name: upgrade-olm-operators
      spec:
        remediationAction: inform
        severity: high
        object-templates:
        - complianceType: musthave
          objectDefinition:
            apiVersion: operators.coreos.com/v1alpha1
            kind: Subscription
            metadata:
              name: sriov-network-operator-subscription
              namespace: openshift-sriov-network-operator
            spec:
              channel: stable-4.20
            status:
              state: AtLatestKnown
----
+
You can include multiple Operator subscriptions in the same policy.
For more details about creating OLM update policies, see "Coordinating OLM operator updates with cluster updates".

. Create a `ClusterGroupUpgrade` custom resource to apply the OLM Operator update policy:
+
[source,yaml]
----
apiVersion: ran.openshift.io/v1alpha1
kind: ClusterGroupUpgrade
metadata:
  name: core-olm-upgrade-4.20
  namespace: openshift-upgrade-policies
spec:
  clusters:
  - spoke1
  managedPolicies:
  - core-upgrade-olm-4.20
  enable: true
  remediationStrategy:
    maxConcurrency: 1
    timeout: 3600
----

. Apply the OLM Operator update policy by running the following command:
+
[source,terminal]
----
$ oc apply -f core-olm-upgrade-policy.yaml
----

. Apply the OLM Operator `ClusterGroupUpgrade` custom resource by running the following command:
+
[source,terminal]
----
$ oc apply -f core-olm-cgu.yaml
----

. Monitor the OLM Operator update progress by running the following command:
+
[source,terminal]
----
$ oc get cgu core-olm-upgrade-4.20 -n openshift-upgrade-policies -w
----

. Verify that worker nodes remain at the previous version by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get nodes
----
+
Worker nodes should still show the previous kubelet version.

. Create a policy to unpause worker nodes:
+
[source,yaml]
----
apiVersion: policy.open-cluster-management.io/v1
kind: Policy
metadata:
  name: core-unpause-worker-nodes
  namespace: openshift-upgrade-policies
spec:
  remediationAction: inform
  disabled: false
  policy-templates:
  - objectDefinition:
      apiVersion: policy.open-cluster-management.io/v1
      kind: ConfigurationPolicy
      metadata:
        name: unpause-worker-mcp
      spec:
        remediationAction: inform
        severity: high
        object-templates:
        - complianceType: musthave
          objectDefinition:
            apiVersion: machineconfiguration.openshift.io/v1
            kind: MachineConfigPool
            metadata:
              name: worker
            spec:
              paused: false
----

. Create a `ClusterGroupUpgrade` CR to unpause workers:
+
[source,yaml]
----
apiVersion: ran.openshift.io/v1alpha1
kind: ClusterGroupUpgrade
metadata:
  name: core-unpause-workers-4.20
  namespace: openshift-upgrade-policies
spec:
  clusters:
  - spoke1
  managedPolicies:
  - core-unpause-worker-nodes
  enable: false
  remediationStrategy:
    maxConcurrency: 1
----

. Apply the unpause `ClusterGroupUpgrade` CR by running the following command:
+
[source,terminal]
----
$ oc apply -f core-unpause-workers-cgu.yaml
----
+
Enable the unpause `ClusterGroupUpgrade` resource by running the following command:
+
[source,terminal]
----
$ oc patch cgu core-unpause-workers-4.20 \
  -n openshift-upgrade-policies \
  --type merge \
  -p '{"spec":{"enable":true}}'
----

. Monitor worker node updates by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get mcp worker -w
----
+
Worker nodes update in a rolling fashion.
The process typically takes 30-60 minutes depending on the number of worker nodes.

.Verification

. Verify all nodes are at the target y-stream version by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get nodes
----

. Verify all cluster operators are healthy by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get co
----

. Verify workload health by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get pods -A | grep -v Running | grep -v Completed
----

. Validate custom resources by running the following command:
+
Verify Performance Profile status by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get performanceprofile
----
+
Verify PTP configuration by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get ptpconfig -n openshift-ptp
----
+
Verify SR-IOV network node policies by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get sriovnetworknodepolicy -n openshift-sriov-network-operator
----

. Run smoke tests for critical workloads.

.Troubleshooting

* If API deprecation warnings appear, update manifests to use new API versions before updating.
* If operator compatibility issues occur, update operator subscriptions to channels that support the target y-stream.
* If worker node updates get stuck, check `MachineConfigPool` resource status and node logs.
