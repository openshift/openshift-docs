= Understanding service discovery

[role="_abstract"]
Service discovery in {ServerlessLogicProductName} enables workflows to automatically resolve Kubernetes resource network endpoints by using custom URI patterns. This feature simplifies service integration by eliminating the need to hard code service URLs in workflow configurations.

Service discovery operates during Operator-managed deployments or standalone Quarkus application startup. The system scans configuration files to locate service discovery URI patterns and automatically resolves them to actual network endpoints.

== Service discovery workflow

When you deploy a workflow that uses service discovery, the following process occurs:

. The Operator scans the workflow ConfigMap for URI patterns enclosed in `${}`.
. The Operator parses the discovered URIs to identify the target Kubernetes resources.
. The Operator queries the Kubernetes API for the specified resources.
. The Operator resolves the URIs to actual network endpoints based on the resource information.
. The Operator populates a managed properties ConfigMap with the resolved values.

The Operator maintains the managed properties ConfigMap throughout the deployment lifecycle. Your workflow accesses the resolved endpoints through this ConfigMap without requiring manual updates when services change locations or endpoints.

== Benefits of service discovery

Service discovery provides the following advantages:

* *Automatic endpoint resolution*: Eliminates manual configuration of service URLs and reduces configuration errors.
* *Dynamic updates*: Adapts to changes in service locations without requiring workflow modifications.
* *Kubernetes-native integration*: Works seamlessly with standard Kubernetes resources and custom resources such as Knative services.
* *Environment portability*: Workflows can move across namespaces and clusters without hard coded endpoint dependencies.
* *Simplified configuration*: Uses declarative URI patterns instead of managing multiple endpoint configurations.
