= Troubleshooting Cluster Upgrader

[role="_abstract"]
This section describes potential problems and their solutions when using legacy installation method for the secured cluster and enabling the automated updates.
The errors described here can be found in the clusters view when upgrader fails.

== Upgrader is missing permissions

.Symptom

The following error is displayed: "Upgrader failed to execute PreflightStage of the roll-forward workflow: executing stage "Run preflight checks": preflight check "Kubernetes authorization" reported errors. This usually means that access is denied. Have you configured this Secured Cluster for automatically receiving updates?"

.Procedure

* Ensure that the bundle for Secured Cluster was generated with enabled option for future upgrades: "Cluster is configured to allow future automatic upgrades" before clicking on "Download YAML file and keys".
* If possible, remove that secured cluster and generate a new bundle making sure that future upgrades are enabled.
* If recreation of the cluster is not desired or possible, one can investigate the secured cluster checking the following:
** Ensure that Service Account "sensor-upgrader" exists in the same namespace as Sensor.
** Ensure that a ClusterRoleBinding exists (default name: "<namespace>:upgrade-sensors") that grants the "cluster-admin" ClusterRole to the "sensor-upgrader" Service Account.

== Upgrader cannot start

.Symptom

The following error is displayed: "Upgrade initialization error: Pod terminated: (Error)"

.Procedure

* Ensure that the upgrader has enough permissions for accessing the cluster objects - see "Upgrader is missing permissions"
* Check the upgrader logs for more insights.
** Usually, the upgrader deployment is acting in the cluster for a short time while doing the upgrades and later it is cleaned up.
** When "Upgrade initialization error" happens, the cleanup is not executed and the logs of the failed deployment may be accessed in the same namespace as sensor.
