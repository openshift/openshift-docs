= Creating a {product-title} cluster using a custom AWS KMS key

[role="_abstract"]
You can create a {product-title} cluster with a customer-provided KMS key that is used to encrypt either node root volumes, the etcd database, or both. A different KMS key ARN can be provided for each option.

[NOTE]
====
{product-title} does not automatically configure the `default` storage class to encrypt persistent volumes with the customer-provided KMS key. This is something that can be configured in-cluster after installation.
====


.Procedure

. Create a custom AWS customer-managed KMS key by running the following command:
+
[source,terminal]
----
$ KMS_ARN=$(aws kms create-key --region $AWS_REGION --description 'Custom ROSA Encryption Key' --tags TagKey=red-hat,TagValue=true --query KeyMetadata.Arn --output text)
----
+
This command saves the Amazon Resource Name (ARN) output of this custom key for further steps.
+
[NOTE]
====
Customers must provide the `--tags TagKey=red-hat,TagValue=true` argument that is required for a customer KMS key.
====


. Verify the KMS key has been created by running the following command:
+
[source,terminal]
----
$ echo $KMS_ARN
----

. Set your AWS account ID to an environment variable.
+
[source,terminal]
----
$ AWS_ACCOUNT_ID=<aws_account_id>
----

. Set your Account Role Prefix to an environment variable.
+
[source,terminal]
----
$ ACCOUNT_ROLE_PREFIX=<account_role_prefix>
----

. Set your Operator Role Prefix to an environment variable.
+
[source,terminal]
----
$ OPERATOR_ROLE_PREFIX=<operator_role_prefix>
----
. Add the ARN for the account-wide installer role and operator roles that you created in the preceding step to the `Statement.Principal.AWS` section in the file. In the following example, the ARN for the default `ManagedOpenShift-HCP-ROSA-Installer-Role` role is added:

+
[NOTE]
====
To encrypt persistent volumes with a custom KMS key, customers must include the EBS CSI driver's IAM role in the KMS key policy
====

. Confirm the details of the policy file created by running the following command:
+
[source,terminal]
----
$ cat rosa-key-policy.json
----

. Apply the newly generated key policy to the custom KMS key by running the following command:
+
[source,terminal]
----
$ aws kms put-key-policy --key-id $KMS_ARN \
--policy file://rosa-key-policy.json \
--policy-name default
----

. Create the cluster by running the following command:
+
--
include::snippets/rosa-long-cluster-name.adoc[]
--
+
[source,terminal]
----
$ rosa create cluster --cluster-name <cluster_name> \ 
--subnet-ids <private_subnet_id>,<public_subnet_id> \
--sts \
--mode auto \
--machine-cidr 10.0.0.0/16 \
--compute-machine-type m5.xlarge \
--hosted-cp \
--region <aws_region> \
--oidc-config-id $OIDC_ID \
--kms-key-arn $KMS_ARN \
--etcd-encryption-kms-arn $KMS_ARN \
--operator-roles-prefix $OPERATOR_ROLES_PREFIX
----
+
--
where:

`--kms-key-arn`:: This KMS key ARN is used to encrypt all worker node root volumes. It is not required if only etcd database encryption is needed.
`--etcd-encryption-kms-arn`:: This KMS key ARN is used to encrypt the etcd database. The etcd database is always encrypted by default with an AES cipher block, but can be encrypted instead with a KMS key. It is not required if only node root volume encryption is needed.
--

.Verification

. Create a cluster administrator user by running the following command:
+
[source,terminal]
----
$ rosa create admin --cluster <cluster_name>
----
+
Note the credentials and API URL in the command output.

. Log in to the cluster using the `oc` CLI by running the following command:
+
[source,terminal]
----
$ oc login <api_url> --username cluster-admin --password <password>
----
+
Replace `<api_url>` and `<password>` with the values returned by the previous command.

. Verify that you are logged in by running the following command:
+
[source,terminal]
----
$ oc whoami
----
+
.Example output
[source,terminal]
----
cluster-admin
----

* Verify node root volume encryption
+
You can verify that worker node root volumes are encrypted with your custom KMS key.

.. Get the worker node instance IDs by running the following command:
+
[source,terminal]
----
$ INSTANCE_IDS=$(oc get nodes -l node-role.kubernetes.io/worker \
-o jsonpath='{.items[*].spec.providerID}' | tr ' ' '\n' | cut -d'/' -f5)
----

.. Check volume encryption for each instance by running the following command:
+
[source,terminal]
----
$ for INSTANCE_ID in ${INSTANCE_IDS}; do
  echo "Instance: ${INSTANCE_ID}"
  aws ec2 describe-volumes \
    --filters "Name=attachment.instance-id,Values=${INSTANCE_ID}" \
    --query "Volumes[*].{VolumeId:VolumeId,Encrypted:Encrypted,KmsKeyId:KmsKeyId}" \
    --region $AWS_REGION \
    --output table
done
----
+
The output should show `Encrypted: True` and your KMS key ARN for each volume.

* Verify PersistentVolume encryption
+
If you have configured a custom `StorageClass` that uses the KMS key, you can verify that new PersistentVolumes are encrypted.

.. Create a test PersistentVolumeClaim by running the following command:
+
[source,terminal]
----
$ cat <<EOF | oc apply -f -
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: test-kms-pvc
  namespace: default
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 1Gi
  storageClassName: <kms_storage_class_name>
EOF
----

.. Create a pod to bind the PVC by running the following command:
+
[source,terminal]
----
$ cat <<EOF | oc apply -f -
apiVersion: v1
kind: Pod
metadata:
  name: test-kms-pod
  namespace: default
spec:
  containers:
  - name: test
    image: registry.access.redhat.com/ubi9/ubi-minimal:latest
    command: ["sleep", "3600"]
    volumeMounts:
    - name: data
      mountPath: /data
  volumes:
  - name: data
    persistentVolumeClaim:
      claimName: test-kms-pvc
EOF
----

.. Wait for the PVC to bind by running the following command:
+
[source,terminal]
----
$ oc get pvc test-kms-pvc -w
----

.. Verify the volume is encrypted with your KMS key by running the following commands:
+
[source,terminal]
----
$ PV_NAME=$(oc get pvc test-kms-pvc -o jsonpath='{.spec.volumeName}')
$ VOLUME_ID=$(oc get pv ${PV_NAME} -o jsonpath='{.spec.csi.volumeHandle}')
$ aws ec2 describe-volumes \
  --volume-ids ${VOLUME_ID} \
  --query "Volumes[0].{Encrypted:Encrypted,KmsKeyId:KmsKeyId}" \
  --region $AWS_REGION
----
+
The output should show `Encrypted: true` and your KMS key ARN.

.. Clean up the test resources by running the following commands:
+
[source,terminal]
----
$ oc delete pod test-kms-pod
$ oc delete pvc test-kms-pvc
----
