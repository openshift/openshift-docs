= Creating IAM resources for {autonode}

[role="_abstract"]
Create the foundational AWS Identity and Access Management (IAM) resources for your {product-title} cluster with {autonode} enabled.

Before you can enable {autonode} on a {product-title} cluster, you must create the Amazon Web Services (AWS) IAM resources that the cluster requires. These include account roles, an OpenID Connect (OIDC) configuration, and operator roles.

.Prerequisites

* You have installed and configured the latest {rosa-cli-first} version 1.2.57 or later on your installation host.
* You have installed and configured the latest AWS command-line interface (CLI) on your installation host.
* You have verified that your AWS account has the required permissions to create IAM resources.
* You have been enrolled in the {autonode} private preview program.

.Procedure

. Set a unique prefix that identifies resources for your cluster:
+
[source,terminal]
----
$ export PREFIX=<your_cluster_prefix>
----
+
Replace `<your_cluster_prefix>` with a unique identifier, such as `autonode-test` or `my-hcp-cluster`.

. Create the account roles for your {product-title} cluster:
+
[source,terminal]
----
$ rosa create account-roles --prefix $PREFIX --mode auto -y
----
+
This command creates the AWS account-wide roles required for {product-title} cluster operations.

. Create an OIDC provider configuration:
+
[source,terminal]
----
$ rosa create oidc-config --managed -y --mode auto
----
+
This command creates an OIDC identity provider that allows Kubernetes service accounts to assume IAM roles by using web identity federation.

. Capture the OIDC configuration ID from the output:
+
[source,terminal]
----
$ export OIDC_CONFIG_ID=<oidc_config_id>
----
+
Replace `<oidc_config_id>` with the ID value from the `rosa create oidc-config` command output.

. Create operator roles for your {product-title} cluster:
+
[source,terminal]
----
$ rosa create operator-roles --prefix $PREFIX --oidc-config-id $OIDC_CONFIG_ID --hosted-cp -y
----
+
This command creates the IAM roles required by OpenShift cluster operators running on the hosted control plane.

. Add additional Amazon Elastic Compute Cloud (Amazon EC2) tagging permissions to the control plane operator role:
+
.. Create a file named `control-plane-operator-addon-policy.json` with the following content:
+
[source,json]
----
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "AllowEC2Tagging",
            "Effect": "Allow",
            "Action": [
                "ec2:CreateTags"
            ],
            "Resource": "*",
            "Condition": {
                "StringEquals": {
                    "aws:ResourceTag/red-hat-managed": "true"
                }
            }
        }
    ]
}
----
+
This policy allows HyperShift to automatically add the {product-title} cluster ID as a tag on security groups, which Karpenter uses for resource discovery.

.. Create the IAM policy:
+
[source,terminal]
----
$ ADDON_POLICY_ARN=$(aws iam create-policy \
  --policy-name ${PREFIX}-control-plane-operator-addon \
  --policy-document file://control-plane-operator-addon-policy.json \
  --query 'Policy.Arn' \
  --output text)
----

.. Attach the policy to the control plane operator role:
+
[source,terminal]
----
$ aws iam attach-role-policy \
  --role-name ${PREFIX}-system-control-plane-operator \
  --policy-arn $ADDON_POLICY_ARN
----

. Verify the IAM resources were created successfully:
+
[source,terminal]
----
$ rosa list account-roles --prefix $PREFIX
$ rosa list operator-roles --prefix $PREFIX
$ aws iam list-attached-role-policies \
  --role-name ${PREFIX}-system-control-plane-operator
----

.Verification

* Verify that the account roles, OIDC configuration, and operator roles appear in the command output.
* Verify that the control plane operator role has both its default policy and the add-on policy attached.

.Next steps
