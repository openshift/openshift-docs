= About the SPIRE Vault UpstreamAuthority plugin

[role="_abstract"]
To use Vault's security features, centralize certificate management, and keep consistent PKI policies across your infrastructure while providing SPIFFE identities to workloads, you can use the Vault UpstreamAuthority plugin. This plugin connects SPIRE Server to HashiCorp Vault's PKI Engine for automated intermediate CA certificate signing.

The SPIRE Vault UpstreamAuthority plugin provides the following features:

* Integration with HashiCorp Vault PKI Engine as an upstream certificate authority

* Support for many Vault authentication methods

* Automatic signing of SPIRE intermediate CA certificates

* Vault Enterprise namespace support

* Secure certificate management using Vault's security features

[IMPORTANT]
====
The Vault UpstreamAuthority plugin does not support the `PublishJWTKey` remote procedure call (RPC) and is not appropriate for use in nested SPIRE topologies where JSON Web Token Secure Verifiable Identity Document (JWT-SVIDs) are used.
====

[id="spire-vault-authentication-methods_{context}"]
== Supported authentication methods

The Vault UpstreamAuthority plugin supports the following authentication methods for connecting to Vault:

Client Certificate authentication:: Uses mutual TLS (mTLS) with client certificates to authenticate to Vault. This method is suitable for environments where certificate-based authentication is preferred.

Token authentication:: Uses a Vault token for authentication. This is the simplest authentication method and is suitable for development environments or scenarios where token management is already established.

AppRole authentication:: Uses Vault's AppRole authentication method with a role ID and secret ID. This method is recommended for automated workflows and service-to-service authentication.

Kubernetes authentication:: Uses Kubernetes service account tokens to authenticate to Vault. This method is ideal for SPIRE Server deployments running in Kubernetes clusters.

[id="spire-vault-requirements_{context}"]
== Prerequisites and requirements

Before configuring the Vault UpstreamAuthority plugin, ensure the following requirements are met:

Vault PKI Engine:: A running HashiCorp Vault instance with the PKI secrets engine enabled at a configured mount point (default: `pki`). The PKI engine must have a root CA certificate configured for signing intermediate certificates.

Vault Policies:: A Vault policy that grants the `update` capability for the `pki/root/sign-intermediate` endpoint. This policy must be attached to the authentication credentials used by SPIRE Server. The policy allows SPIRE to submit certificate signing requests (CSRs) to Vault for signing.
+
.Example Vault policy
[source,hcl]
----
path "pki/root/sign-intermediate" {
  capabilities = ["update"]
}
----

Authentication credentials:: Valid credentials for one of the supported authentication methods: token authentication, AppRole authentication, Kubernetes authentication, or client certificate authentication. The credentials must be associated with the Vault policy that grants signing permissions.

TTL configuration:: SPIRE Server's `ca_ttl` configuration must not exceed the Vault PKI engine's maximum lease TTL. Mismatched TTL values will cause certificate signing failures.

Network connectivity:: SPIRE Server must be able to reach the Vault server over the network. For TLS-enabled Vault servers (recommended), SPIRE Server needs access to the CA certificate that signed Vault's TLS certificate.

Kubernetes RBAC (when using Kubernetes authentication):: If SPIRE Server runs in Kubernetes and uses Kubernetes authentication, a ServiceAccount must be created for SPIRE Server and bound to the Vault Kubernetes auth role. No additional Kubernetes RBAC permissions are required beyond the ServiceAccount creation.
