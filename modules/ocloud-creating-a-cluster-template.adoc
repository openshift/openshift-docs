= Create a ClusterTemplate for cluster provisioning

[role="_abstract"]
You can define a `ClusterTemplate` custom resource that serves as a reusable blueprint for provisioning clusters. The template specifies the cluster topology, hardware requirements, default configuration, and the parameter schema that site engineers must supply in a `ProvisioningRequest`.

A `ClusterTemplate` references two ConfigMaps that contain defaults for cluster installation and Day-2 policy configuration. When a site engineer creates a `ProvisioningRequest`, the O-Cloud Manager merges the site-specific parameters from the request with these defaults to produce the final cluster and policy definitions.

.Prerequisites

* The O-Cloud Manager Operator is installed on the hub cluster.
* The GitOps repository is configured and synchronized with ArgoCD.

.Procedure

. Create a namespace for the `ClusterTemplate` and its associated resources by running the following command:
+
[source,terminal]
----
$ oc create namespace <template_namespace>
----
+
where:
+
`<template_namespace>`:: Specifies the target namespace for the `ClusterTemplate` resource and associated `ConfigMap` resources.

. Create a `ConfigMap` containing default values for cluster installation parameters:
+
[source,yaml]
----
apiVersion: v1
kind: ConfigMap
metadata:
  name: clusterinstance-defaults-v1
  namespace: <template_namespace>
data:
  clusterinstance-defaults: |
    clusterImageSetNameRef: "4.Y.Z"
    pullSecretRef:
      name: pull-secret
    networkType: OVNKubernetes
    templateRefs:
      - name: ai-cluster-templates-v1
        namespace: open-cluster-management
    nodes:
      - role: master
        bootMode: UEFI
        ironicInspect: "disabled"
        automatedCleaningMode: disabled
        nodeNetwork:
          interfaces:
            - name: ens3f0
              label: boot-interface
        templateRefs:
          - name: ai-node-templates-v1
            namespace: open-cluster-management
----
+
* `clusterImageSetNameRef` specifies the {product-title} release version to install.
* `templateRefs` references the SiteConfig Operator templates used for cluster installation.
* The `boot-interface` label on a network interface is required so the O-Cloud Manager can resolve the boot NIC MAC address from hardware inventory data.

. Create a `ConfigMap` containing default values for Day-2 configuration policy parameters:
+
[source,yaml]
----
apiVersion: v1
kind: ConfigMap
metadata:
  name: policytemplates-defaults-v1
  namespace: <template_namespace>
data:
  policytemplate-defaults: |
    cpu-isolated: "2-63"
    cpu-reserved: "0-1"
    hugepages-default: "1G"
    hugepages-size: "1G"
    hugepages-count: "32"
----
+
The ConfigMap must contain a `policytemplate-defaults` key. The value is a YAML string defining the default policy parameters that are merged with site-specific values from the `ProvisioningRequest`.

. Create the `ClusterTemplate` custom resource:
+
[source,yaml]
----
apiVersion: clcm.openshift.io/v1alpha1
kind: ClusterTemplate
metadata:
  name: sno-ran-du.v4-Y-Z-1
  namespace: <template_namespace>
spec:
  name: sno-ran-du
  version: v4-Y-Z-1
  release: 4.Y.Z
  templateDefaults:
    hwMgmtDefaults:
      hardwareProvisioningTimeout: "90m"
      nodeGroupData:
        - name: controller
          role: master
          resourceSelector:
            resourceselector.clcm.openshift.io/server-type: XR8620t
    clusterInstanceDefaults: clusterinstance-defaults-v1
    policyTemplateDefaults: policytemplates-defaults-v1
  templateParameterSchema:
    properties:
      nodeClusterName:
        type: string
      oCloudSiteId:
        type: string
      policyTemplateParameters:
        properties:
          cpu-isolated:
            type: string
          cpu-reserved:
            type: string
        type: object
      clusterInstanceParameters:
        properties:
          clusterName:
            type: string
          baseDomain:
            type: string
          nodes:
            items:
              properties:
                hostName:
                  type: string
              required:
                - hostName
              type: object
            type: array
        required:
          - clusterName
          - nodes
        type: object
    required:
      - nodeClusterName
      - oCloudSiteId
      - policyTemplateParameters
      - clusterInstanceParameters
    type: object
----
+
* `metadata.name` must follow the format `<name>.<version>` and must be unique across all namespaces.
* `spec.release` must match the {product-title} release version in the `clusterinstance-defaults` ConfigMap.
* `spec.templateDefaults.clusterInstanceDefaults` references the `ConfigMap` that contains default cluster installation parameters. These defaults are merged with the site-specific values from the `ProvisioningRequest` to produce the final `ClusterInstance` definition.
* `spec.templateDefaults.policyTemplateDefaults` references the `ConfigMap` that contains default Day-2 configuration policy parameters. These defaults are merged with the site-specific policy values from the `ProvisioningRequest`.
* `spec.templateDefaults.hwMgmtDefaults.nodeGroupData` defines the node groups and resource selectors used to allocate bare-metal hosts during provisioning.
* `spec.templateParameterSchema` defines the OpenAPI v3 schema that validates parameters supplied in a `ProvisioningRequest`. The schema must include `nodeClusterName`, `oCloudSiteId`, `policyTemplateParameters`, and `clusterInstanceParameters`.

. Apply the resources to the hub cluster, or commit them to the GitOps repository for ArgoCD synchronization.

.Verification

. Verify that the `ClusterTemplate` is validated by running the following command:
+
[source,terminal]
----
$ oc get clustertemplates.clcm.openshift.io -A
----
+
+
.Example output
[source,terminal]
----
NAMESPACE          NAME                   AGE   STATE       DETAILS
ocloud-templates   sno-ran-du.v4-Y-Z-1   18s   Completed   The cluster template validation succeeded
----
+
A `STATE` of `Completed` with a `ClusterTemplateValidated` condition of `True` indicates that the template passed validation and is ready for use. After successful validation, the `ClusterTemplate` and its referenced ConfigMaps become immutable.
