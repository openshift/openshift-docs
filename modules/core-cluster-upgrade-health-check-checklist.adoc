= Health check command reference for bare metal Core upgrades

[role="_abstract"]
You can use this reference checklist to quickly run pre-upgrade health checks with copy-paste ready commands.
This checklist provides the essential commands, expected outputs, and pass or fail criteria for each health check.

[id="core-cluster-health-checks_{context}"]
== Core cluster health checks

.Control plane and worker nodes
[cols="2,3,2",options="header"]
|===
|Component
|Command
|Expected result

|Control plane nodes
|`oc get nodes -l node-role.kubernetes.io/master=`
|All nodes `Ready`, none `SchedulingDisabled`

|Worker nodes
|`oc get nodes -l node-role.kubernetes.io/worker=`
|All nodes `Ready`, none `SchedulingDisabled`

|Node resource utilization
|`oc adm top nodes`
|CPU < 80%, Memory < 85%
|===

.Cluster operators
[source,terminal]
----
$ oc get co
----

*Pass criteria:* All operators show `AVAILABLE=True`, `PROGRESSING=False`, `DEGRADED=False`

*Quick check for degraded operators:*
[source,terminal]
----
$ oc get co | grep -v "True.*False.*False"
----

If this command returns any operators, investigate before upgrading.

.etcd cluster health
[source,terminal]
----
$ oc get etcd -o=jsonpath='{range .items[0].status.conditions[?(@.type=="EtcdMembersAvailable")]}{.message}{"\n"}{end}'
----

*Expected output:* `3 members are available` (or the number of control plane nodes)

*Verify etcd pod status:*
[source,terminal]
----
$ oc get pods -n openshift-etcd -l app=etcd
----

*Pass criteria:* All etcd pods are `Running` with `3/3` or `4/4` containers ready.

[id="machine-configuration-and-storage_{context}"]
== Machine configuration and storage

.Machine config pools
[source,terminal]
----
$ oc get mcp
----

*Pass criteria:*

* `UPDATED=True`
* `UPDATING=False`
* `DEGRADED=False`
* `MACHINECOUNT` = `READYMACHINECOUNT` = `UPDATEDMACHINECOUNT`

.Persistent volumes
[source,terminal]
----
$ oc get pv
----

*Pass criteria:* All PVs show `Bound` or `Available` status, none show `Failed`.

.Storage classes
[source,terminal]
----
$ oc get storageclass
----

*Pass criteria:* At least one storage class exists and is marked as default (indicated by `(default)` annotation).

[id="operator-and-application-health_{context}"]
== Operator and application health

.OLM operators
[source,terminal]
----
$ oc get csv -A
----

*Pass criteria:* All ClusterServiceVersions show `Succeeded` phase.

*Quick check for failed operators:*
[source,terminal]
----
$ oc get csv -A | grep -v Succeeded
----

.Pods not in Running or Completed state
[source,terminal]
----
$ oc get pod -A | grep -E -i -v 'complete|running'
----

*Pass criteria:* The command produces no output. If any pods are listed, review their status before upgrading.

.Pod Disruption Budgets
[source,terminal]
----
$ oc get pdb -A
----

*Pass criteria:*

* PDBs exist for all critical workloads
* `ALLOWED DISRUPTIONS` is greater than 0
* `MIN AVAILABLE` is not set to 100% of replicas

[id="deployment-specific-health-checks_{context}"]
== Deployment-specific health checks

.Performance profiles
[source,terminal]
----
$ oc get performanceprofile
----

*Verify applied status:*
[source,terminal]
----
$ oc get performanceprofile -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.status.conditions[?(@.type=="Applied")].status}{"\n"}{end}'
----

*Pass criteria:* All Performance Profiles show `Applied` status as `True`.

.SR-IOV network node policies
[source,terminal]
----
$ oc get sriovnetworknodepolicy -n openshift-sriov-network-operator
----

*Pass criteria:* All policies show `Succeeded` status.

*Detailed status check:*
[source,terminal]
----
$ oc get sriovnetworknodepolicy -n openshift-sriov-network-operator -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.status.syncStatus}{"\n"}{end}'
----

[id="api-and-authentication_{context}"]
== API and authentication

.API server responsiveness
[source,terminal]
----
$ oc version
----

*Pass criteria:* Command completes within 5 seconds.

.Authentication
[source,terminal]
----
$ oc whoami
----

*Pass criteria:* Command returns your username successfully.

.Certificate expiration
[source,terminal]
----
$ oc get certificates -A
----

*Pass criteria:* No certificates expire within 30 days.

[id="backup-verification_{context}"]
== Backup verification

*Recent etcd backup check*

Verify that an etcd backup exists from within the past 24 hours.

*For automated backups:*

Check your backup storage location (file system, S3 bucket, or other) for recent backup files with timestamps from the past 24 hours.

*To create a manual backup:*
[source,terminal]
----
$ oc debug node/<control-plane-node> -- chroot /host /usr/local/bin/cluster-backup.sh /home/core/assets/backup
----

Document the backup location and timestamp for potential disaster recovery scenarios.

[id="common-failure-modes-and-quick-fixes_{context}"]
== Common failure modes and quick fixes

*Degraded cluster operator*

*Symptom:* Operator shows `DEGRADED=True` in `oc get co` output.

*Quick fix:*

. Check operator logs:
+
[source,terminal]
----
$ oc logs -n openshift-<operator-namespace> deployment/<operator-deployment>
----

. Check operator events:
+
[source,terminal]
----
$ oc get events -n openshift-<operator-namespace> --sort-by='.lastTimestamp'
----

. Restart operator pod if logs indicate transient failure:
+
[source,terminal]
----
$ oc delete pod -n openshift-<operator-namespace> <operator-pod>
----

*etcd member unhealthy*

*Symptom:* `oc get etcd` shows fewer members available than expected.

*Quick fix:*

. Check etcd pod logs:
+
[source,terminal]
----
$ oc logs -n openshift-etcd <etcd-pod>
----

. Verify etcd pod status:
+
[source,terminal]
----
$ oc get pods -n openshift-etcd -l app=etcd
----

. If an etcd pod is not running, investigate node issues on the corresponding control plane node.

*PDB blocking updates*

*Symptom:* PDB shows `ALLOWED DISRUPTIONS` of 0.

*Quick fix:*

. Review PDB configuration:
+
[source,terminal]
----
$ oc get pdb <pdb-name> -n <namespace> -o yaml
----

. Adjust `minAvailable` or `maxUnavailable` to allow at least one disruption.

. For example, if you have 3 replicas and `minAvailable: 3`, change to `minAvailable: 2`:
+
[source,terminal]
----
$ oc patch pdb <pdb-name> -n <namespace> --type merge -p '{"spec":{"minAvailable":2}}'
----

*Node NotReady*

*Symptom:* Node shows `NotReady` status in `oc get nodes` output.

*Quick fix:*

. Check node conditions:
+
[source,terminal]
----
$ oc describe node <node-name> | grep -A 10 Conditions
----

. Check kubelet logs:
+
[source,terminal]
----
$ oc debug node/<node-name> -- chroot /host journalctl -u kubelet
----

. Common causes: disk pressure, memory pressure, network connectivity issues.
