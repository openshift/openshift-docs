= TNF control-plane node replacement overview

[role="_abstract"]
Two-Node with Fencing (TNF) cluster architecture does not require shared storage and supports both `active-active` and `active-passive` workloads. When a control plane node fails, pacemaker automatically triggers fencing via the Redfish protocol and failover occurs within minutes.

The following components work together during a node failure and replacement:

etcd:: Uses Raft consensus, which requires quorum. When one node fails, the surviving node enters a degraded state until quorum is restored.

Pacemaker:: Manages fencing via Redfish. Coordinates the lifecycle of etcd and kubelet on each node.

Cluster ETCD Operator (CEO):: Orchestrates etcd recovery. Executes quorum restoration, and manages node replacement jobs including `tnf-auth` and `tnf-update-setup`.

PacemakerCluster CR:: Monitors and reports pacemaker cluster health. Provides structured status for nodes, resources, and fencing agents.
