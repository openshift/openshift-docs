= Control plane node replacement in two-node OpenShift cluster with fencing

[role="_abstract"]
Designed for simplicity and resilience, a two-node OpenShift cluster with Fencing (TNF) eliminates the cost and complexity of shared storage while supporting both `active-active` and `active-passive` workloads. If a control plane node ever fails, Pacemaker uses the Redfish protocol to automatically isolate the issue and safely restore your operations within minutes, keeping your downtime to an absolute minimum.

The following components work together during a node failure and replacement:

etcd:: The etcd relies on Raft consensus to keep your data safe and synchronized, which requires a quorum to function. If you lose one node, the surviving node automatically enters a `degraded` state to protect data integrity, remaining in this safe mode until you restore the quorum.

Pacemaker:: The Pacemaker manages fencing by using Redfish protocol. It coordinates the lifecycle of etcd and kubelet on each node.

Cluster etcd Operator (CEO):: The CEO handles the heavy lifting during disruptions by orchestrating etcd recovery. It automatically restores your data quorum and safely manages node replacement workflows by running the `tnf-auth` and `tnf-update-setup` configuration jobs.

`PacemakerCluster` custom resource:: This custom resource gives you complete visibility into system resilience by monitoring and reporting the health of the Pacemaker cluster. It translates complex system telemetry into a clear, structured status view of your nodes, active workloads, and fencing agents, helping you to troubleshoot faster.
