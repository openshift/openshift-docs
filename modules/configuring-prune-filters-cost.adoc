= Configuring prune filters for cost optimization

[role="_abstract"]
Prune filters remove specific fields from log records while preserving the records themselves. Reducing log payload size lowers storage costs and network bandwidth consumption. You do not lose the record count that is needed for analytics and alerting.

.Prerequisites

* You have installed the {clo}.
* You have administrator permissions.
* You have created a `ClusterLogForwarder` custom resource.

.Procedure

. Choose the pruning scenario that matches your environment:
+
** *Scenario 1*: Prune verbose Kubernetes metadata
** *Scenario 2*: Prune high-cardinality fields for external forwarding
** *Scenario 3*: Apply selective pruning per pipeline

. Edit your `ClusterLogForwarder`:
+
[source,terminal]
----
$ oc edit clusterlogforwarder instance -n openshift-logging
----

. Add the appropriate prune filter configuration:
+
*Scenario 1: Prune verbose Kubernetes metadata*
+
Kubernetes adds extensive metadata to every log record. This filter removes high-cardinality fields that add size but provide limited value. It preserves labels that are needed for queries and annotations that are needed for operations.
+
[source,yaml]
----
apiVersion: observability.openshift.io/v1
kind: ClusterLogForwarder
metadata:
  name: instance
  namespace: openshift-logging
spec:
  filters:
  - name: prune-k8s-metadata
    type: prune
    prune:
      in:
      - .kubernetes.annotations
      - .kubernetes.namespace_id
      - .kubernetes.pod_id
      - .kubernetes.container_id

  pipelines:
  - name: application-logs
    inputRefs:
    - application
    filterRefs:
    - prune-k8s-metadata
    outputRefs:
    - default-lokistack
----
+
This filter removes annotations and ID fields (namespace ID, pod ID, container ID) that are high-cardinality but rarely needed for queries. The filter preserves labels because they are the foundation for Kubernetes queries and service-to-pod matching.
+
*Scenario 2: Prune high-cardinality fields for external forwarding*
+
When forwarding to external systems with per-GB pricing, reducing payload size directly reduces costs. This filter strips fields that add volume but limited value.
+
[source,yaml]
----
apiVersion: observability.openshift.io/v1
kind: ClusterLogForwarder
metadata:
  name: instance
  namespace: openshift-logging
spec:
  filters:
  - name: prune-for-splunk
    type: prune
    prune:
      in:
      - .kubernetes.annotations
      - .kubernetes.container_image
      - .kubernetes.namespace_id
      - .kubernetes.pod_id
      - .kubernetes.container_id

  pipelines:
  - name: application-to-splunk
    inputRefs:
    - application
    filterRefs:
    - prune-for-splunk
    outputRefs:
    - external-splunk
----
+
Container images are immutable. Recording them in every log adds size without providing value. ID fields (namespace ID, pod ID, container ID) are high-cardinality strings that duplicate information available through name fields. This filter reduces payload size by approximately 30-40% for typical application logs.
+
*Scenario 3: Apply selective pruning per pipeline*
+
Different destinations might need different field sets. This configuration applies different pruning strategies to pipelines routing to different outputs.
+
[source,yaml]
----
apiVersion: observability.openshift.io/v1
kind: ClusterLogForwarder
metadata:
  name: instance
  namespace: openshift-logging
spec:
  filters:
  - name: prune-minimal
    type: prune
    prune:
      in:
      - .kubernetes.pod_ip
      - .kubernetes.container_id

  - name: prune-aggressive
    type: prune
    prune:
      in:
      - .kubernetes.annotations
      - .kubernetes.container_image
      - .kubernetes.namespace_id
      - .kubernetes.pod_id
      - .kubernetes.container_id
      - .kubernetes.pod_ip

  pipelines:
  - name: to-lokistack
    inputRefs:
    - application
    filterRefs:
    - prune-minimal
    outputRefs:
    - default-lokistack

  - name: to-s3-archive
    inputRefs:
    - application
    filterRefs:
    - prune-aggressive
    outputRefs:
    - long-term-s3
----
+
The `prune-minimal` filter applies minimal pruning for interactive queries in `LokiStack`, where users need full context. The `prune-aggressive` filter applies aggressive pruning for long-term archival, where cost optimization is the priority.

.Verification

. Compare log record sizes before and after pruning by examining sample records:
+
.. Query a log record without pruning:
+
[source,terminal]
----
$ oc logs -n openshift-logging _<collector-pod-name>_ --tail=1 | jq 'length'
----
+
.Example output before pruning
[source,terminal]
----
1847
----

.. Query a log record with pruning applied:
+
[source,terminal]
----
$ oc logs -n openshift-logging _<collector-pod-name>_ --tail=1 | jq 'length'
----
+
.Example output pruned
[source,terminal]
----
1124
----
+
This example shows approximately 39% reduction in payload size.


. Monitor storage consumption over time:
+
For `LokiStack`, check the object storage bucket size before and after applying prune filters. For external destinations, review billing reports to confirm reduced data transfer and storage costs.
