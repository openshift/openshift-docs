= {oadp-short} use case for two `BackupStorageLocation`

In this use case, you configure a DPA with two `BackupStorageLocation` by using two cloud credentials. You then backup and restore a `MySQL` database application by using the first `BackupStorageLocation`. In a later section, you run a backup and restore by using the second `BackupStorageLocation`.

. Login to openshift cluster
Create credentials to access the MCG bucket
Create credentials to access the AWS bucket
Update DPA fields such as bucket and s3Url (Note: Specify the endpoint from where you can access the bucket such as route or ip)
Create dpa and verify it's reconciled
Verify the BSL status is available

.Prerequisites

* You must install the {oadp-short} Operator.
* You must configure two backup storage locations, AWS S3 and MCG object storage.
* You must have a `MySQL` database deployed on the {OpenShift} cluster.

. Create the first `Secret` for AWS S3 storage provider  with the default name by running the following command.

[source, terminal]
----
oc create secret generic cloud-credentials -n openshift-adp --from-file cloud=<aws_credentials_file_name> # <1>
----
<1> Specify the name of the cloud credentials file for AWS S3.

. Create the second `Secret` for MCG with a custom name by running the following command.

[source, terminal]
----
oc create secret generic mcg-secret -n openshift-adp --from-file cloud=<MCG_credentials_file_name> # <1>
----
<1> Specify the name of the cloud credentials file for MCG. Note the name of the custom secret `mcg-secret`

. Configure the DPA with the two `BackupStorageLocations` as shown in the following example. The name of the DPA is set as `two-bsl-dpa`.

[source, yaml]
----
apiVersion: oadp.openshift.io/v1alpha1
kind: DataProtectionApplication
metadata:
  name: two-bsl-dpa
  namespace: openshift-adp  
spec:
  backupLocations:
  - name: aws
    velero:
      config:
        profile: default
        region: <region> # <1>
      credential:
        key: cloud
        name: cloud-credentials
      default: true
      objectStorage:
        bucket: <bucket_name> # <2>
        prefix: velero
      provider: aws
  - name: mcg
    velero:
      config:
        insecureSkipTLSVerify: "true"
        profile: noobaa
        region: noobaa
        s3ForcePathStyle: "true"
        s3Url: <s3_url> # <3>
      credential:
        key: cloud
        name: mcg-secret # <4>
      objectStorage:
        bucket: <bucket_name_mcg> # <5>
        prefix: velero
      provider: aws
  configuration:
    nodeAgent:
      enable: true
      uploaderType: kopia
    velero:
      defaultPlugins:
      - openshift
      - aws
----
<1> Specify the AWS region for the bucket.
<2> Specify the AWS S3 bucket name.
<3> Specify the URL of the S3 endpoint for MCG.
<4> Specify the name of the custom secret for MCG storage.
<5> Specify the MCG bucket name.

. Create the DPA by running the following command.

[source, terminal]
----
oc create -f <dpa_file_name> # <1>
----
<1> Specify the file name of the DPA you updated in an earlier step.

. Verify if the DPA has reconciled by running the following command.

[source, terminal]
----
oc get dpa -o yaml
----

. Verify if the `BackupStorageLocation` status is available by running the following command.

[source, terminal]
----
oc get bsl
----

. Create a backup `CR` with default `BSL`. Note in the following example that, the `storageLocation` field is not specified in the backup `CR`. 

[source, yaml]
----
apiVersion: velero.io/v1
kind: Backup
metadata:
  name: test-backup1
  namespace: openshift-adp
spec:
  includedNamespaces:
  - <mysql_namespace>> # <1>
  defaultVolumesToFsBackup: true
----
<1> Specify the namespace for the `MySQL` application installed in the cluster.

. Create a backup by running the following command.

[source, terminal]
----
oc apply -f <backup_file_name> # <1>
----
<1> Specify the name of the backup `CR` file.

. Verify that the backup completed with the default `BSL` by running the following command.

[source, terminal]
----
oc get backup <backup_name> -o yaml # <1>
----
<1> Specify the name of the backup from an earlier step.

.Example output

. Create a backup `CR` by using `MCG` as the `BSL`. Note in the following example that, the second `storageLocation` value is specified at the time of backup `CR` creation.

[source,yaml]
----
apiVersion: velero.io/v1
kind: Backup
metadata:
  name: test-backup1
  namespace: openshift-adp
spec:
  includedNamespaces:
  - <mysql_namespace>> # <1>
  storageLocation: mcg # <2>
  defaultVolumesToFsBackup: true
----
<1> Specify the namespace for the `MySQL` application installed in the cluster.
<2> Specify the second storage location.

. Create the backup by running the following command.

[source, terminal]
----
oc apply -f <backup_file_name> # <1>
----
<1> Specify the name of the backup `CR` file.


. Verify that the backup completed with the second `BSL` by running the following command.

[source, terminal]
----
oc get backup <backup_name> -o yaml # <1>
----
<1> Specify the name of the backup.

. To run the restore operation, delete the `MySQL` application namespace and the backup created in an earlier step. Run the following commands.

[source, terminal]
----
oc delete ns <mysql_namespace>
velero delete backup <backup_name> -n openshift-adp
----

. Create the restore `CR`. 

[source, yaml]
----
apiVersion: velero.io/v1
kind: Restore
metadata:
  name: test-restore 
  namespace: openshift-adp
spec:
  backupName: <backup_name> # <1> 
  includedResources: [] 
  excludedResources:
  - nodes
  - events
  - events.events.k8s.io
  - backups.velero.io
  - restores.velero.io
  - resticrepositories.velero.io
  restorePVs: true

----
<1> Specify the name of the backup which used `MCG` as the backup location.

. Restore the `MySQL` application namespace running the following command.

[source, terminal]
----
oc apply -f <restore_file_name> # <1>
----
<1> Specify the name of the restore `CR` file.

. Verify the restore operation by running the following command.

[source, terminal]
----
oc get restore <restore_name> -o yaml # <1>
----
<1> Specify the name of restore.
