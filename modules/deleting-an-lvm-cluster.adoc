= Deleting an LVMCluster

[role="_abstract"]
When you delete an `LVMCluster`, the operator enforces deletion gates to prevent data loss. The gates that apply depend on the reclaim policy in use.

.procedure

.Gate 1: Active PVCs (applies to all reclaim policies)

The `LVMCluster` cannot be deleted while PVCs that reference LVMS StorageClasses still exist. You see a `DeletionPending` event:

[source,terminal]
----
Warning  DeletionPending  found PVCs provisioned by LVMS, waiting for their deletion
----

Delete all PVCs that use the LVMS StorageClass. Back up data before deletion if needed.

[source,terminal]
----
# List PVCs using the LVMS StorageClass
$ oc get pvc -A --field-selector spec.storageClassName=lvms-vg1

# Delete the PVCs
$ oc delete pvc <pvc-name> -n <namespace>
----

With the default `Delete` reclaim policy, deleting the PVCs automatically deletes the PVs and on-disk logical volumes. Once all PVCs are removed, `LVMCluster` deletion proceeds without further action.

.Gate 2: Retained PVs (Retain policy only)

After PVCs are deleted, if the reclaim policy is `Retain`, the PVs still exist and the `LVMCluster` deletion remains blocked:

[source,terminal]
----
Warning  DeletionPending  found PVs with Retain policy from LVMS, waiting for manual cleanup
----

.Delete the retained PVs.

[source,terminal]
----
# List retained PVs
$ oc get pv --field-selector spec.storageClassName=lvms-vg1

# Delete the PVs
$ oc delete pv <pv-name>
----

.Gate 3: On-disk logical volumes (Retain policy only)

Even after PV objects are deleted from Kubernetes, the underlying logical volumes still exist on disk because the `Retain` policy preserved them. The VG Manager detects this and emits a `ManualCleanupRequired` event:

[source,terminal]
----
Warning  ManualCleanupRequired  volume group vg1 has retained logical volumes [pvc-abc123]; manual cleanup required before deletion can proceed
----

Delete the TopoLVM LogicalVolume custom resources to trigger on-disk logical volume cleanup.

[source,terminal]
----
# List LogicalVolume CRs
$ oc get logicalvolumes

# Delete LogicalVolume CRs for your device class
$ oc delete logicalvolume <lv-name>
----

.Once all LogicalVolume CRs are removed and the on-disk logical volumes are cleaned up, the `LVMCluster` deletion completes automatically.

.Summary of the deletion sequence

With `Delete` reclaim policy::

. Delete all PVCs using the LVMS StorageClass.
. `LVMCluster` deletion completes automatically.

With `Retain` reclaim policy::

. Delete all PVCs using the LVMS StorageClass.
. Delete all retained PVs.
. Delete the LogicalVolume CRs to clean up on-disk logical volumes.
. `LVMCluster` deletion completes automatically.
