= Command-line options for fencing validator script

[role="_abstract"]
You can see different command-line options for the `fencing_ validator` script. 

[source,terminal]
----
$ oc debug node/<node_name> --chroot /host /usr/local/bin/fencing_validator --help
----

The following table details command-line options for the `oc debug node` command:

[cols="1,1,2", options="header"]
|===
| Flag | Argument | Description

| `--user`
| `<ssh_user>`
| SSH username for remote node access. The default value is `core`.

| `--ssh-key`
| `<path>`
| Path to SSH private key.

| `--kubeconfig`
| `<path>`
| Path to kubeconfig file.

| `--transport`
| `auto`, `ssh`, or `ocdebug`
| How the script connects to the other node. For more information, see `auto` tries SSH first, falls back to `oc debug`.

| `--timeout`
| `<seconds>`
| Maximum wait time for recovery loops. By default, it is 1200 seconds or 20 minutes.

| `--hosts`
| `"nodeA,nodeB"`
| Comma-separated pair of node hostnames or IP addresses.

| `--host-a`
| `<host>`
| Explicitly set the first node.

| `--host-b`
| `<host>`
| Explicitly set the second node.

| `--disruptive`
| *(none)*
| Enable destructive fencing tests (reboots nodes)

| `--dry-run`
| *(none)*
| Show what would be done without doing actual validations.

| `-h`, `--help`
| *(none)*
| Show usage information.
|===

You can set all the flags by using the following environment variables:

* `SSH_USER`
* `SSH_KEY`
* `KUBECONFIG`
* `TRANSPORT`
* `TIMEOUT`
* `DISRUPTIVE`
* `DRY_RUN`
* `IP_A / IP_B`: set host addresses directly
* `OC_BIN`: custom `oc` binary path
* `OC_REQ_TIMEOUT`: per-API-call timeout. The default value is 10 seconds.
* `CMD_EXEC_TIMEOUT_SECS`: per-command timeout. The default value is 60 seconds.
