= GitOps and {cgu-operator} for bare-metal core cluster updates

[role="_abstract"]
You can use GitOps workflows with {cgu-operator-full} ({cgu-operator}) to automate cluster updates across your bare-metal core deployment.
This approach provides consistency, auditability, and automation for managing updates at scale.

The GitOps approach treats cluster configuration as code stored in Git repositories.
{cgu-operator} extends this model to orchestrate policy-based updates across hundreds of clusters with topology awareness and intelligent batching.

[id="gitops-approach-to-cluster-lifecycle-management_{context}"]
== GitOps approach to cluster lifecycle management

GitOps provides a declarative approach to managing cluster configuration and lifecycle operations.
You define the desired state of your clusters in Git, and automation tools ensure the actual state matches the desired state.

For cluster updates, the GitOps workflow follows these steps:

. Define the desired cluster version in a Git repository as a policy.
. Commit and push the policy changes through your review process.
. Apply the policy to the {rh-rhacm-first} hub cluster.
. {cgu-operator} orchestrates the rollout to target clusters according to your batching strategy, using `ClusterGroupUpgrade` custom resources.
. Monitor progress and verify successful updates.

The following list describes the benefits of using GitOps for bare-metal core updates:

* Audit trail: All update decisions are tracked in Git commit history.
* Review process: Policy changes go through pull request reviews before application.
* Policy versioning: Revert to previous policy versions in Git if issues occur.
* Consistency: All clusters receive identical update configurations.
* Automation: Eliminate manual update steps and reduce human error.

[id="clustergroupupgrade-custom-resource_{context}"]
== ClusterGroupUpgrade custom resource

The `ClusterGroupUpgrade` custom resource is the primary mechanism for orchestrating updates with {cgu-operator}.
A `ClusterGroupUpgrade` resource defines which clusters to update, which policies to apply, and how to batch the rollout.

IMPORTANT: {rh-rhacm} policies used for updates must be set to `inform` mode. The `ClusterGroupUpgrade` resource temporarily changes these policies to `enforce` mode at the scheduled update time, applying them to the target clusters. Once the `ClusterGroupUpgrade` resource completes or times out, the policies automatically revert to `inform` mode. This ensures that update policies are only enforced at the specific time you intend, rather than immediately when the policy is created.

The following list describes the key `ClusterGroupUpgrade` fields:

* `clusters`: List of target clusters or a `ManagedClusterSet` selector.
* `managedPolicies`: Policies to apply during the update.
* `enable`: Controls whether the `ClusterGroupUpgrade` resource starts immediately (`true`) or waits (`false`).
* `remediationStrategy`: Batching, concurrency, and timeout settings.
* `actions`: Hooks for pre-update and post-update automation.

[id="clustergroupupgrade-lifecycle_{context}"]
== ClusterGroupUpgrade lifecycle

A `ClusterGroupUpgrade` resource progresses through several states during an update:

. `Preparing`: {cgu-operator} evaluates cluster readiness and policy compliance.
. `InProgress`: {cgu-operator} applies policies to clusters in batches.
. `Complete`: All clusters successfully updated and compliant.
. `Failed`: One or more clusters failed to update.

[id="benefits-of-cgu-operator-for-large-scale-deployments_{context}"]
== Benefits of {cgu-operator} for large-scale deployments

{cgu-operator} is specifically designed for large-scale deployments with many clusters across many sites.

The following list describes the key benefits for bare-metal core clusters:

* Scale: Update hundreds or thousands of clusters from a central hub.
* Canary deployments: Test updates on a single cluster before rolling out to the fleet.
* Maintenance windows: Schedule updates during planned downtime using blocking custom resources.
* Integration: Works with existing GitOps tools such as ArgoCD and Flux.

[id="comparison-to-manual-upgrades_{context}"]
== Comparison to manual updates

Manual cluster updates involve logging into each cluster individually and running update commands.
This approach does not scale for large deployments with many clusters at many sites.

.Manual compared to {cgu-operator}-based updates
[cols="1,2,2",options="header"]
|===
|Aspect
|Manual approach
|{cgu-operator} approach

|Scale
|One cluster at a time
|Hundreds of clusters in batches

|Consistency
|Varies by operator
|Identical policy applied to all clusters

|Auditability
|Manual logs
|Git commit history and {rh-rhacm} event tracking

|Update halting
|Manual intervention
|Automated halt on policy violations
|===

Use manual updates only for single-cluster test environments or when troubleshooting specific cluster issues.
For any production deployment managing multiple clusters, {cgu-operator}-based updates are essential.
