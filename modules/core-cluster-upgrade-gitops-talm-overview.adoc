= GitOps and {cgu-operator} for bare metal Core cluster upgrades

[role="_abstract"]
You can use GitOps workflows with {cgu-operator-full} ({cgu-operator}) to automate cluster upgrades across your bare metal Core deployment.
This approach provides consistency, auditability, and automation for managing upgrades at scale.

The GitOps approach treats cluster configuration as code stored in Git repositories.
{cgu-operator} extends this model to orchestrate policy-based upgrades across hundreds of edge clusters with topology awareness and intelligent batching.

[id="gitops-approach-to-cluster-lifecycle-management_{context}"]
== GitOps approach to cluster lifecycle management

GitOps provides a declarative approach to managing cluster configuration and lifecycle operations.
You define the desired state of your clusters in Git, and automation tools ensure the actual state matches the desired state.

For cluster upgrades, the GitOps workflow follows these steps:

. Define the desired cluster version in a Git repository as a policy
. Commit and push the policy changes through your review process
. Apply the policy to the {rh-rhacm} hub cluster
. {cgu-operator} orchestrates the rollout to target clusters according to your batching strategy, using ClusterGroupUpgrade (CGU) custom resources
. Monitor progress and verify successful upgrades

*Benefits of GitOps for bare metal Core upgrades:*

* *Audit trail*: All upgrade decisions are tracked in Git commit history
* *Review process*: Policy changes go through pull request reviews before application
* *Rollback capability*: Revert to previous policy versions if issues occur
* *Consistency*: All clusters receive identical upgrade configurations
* *Automation*: Eliminate manual upgrade steps and reduce human error

[id="cgu-operator-fundamentals_{context}"]
== {cgu-operator} fundamentals

{cgu-operator-full} is an operator that orchestrates policy-based upgrades across multiple clusters managed by {rh-rhacm}.
{cgu-operator} respects cluster groupings, enforces upgrade ordering, and manages concurrency limits to ensure safe rollouts.

{cgu-operator} provides these key capabilities:

* *Batch orchestration*: Upgrade clusters in waves with controlled concurrency
* *Topology awareness*: Respect site locations and cluster groupings when scheduling upgrades
* *Automated rollback*: Stop upgrades if policy violations occur
* *Progress tracking*: Monitor upgrade status across all clusters in real time

[id="clustergroupupgrade-custom-resource_{context}"]
== ClusterGroupUpgrade custom resource

The ClusterGroupUpgrade (CGU) custom resource is the primary mechanism for orchestrating upgrades with {cgu-operator}.
A CGU defines which clusters to upgrade, which policies to apply, and how to batch the rollout.

IMPORTANT: {rh-rhacm} policies used for upgrades must be set to `inform` mode. The CGU temporarily changes these policies to `enforce` mode at the scheduled upgrade time, applying them to the target clusters. Once the CGU completes or times out, the policies automatically revert to `inform` mode. This ensures that upgrade policies are only enforced at the specific time you intend, rather than immediately when the policy is created.

*Key CGU fields:*

* `clusters`: List of target clusters or a ManagedClusterSet selector
* `managedPolicies`: Policies to apply during the upgrade
* `enable`: Controls whether the CGU starts immediately (true) or waits (false)
* `remediationStrategy`: Batching, concurrency, and timeout settings
* `actions`: Hooks for pre-upgrade and post-upgrade automation

[id="clustergroupupgrade-lifecycle_{context}"]
== ClusterGroupUpgrade lifecycle

A ClusterGroupUpgrade progresses through several states during an upgrade:

. *Preparing*: {cgu-operator} evaluates cluster readiness and policy compliance
. *InProgress*: {cgu-operator} applies policies to clusters in batches
. *Complete*: All clusters successfully upgraded and compliant
. *Failed*: One or more clusters failed to upgrade

You can monitor the CGU status to track progress:

[source,terminal]
----
$ oc get cgu <cgu-name> -n <namespace> -o yaml
----

The CGU status includes detailed information about each cluster's upgrade state and any errors encountered.

[id="gitops-workflow-for-cgu-operator-based-upgrades_{context}"]
== GitOps workflow for {cgu-operator}-based upgrades

The complete workflow for a {cgu-operator}-based cluster upgrade includes these steps:

. *Define desired state in Git*: Create or update a policy that specifies the target cluster version
+
[source,yaml]
----
apiVersion: config.openshift.io/v1
kind: ClusterVersion
metadata:
  name: version
spec:
  channel: stable-4.20
  desiredUpdate:
    version: 4.20.1
----

. *Apply policies to {rh-rhacm} hub*: Push the policy to your {rh-rhacm} hub cluster where {cgu-operator} monitors for changes
+
[source,terminal]
----
$ oc apply -f core-upgrade-policy.yaml
----

. *Create CGU CR to orchestrate rollout*: Define which clusters receive the upgrade and how to batch the rollout
+
[source,yaml]
----
apiVersion: ran.openshift.io/v1alpha1
kind: ClusterGroupUpgrade
metadata:
  name: core-upgrade-4.20.1
  namespace: default
spec:
  clusters:
  - spoke1
  - spoke2
  managedPolicies:
  - core-upgrade-4.20.1-policy
  enable: true
  remediationStrategy:
    maxConcurrency: 2
    timeout: 3600
----

. *{cgu-operator} applies policies in batches*: {cgu-operator} enforces the policy on target clusters according to the batching strategy
+
{cgu-operator} upgrades clusters in batches, waiting for each batch to complete successfully before proceeding to the next batch.

. *Monitor progress and verify success*: Track CGU status and verify cluster health after upgrades complete
+
[source,terminal]
----
$ oc get cgu core-upgrade-4.20.1 -w
----

[id="benefits-of-cgu-operator-for-large-scale-deployments_{context}"]
== Benefits of {cgu-operator} for large-scale deployments

{cgu-operator} is specifically designed for large-scale deployments with many clusters across many sites.

*Key benefits for bare metal Core clusters:*

* *Scale*: Upgrade hundreds or thousands of edge clusters from a central hub
* *Canary deployments*: Test upgrades on a single cluster before rolling out to the fleet
* *Maintenance windows*: Schedule upgrades during planned downtime using blocking custom resources
* *Integration*: Works with existing GitOps tools such as ArgoCD and Flux

[id="comparison-to-manual-upgrades_{context}"]
== Comparison to manual upgrades

Manual cluster upgrades involve logging into each cluster individually and running upgrade commands.
This approach does not scale for large deployments with many clusters at many sites.

.Manual compared to {cgu-operator}-based upgrades
[cols="1,2,2",options="header"]
|===
|Aspect
|Manual approach
|{cgu-operator} approach

|Scale
|One cluster at a time
|Hundreds of clusters in batches

|Consistency
|Varies by operator
|Identical policy applied to all clusters

|Auditability
|Manual logs
|Git commit history and {rh-rhacm} event tracking

|Rollback
|Manual intervention
|Automated on policy violations
|===

Use manual upgrades only for single-cluster test environments or when troubleshooting specific cluster issues.
For any production deployment managing multiple clusters, {cgu-operator}-based upgrades are essential.
Consistent, auditable, and automated upgrade processes are a requirement for all large organizations operating at scale, not an optional enhancement.
{cgu-operator} provides the policy-driven orchestration, batch management, and progress tracking that production environments demand.
