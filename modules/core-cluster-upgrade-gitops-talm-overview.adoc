= Policy-based cluster updates with {rh-rhacm} and {cgu-operator}

[role="_abstract"]
You can use {rh-rhacm-first} and {cgu-operator-first} to automate cluster updates across spoke clusters managed from a hub cluster.
Optionally, you can manage {rh-rhacm} policies through a GitOps workflow, storing them in a Git repository for versioning, review, and auditability.

{rh-rhacm}:: Provides the policy framework for managing cluster configuration across a fleet of spoke clusters from a central hub cluster. You define {rh-rhacm} policies that declare the desired cluster version, and {rh-rhacm} evaluates compliance across target clusters.

{cgu-operator}:: Orchestrates the rollout of {rh-rhacm} policies to target clusters according to your batching strategy. {cgu-operator} uses `ClusterGroupUpgrade` custom resources to coordinate which clusters to update, which policies to apply, and how to manage concurrency. A `ClusterGroupUpgrade` resource progresses through several states during an update:
+
* `Preparing`: {cgu-operator} evaluates cluster readiness and policy compliance.
* `InProgress`: {cgu-operator} applies policies to clusters in batches.
* `Complete`: All clusters successfully updated and compliant.
* `Failed`: One or more clusters failed to update.
