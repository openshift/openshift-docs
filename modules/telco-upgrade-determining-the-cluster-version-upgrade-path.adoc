= Determining the cluster version upgrade path

Use the link:https://access.redhat.com/labs/ocpupgradegraph/update_path/[Red Hat {product-title} Update Graph] tool to determine if the path is valid for the z-stream release you want to upgrade to.
You should also always verify the upgrade with your Red Hat Technical Account Manager to ensure that the upgrade path is valid for telco implementations.

[IMPORTANT]
====
The <4.y+1.z> or <4.y+2.z> version that you upgrade to must have the same patch level as the <4.y.z> release you are upgrading from.

The OpenShift upgrade process mandates that if a fix is present in a specific <4.y.z> release, then the that fix must be present in the <4.y+1.z> release that you upgrade to.
====

.Bug fix backporting and the upgrade graph
image::openshift-bug-fix-backporting-upgrade-graph.png[Bug fix backporting and the upgrade graph]

[IMPORTANT]
====
OpenShift development has a strict backport policy that prevents regressions.
For example, a bug must be fixed in 4.16.z before it is fixed in 4.15.z.
This means that the upgrade graph does not allow for upgrades to chronologically older releases even if the minor version is greater, for example, upgrading from 4.15.24 to 4.16.2.
====
