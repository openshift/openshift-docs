= When to use the O-Cloud Manager

[role="_abstract"]
{product-title} provides two approaches for managing the lifecycle of bare-metal clusters at scale: the O-Cloud Manager and {ztp-first}. Both use {rh-rhacm-first} and the SiteConfig Operator to install clusters, but they differ in how you define and operate those clusters over time.

The O-Cloud Manager adds a template abstraction layer on top of `ClusterInstance` resources. Instead of defining each cluster individually, platform engineers create reusable `ClusterTemplate` blueprints and site engineers instantiate clusters by submitting a `ProvisioningRequest` with site-specific values. The Operator handles hardware allocation, firmware preparation, cluster installation, and ongoing lifecycle operations through that single resource.

This approach is designed for the following use cases:

* Manage a large fleet of clusters with similar configurations that differ only in site-specific parameters such as network addressing and cluster names.
* Require integrated hardware management with declarative firmware and BIOS configuration as part of the provisioning flow.
* Want a single custom resource as the control point for provisioning, Day-2 configuration, scaling, and upgrades.
* Need O-RAN compliant infrastructure with SMO integration through standardized O2 IMS REST APIs.

{ztp} takes a different approach. Cluster definitions are committed directly as `ClusterInstance` resources in a Git repository and synchronized to the hub cluster through ArgoCD. Each cluster is fully specified in its own resource, and lifecycle operations are managed through Git commits and ACM policy updates rather than through a single orchestrating custom resource.

{ztp} is appropriate in the following use cases:

* You prefer to manage cluster state entirely through Git and do not need the template-based separation between blueprints and site-specific parameters.
* Hardware allocation and firmware management are handled outside the provisioning workflow.
* Your fleet is small enough that defining each cluster individually does not create significant maintenance overhead.
