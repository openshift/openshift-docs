= About in-place pod resizing

In-place pod resize allows you to changing the CPU and memory resources for containers within a running pod without application disruption. The standard methods for changing a pod's resources require you to modify the pod specification and re-create the pod, potentially causing disruption. Only CPU and memory resources can be resized.

Using in-place pod resizing requires that you use the ` --subresource resize` flag when editing the pod in the {oc-first}, as shown in the following example:

.Example commands
[source,terminal]
----
$ oc edit pod resize-demo  --subresource resize
----

[source,terminal]
----
$ apply -f <updated_file_name>.yaml --subresource resize
----

[source,terminal]
----
$ patch pod resize-demo --subresource resize --patch \
  '{"spec":{"containers":[{"name":"pause", "resources":{"requests":{"cpu":"800m"}, "limits":{"cpu":"800m"}}}]}}'
----

When using in-place pod resizing to change CPU or memory resources, you can control whether a pod is restarted when by configuring a resize policy in the pod specification. This allows fine-grained control based on resource type (CPU or memory). The following example resize policy requires a pod restart upon changing the memory resources, but prevents a restart for CPU resource changes.

.Example resource policy
[source,terminal]
----
apiVersion: v1
kind: Pod
metadata:
  name: resize-demo
spec:
  securityContext:
    runAsNonRoot: true
    seccompProfile:
      type: RuntimeDefault
  containers:
  - name: pause
# ...
    resizePolicy:
    - resourceName: cpu
      restartPolicy: NotRequired
    - resourceName: memory
      restartPolicy: RestartContainer
----

[NOTE]
====
Memory limits cannot be decreased unless the resize policy for `memory` is `RestartContainer`.
====



After making the resource changes, the pod status conditions indicates the state of a resize request by using the following messages:

* `PodResizePending`: The kubelet cannot immediately make the change for one of the following reasons:
** `Infeasible`: The requested resize cannot be executed on the current node. For example, you are requesting more resources than the node has available.
** `Deferred`: The requested resize is currently not possible, but might become feasible at a later time. For example, if another pod is removed is removed from the node, the requested resources might become avilable. The kubelet retries the resize.

* `PodResizeInProgress`: The kubelet is able to allocate the requested resources and the change is being applied.

* `Error`: In case of an error during the resource allocation are reported in the message field along with reason.

.Example status for an infeasible change
[source,terminal]
----
apiVersion: v1
kind: Pod
metadata:
  name: resize-demo
# ...
status:
  conditions:
  - lastProbeTime: "2025-09-03T15:00:50Z"
    lastTransitionTime: "2025-09-03T15:00:50Z"
    message: 'Node didn''t have enough capacity: cpu, requested: 1000000, capacity:
      3500'
    reason: Infeasible
    status: "True"
    type: PodResizePending
----

[NOTE]
====
* Non-restartable init containers and ephemeral containers cannot be resized.
* Pods managed by static CPU or Memory manager policies cannot be resized in-place.
* Pods utilizing swap memory cannot resize memory requests unless the resize policy for `memory` is `RestartContainer`.
====
