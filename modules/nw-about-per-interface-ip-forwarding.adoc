= About per-interface IP forwarding

[role="_abstract"]
Per-interface IP forwarding allows you to enable IPv4 packet forwarding on specific network interfaces rather than globally across all interfaces. This feature is particularly useful in telco and multi-network environments where you need to balance security requirements with functional needs for services like MetalLB-backed load balancers.

== Understanding IP forwarding modes

From {product-title} 4.14 onward, OVN-Kubernetes uses the `Restricted` IP forwarding mode by default. This mode disables global IP forwarding and only permits forwarding between OVN-Kubernetes-managed interfaces, such as the `br-ex` bridge and pod network interfaces. While this enhances security by preventing nodes from acting as general-purpose routers, it can interfere with certain features that require IP forwarding on secondary network interfaces.

The following IP forwarding modes are available:

`Restricted`:: The default mode. Global IP forwarding is disabled. Only forwarding between OVN-Kubernetes-managed interfaces is permitted. This mode provides better security but can break services that require forwarding on secondary interfaces.

`Global`:: Enables IP forwarding across all interfaces on the node. While this mode supports all forwarding use cases, it effectively turns the node into a router, which introduces security risks in untrusted network environments.

`Per-interface`:: Enables IP forwarding only on specific interfaces that you designate. This mode provides a balanced approach that enables required functionality while maintaining security on other interfaces.

== Use cases for per-interface IP forwarding

The primary use case for per-interface IP forwarding is enabling MetalLB-backed load balancers on secondary network interfaces. When you configure MetalLB to advertise IP addresses on a secondary interface, traffic arriving at that interface must be forwarded to the appropriate pods. With `Restricted` mode, this forwarding is blocked by default.

Other use cases include:

* Multi-homed nodes where specific interfaces require routing capabilities
* Network segmentation scenarios where only certain interfaces should participate in routing
* Telco workloads that require traffic forwarding on specific network interfaces

== Security considerations

Per-interface IP forwarding provides a more secure alternative to enabling global forwarding. Consider the following security implications:

* Enabling forwarding on an interface allows traffic to be routed through that interface to other destinations on the node.
* If you enable forwarding on multiple interfaces, traffic can potentially be routed between those interfaces, which might not align with your network segmentation goals.
* Review your network topology carefully before enabling forwarding on any interface to ensure that traffic paths align with your security policies.

== IPv4 only support

Per-interface IP forwarding is supported for IPv4 only. IPv6 per-interface forwarding is not supported because enabling IPv6 forwarding at the interface level also requires enabling global IPv6 forwarding. Additionally, enabling IPv6 forwarding changes the behavior of Router Advertisements (`accept_ra`), which can expose nodes to unintended traffic and security risks.

== Requirements

To use per-interface IP forwarding with the nmstate API, ensure that your cluster meets the following requirements:

* The Kubernetes NMState Operator is installed.
* Cluster nodes are running {op-system-first} with NetworkManager version 1.30 or later.
* The nmstate package on nodes supports the `forwarding` property in the IPv4 interface configuration.
