= Recovering when a node registers from a backup image before reprovisioning

[role="_abstract"]
If the replacement node boots from an old disk image and registers to Kubernetes before Ironic reprovisions it, you must perform additional recovery steps to resolve identity and networking conflicts.

It has the following symptoms:

* A node object appears with an old UID.
* OVN chassis identity conflicts occur.
* CSRs are not automatically approved.
* East-west connectivity failures occur.

[NOTE]
====
Ensure the failed node is fully destroyed or its disk is wiped before powering it on for reprovisioning.
====

.Procedure

. Delete the stale OVN chassis entry before reprovisioning completes. SSH to the surviving node, then run the following commands from inside the `ovnkube-node` pod:
+
[source,terminal]
----
$ oc -n openshift-ovn-kubernetes exec -it <ovnkube-node-pod-survivor> -c ovnkube-controller -- bash
----

. List all chassis to find the stale entry by running the following command:
+
[source,terminal]
----
$ ovn-sbctl list Chassis
----

. Delete the chassis that matches the old node name or chassis ID by running the following command:
+
[source,terminal]
----
$ ovn-sbctl del Chassis <old_chassis_uuid>
----

. If the `cluster-machine-approver` did not automatically approve the node certificates, manually approve pending CSRs by running the following command:
+
[source,terminal]
----
$ oc get csr | grep Pending
$ oc adm certificate approve <csr_name>
----

. If internal connectivity fails after reprovisioning, restart the `ovnkube-node` pod on the surviving node to force OVN to resync remote chassis and encap state by running the following command:
+
[source,terminal]
----
$ oc delete pod -n openshift-ovn-kubernetes \
  -l app=ovnkube-node --field-selector spec.nodeName=<survivor-name>
----
