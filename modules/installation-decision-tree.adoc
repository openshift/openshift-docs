[id="installation-decision-tree-step2_{context}"]
== Step 2: What is your network environment?

After you select a platform, determine which network scenario applies.

[cols="1,2a,2a",options="header"]
|===
|Network type |Description |What you need to do

|*Connected*
|Your cluster nodes have direct access to the internet or through a proxy.
|Follow the standard installation procedure for your platform.

|*Disconnected or restricted network*
|Your cluster nodes have limited or no access to the internet.
|Before you install, mirror the required container images to a local registry. See xref:../../disconnected/installing-mirroring-installation-images.adoc#installing-mirroring-installation-images[Mirroring images for a disconnected installation].

Then follow the disconnected installation procedure for your platform.

|*Private*
|Your cluster does not expose external endpoints. Available for AWS, Azure, and GCP.
|Follow the private cluster installation procedure for your platform.
|===

[id="installation-decision-tree-step3_{context}"]
== Step 3: What installation method do you want to use?

{product-title} supports four installation methods. Choose the method that best fits your level of automation and control.

[cols="1,2a,1a,1a",options="header"]
|===
|Method |Best for |Automation level |Platform control

|*Assisted Installer*
|Getting started quickly with smart defaults and pre-flight validation. Ideal for connected environments.

link:https://access.redhat.com/documentation/en-us/assisted_installer_for_openshift_container_platform[Assisted Installer documentation]
|High
|Low

|*Agent-based Installer*
|Disconnected or restricted environments where the Assisted Installer service is unavailable. Provides a self-contained, bootable installation image.

xref:../../installing/installing_with_agent_based_installer/preparing-to-install-with-agent-based-installer.adoc#preparing-to-install-with-agent-based-installer[Agent-based Installer documentation]
|High
|Medium

|*Installer-provisioned infrastructure (IPI)*
|Automated provisioning on supported cloud and virtualized platforms. The installer creates and manages all required infrastructure.

See the IPI guides under your platform.
|High
|Low

|*User-provisioned infrastructure (UPI)*
|Full control over infrastructure provisioning. You create and manage nodes, networking, and load balancers manually or with your own automation.

See the UPI guides under your platform.
|Low
|High
|===

Not every method is available on every platform. See xref:../../installing/overview/installing-preparing.adoc#supported-installation-methods-for-different-platforms[Supported installation methods for different platforms] for a complete list.

[id="installation-decision-tree-step4_{context}"]
== Step 4: What cluster topology do you need?

Most installations deploy a standard multi-node cluster with three control plane nodes and two or more compute nodes. If your environment requires a different topology, choose from the following options:

[cols="1,2a,1a",options="header"]
|===
|Topology |When to use |Documentation

|*Standard multi-node*
|Production workloads with high availability. This is the default topology.
|Follow the standard installation guide for your platform.

|*Three-node compact cluster*
|Reduced infrastructure costs. Control plane nodes also run workloads. Available on AWS, Azure, GCP, vSphere, Nutanix, and OpenStack.
|Look for "Installing a three-node cluster" under your platform.

|*Single-node OpenShift (SNO)*
|Edge deployments or resource-constrained environments where a single server runs both the control plane and workloads.
|xref:../../installing/installing_sno/install-sno-installing-sno.adoc#installing-sno[Installing single-node OpenShift]

|*Two-node cluster*
|Minimal high-availability deployments at the edge using fencing for quorum.
|xref:../../installing/installing_two_node_cluster/installing_tnf/installing-two-node-fencing.adoc#installing-two-node-fencing[Installing a two-node cluster with fencing]
|===
