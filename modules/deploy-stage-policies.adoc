= Deploy stage policies

{product-title-short} uses deploy stage policies when inspecting workloads during an deployment attempt. These policies are composed of workload criteria, such as the following criteria:

* Container configuration, including environment variables and privileges
* Deployment metadata, such as required or disallowed annotations and labels
* Storage volume information, such as volume name and mount configuration
* Networking configuration, such as port exposure and ingress or egress policy
* _Kubernetes Access_ configuration, which consists of criteria such as service account configuration and RBAC permissions

You can also include image criteria, such as image registry, image content, and CVE data, when building deploy stage policies. This is helpful if you want to evaluate the same requirements in both pipeline build and deployment attempts. This eliminates the need to duplicate policies, because policy can include both image and workload criteria and be used for both the build and deploy stages.

When enforced, a policy violation results in the following actions:

* The {product-title-short} Admission Controller rejects the deployment attempt.
* If the Admission Controller fails, {product-title-short} attempts to scale the deployment number to zero.
