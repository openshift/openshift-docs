= Understanding the image-based upgrade with {ztp}

When you perform an image-based upgrade through {ztp}, you need to specify your desired stage transition policies in your group `PolicyGenTemplate` CR.
Then, TALM reconciles the policies through the `ClusterGroupUpgrade` CR.
This requires you to create `ClusterGroupUpgrade` CRs for each stage, which is error-prone and time-consuming.
To improve and simplify the process, you can use the `ImageBasedGroupUpgrade` CR, as a single resource, to define your stage transition policies and provide instructions for TALM to apply the policies when conditions are met. 
The `ImageBasedGroupUpgrade` CR supports both a manually controlled, progressive and a fully automated upgrade flow.

The `ImageBasedGroupUpgrade` CR combines the `ImageBasedUpgrade` and `ClusterGroupUpgrade` APIs, which means that it shares certain mechanisms from the the `ImageBasedUpgrade` and `ClusterGroupUpgrade` APIs. For example, you can use the cluster selection and rollout strategy the same way as with the `ClusterGroupUpgrade` API.

The following example shows an `ImageBasedGroupUpgrade` CR that upgrades target clusters with the `common` label selector to {product-title} 4.16 in two steps:

. First, the {lcao} performs the `Prep` and `Upgrade` stages.
The `Upgrade` stage is only performed if the `Prep` stage is successful.

. Then, the {lcao} finalizes the upgrade on all clusters that successfully passed the previous step.

.Example ImageBasedGroupUpgrade CR
[source,yaml]
----
apiVersion: lcm.openshift.io/v1alpha1
kind: ImageBasedGroupUpgrade
metadata:
  name: upgrade-4.15.0
  namespace: ibu-upgrade
spec:
  clusterLabelSelectors:
    - matchLabels:
        common: "true"
  ibuSpec:
    seedImageRef:
      image: quay.io/repo/seed:4.16.0
      version: 4.16.0
    oadpContent:
      - name: oadp-cm
        namespace: openshift-adp
  plan:
    - actions: ["Prep", "Upgrade"]
      rolloutStrategy:
        maxConcurrency: 200
        timeout: 60
    - actions: ["FinalizeUpgrade"]
      rolloutStrategy:
        maxConcurrency: 500
        timeout: 10
----


//Status Upgrade/Monitoring
The CR ensures better monitoring experience by introducing the a comprehensive status reporting for all clusters, aggregated in one place.
You can check for the following actions:

`status.completedActions`:: Shows all completed actions defined in the `plan` section.
`status.currentAction`:: Shows all actions that are currently in progress.
`status.failedActions`:: Shows all failed actions along with a detailed error message.

//Supported action combinations
Each item in the `plan` section is a separate step but one step can have several actions.
If you define more than one action in a step, the rollout strategy applies to all of the steps together.

In the following example, you can see that the `Prep` and `Upgrade` stages must complete within 60 minutes all together, while finalizing the upgrade, which is a single action, must complete within 10 minutes. 
//check if it's minutes

[source,yaml]
----
plan:
- actions: ["Prep", "Upgrade"]
    rolloutStrategy:
    maxConcurrency: 200
    timeout: 60
- actions: ["FinalizeUpgrade"]
    rolloutStrategy:
    maxConcurrency: 500
    timeout: 10
----

The {lcao} accepts the following actions:

`Prep`:: Move to the `Prep` stage.
`Upgrade`:: Move to the `Upgrade` stage.
`FinalizeUpgrade`:: Move to the `Idle` stage after a successful upgrade.
`Rollback`:: Move to the `Rollback` stage.
`FinalizeRollback`:: Move to the `Idle` stage after a rollback.
`AbortOnFailure`:: Cancel the upgrade after a failed `Prep` or `Upgrade` actions by moving to the `Idle` stage.
`Abort`:: Cancel the upgrade after the previous action by moving to the `Idle` stage.

[IMPORTANT]
====
To avoid mistakenly cancelling or rolling back an upgrade, ensure that either the `Abort` or `Rollback` action is the first step in a separate plan.
====

The following action combinations are supported:

[NOTE]
====
The brackets signify one separate step in the `plan` section, where one step can contain one or more actions.
====

* `['Prep'], ['Abort']`
+
[source,yaml]
----
plan:
- actions: ["Prep"]
    [...]
- actions: ["Abort"]
    [...]
----

* `['Prep','Upgrade','FinalizeUpgrade']`

+
[source,yaml]
----
plan:
- actions: ["Prep", "Upgrade", "FinalizeUpgrade"]
    [...]
----

* `['Prep'], ['AbortOnFailure'], ['Upgrade'], ['AbortOnFailure'], ['FinalizeUpgrade']`
* `['Prep'], ['AbortOnFailure'], ['Upgrade'], ['AbortOnFailure'], ['Rollback', 'FinalizeRollback']`
* `['Prep', 'Upgrade'], ['Rollback', 'FinalizeRollback']`
* `['Rollback', 'FinalizeRollback']`

The following combinations are for cases where you need to resume or abort an ongoing upgrade from a completely new `ImageBasedGroupUpgrade` CR:

* `['Upgrade','FinalizeUpgrade']`
* `['FinalizeUpgrade']`
* `['FinalizeRollback']`
* `['Abort']`
* `['AbortOnFailure']`

You can append subsequent steps later, which offers more control over the upgrade plan.
[WARNING]
====
You cannot remove actions or steps from an ongoing plan
