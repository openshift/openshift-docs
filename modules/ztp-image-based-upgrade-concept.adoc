= Using the `ImageBasedGroupUpgrade` CR for the image-based upgrade with {ztp}

You can use a single resource, the `ImageBasedGroupUpgrade` CR, to define your stage transition policies and provide instructions for {cgu-operator} to reconcile the policies.

The `ImageBasedGroupUpgrade` CR combines the `ImageBasedUpgrade` and `ClusterGroupUpgrade` APIs, sharing certain mechanisms with them.
For example, you can define the cluster selection and rollout strategy with the `ImageBasedGroupUpgrade` API the same way that you can do with the `ClusterGroupUpgrade` API.

The `ImageBasedGroupUpgrade` CR supports both a manually controlled, and a fully automated upgrade flow.

[id="ztp-image-based-upgrade-supported-combinations_{context}"]
== Supported action combinations

You can specify stage transitions by using the `actions` field in the `plan` section of `ImageBasedGroupUpgrade` CR.
These actions can be combined differently in your upgrade plan.
Each `plan.actions` field is a separate step but one step can contain several actions that all share the same rollout strategy.

You can append subsequent steps later, which offers more control over the upgrade plan.

[WARNING]
====
You cannot remove actions or steps from an ongoing plan.
====
//Until the IBGU is applied?

The following example demonstrates that the `Prep` and `Upgrade` actions must complete within 60 minutes all together.
After the first step is complete, the next step includes one action, finalizing the upgrade, that must complete within 10 minutes.
//check if it's minutes

[source,yaml]
----
plan:
- actions: ["Prep", "Upgrade"]
    rolloutStrategy:
    maxConcurrency: 200
    timeout: 60
- actions: ["FinalizeUpgrade"]
    rolloutStrategy:
    maxConcurrency: 500
    timeout: 10
----

[IMPORTANT]
====
The clusters that fail one of the actions will skip the rest of the actions in the plan.
====
//in the plan or in the step?

The {lcao} accepts the following actions:

`Prep`:: Move to the `Prep` stage.
`Upgrade`:: Move to the `Upgrade` stage.
`FinalizeUpgrade`:: Move to the `Idle` stage to finalize a successful upgrade.
`Rollback`:: Move to the `Rollback` stage.
`FinalizeRollback`:: Move to the `Idle` stage to finalize a rollback.
`AbortOnFailure`:: Move to the `Idle` stage to cancel the upgrade after a failed `Prep` or `Upgrade` action.
`Abort`:: Move to the `Idle` stage to cancel the upgrade after the previous action.

[IMPORTANT]
====
To avoid cancelling or rolling back an upgrade, ensure that either the `Abort` or `Rollback` action is the first step in a separate `ImageBasedGroupUpgrade` CR.
====

The following action combinations are supported, where one pair of brackets signify one step in the `plan` section:

* `["Prep"]`, `["Abort"]`
* `["Prep","Upgrade","FinalizeUpgrade"]`
* ``["Prep"]`, `["AbortOnFailure"]`, `["Upgrade"]`, `["AbortOnFailure"]`, `["FinalizeUpgrade"]`
* `["Prep"]`, `["AbortOnFailure"]`, `["Upgrade"]`, `["AbortOnFailure"]`, `["Rollback", "FinalizeRollback"]`
* `["Prep", "Upgrade"]`, `["Rollback", "FinalizeRollback"]`
* `["Rollback", "FinalizeRollback"]`

The following combinations are for cases where you need to resume or cancel an ongoing upgrade from a completely new `ImageBasedGroupUpgrade` CR:

* `["Upgrade","FinalizeUpgrade"]`
* `["FinalizeUpgrade"]`
* `["FinalizeRollback"]`
* `["Abort"]`
* `["AbortOnFailure"]`

[id="ztp-image-based-upgrade-status-monitoring_{context}"]
== Status monitoring

The `ImageBasedGroupUpgrade` CR ensures better monitoring experience by introducing a comprehensive status reporting for all clusters that is aggregated in one place.
You can monitor the following actions:

`status.completedActions`:: Shows all completed actions defined in the `plan` section.
`status.currentAction`:: Shows all actions that are currently in progress.
`status.failedActions`:: Shows all failed actions along with a detailed error message.

[id="ztp-image-based-upgrade-cluster-labeling_{context}"]
== Cluster labeling for action completion

If the the `Prep` stage fails or the upgrade is automatically rolled back after a failed upgrade, the {lcao} labels the affected managed clusters with the following labels:

* `lcm.openshift.io/ibgu-<stage>-completed`
* `lcm.openshift.io/ibgu-<stage>-failed`

These labels are removed after successfully cancelling or finalizing the change.
