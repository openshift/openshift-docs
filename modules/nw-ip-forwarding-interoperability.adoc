= IP forwarding interoperability considerations

[role="_abstract"]
When configuring per-interface IP forwarding, consider how it interacts with other {product-title} networking features. Some features have dependencies on IP forwarding settings that can affect their functionality.

== Egress services

Egress services use iptables rules to manage source IP address translation for traffic leaving the cluster. When you configure the Cluster Network Operator with `ipForwarding: Restricted`, these iptables rules might not function as expected because global forwarding is disabled.

To ensure egress services continue to function when using per-interface forwarding instead of global forwarding:

* Verify that the interfaces used by egress traffic have forwarding enabled.
* If you use egress services with MetalLB, ensure that the secondary interfaces used for load balancer traffic have forwarding configured.
* Consider adding the pod CIDR to any allow lists if you implement network policies that might block forwarded traffic.

== MetalLB on secondary networks

MetalLB-backed load balancers on secondary network interfaces require IP forwarding to route incoming traffic to backend pods. When using `ipForwarding: Restricted` mode:

* Traffic arriving on secondary interfaces cannot be forwarded to pods without explicit per-interface forwarding configuration.
* You must enable forwarding on each secondary interface that MetalLB uses for load balancer services.
* Both the `routingViaHost` gateway configuration and per-interface forwarding might be required depending on your network topology.

== OVN-Kubernetes managed interfaces

The following interfaces are managed by OVN-Kubernetes and have their forwarding behavior controlled by the network operator:

* `br-ex` (external bridge)
* Pod network interfaces
* Service network interfaces

Do not configure per-interface forwarding directly on OVN-Kubernetes managed interfaces. Instead, use the Cluster Network Operator `gatewayConfig.ipForwarding` setting to control forwarding behavior for these interfaces.

== Virtual routing and forwarding (VRF)

If you use VRF instances for network segmentation, consider the following interactions with IP forwarding:

* VRF interfaces have their own forwarding domains that are isolated from the main routing table.
* Enabling forwarding on a VRF interface affects traffic within that VRF's routing context only.
* Traffic cannot be forwarded between VRF instances unless you explicitly configure route leaking.

== SR-IOV interfaces

For SR-IOV virtual functions (VFs) used as secondary network interfaces:

* Each VF is a separate interface that might require its own forwarding configuration.
* If pods attached to SR-IOV VFs need to receive forwarded traffic, ensure forwarding is enabled on the VF interface.
* Use the nmstate API with the VF interface name to configure forwarding.

== Network policies

Network policies operate at the pod level and are evaluated before IP forwarding decisions. Consider the following:

* Network policies that deny ingress or egress traffic are enforced regardless of interface forwarding settings.
* Traffic that is permitted by network policies can still be dropped if forwarding is not enabled on the appropriate interfaces.
* Review your network policies to ensure they allow the traffic flows that forwarding enables.

== Troubleshooting forwarding issues

If services are not reachable after configuring per-interface forwarding, verify the following:

. Check that forwarding is enabled on the correct interface:
+
[source,terminal]
----
$ cat /proc/sys/net/ipv4/conf/<interface_name>/forwarding
----

. Verify that the interface has the expected IP configuration:
+
[source,terminal]
----
$ ip addr show <interface_name>
----

. Check for iptables rules that might block forwarded traffic:
+
[source,terminal]
----
$ iptables -L -v -n
----

. Review the routing table to ensure traffic is directed to the correct interface:
+
[source,terminal]
----
$ ip route show
----
