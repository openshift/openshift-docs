[id="rosa-nodes-autonode-prerequisites_{context}"]
= {autonode} prerequisites

[role="_abstract"]
Before you can enable and use the {autonode} feature on {product-title}, you must meet the following requirements.

[id="rosa-nodes-autonode-software-requirements_{context}"]
== Software requirements

{rosa-cli} version::
You must have {rosa-cli} version 1.2.57 or later installed. To check your installed version:
+
[source,terminal]
----
$ rosa version
----
+
To upgrade the {rosa-cli}, see link:https://docs.openshift.com/rosa/rosa_install_access_delete_clusters/rosa-sts-installing-cli.html[Installing the {rosa-cli}].

OpenShift version::
Your {product-title} cluster must run {product-title} version 4.19 or later. {autonode} is not supported on earlier versions.

AWS CLI::
You must have the Amazon Web Services (AWS) command-line interface (CLI) installed and configured to create AWS Identity and Access Management (IAM) policies and roles, and to tag AWS resources. See the link:https://docs.aws.amazon.com/cli/latest/userguide/getting-started-install.html[AWS CLI installation guide].

OpenShift CLI::
You must have the OpenShift CLI (`oc`) installed to verify {autonode} resources and deploy workloads. See link:https://docs.openshift.com/rosa/cli_reference/openshift_cli/getting-started-cli.html[Getting started with the OpenShift CLI].

[id="rosa-nodes-autonode-cluster-requirements_{context}"]
== Cluster requirements

Cluster type::
{autonode} is supported only on {product-title} clusters with hosted control planes (HCP). It is not supported on {product-title} clusters with classic control plane architecture.

Cluster state::
Your cluster must be in the `ready` state before you can enable {autonode}. You can check the cluster state with:
+
[source,terminal]
----
$ rosa describe cluster -c <cluster_name> | grep State
----

Private preview requirements::
During the {autonode} private preview, additional requirements apply:
+
* Your cluster must be deployed to the private preview shard (shard ID: `9f11dd2b-98c1-11f0-8fe5-0a580a830a08`)
* Your cluster must be deployed in the `us-east-1` AWS region
* You must be enrolled in the {autonode} private preview program
+
[NOTE]
====
These restrictions apply only during the private preview period. General availability releases of {autonode} support all {product-title} regions and do not require a specific shard.
====

[id="rosa-nodes-autonode-iam-requirements_{context}"]
== AWS IAM requirements

IAM policy creation permissions::
You must have AWS IAM permissions to create custom IAM policies. The {autonode} feature requires a custom IAM policy with approximately 200 lines of permissions for Amazon Elastic Compute Cloud (Amazon EC2), IAM, Systems Manager (SSM), pricing, and Simple Queue Service (SQS) services.

IAM role creation permissions::
You must have permissions to create IAM roles and attach policies to roles. {autonode} requires a dedicated IAM role that the Karpenter controller assumes through IAM Roles for Service Accounts (IRSA).

OIDC provider access::
You must be able to query the OpenID Connect (OIDC) provider URL for your cluster's OIDC configuration. This is required to create the trust policy for the {autonode} IAM role.

[id="rosa-nodes-autonode-aws-resource-requirements_{context}"]
== AWS resource requirements

VPC and networking::
Your {product-title} cluster must be deployed in an AWS Virtual Private Cloud (VPC) with:
+
* At least one private subnet
* A security group for worker nodes
* Internet connectivity for nodes to pull container images (via Network Address Translation (NAT) gateway or VPC endpoints)

Resource tagging permissions::
You must have permissions to create tags on Amazon EC2 resources, specifically:
+
* Subnets
* Security groups
+
{autonode} uses the `karpenter.sh/discovery` tag to identify which subnets and security groups to use when provisioning nodes.

[id="rosa-nodes-autonode-operator-role-permissions_{context}"]
== Operator role permissions

Control plane operator role::
The operator role for your cluster's control plane must have permissions to create Amazon EC2 tags on resources tagged with `red-hat-managed: true`. During the private preview, you must manually add this permission to the control plane operator role.
+
The required permission is:
+
[source,json]
----
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "VisualEditor0",
            "Effect": "Allow",
            "Action": [
                "ec2:CreateTags"
            ],
            "Resource": "*",
            "Condition": {
                "StringEquals": {
                    "aws:ResourceTag/red-hat-managed": "true"
                }
            }
        }
    ]
}
----
+
[NOTE]
====
This permission allows the HyperShift control plane to tag the cluster's security group with the cluster ID, which {autonode} requires for discovery.
====

[id="rosa-nodes-autonode-knowledge-prerequisites_{context}"]
== Knowledge prerequisites

Kubernetes scheduling concepts::
You should understand Kubernetes pod scheduling concepts, including:
+
* Resource requests and limits
* Node selectors and node affinity
* Pod affinity and anti-affinity
* Taints and tolerations
* Topology spread constraints
+
These concepts are used to express workload placement requirements that {autonode} honors when provisioning nodes.

{product-title} machine pools::
You should understand {product-title} machine pools and when to use them versus {autonode}.

AWS EC2 instance types::
You should be familiar with AWS Amazon EC2 instance families and types to make informed decisions about which instance types to allow in your `NodePool` configurations.

[id="rosa-nodes-autonode-cost-quota-considerations_{context}"]
== Cost and quota considerations

AWS service quotas::
Ensure your AWS account has sufficient service quotas for:
+
* Amazon EC2 instances in the regions where you will provision nodes
* Amazon EC2 volumes
* Elastic network interfaces
+
{autonode} can provision many instances quickly in response to workload demand. Insufficient quotas will cause node provisioning failures.

Cost monitoring::
Plan for cost monitoring and budgeting. {autonode} can provision significant capacity in response to workload demand. Consider:
+
* Setting AWS budget alerts
* Configuring `NodePool` limits to cap maximum capacity
* Monitoring CloudWatch metrics for Amazon EC2 usage
* Using spot instances for cost savings on fault-tolerant workloads
