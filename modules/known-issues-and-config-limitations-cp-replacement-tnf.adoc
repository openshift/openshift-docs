= Known issues and configuration limitations for control plane replacement in TNF

[role="_abstract"]
There are some known issues and configuration limitations you might encounter when performing a control-plane node replacement in a TNF cluster.

Stale installer pods block new installers:: After deleting the node object, you must delete `app=installer` pods in the static-pod operator namespaces (`openshift-etcd`, `openshift-kube-apiserver`, `openshift-kube-controller-manager`, `openshift-kube-scheduler`). See Phase 3, step 4 in the replacement procedure.

Node registration from backup scenarios:: If the replacement node boots from an old disk before Ironic reprovisions it, you must perform additional recovery steps:
+
* OVN chassis cleanup is required (link:https://issues.redhat.com/browse/OCPBUGS-80960[OCPBUGS-80960]).
* CSR approval may need to be performed manually.
* OVN-Kubernetes may need to be restarted on the surviving node to restore connectivity.
+
See xref:tnf-special-recovery-node-registers-from-backup_{context}[Recovering when a node registers from a backup image before reprovisioning] for the full procedure.

CEO degraded after temporary Redfish outage:: If the Redfish interface is temporarily unavailable during the fencing-job window, the CEO may remain in a degraded state. The expected behavior is that the cluster retries the fencing job after Redfish connectivity is restored. If the cluster does not recover automatically, manually clear the degraded condition or restart the CEO.

The Supported configurations are specified in the following table:

[cols="1,2", options="header"]
|===
|Configuration |Support status

|Platform
|Baremetal IPI only. Other platforms are not tested.

|ODF storage
|Not supported on TNF.

|LVMS storage
|Does not support ReadWriteMany (RWX). Live migration features that require RWX storage are not available.

|OpenShift Virtualization (CNV)
|VM high availability requires testing. Live migration requires RWX storage.
|===
