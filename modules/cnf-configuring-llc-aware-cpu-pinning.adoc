= Configuring last-level cache (LLC) aware CPU pinning for low latency workloads

You can configure LLC-aware CPU pinning to improve the performance of low latency workloads. Traditional CPU architectures feature a single LLC, or layer 3 cache, which all cores on the CPU can use. Some modern CPU architectures split this LLC into two or more physical caches to reduce the distance, and therefore latency, between the cores and the LLC.

To take advantage of this split LLC architecture, you can configure a CPU Manager policy to pin workloads to cores that have the lowest latency to the LLC. This configuration can significantly improve performance for low latency workloads.

Without configuring CPU scheduling policies to take advantage of this split LLC architecture you might see reduced performance because of inter-cache access contention and resulting noisy neighbor issues. For example, workloads that are scheduled by the CPU Manager might be assigned to cores that have different access latencies to the LLC.

You can configure the `prefer-align-cpus-by-uncorecache` CPU Manager option in the `KubletConfig` resource to enable LLC-aware pinning. This feature uses a best-effort policy that permits guaranteed containers larger than a LLC domain. If the CPU Manager cannot find a suitable core, it admits the pod to the cluster without LLC-aware pinning. You must set `cpuManagerPolicy: static` to use LLC-aware CPU pinning. 

.Prerequisites

* You have `cluster-admin` role access.
* You installed the OpenShift CLI (`oc`).
* You have a CPU architecture with a split LLC cache.
