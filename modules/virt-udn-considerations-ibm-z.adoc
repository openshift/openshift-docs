= Considerations when running {VirtProductName} on {ibm-z-name}

[role="_abstract"]
When running {VirtProductName} on {ibm-z-name}, the required network configuration depends on the hardware generation and the network adapter in use. Network interfaces on {ibm-z-name} behave differently from standard Ethernet devices, which affects how virtual machine traffic is forwarded in bridge-based setups.

== RoCE adapters

On {ibm-z-name} z17, RoCE adapters support promiscuous mode at the hardware level. When enabled, the RoCE adapter forwards traffic for all virtual machine MAC addresses, eliminating the need to manually register individual MAC addresses.

On earlier {ibm-z-name} generations, RoCE adapters do not support promiscuous mode. Each virtual machine MAC address must be manually registered with the RoCE interface. Traffic destined for unregistered MAC addresses is not forwarded.

To use promiscuous mode on {ibm-z-name} z17, complete the following steps:

. Enable promiscuous mode on the RoCE adapter at the hardware level in the LPAR.

. Enable promiscuous mode on the corresponding network interface by running the following command:
+
[source,terminal]
----
$ ip link set dev <interface> promisc on
----
+
where `<interface>` is the name of the RoCE network interface, for example `ens329`.

* Verify that promiscuous mode is enabled by running the following command:
+
[source,terminal]
----
$ ip link show dev <interface>
----
+
Example output:
+
[source,terminal]
----
3: ens329: <BROADCAST,MULTICAST,PROMISC,UP,LOWER_UP> mtu 1500 qdisc mq state UP mode DEFAULT group default qlen 1000
    link/ether 22:4b:c0:53:05:be brd ff:ff:ff:ff:ff:ff
    altname enp0s0
----
+
The presence of the `PROMISC` flag confirms that promiscuous mode is active.

== OSA and HiperSockets adapters

OSA and HiperSockets interfaces on {ibm-z-name} z17 and earlier generations do not require explicit MAC address registration. However, Linux bridge and Open vSwitch (OVS) bridge setups require additional configuration for virtual machine networking.

In bridge-based configurations, each virtual machine uses its own MAC address. By default, the qeth driver does not forward traffic to these MAC addresses unless you enable specific Virtual NIC Characteristics (VNICC) attributes. If you do not enable these attributes, the qeth driver silently drops packets whose destination MAC address does not match the physical NIC MAC address. This can result in failed or incomplete network connectivity for virtual machines.

You can enable the required VNICC attributes manually through sysfs by running the following commands:

[source,terminal]
----
$ echo 1 > /sys/devices/qeth/0.0.1100/vnicc/flooding
----

[source,terminal]
----
$ echo 1 > /sys/devices/qeth/0.0.1100/vnicc/mcast_flooding
----

[source,terminal]
----
$ echo 1 > /sys/devices/qeth/0.0.1100/vnicc/learning
----

You can also enable MAC address learning by running the following command:

[source,terminal]
----
$ sudo chzdev <device_bus_id> vnicc/learning=1
----

where:

`chzdev`:: Specifies the tool to configure {ibm-z-name} devices.
`<device_bus_id>`:: Specifies the ID of the qeth network device, for example `0.0.1100`.
`vnicc/learning=1`:: Enables MAC address learning.
