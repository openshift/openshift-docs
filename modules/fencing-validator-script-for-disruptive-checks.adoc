= Fencing validator script for disruptive checks

[role="_abstract"]
You can validate your cluster's resilience and perform disruptive checks from a peer node by using the `fencing_validator` script. By executing these simulated failures, you can ensure your high-availability environment correctly isolates and recovers from errors.

[source,terminal]
----
$ pcs stonith fence <node>
----

You can check whether Two-Node with Fencing (TNF) cluster setup actually works by running the following command: 

[source,terminal]
----
$ oc debug node/<node-name> --chroot /host /usr/local/bin/fencing_validator --disruptive
----

[WARNING]
====
The `--disruptive` flag fences each control-plane node one at a time and verifies recovery. The flag causes temporary workload disruption. Only use this during maintenance windows or initial validation. The `--disruptive` flag actually performs the Shoot The Other Node In The Head (STONITH) fence operations such as power cycle or VM reset, and not a graceful shutdown.
====

The fencing validator script with `--disruptive` flag runs the following checks: 

. **Fence Node A** - Triggers STONITH to reboot the first control-plane node.
. **Verify NotReady** - Waits for Kubernetes to report the node as `NotReady`, which confirms the reboot happened.
. **Verify recovery** - Waits for the node to come back to the `Ready` state, rejoin the Pacemaker cluster as `ONLINE`, and for etcd to regain quorum.
. **Post-recovery daemon check** - Re-checks all daemons are healthy after recovery.
. **Fence Node B** - Repeats the same process for the second node.
. **Verify NotReady, recovery, and daemons** - Same recovery validation for Node B.
