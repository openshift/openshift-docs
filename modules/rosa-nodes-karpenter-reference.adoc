= Karpenter API reference

[role="_abstract"]
When you create or edit `NodePool` and `OpenShiftEC2NodeClass` custom resources, you must conform to the following engineering validations, structural constraints, and resources hierarchy.

For `NodePool` resource constraints and scheduling validations. see the following list:

* *Instance sizing validation:* Select from the supported list of Amazon `EC2` instances. The selected `EC2` instance must meet the minimum resource requirements of 4 vCPUs or more. To enforce this validation on your cluster, include the `karpenter.k8s.aws/instance-cpu` key in your `spec.template.spec.requirements` block.
* *Reserved Kubernetes labels:* In the `NodePool` configurations that you create, avoid using the following reserved Kubernetes labels:
** `node-role.kubernetes.io/master`
** `node-role.kubernetes.io/infra`
** `node-role.kubernetes.io/control-plane`
* *Static capacity restrictions:* Setting the `spec.replicas` field or `StaticCapacity` is not supported. Karpenter must control scheduling dynamically.
* *Workload interruption defaults:* Review the default values of the custom resources managed by Karpenter. For example, Karpenter implements the spot market type and automatically expires nodes after 30 days by default. If your workloads must not be interrupted, you must configure the custom resources explicitly to modify these behaviors.

For `OpenShiftEC2NodeClass` immutability and lifecycle rules, see the following list: 

* *Resource immutability:* The `OpenShiftEC2NodeClass` resource is immutable. Use this resource only for referencing purposes inside the `NodePool` resources that you create.
* *Default node classes:* A default `OpenShiftEc2NodeClass` is created automatically when you enable the AutoNode feature in the cluster. This default resource is completely immutable. All `NodePool` resources that reference this default `OpenShiftEc2NodeClass` upgrade automatically when the hosted control plane upgrades.
* *Control plane version pinning:* All `NodePool` resources that do not specify a `spec.version` field upgrade automatically alongside the hosted control plane. If you prefer to prevent automated upgrades for specific node pools, you must pin the version of the `OpenShiftEc2NodeClass` resource by using the `spec.version` field.
* *Reserved AWS tags:* Avoid using reserved AWS tags within your node class configurations. The following tag patterns are strictly prohibited:
** `red-hat-managed`
** `api.openshift.com/*`
** `kubernetes.io/cluster/*`

For storage and networking infrastructure requirements, see the following list:

* *Root volume constraints:* Within the `OpenShiftEC2NodeClass` specification, the minimum allowed root volume size is `100 GiB`.
* *Volume encryption validation:* When you customize the root volume disk settings, ensure that disk encryption is turned on. You can use a custom key managed by the AWS Key Management Service (KMS) to encrypt these root volumes.
* *Public IP address support:* The association of public IP addresses for worker nodes is not supported. Avoid setting the `associatePublicIPAddress` field to `true` on any `OpenShiftEC2NodeClass` that you create.

For resource hierarchy, see the following list:

* *De-provisioning sequence:* When you remove nodes managed by Karpenter, clean up the `NodePool` custom resources and the applications running on them before cleaning up any custom `OpenShiftEc2NodeClass` that the `NodePool` reference.

For more information about Karpenter API specifications, see the following resources:

* Karpenter `NodePool` specification: The structural definitions, core scheduling constraints, and expiration properties for the `NodePool` type are managed by the core open-source schema tracking engines. For more information, see link:https://karpenter.sh/docs/concepts/nodepools/#spectemplatespecterminationgraceperiod[Karpenter NodePool specifications].

* Kaprenter `NodePool` concepts: For general information on Karpenter `NodePools` see link:https://karpenter.sh/docs/concepts/nodepools/[Karpenter NodePool overview].

* Red{nbsp}Hat {OCP-short} `EC2NodeClass` specification: The platform infrastructure layer constraints, security group configurations, and subnet selectors are defined by engineering for {hcp}. To validate downstream schema parameters, data types, and structural changes inside the Go engine models, see the link:https://github.com/openshift/hypershift/tree/main/api[HyperShift API Definitions Repository].
