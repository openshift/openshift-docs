= Enabling OVS balance-slb mode for your cluster

You can enable the Open vSwitch (OVS) `balance-slb` mode on infrastructure where your cluster runs so that two or more physical interfaces can share their network traffic. A `balance-slb` mode interface provides source load balancing (SLB) capabilities for a cluster that runs virtualization workloads, where the interface can act independently without needing to communicate with a network switch.

Source load balancing works by assigning a Media Access Control (MAC) address and a virtual local area network (vLAN), if required, to a bond interface, such as `br-phy`. Because of the shared MAC address and vLAN between interfaces, using `balance-slb` mode to share pod traffic has no benefit.

The following diagram shows `balance-slb` mode on a simple cluster infrastructure layout. Virtual machines (VMs) connect to specific localnet `NetworkAttachmentDefinition` (NAD) custom resource definition (CRDs), `NAD 0` or `NAD 1`. Each NAD provides VMs with access to network traffic, such as VLAN ID tags. A `br-ex` OVS bridge receives traffic from VMs and passes the traffic to the next OVS bridge, `br-phy`. The `br-phy` bridge functions as the controller for the SLB bond, `bond 0`. The SLB bond balances traffic from different VM ports over the physical interface links, such as `eno0` and `eno1`. Additionally, ingress traffic from either physical interface can pass through the set of OVS bridges to reach the VMs.

.OVS `balance-slb` mode ` operating on a localnet with two NADs
image::552_OpenShift_slb_mode_0525.png[OVS `balance-slb` mode ` operating on a localnet with two NADs]

You can integrate the `balance-slb` mode interface into primary or secondary network types by using OVS bonding. Note the following points about OVS bonding:

* Supports the OVN-Kubernetes CNI plugin and easily integrates with the plugin.
* Natively supports `balance-slb` mode. 
* Diverts all node traffic to a single `balance-slb` mode interface. 
* Cannot use the method external to your {product-title} cluster. 

[IMPORTANT]
====
If you use the kernel bonding to enable `balance-slb` mode for your cluster, Network Manager might experience Gratuitous Address Resolution Protocol (GARP) handling issues. These issues occur when the Network Manager cannot retrieve source MAC addresses from the OVS bridge interface because of a broken link to a port disables the port.
====

.Prerequisites

* You have logged in to the {product-title} web console.
* You have installed the {oc-first}.
* You have logged into {product-title} as a cluster administrator.
* You have more than one physical interfaces attached to your primary network and you defined the interfaces in the `MachineConfig` file.
ifndef::postinstall-bare-metal-ipi[]
* You created a manifest object and defined a customized `br-ex` bridge in the object configuration file.
endif::postinstall-bare-metal-ipi[]
ifdef::postinstall-bare-metal-ipi[]
* You have more than one physical interfaces attached to your primary network and you defined the interfaces in the NNCP or NAD file.
* You created a NNCP CR and defined a customized `br-ex` bridge network configuration in the object configuration file.
endif::postinstall-bare-metal-ipi[]

.Procedure

ifndef::postinstall-bare-metal-ipi[]
. Configure the `install-config.yaml` configuration file by adding a `networkConfig` section similar to the following example to the file. The `networkConfig` section outlines interfaces that interact with the bond interface.
+
[source,yaml]
----
networkConfig: &BOND
  interfaces:
    - name: enp1s0 <1>
      type: interface
      state: up
      ipv4:
        enabled: true
        dhcp: true
      ipv6:
        enabled: false
    - name: enp2s0
      type: interface
      state: up
      ipv4:
        dhcp: true
        enabled: true
      ipv6:
        enabled: false
    - name: enp3s0
      type: interface
      state: down
      ipv4:
        enabled: false
      ipv6:
        enabled: false
# ...
----
<1> The interface that pulls in the Ignition config file for the bond interface.

. Go to the `./<installation_directory>/auth` directory and modify the `master.ign` and `worker.ign` files and add a `balance-slb` configuration to these files.
endif::postinstall-bare-metal-ipi[]

ifdef::postinstall-bare-metal-ipi[]
. Create a `pull-secret.txt` configuration file and add a configuration similar to the following example to the file:
+
[source,yaml]
----
networkConfig: &BOND
  interfaces:
    - name: enp1s0
      type: interface
      state: up
      ipv4:
        enabled: true
        dhcp: true
      ipv6:
        enabled: false
    - name: enp2s0
      type: interface
      state: up
      ipv4:
        dhcp: true
        enabled: true
      ipv6:
        enabled: false
    - name: enp3s0
      type: interface
      state: down
      ipv4:
        enabled: false
      ipv6:
        enabled: false
# ...
----

. Specify the path to your `pull-secret.txt` in the `pullSecret` parameter of your NNCP or NAD file.
endif::postinstall-bare-metal-ipi[]

. Open your NMState configuration file in your editor and add the following configuration in the file:
+
[source,yaml]
----
# ...
interfaces:
# ...
- name: br-phy <1>
  type: ovs-bridge
  state: up
  ipv4:
    enabled: false
    dhcp: false
  ipv6:
    enabled: false
    dhcp: false
  bridge:
    allow-extra-patch-ports: true
    port:
    - name: patch-phy-to-ex
    - name: ovs-bond
      link-aggregation:
        mode: balance-slb <2>
        port:
        - name: eno1
        - name: eno2
- name: patch-ex-to-phy <3>
  type: ovs-interface
  state: up
  patch:
    peer: patch-phy-to-ex
- name: patch-phy-to-ex <4>
  type: ovs-interface
  state: up
  patch:
    peer: patch-ex-to-phy
- name: eno1 <5>
  type: ethernet
  state: up
  ipv4:
    enabled: false
  ipv6:
    enabled: false
- name: eno2 <5>
  type: ethernet
  state: up
  ipv4:
    enabled: false
  ipv6:
    enabled: false
# ...
----
<1> Specify an additional bridge, such as `br-phy`, that isolates the `br-ex` bond interface from any changes to the original network rules that OVN-Kubernetes might make to the network environment.
<2> Enables `balance-slb` mode for the OVS bridge interface.
<3> Specify the OVS virtual interface that uses an uplink switch to connect the additional bridge with the br-ex bridge.
<4> Specify the OVS virtual interface that uses an uplink switch to connect the br-ex bridge with the additional bridge.
<5> Specify the physical interfaces that exist on your primary network.

. For static IP address assignment, manually reboot each node that exists in your cluster. For DHCP address assignment, you can reboot all nodes in one reboot operation.

. Optional: If after the reboot operation and you waited a few minutes, a node or some nodes do not have the `READY` status, enter the following commands:
+
.. Check the current state of an Open vSwitch (OVS) configuration, especially details about bridges, ports, and interfaces, by entering the following command:
+
[source,terminal]
----
$ ovs-vsctl show
----
+ 
.. If you identify the `Bridge br-ex` entry in the output, delete the entry by running the following command:
+
[source,terminal]
----
$ ovs-vsctl del-br br-ex
----
