= Manually recovering from a disruption event when fencing is unavailable

You might need to perform manual recovery steps if a disruption event prevents fencing from functioning correctly. In these cases, you can run commands directly on the control plane nodes to recover the cluster.

.Prerequisites

* You have administrative access to the control plane nodes.
* You can connect to the nodes by using SSH.

.Procedure

. Start the `podman-etcd` agent manually on a control plane node by running the following command:
+
[source,terminal]
----
$ sudo systemctl start podman-etcd
----
+
[NOTE]
====
Run this command only if a node cannot be fenced automatically and etcd is unavailable. Perform manual intervention only when automated recovery is not possible.
====

. Troubleshoot issues that are specific to two-node fencing. You can run the following commands to verify whether the fencing components are running correctly and generate logs for diagnosis:
+
[source,terminal]
----
$ oc get pods -n openshift-machine-api
$ oc logs -n openshift-machine-api <fencing_pod_name>
----

. Update the fencing secret if the BMC credentials change by running the following command:
+
[source,terminal]
----
$ oc -n openshift-machine-api delete secret cluster-fencing-credentials
$ oc -n openshift-machine-api create secret generic cluster-fencing-credentials \
  --from-file=credentials.yaml=<path_to_updated_credentials>
----
+
This action refreshes the fencing configuration with the updated BMC credentials.
