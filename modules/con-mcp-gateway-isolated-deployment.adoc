= Isolated {mcpg} deployment overview

[role="_abstract"]
You can deploy multiple {mcpg} instances within a single cluster to isolate traffic between teams. Each deployment uses its own `MCPGatewayExtension` custom resource (CR) to define which `Gateway` object the {mcpg} instance is responsible for. This allows different teams to manage their own MCP servers independently.

The `MCPGatewayExtension` CR has the following responsibilities:

* Defines which `Gateway` object the {mcpg} instance is responsible for.
* Determines where configuration `Secret` objects are created, which is the same namespace as the `MCPGatewayExtension` CR.
* Enables isolation by allowing multiple {mcpg} instances in different namespaces to target different `Gateway` objects.

The following diagram shows the architecture of an isolated deployment with two teams:

----
┌──────────────────────────────────────────────────────────────────────┐
│                     MCP Gateway Controller                           │
│                        (cluster-wide)                                │
└──────────────────────────────────────────────────────────────────────┘
                                    │
              ┌─────────────────────┴─────────────────────┐
              ▼                                           ▼
┌───────────────────────────────┐     ┌───────────────────────────────┐
│          Team A NS            │     │          Team B NS            │
├───────────────────────────────┤     ├───────────────────────────────┤
│ MCPGatewayExtension           │     │ MCPGatewayExtension           │
│   → Gateway A                 │     │   → Gateway B                 │
│ MCP Broker/Router             │     │ MCP Broker/Router             │
│ Config Secret                 │     │ Config Secret                 │
│ MCPServerRegistrations        │     │ MCPServerRegistrations        │
└───────────────────────────────┘     └───────────────────────────────┘
----

Each `MCPGatewayExtension` CR must target a different `Gateway` object. The controller creates configuration `Secret` objects in the same namespace as each valid `MCPGatewayExtension` CR. Those secrets are mounted into the broker and router deployments.

For cross-namespace `Gateway` references, a `ReferenceGrant` CR must exist in the target `Gateway` object namespace.

[id="con-mcp-gateway-isolated-deployment-limitations_{context}"]
== Limitations

Review the following limitations before you deploy multiple isolated {mcpg} instances.

Broker and router must be co-located with MCPGatewayExtension:: The {mcpg} instance, including the broker and router, must be deployed in the same namespace as the `MCPGatewayExtension` CR. The controller writes the configuration `Secret` object to the `MCPGatewayExtension` CR namespace, and the broker and router mount this secret.

One MCPGatewayExtension per namespace:: Each namespace can have only one `MCPGatewayExtension` CR. The controller writes configuration to a well-known `Secret` object name, so multiple extensions in the same namespace overwrite each other.

One MCPGatewayExtension per Gateway:: Only one `MCPGatewayExtension` CR can target a given `Gateway` object. If multiple extensions target the same `Gateway` object, the controller marks newer ones as conflicted. The oldest extension by creation timestamp takes precedence.
