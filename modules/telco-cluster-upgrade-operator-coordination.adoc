= Coordinate OLM operator upgrades with cluster upgrades

[role="_abstract"]
You can coordinate Operator Lifecycle Manager (OLM) operator upgrades with {product-title} cluster upgrades to ensure operator compatibility.
Incompatible operators can block cluster upgrades or cause degraded cluster operators after upgrades complete.

Update operator subscriptions before or immediately after control plane upgrades to maintain operator compatibility with the new cluster version.

.Prerequisites

* Understanding of OLM concepts including subscriptions, channels, and ClusterServiceVersions (CSVs).
* Access to clusters with cluster-admin privileges.
* Cluster upgrade is planned or in progress.

.Procedure

. Check installed operator versions before upgrading the cluster:
+
[source,terminal]
----
$ oc get csv -A
----
+
.Example output
[source,terminal]
----
NAMESPACE                              NAME                                          VERSION   PHASE
openshift-sriov-network-operator       sriov-network-operator.v4.19.0                4.19.0    Succeeded
openshift-ptp                          ptp-operator.v4.19.0                          4.19.0    Succeeded
openshift-performance-addon-operator   performance-addon-operator.v4.19.0            4.19.0    Succeeded
----

. Check current operator subscription channels:
+
[source,terminal]
----
$ oc get subscription -A
----
+
.Example output
[source,terminal]
----
NAMESPACE                              NAME                                PACKAGE                  SOURCE             CHANNEL
openshift-sriov-network-operator       sriov-network-operator-subscription sriov-network-operator   redhat-operators   stable-4.19
openshift-ptp                          ptp-operator-subscription           ptp-operator             redhat-operators   stable-4.19
----

. Review operator compatibility matrices in the {product-title} release notes:
+
Identify which operator channels support your target {product-title} version.
+
.Example compatibility
[cols="2,2,2",options="header"]
|===
|Operator
|{product-title} 4.19
|{product-title} 4.20

|SR-IOV Network Operator
|stable-4.19
|stable-4.20

|PTP Operator
|stable-4.19
|stable-4.20

|Performance Addon Operator
|stable-4.19
|stable-4.20 (deprecated, use Node Tuning Operator)
|===

. Identify operators that need channel updates before the cluster upgrade:
+
For a cluster upgrading from 4.19 to 4.20, identify operators still on 4.19 channels.

. Update operator subscription channels for the target {product-title} version:
+
For example, update the SR-IOV operator channel:
+
[source,terminal]
----
$ oc patch subscription sriov-network-operator-subscription \
  -n openshift-sriov-network-operator \
  --type merge \
  -p '{"spec":{"channel":"stable-4.20"}}'
----
+
Update the PTP operator channel:
+
[source,terminal]
----
$ oc patch subscription ptp-operator-subscription \
  -n openshift-ptp \
  --type merge \
  -p '{"spec":{"channel":"stable-4.20"}}'
----
+
Verify the subscription channel updates:
+
[source,terminal]
----
$ oc get subscription -n openshift-sriov-network-operator -o jsonpath='{.items[0].spec.channel}'
----
+
.Example output
[source,terminal]
----
stable-4.20
----

. Monitor operator upgrades after updating channels:
+
[source,terminal]
----
$ oc get csv -n openshift-sriov-network-operator -w
----
+
The operator creates a new ClusterServiceVersion and transitions to the new version.
+
Wait for the new CSV to appear and reach Succeeded phase:
+
[source,terminal]
----
$ oc get csv -n openshift-sriov-network-operator -o jsonpath='{.items[?(@.spec.version=="4.20.0")].status.phase}'
----
+
.Example output
[source,terminal]
----
Succeeded
----

. Verify operators are at compatible versions before upgrading the cluster:
+
[source,terminal]
----
$ oc get csv -A | grep -v Succeeded
----
+
All operators should show `Succeeded` phase.

. After the control plane upgrade completes, check for pending operator updates:
+
[source,terminal]
----
$ oc get csv -A
----
+
Some operators may not update until after the cluster upgrade completes.

. Approve manually-approved operators if necessary:
+
If operators are configured with manual approval, approve pending install plans:
+
[source,terminal]
----
$ oc get installplan -A
----
+
[source,terminal]
----
$ oc patch installplan <install-plan-name> \
  -n <namespace> \
  --type merge \
  -p '{"spec":{"approved":true}}'
----

. Verify operator health after cluster upgrade:
+
[source,terminal]
----
$ oc get csv -A
----
+
All operators should show `Succeeded` phase.

. Check operator pods are running:
+
[source,terminal]
----
$ oc get pods -n openshift-sriov-network-operator
$ oc get pods -n openshift-ptp
$ oc get pods -n openshift-performance-addon-operator
----

. Verify Telco-specific custom resources are reconciled successfully:
+
Check Performance Profiles:
+
[source,terminal]
----
$ oc get performanceprofile -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.status.conditions[?(@.type=="Applied")].status}{"\n"}{end}'
----
+
Check PTP configurations:
+
[source,terminal]
----
$ oc get ptpconfig -n openshift-ptp
----
+
Check SR-IOV network node policies:
+
[source,terminal]
----
$ oc get sriovnetworknodepolicy -n openshift-sriov-network-operator
----
+
All custom resources should show expected status.

== Telco-specific operator considerations

.Performance Addon Operator

The Performance Addon Operator is deprecated starting with {product-title} 4.20.
The Node Tuning Operator now provides performance tuning capabilities.

If upgrading to 4.20 or later, plan to migrate from Performance Addon Operator to Node Tuning Operator:

. Keep Performance Profiles until the migration is complete.
. Node Tuning Operator automatically manages existing Performance Profiles.
. Verify that performance tuning remains effective after the upgrade.

.SR-IOV Network Operator

The SR-IOV Network Operator may require node reboots when upgrading to new versions.
Schedule operator upgrades during maintenance windows to avoid unexpected node reboots.

Check for pending node reboots:
[source,terminal]
----
$ oc get sriovnetworknodepolicy -n openshift-sriov-network-operator -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.status.syncStatus}{"\n"}{end}'
----

.PTP Operator

PTP configurations are sensitive to version changes.
Validate PTP synchronization after operator upgrades:

[source,terminal]
----
$ oc get pods -n openshift-ptp -l app=linuxptp-daemon -o wide
$ oc logs -n openshift-ptp <linuxptp-daemon-pod>
----

Check PTP synchronization status in pod logs.

== Operator upgrade workflow example

The recommended workflow for coordinating operator and cluster upgrades:

. *Pre-upgrade (before cluster upgrade)*:
+
Update operator subscription channels to versions compatible with the target {product-title} release.

. *During control plane upgrade*:
+
Do not modify operator subscriptions or approve pending install plans while the control plane is upgrading.

. *After control plane upgrade*:
+
Verify operators updated automatically, or approve manual install plans for manually-approved operators.

. *After worker unpause*:
+
Verify all operators are healthy and custom resources are reconciled successfully.

.Verification

. Verify all operators are at expected versions:
+
[source,terminal]
----
$ oc get csv -A
----

. Verify operator pods are running without errors:
+
[source,terminal]
----
$ oc get pods -A | grep -E "openshift-sriov|openshift-ptp|openshift-performance"
----

. Verify custom resources are applied successfully:
+
[source,terminal]
----
$ oc get performanceprofile,ptpconfig,sriovnetworknodepolicy -A
----

.Troubleshooting

* If an operator blocks the cluster upgrade, update the operator subscription channel before upgrading the cluster.
* If an operator is stuck in `Installing` phase, check catalog source availability:
+
[source,terminal]
----
$ oc get catalogsource -n openshift-marketplace
----

* If a custom resource is not reconciling, check operator logs:
+
[source,terminal]
----
$ oc logs -n <operator-namespace> deployment/<operator-deployment>
----

* If operator pods are in `CrashLoopBackOff`, check for compatibility issues with the new cluster version.
