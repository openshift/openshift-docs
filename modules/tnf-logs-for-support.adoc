= TNF logs for support

[role="_abstract"]
When a Two-Node with Fencing (TNF) control plane node fails, collect the logs to diagnose etcd, networking, and high-availability issues before contacting Red Hat support.

etcd logs::

Records consensus, leader election, and key-value store activity for the etcd cluster.

[source,bash]
----
# TNF: etcd runs as podman container managed by pacemaker, not Kubernetes pod
# SSH to node
$ sudo podman logs etcd
# Or from journalctl (pacemaker manages the container)
$ sudo journalctl -u pacemaker -n 1000 | grep etcd
----

CEO logs::
Captures the etcd operator's actions for managing and reconciling etcd cluster state.

[source,bash]
----
$ oc logs -n openshift-etcd-operator deployment/etcd-operator --all-containers=true
----

Pacemaker logs::

Tracks the high-availability resource manager controlling containerized etcd on TNF nodes.

[source,bash]
----
# SSH to node
$ sudo journalctl -u pacemaker -n 500
----

OVN-Kubernetes logs::

Shows the OVN network controller's activity for pod networking and traffic policy enforcement.

[source,bash]
----
$ oc logs -n openshift-ovn-kubernetes <ovnkube-node-pod> -c ovnkube-controller
----

Must-gather::

Bundles a full snapshot of cluster diagnostics, logs, and configs for Red Hat support.

[source,bash]
----
$ oc adm must-gather --dest-dir=./must-gather
----

For TNF-specific debugging, gather pacemaker status and logs from both nodes from SSH.
