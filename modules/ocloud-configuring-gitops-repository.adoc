= Configure the GitOps repository for cluster templates

[role="_abstract"]
Configure a Git repository with a directory layout for cluster and policy templates, configuration policies, and hardware profiles. ArgoCD synchronizes this repository content to the hub cluster where the O-Cloud Manager uses it for cluster provisioning.

[NOTE]
====
This repository layout differs from the {ztp-first} GitOps layout. In {ztp}, the Git repository contains a `ClusterInstance` CR for each site, meaning every cluster has its own CR in the repository. In the O-Cloud Manager flow, the repository contains reusable blueprints and configuration resources that define how clusters are provisioned by default. Individual clusters are instantiated with site-specific parameters by creating `ProvisioningRequest` resources on the hub cluster, not by committing site-specific YAML to Git.
====

.Prerequisites

* The O-Cloud Manager Operator is installed on the hub cluster.
* The {gitops-title} Operator is installed.
* You have a Git repository accessible by ArgoCD on the hub cluster.

.Procedure

. Create your Git repository. See the following recommended directory layout:
+
[source,text]
----
├── clustertemplates/
│   ├── hardwareprofiles/
│   ├── inventory/
│   └── version_4.Y.Z/
│       ├── sno-ran-du/
│       │   ├── clusterinstance-defaults-v1.yaml
│       │   ├── policytemplates-defaults-v1.yaml
│       │   ├── pull-secret.yaml
│       │   ├── extra-manifest/
│       │   ├── ns.yaml
│       │   └── sno-ran-du-v4-Y-Z-1.yaml
│       └── std-ran-du/
│           ├── clusterinstance-defaults-v1.yaml
│           ├── policytemplates-defaults-v1.yaml
│           ├── pull-secret.yaml
│           ├── ns.yaml
│           └── std-ran-du-v4-Y-Z-1.yaml
└── policytemplates/
    └── version_4.Y.Z/
        ├── sno-ran-du/
        │   ├── ns.yaml
        │   ├── msc-binding.yaml
        │   └── sno-ran-du-pg-v4-Y-Z-v1.yaml
        ├── std-ran-du/
        │   ├── ns.yaml
        │   ├── msc-binding.yaml
        │   └── std-ran-du-pg-v4-Y-Z-v1.yaml
        └── source-crs/
----
+
* `clustertemplates/` contains resources that define how clusters are provisioned:
** `hardwareprofiles/` contains `HardwareProfile` custom resources that define firmware levels and BIOS settings for classes of bare-metal servers.
** `inventory/` contains `BareMetalHost` resources for sites where server inventory is managed through GitOps.
** `version_4.Y.Z/` organizes content by the target {product-title} release version.
** `sno-ran-du/` and `std-ran-du/` are template directories for single-node and standard multi-node cluster topologies respectively.
** `sno-ran-du-v4-Y-Z-1.yaml` is a `ClusterTemplate` custom resource that defines the cluster blueprint, including the parameter schema, node group definitions, and references to default ConfigMaps.
** `clusterinstance-defaults-v1.yaml` is a `ConfigMap` containing default values for cluster installation parameters such as network configuration, image sets, and node settings. The `ProvisioningRequest` can override or supplement these values with site-specific parameters.
** `policytemplates-defaults-v1.yaml` is a `ConfigMap` containing default values for Day-2 configuration policy parameters such as CPU allocation, hugepages, and SR-IOV settings.
** `extra-manifest/` contains additional manifests applied during cluster installation.
* `policytemplates/` contains `PolicyGenerator` resources that define Day-2 configuration policies applied to managed clusters after provisioning:
** `sno-ran-du-pg-v4-Y-Z-v1.yaml` is a `PolicyGenerator` resource that references source CRs and generates the policies enforced on the spoke cluster.
** `source-crs/` contains the source CRs, such as subscriptions, performance profiles, and SR-IOV configurations, referenced by the policy generators.

. Create an ArgoCD `Application` resource for cluster templates that points to the `clustertemplates/` directory in your Git repository:
+
[source,yaml]
----
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: clusters
  namespace: openshift-gitops
spec:
  destination:
    server: https://kubernetes.default.svc
  project: default
  source:
    path: clustertemplates
    repoURL: https://git.example.com/your-org/ocloud-templates.git
    targetRevision: main
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
  ignoreDifferences:
  - group: metal3.io
    jsonPointers:
    - /spec/preprovisioningNetworkDataName
    - /spec/online
    kind: BareMetalHost
----
+
* `spec.source.path` points to the `clustertemplates/` directory that contains `ClusterTemplate` resources, defaults, and hardware profiles.
* `spec.ignoreDifferences` prevents ArgoCD from reverting fields on `BareMetalHost` resources that the Bare Metal Operator updates dynamically during provisioning.

. Create an ArgoCD `Application` resource for policy templates that points to the `policytemplates/` directory:
+
[source,yaml]
----
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: policies
  namespace: openshift-gitops
spec:
  destination:
    server: https://kubernetes.default.svc
  project: default
  source:
    path: policytemplates
    repoURL: https://git.example.com/your-org/ocloud-templates.git
    targetRevision: main
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
----
+
* `spec.source.path` points to the `policytemplates/` directory that contains ACM policy generator resources and source custom resources applied to managed clusters after provisioning.

. Add the O-Cloud Manager CRs to the ArgoCD `AppProject` resource under `spec.clusterResourceWhitelist`:
+
[source,yaml]
----
spec:
  clusterResourceWhitelist:
  - group: clcm.openshift.io
    kind: ClusterTemplate
  - group: clcm.openshift.io
    kind: HardwareProfile
----

. Apply the ArgoCD application resources:
+
[source,terminal]
----
$ oc apply -f <clusters_application_filename>.yaml
$ oc apply -f <policies_application_filename>.yaml
----

.Verification

* Verify that ArgoCD has synchronized the cluster templates to the hub cluster:
+
[source,terminal]
----
$ oc get clustertemplates.clcm.openshift.io -A
----
+
The output lists the `ClusterTemplate` resources with their validation status. A `ClusterTemplateValidated` condition of `True` indicates that the template is ready for use.
