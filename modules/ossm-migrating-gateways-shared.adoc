= Shared gateway migration (In Place)

Follow this procedure if your environment is using a centralized gateway shared across multiple namespaces. This procedure uses `istio-ingress` as an example.

[NOTE]
====
The label for the gateway namespace differs between mulitenant and cluster-wide meshes. Return to the either the multitenant or cluster-wide migration guide to understand which labels to use in your specific migration case, and to see if the `maistra.io/ignore-namespace: "true"` label should be removed.
====

.Procedure

. Label the gateway namespace to ensure injection from the new mesh is enabled for the namespace

. Add the `maistra.io/ignore-namespace: "true"` label by running the following command:
+
[source,terminal]
----
$ oc label namespace istio-ingress istio.io/rev=${ISTIO_REVISION} maistra.io/ignore-namespace="true"
----

. If necessary, remove `istio-injection=enabled` label.

. Restart the gateway deployment by running the following command:
+
[source,terminal]
----
$ oc -n istio-ingress rollout restart deployment ${GATEWAY_NAME}
----

. Check that the gateway is running the new revision by running the following command:
+
[source,terminal]
----
$ oc `istioctl ps -n istio-ingress`
----

. Verify gateway pod is running with new revision.

. Test application-specific routes.
