= Performing node maintenance on a hosting cluster

[role="_abstract"]
Perform maintenance on a hosting cluster node, such as applying operating system updates or hardware repairs, without disrupting the target clusters.

When you drain a hosting cluster node, the control plane VM on that node shuts down.
If pod disruption budgets and anti-affinity rules are configured correctly, only one control plane VM for each target cluster is affected, and `etcd` quorum is preserved.

.Prerequisites

* Control plane VMs are distributed across hosting cluster nodes by using anti-affinity rules.
* A `PodDisruptionBudget` with `minAvailable: 2` is configured for each target cluster's control plane VMs.
* The VM eviction strategy is set to `None` for all control plane VMs.

.Procedure

. Identify which control plane VMs are running on the node you plan to maintain by running the following command:
+
[source,terminal]
----
$ oc get pods --field-selector spec.nodeName=<node_name> -n <vm_namespace> -l vcp-cluster=<cluster_name>
----
+
[NOTE]
====
Use `oc get pods` with the `spec.nodeName` field selector to filter by node.
The `oc get vmi` command does not support the `status.nodeName` field selector.
====

. Verify that all target clusters affected by this node have healthy `etcd` quorum before proceeding by running the following command:
+
[source,terminal]
----
$ oc --kubeconfig=<target_cluster_kubeconfig> get etcd -o=jsonpath='{range .items[0].status.conditions[?(@.type=="EtcdMembersAvailable")]}{.message}{end}'
----

. Cordon the node to prevent new workloads from being scheduled by running the following command:
+
[source,terminal]
----
$ oc adm cordon <node_name>
----

. Drain the node to evict all pods, including control plane VM pods, by running the following command:
+
[source,terminal]
----
$ oc adm drain <node_name> --ignore-daemonsets --delete-emptydir-data
----
+
The `PodDisruptionBudget` ensures that only one control plane VM per target cluster is evicted.
The drain operation blocks until the PDB allows eviction.

. Verify that the remaining control plane VMs maintain `etcd` quorum on each affected target cluster by running the following command:
+
[source,terminal]
----
$ oc --kubeconfig=<target_cluster_kubeconfig> get etcd -o=jsonpath='{range .items[0].status.conditions[?(@.type=="EtcdMembersAvailable")]}{.message}{end}'
----

. Perform the required maintenance on the physical node.

. Uncordon the node to allow workloads to be scheduled again by running the following command:
+
[source,terminal]
----
$ oc adm uncordon <node_name>
----

. Verify that the control plane VM restarts on the node and rejoins the target cluster by running the following command:
+
[source,terminal]
----
$ oc get vmi -A -l redfish-enabled=true
----
+
All control plane VMs must show a `Running` phase.
+
[NOTE]
====
Use `oc get vmi` (VirtualMachineInstance) instead of `oc get vm` (VirtualMachine) to verify the actual runtime state.
The `VirtualMachine` resource reflects the desired state, while the `VirtualMachineInstance` resource reflects whether the VM is currently running.
====

.Verification

* Verify `etcd` health on each affected target cluster:
+
[source,terminal]
----
$ oc --kubeconfig=<target_cluster_kubeconfig> get etcd -o=jsonpath='{range .items[0].status.conditions[?(@.type=="EtcdMembersAvailable")]}{.message}{end}'
----
