= Control who can access your published API

[role="_abstract"]
You can review access requests to verify that consumers have legitimate use cases before granting credentials when you publish an API with manual approval. This helps you know who is using your API so that you can prevent unauthorized or inappropriate usage.

.Prerequisites

* You enabled the {prodname} {ocp} web console plugin.
* You have either the `api-owner` role bound in your namespace, or the `api-admin` role bound cluster-wide.
* You published an API product with manual approval mode enabled.
* Developers have submitted access requests for your API.

.Procedure

. Review pending requests in the web console to understand who wants to use your API:
+
.. In the {ocp} web console, navigate to *{prodname} API Catalog* -> *API Key Approvals*.
.. Review the pending API key requests:
+
* *Requester*: Who submitted the request
* *API Product*: Which of your APIs they want to access
* *Use Case*: How they plan to use the API
* *Tier*: Which service tier they selected (if you offer multiple tiers)

. Decide whether to approve or reject based on the use case:
+
*Approve requests when:*
+
* The use case is legitimate and appropriate for your API
* The requester's team has a valid business need
* The selected service tier matches their usage requirements
+
*Reject requests when:*
+
* The use case is unclear or does not match your API's purpose
* The requester should use a different API or approach
* You need more information before approving

. For a single request, take action:
+
*To approve:*
+
.. Click the request.
.. Review the details to confirm the use case makes sense.
.. Click *Approve*.
.. Optional: Add a comment welcoming the consumer or providing guidance.
.. Click *Confirm*.
+
*To reject:*
+
.. Click the request.
.. Click *Deny*.
.. Add a comment explaining why the request was rejected and what the consumer should do instead (for example, "This API is for payment processing. For order status, use the Orders API").
.. Click *Confirm*.

. For many similar requests, use bulk actions:
+
*To bulk approve requests:*
+
.. Select the checkboxes for requests with similar use cases.
.. Click *Bulk Approve*.
.. Optional: Add a comment that applies to all selected requests.
.. Click *Confirm*.
+
*To bulk reject requests:*
+
.. Select the checkboxes for requests that share the same issue.
.. Click *Bulk Deny*.
.. Add a comment explaining the rejection reason.
.. Click *Confirm*.

.Verification

. Navigate to *{prodname} API Catalog* -> *API Key Approvals*.

. Verify that approved requests show `Approved` status and rejected requests show `Denied` status.

. For approved requests, confirm that consumers can retrieve credential by asking an approved API consumer to check their *My API Keys* page and verify that they can view and copy their API key.

.Next steps

* Approved consumers can now call your API by using their credentials.
* Rejected consumers receive your feedback comment explaining why their request was denied and what alternative actions they should take.
