= Configuring sigstore support

You can use the `ClusterImagePolicy` and `ImagePolicy` custom resource (CR) objects to enable and configure sigstore support at the cluster or namespace scope. These policies specify the images and repositories to be verified by using sigstore tooling.

* Cluster image policy. A cluster image policy object enables a cluster administrator to configure sigstore support for the entire cluster. When enabled, the Machine Config Operator (MCO) watches the `ClusterImagePolicy` object and updates the `/etc/containers/policy.json` file in the `99-<pool>-generated-registries` machine config and all nodes in the cluster.

* Image policy. An image policy enables a cluster administrator or application developer to configure sigstore support for a specific namespace. The MCO watches `ImagePolicy` instance in different namespaces and merges the instances for each namespace into a `<NAMESPACE>.json` file in the `99-<pool>-generated-registries` machine-config and all nodes in the cluster. A cluster image policy overrides an image policy.

The following parameters apply to cluster and/or image policies. For information on using these parameters, see _Creating a cluster image policy CR_ and _About Creating an image policy CR_.

`scopes`:: Defines a list of image identities assigned to a policy. An image identity can refer to any of the following identities: 
* An individual image, by using a tag or digest, such as `docker.io/library/busybox:latest`. 
* A repository by omitting the tag or digest
* A repository namespace
* A registry host by specifying only the host name and port number or a wildcard expression starting with `*.` 
+
If multiple image identities match a single image, the policy for only the most specific image identity applies. 
+
In a cluster image policy, make sure that the policy allows deployment of the {product-title} images in the `quay.io/openshift-release-dev/ocp-release` and `quay.io/openshift-release-dev/ocp-v4.0-art-dev` repositories. Images in these repositories are required for cluster operation.

`policy`:: Contains configuration to allow scopes to be verified, and defines how images not matching the verification policy will be treated. You must configure a `rootOfTrust` and `signedIdentity`.
* `rootOfTrust`: Specifies the root of trust for the policy. Configure for either a puclic key or Fulcio certification.
** `publicKey`:  Indicates that the policy relies on a sigstore public key. You must specify a base64-encoded PEM format public key. You can optionally include Rekor verification.
** `FulcioCAWithReko`: Indicates that the policy is based on the Fulcio certification. You must specify the following parameters:
*** A base64-encoded PEM format fulcio CA.
*** An OpenID Connect (OIDC) issuer and the email of the Fulcio authentication configuration.
*** You can optionally include Rekor verification.
* `signed identity`: Specifies the approach used in the verification process to verify the identity in the signature and the actual image identity. You must specify the following parameters for the match policy:
** `MatchRepoDigestOrExact`. If the image identity uses a digest reference, the identity in the signature must be in the same repository as the image identity. If the image identity carries a tag, the identity in the signature must exactly match. This is the default.
** `MatchRepository`. The identity in the signature must be in the same repository as the image identity. The `exactRepository` parameter must be specified. This is useful to pull an image that contains the `latest` tag if the image is signed with a tag specifying an exact image version.
** `ExactRepository`. The identity in the signature must be in the same repository as a specified identity For example, `docker.io/library/busybox`.
** `RemapIdentity`. If the image identity matches the specified prefix, that prefix is replaced by the specified `signedPrefix`. If the image identity does not match, the `prefix` is unchanged and no remapping takes place. This useful when verifying signatures for a mirror of some other repository namespace that preserves the vendor’s repository structure. You must specify the following parameters:
*** `prefix`: Specifies the prefix of the image identity to be matched. If the image identity matches the specified prefix, that prefix is replaced by the specified `signedPrefix`. The prefix can be either `host[:port]` values that matching the exact `host[:port], string`, repository namespaces, or repositories. The prefix must not contain tags or digests. For example, to specify a single repository, use `docker.io/library/busybox` and not `busybox`. To specify the parent namespace of `docker.io/library/busybox`, you can use `docker.io/library`. The maximum length is 512 characters.
*** `signedPrefix`: Specifies the prefix of the image identity to be matched in the signature. The format is the same as "prefix".
