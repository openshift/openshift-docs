= Image data model refactoring for accuracy and performance

[role="_abstract"]

// JIRA: ROX-32865
The internal image data model has been refactored for improved accuracy and performance.

This refactoring optimizes how {product-title-short} stores and processes image vulnerability data, resulting in faster query performance, reduced storage requirements, and more accurate vulnerability tracking across image layers.

Images are now uniquely identified by the combination of name and digest, rather than by digest alone. This new data model resolves several long-standing issues when multiple images share the same digest but have different names, for example, different registries or tags. Note the following behaviors:

* If an environment has multiple images that share the same digest, then the total number of images will increase in the *Vulnerability Management* views to distinguish between them. 

* Deployments can now correctly distinguish images with the same digest but different names, so each deployment will show its own image reference in *Vulnerability Management* dashboards and vice-versa.

* Vulnerability exceptions, such as deferrals and false positives, can now be correctly scoped to a specific image name. Previously, a deferral or false-positive created for one image name was leaked to all images sharing the same digest.

* Policies will evaluate correctly per the deployed or checked image name and respect its vulnerability exceptions, rather than being affected by shared-digest exception leakage.

* When Central and Sensor are both upgraded to 4.11, enforcement will evaluate against the exact image name being deployed. For example, if `registry-a/app:latest` has a vulnerability exception but `registry-b/app:latest` shares the same digest without the exception, deploying `registry-a/app:latest` will skip enforcement for the deferred or false positive CVEs, while deploying `registry-b/app:latest` will still trigger enforcement.
