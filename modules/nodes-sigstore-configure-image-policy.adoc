= Creating an image policy

.Example image policy object with an exact repository match policy
[source,yaml]
----
apiVersion: config.openshift.io/v1alpha1
kind: ImagePolicy
metadata:
  name: p0
  namespace: mynamespace
spec:
  scopes: <1>
  - example.com/global/image # ignore policy for this scope in the namespace.json
  - example.com
  policy: <2>
    rootOfTrust: <3>
      policyType: PublicKey <4>
      publicKey: 
        keyData: <5> LS0tLS1CRUdJTiBQVUJMSUMgS0VZLS0tLS0KTUZ
        rekorKeyData: <6>
    signedIdentity:
      matchPolicy: ExactRepository <7>
      exactRepository:
        repository: example.com/foo/bar <8>
----
<1> Defines a list of image identities assigned to this policy. Each item refers to a scope in a registry.
<2> policy contains configuration to allow scopes to be verified, and defines how images not matching the verification policy will be treated.
<3> Specifies the root of trust for the policy.
<4> Indicates that the policy relies on a sigstore publicKey and may optionally use a Rekor verification. 
<5> Contains inline base64-encoded data for the PEM format public key. Maximum length is 8192 characters.
<6> Optional: contains inline base64-encoded data for the PEM format from the Rekor public key. Maximum length is 8192 characters.
<7> the approach used in the verification process to verify the identity in the signature and the actual image identity:
* `matchRepoDigestOrExact`. the identity in the signature must be in the same repository as the image identity if the image identity is referenced by a digest. Otherwise, the identity in the signature must be the same as the image identity. This is the default.
* `matchRepository`. the identity in the signature must be in the same repository as the image identity. the exactRepository must be specified
* `exactRepository`. the exactRepository must be specified.  the identity in the signature must be in the same repository as a specific identity specified by "repository".
* `remapIdentity`. the signature must be in the same as the remapped image identity. Remapped image identity is obtained by replacing the "prefix" with the specified “signedPrefix” if the the image identity matches the specified remapPrefix. the remapIdentity must be specified
<8> reference of the image identity to be matched. The value should be a repository name (by omitting the tag or digest) in a registry implementing the "Docker Registry HTTP API V2". For example, docker.io/library/busybox

.Example cluster image policy object with a Fulcio certificate policy and the `remapIdentity` match policy
[source,yaml]
----
apiVersion: config.openshift.io/v1alpha1
kind: ClusterImagePolicy 
metadata:
  name: p1
spec:
  scopes: <1>
    - example.com/global
    - quay.io/openshift-release-dev/ocp-release
    - quay.io/openshift-release-dev/ocp-v4.0-art-dev
  policy:
    rootOfTrust: <2>
      policyType: FulcioCAWithReko <3>
      publicKey:
        fulcioCAData: <4>
        fulcioSubject: <5>
          oidcIssuer: <6>
          signedEmail: <7>
        rekorKeyData: Zm9vIGJhcg== <8>
    signedIdentity: <9>
      matchPolicy: matchRepository <10>
      matchRepository:
        repository: example.com/foo/bar <11>  
----
<1> Defines the list of image identities assigned to a policy. A scope is a prefix of individual-image scopes, and specify a repository (by omitting the tag or digest), a repository namespace, or a registry host (by only specifying the host name and possibly a port number) or a wildcard expression starting with `*.`, for matching all subdomains (not including a port number). Wildcards are only supported for subdomain matching, and may not be used in the middle of the host, i.e.  /*.example.com is a valid case, but example/*./*.com is not. In addition to setting a policy appropriate for your own deployed applications, make sure that a policy on the OpenShift image repositories `quay.io/openshift-release-dev/ocp-release`, `quay.io/openshift-release-dev/ocp-v4.0-art-dev` allows deployment of the OpenShift images required for cluster operation.
<2> Defines the root of trust based on the selected policyType
<3> Indicates that the policy is based on the Fulcio certification and incorporates a Rekor verification. 
<4> contains inline base64-encoded data for the PEM format fulcio CA.
<5> Specifies OIDC issuer and the email of the Fulcio authentication configuration.
<6> contains the expected OIDC issuer as a valid URL. It will be verified that the Fulcio-issued certificate contains a (Fulcio-defined) certificate extension pointing at this OIDC issuer URL. When Fulcio issues certificates, it includes a value based on an URL inside the client-provided ID token.
<7>  holds the email address the the Fulcio certificate is issued for.
<8> Contains inline base64-encoded data for the PEM format from the Rekor public key. Maximum length is 8192 characters.
<9> specifies what image identity the signature claims about the image. The default is `MatchRepoDigestOrExact`.
<10> the approach used in the verification process to verify the identity in the signature and the actual image identity:
* `matchRepoDigestOrExact`. the identity in the signature must be in the same repository as the image identity if the image identity is referenced by a digest. Otherwise, the identity in the signature must be the same as the image identity. This is the default.
* `matchRepository`. the identity in the signature must be in the same repository as the image identity. the exactRepository must be specified
* `exactRepository`. the exactRepository must be specified.  the identity in the signature must be in the same repository as a specific identity specified by "repository".
* `remapIdentity`. the signature must be in the same as the remapped image identity. Remapped image identity is obtained by replacing the "prefix" with the specified “signedPrefix” if the the image identity matches the specified remapPrefix. the remapIdentity must be specified
<11> For the `matchRepository` match policy, specifies the repository that contains the image identity and signature.
