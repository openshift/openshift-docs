= Creating an image policy CR

An `ImagePolicy` custom resource (CR) enables a cluster administrator or application developer to configure sigstore support for a specific namespace. The MCO watches `ImagePolicy` instance in different namespaces and merges the instances for each namespace into a `<NAMESPACE>.json` file in the `99-<pool>-generated-registries machine-config`. A cluster image policy overrides an image policy. 

The following example shows general guidelines on how to configure an `ImagePolicy` object. For more details on the parameters, see _Configuring sigstore support_. 

.Prerequisites
// Taken from https://issues.redhat.com/browse/OCPSTRAT-918
* You have a supported public key infrastructure (PKI) and private keys or are willing to use OpenID Connect (OIDC) to generate a certificate for the signature.
* You have a signing process in place to sign your images.
* You have access to a registry that supports cosign signatures.
* You enabled the required Technology Preview features for your cluster by editing the `FeatureGate` CR named `cluster`:
+
[source,terminal]
----
$ oc edit featuregate cluster
----
+
.Example `FeatureGate` CR
[source,yaml]
----
apiVersion: config.openshift.io/v1
kind: FeatureGate
metadata:
  name: cluster
spec:
  featureSet: TechPreviewNoUpgrade <1>
----
<1> Enables the required `UserNamespacesSupport` and `ProcMountType` features.
+
[WARNING]
====
Enabling the `TechPreviewNoUpgrade` feature set on your cluster cannot be undone and prevents minor version updates. This feature set allows you to enable these Technology Preview features on test clusters, where you can fully test them. Do not enable this feature set on production clusters.
====
+
After you save the changes, new machine configs are created, the machine config pools are updated, and scheduling on each node is disabled while the change is being applied.

.Procedure

. Create an image policy object similar to the following examples:
+
--
.Example image policy object with an exact repository match policy
[source,yaml]
----
apiVersion: config.openshift.io/v1alpha1
kind: ImagePolicy <1>
metadata:
  name: p0
  namespace: mynamespace <2>
spec:
  scopes: <3>
    - docker.io
  policy: <4>
    rootOfTrust: <5>
      policyType: PublicKey <6>
      publicKey: 
        keyData: a2V5RGF0YQ== <7>
        rekorKeyData: cmVrb3JLZXlEYXRh <8>
    signedIdentity:
      matchPolicy: ExactRepository <9>
      exactRepository:
        repository: example.com/repository/image <10>
----
<1> Creates an `ImagePolicy` object.
<2> Specifies the namespace where the image policy is applied. 
<3> Defines a list of image IDs assigned to this policy.
<4> Specifies the parameters that define how the image IDs are verified.
<5> Defines a root of trust for the policy.
<6> Specifies the policy types that define the root of trust, either a public key or a Fulcio certification. You can optionally include a Rekor verification.
<7> For a public key policy, specifies a base64-encoded public key in the PEM format. The maximum length is 8192 characters.
<8> Optional: Specifies a base64-encoded Rekor public key in the PEM format. The maximum length is 8192 characters.
<9> Specifies one of the following processes to verify the identity in the signature and the actual image ID:
* `matchRepoDigestOrExact` 
* `matchRepository`.
* `exactRepository`
* `remapIdentity`
<10> For `ExactRepository`, specify the repository with the image ID to be matched.
--
+
--
.Example image policy object with a Fulcio certificate policy and the `matchRepository` match policy
[source,yaml]
----
apiVersion: config.openshift.io/v1alpha1
kind: ImagePolicy <1>
metadata:
  name: p1
  namespace: mynamespace <2>
spec:
  scopes: <3>
    - docker.io
  policy: <4>
    rootOfTrust: <5>
      policyType: FulcioCAWithReko <6>
      publicKey: <7>
        fulcioCAData: nMIICGjCCAaGgAwIBAgIUALnViVfnU
        fulcioSubject:
          oidcIssuer: https://OIDC.example.com
          signedEmail: username@example.com
        rekorKeyData: cmVrb3JLZXlEYXRh <8>
    signedIdentity:
      matchPolicy: exactRepository <9>
      exactRepository:
        repository: example.com/repository/image <10>  
----
<1> Creates an `ImagePolicy` object.
<2> Specifies the namespace where the image policy is applied. 
<3> Defines a list of image IDs assigned to this policy.
<4> Specifies the parameters that define how the image IDs are verified.
<5> Defines a root of trust for the policy.
<6> Specifies the policy types that define the root of trust, either a public key or a Fulcio certification. You can optionally include a Rekor verification.
<7> For a Fulcio certificate policy, the following parameters are required:
* `fulcioCAData`: Specifies a base64-encoded Fulcio CA in the PEM format. The maximum length is 8192 characters.
* `fulcioSubject`: Specifies the OIDC issuer and the email of the Fulcio authentication configuration.
<8> Optional: specifies a base64-encoded Rekor public key in the PEM format. The maximum length is 8192 characters.
<9> Specifies one of the following processes to verify the identity in the signature and the actual image ID:
* `matchRepoDigestOrExact` 
* `matchRepository`.
* `exactRepository`
* `remapIdentity`
<10> For the `exactRepository` match policy, specifies the repository that contains the image ID and signature.
--

. Create the cluster image policy object:
+
[source,terminal]
----
$ oc create -f <file-name>.yaml
----
+
Scheduling is disabled on each node as it is updated.

.Verification

. After the nodes in your cluster are updated, you can verify that the image policy has been configured:

.. Enter into the node in debug mode by running the following command::
+
[source,terminal]
----
$ oc debug node/<node_name>
----

..  Check the crio log for sigstore signature verification for namespaced policy by running the following command:
+
[source,terminal]
----
sh-5.1#  journalctl -u crio | grep -A 100 "Pulling image: docker.io"
----
+
