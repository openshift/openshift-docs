= Acknowledging the upgrade for EUS-EUS and y-stream upgrades

When you upgrade to all versions from 4.11 and later, you need to manually acknowledge that the upgrade can continue.

.Procedure
. Run the following command:
+
[source,terminal]
----
$ oc -n openshift-config patch cm admin-acks --patch '{"data":{"ack-<upgrade_version_from>-kube-<kube_api_version>-api-removals-in-<upgrade_version_to>":"true"}}' --type=merge
----
+
where:
+
--
<upgrade_version_from> :: Is the cluster version you are moving from, for example, `4.14`.
<kube_api_version> :: Is kube API version, for example, `1.28`.
<upgrade_version_to> :: Is the cluster version you are moving to, for example, `4.15`.
--

.Verification
* Verify the update.
Run the following command:
+
[source,terminal]
----
$ oc get configmap admin-acks -n openshift-config -o json | jq .data
----
+
.Example output
[source,json]
----
{
  "ack-4.14-kube-1.28-api-removals-in-4.15": "true",
  "ack-4.15-kube-1.29-api-removals-in-4.16": "true"
}
----
+
[NOTE]
====
In this example, the cluster is upgraded from version 4.14 to 4.15, and then from 4.15 to 4.16 in an EUS to EUS upgrade.
====
