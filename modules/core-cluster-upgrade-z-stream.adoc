= Updating clusters with z-stream releases

[role="_abstract"]
You can update clusters to z-stream patch releases by using {rh-rhacm} policies and {cgu-operator-full} ({cgu-operator}).
Z-stream updates apply security fixes and bug patches with minimal risk and should be performed regularly to maintain cluster security.

Apply z-stream updates as soon as they become available to address critical vulnerabilities and maintain security compliance.

.Prerequisites

* Pre-update health check completed successfully.
* {cgu-operator} is installed and configured on the {rh-rhacm-first} hub cluster.
* Z-stream patch release is available in the Red{nbsp}Hat errata.
* Recent etcd backup exists (within 24 hours).

.Procedure

. Verify the current cluster version by running the following command:
+
[source,terminal]
----
$ oc get clusterversion
----
+
.Example output
[source,terminal]
----
NAME      VERSION   AVAILABLE   PROGRESSING   SINCE   STATUS
version   4.20.0    True        False         30d     Cluster version is 4.20.0
----

. Check for available z-stream updates by running the following command:
+
[source,terminal]
----
$ oc adm upgrade
----
+
.Example output
[source,terminal]
----
Cluster version is 4.20.0

Upstream is unset, so the cluster will use an appropriate default.
Channel: stable-4.20 (available channels: candidate-4.20, eus-4.20, fast-4.20, stable-4.20)

Recommended updates:

  VERSION     IMAGE
  4.20.1      quay.io/openshift-release-dev/ocp-release@sha256:abc123...
  4.20.2      quay.io/openshift-release-dev/ocp-release@sha256:def456...
----

. Review the z-stream release notes for the target version:
+
Navigate to the {product-title} documentation and review the release notes for security fixes and bug fixes included in the z-stream release.

. Create an {rh-rhacm} policy for the z-stream update:
+
[source,yaml]
----
apiVersion: policy.open-cluster-management.io/v1
kind: Policy
metadata:
  name: core-upgrade-4.20.1-policy
  namespace: openshift-upgrade-policies
spec:
  remediationAction: inform
  disabled: false
  policy-templates:
  - objectDefinition:
      apiVersion: policy.open-cluster-management.io/v1
      kind: ConfigurationPolicy
      metadata:
        name: upgrade-cluster-version
      spec:
        remediationAction: inform
        severity: high
        object-templates:
        - complianceType: musthave
          objectDefinition:
            apiVersion: config.openshift.io/v1
            kind: ClusterVersion
            metadata:
              name: version
            spec:
              channel: stable-4.20
              desiredUpdate:
                version: 4.20.1
----
+
Replace `4.20.1` with your target z-stream version.
The policy is set to `inform` so it does not immediately push the update.
The `ClusterGroupUpgrade` CR enforces the policy at the scheduled update time.

. Commit the policy to your Git repository and apply it to the {rh-rhacm} hub cluster by running the following command:
+
[source,terminal]
----
$ oc apply -f core-upgrade-4.20.1-policy.yaml
----
+
Verify the policy was created by running the following command:
+
[source,terminal]
----
$ oc get policy core-upgrade-4.20.1-policy -n openshift-upgrade-policies
----
+
.Example output
[source,terminal]
----
NAME                           REMEDIATION ACTION   COMPLIANCE STATE   AGE
core-upgrade-4.20.1-policy    enforce              NonCompliant       30s
----
+
Verify the placement binding exists by running the following command:
+
[source,terminal]
----
$ oc get placementbinding -n openshift-upgrade-policies
----
+
Verify the placement rule exists by running the following command:
+
[source,terminal]
----
$ oc get placementrule -n openshift-upgrade-policies
----
+
If no `PlacementBinding` or `PlacementRule` exists, the policy will not apply to any clusters.
Create appropriate placement resources for your target clusters.

. Create a `ClusterGroupUpgrade` CR for the z-stream update:
+
[source,yaml]
----
apiVersion: ran.openshift.io/v1alpha1
kind: ClusterGroupUpgrade
metadata:
  name: core-zstream-upgrade-4.20.1
  namespace: openshift-upgrade-policies
spec:
  clusters:
  - spoke1
  - spoke2
  managedPolicies:
  - core-upgrade-4.20.1-policy
  enable: false
  remediationStrategy:
    maxConcurrency: 2
    timeout: 3600
----
+
Adjust the following values:
+
--
* `clusters`: List your target cluster names
* `maxConcurrency`: Number of clusters to update simultaneously
* `timeout`: Maximum time in seconds (3600 = 1 hour per cluster)
--

. Apply the `ClusterGroupUpgrade` CR by running the following command:
+
[source,terminal]
----
$ oc apply -f core-zstream-cgu.yaml
----

. Enable the `ClusterGroupUpgrade` resource to start the update by running the following command:
+
[source,terminal]
----
$ oc patch cgu core-zstream-upgrade-4.20.1 \
  -n openshift-upgrade-policies \
  --type merge \
  -p '{"spec":{"enable":true}}'
----

. Monitor the `ClusterGroupUpgrade` progress by running the following command:
+
[source,terminal]
----
$ oc get cgu core-zstream-upgrade-4.20.1 -n openshift-upgrade-policies -w
----
+
The `ClusterGroupUpgrade` resource progresses through these states:
+
--
* `Preparing`: {cgu-operator} validates cluster readiness
* `InProgress`: Clusters are updating
* `Complete`: All clusters successfully updated
--

. Monitor the cluster version on a target cluster by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get clusterversion -w
----
+
The cluster version progresses through the update, showing `PROGRESSING=True` during the update.

. Monitor cluster operator status during the update by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get co
----
+
Operators update sequentially during the z-stream update.

.Verification

. Verify the cluster version updated successfully by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get clusterversion -o jsonpath='{.items[0].status.desired.version}'
----
+
.Example output
[source,terminal]
----
4.20.1
----

. Verify all cluster operators are available and not degraded by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get co
----
+
All operators should show `AVAILABLE=True`, `PROGRESSING=False`, `DEGRADED=False`.

. Check workload health by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get pods -A | grep -v Running | grep -v Completed
----
+
No pods should show unexpected failures or crash loops.

. Validate custom resources by running the following command:
+
Verify Performance Profile status by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get performanceprofile
----
+
Verify PTP configuration by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get ptpconfig -n openshift-ptp
----
+
Verify SR-IOV network node policies by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get sriovnetworknodepolicy -n openshift-sriov-network-operator
----
+
All resources should show expected status.

. Run smoke tests for critical workloads to ensure functionality after the update.

.Troubleshooting

* If the `ClusterGroupUpgrade` resource is stuck in `Preparing` state, check policy compliance in {rh-rhacm} by running the following command:
+
[source,terminal]
----
$ oc get policy core-upgrade-4.20.1-policy -n openshift-upgrade-policies
----

* If a cluster operator becomes degraded during the update, check operator logs by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 logs -n openshift-<operator_namespace> deployment/<operator_deployment>
----

* If the update exceeds the timeout, investigate slow nodes or image pull issues by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get events -A --sort-by='.lastTimestamp' | tail -20
----

.Next steps

* Document the successful update in your change management system.
* Update cluster inventory with the new version.
* Schedule the next z-stream check for the following week.
