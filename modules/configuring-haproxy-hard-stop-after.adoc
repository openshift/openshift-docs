= Configuring the HAProxy hard-stop-after setting

[role="_abstract"]
To cap how long an old HAProxy process keeps running after a router reload, set `hard-stop-after`. By doing this task, you avoid wasting memory and CPU when connections stay open long enough that several old processes pile up.

During a reload, the router starts a new HAProxy process for new traffic and keeps the old process until its connections drain. The global `hard-stop-after` value sets the maximum time allowed for that soft stop. This behavior differs from `spec.tuningOptions.tunnelTimeout`, which sets how long idle tunnel connections (including `WebSockets`) can remain open during steady operation.

Apply the `ingress.operator.openshift.io/hard-stop-after` annotation to `IngressController` or to `ingresses.config/cluster`. If you set both, the `IngressController` annotation takes precedence. The Ingress Operator redeploys the router and sets `ROUTER_HARD_STOP_AFTER`.

[NOTE]
====
Use this setting together with xref:#configuring-haproxy-interval_{context}[Configuring the HAProxy reload interval] when you change how often the router reloads and you want a clear upper bound on old processes.
====

.Procedure

* Set the annotation on the default Ingress Controller by running the following command. Replace `30m` with a duration that fits your workload.
+
[source,terminal]
----
$ oc -n openshift-ingress-operator annotate ingresscontrollers/default ingress.operator.openshift.io/hard-stop-after=30m --overwrite
----

* To use the cluster ingress configuration instead, run the following command:
+
[source,terminal]
----
$ oc annotate ingresses.config/cluster ingress.operator.openshift.io/hard-stop-after=1h --overwrite
----
