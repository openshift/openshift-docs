= Sample YAML for a Cluster API infrastructure resource on {rh-openstack}

The infrastructure resource is provider-specific and defines properties that are shared by all the compute machine sets in the cluster, such as the region and subnets.
The compute machine set references this resource when creating machines.

[IMPORTANT]
====
Only the parameters in the following example are validated to be compatible with {product-title}.
Other parameters, such as those documented in the upstream link:https://cluster-api-openstack.sigs.k8s.io/clusteropenstack/configuration#optional-configuration({cap-openstack-first}) book, might cause undesired behavior.
====

[source,yaml]
----
apiVersion: infrastructure.cluster.x-k8s.io/v1beta1
kind: OpenStackCluster # <1>
metadata:
  name: <cluster_name> # <2>
  namespace: openshift-cluster-api
  labels:
    cluster.x-k8s.io/cluster-name: <cluster_name>
spec:
  apiServerLoadBalancer: # <3>
    enabled: true
  disableExternalNetwork: false # <4>
  externalNetwork: # <5>
    id: <openstack_network_id>
  identityRef: # <6>
    cloudName: <openstack_cloud> # <7>
    name: <cluster_name>-cloud-config
  managedSecurityGroups: # <8>
    allNodesSecurityGroupRules: # <9>
    - description: Created by cluster-api-provider-openstack - BGP (calico)
      direction: ingress
      etherType: IPv4
      name: BGP (Calico)
      portRangeMax: 179
      portRangeMin: 179
      protocol: tcp
      remoteManagedGroups:
      - controlplane
      - worker
    - description: Created by cluster-api-provider-openstack - IP-in-IP (calico)
      direction: ingress
      etherType: IPv4
      name: IP-in-IP (calico)
      protocol: "4"
      remoteManagedGroups:
      - controlplane
      - worker
  managedSubnets: # <10>
  - cidr: 10.6.0.0/24
    dnsNameservers:
    - <openstack_dns_nameserver>
----
<1> Specify the infrastructure kind for the cluster.
This value must match the value for your platform.
<2> Specify the cluster ID as the name of the cluster.
<3> Optional: Include this stanza as shown to create an API server load balancer.
If you do not want to create an API server load balancer, omit this stanza.
<4> Optional: Specify whether to attempt to connect the cluster to an external network.
The valid values for this field are `false` (attempt to connect) or `true` (do not attempt to connect).
<5> Optional: Specify the {rh-openstack} network that provides public internet to the VMs.
If specified, this field must refer to a single external network.
+
If not specified, and only one external network exists, the controller uses the existing external network.
If not specified, and multiple external networks exist, you must set `disableExternalNetwork: true` to prevent attempts to connect the cluster to an external network.
<6> Specify a reference to a secret that has the {rh-openstack} credentials to use when reconciling this cluster.
This secret also reconciles machines unless overridden in the machine spec.
<7> Specify the {rh-openstack} cloud environment. #What does this actually mean? Valid values?#
The default value is `capo-e2e`. #What does that do/mean?#
<8> Optional: Specify this field but leave it empty to create {rh-openstack} provider-managed security groups with default rules that allow the kubelet, etcd, and the Kubernetes API server to function correctly. #What are the defaults?#
Specify additional values to add any required rules to the managed security groups.
<9> Specify the information associated with the {rh-openstack} security group rule for the cluster.
You can include the following parameters:
+
--
`description`:: Optional: Specify a description of the security group rule.
`direction`:: Specify the direction of traffic on which to apply the security group rule.
Valid values for this field are `ingress` or `egress`.
An `ingress` direction rule applies to incoming traffic and an `egress` direction rule applies to outgoing traffic.
`etherType`:: Optional: Specify the Internet Protocol type to use.
Valid values for this field are `IPv6` or `IPv4`.
`name`:: Specify the name of the security group rule.
`portRangeMax`:: Optional: Specify a number in the range that matches the security group rule.
If the protocol is TCP or UDP, this value must be less than or equal to the value of the `portRangeMax` attribute.
`portRangeMin`:: Optional: Specify a number in the range that matches the security group rule.
The `portRangeMin` attribute constrains the `portRangeMax` attribute.
`protocol`:: Optional: Specify the communication protocol to use.
For example, to use the Transmission Control Protocol (TCP), specify `tcp`.
#Need all valid values#
`remoteGroupID`:: Optional: Specify the remote group ID to associate with this security group rule.
If you use `remoteGroupID`, do not use `remoteIPPrefix` or `remoteManagedGroups` in the same rule.
`remoteIPPrefix`:: Optional: Specify the remote IP prefix to associate with this security group rule.
If you use `remoteIPPrefix`, do not use `remoteGroupID` or `remoteManagedGroups` in the same rule.
`remoteManagedGroups`:: Optional: Specify the remote managed groups to associate with this security group rule.
If you use `remoteManagedGroups`, do not use `remoteGroupID` or `remoteIPPrefix` in the same rule.
--
<10> Specify the parameters that define your subnet:
+
--
`allocationPools`:: Optional: Specify an array of `AllocationPool` objects to apply to the subnet.
If specified, {rh-openstack} only allocates these IP addresses for machines, but you can create ports from outside of these ranges manually.
`cidr`:: Specify the IP address range used to create the subnet in Classless Inter-Domain Routing (CIDR) notation, for example `10.0.0.0/24`.
`dnsNameservers`:: Specify a list of DNS server addresses to use when creating the subnet.
These addresses must use the same IP version as the `cidr` range.
