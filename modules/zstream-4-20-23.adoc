== Known issues

* Booting the `Baremetalhost` object into the correct `operating system` repeatedly fails on certain Baseboard Management Controller (BMC) firmware versions due to the SuperMicro ARS-111GL-NHR server booting into an existing hard drive instead of virtual media. This issue arises from an updated BIOS and bare-metal host firmware, making CD unsupported while USB CD works. Consequently, the node inspection fails. If impacted, to work around this problem, manually set the `BootSourceOverrideTarget` to be USB CD instead of CD boot the node from the correct virtual media. (link:https://redhat.atlassian.net/browse/OCPBUGS-82299[OCPBUGS-82299])

[id="zstream-4-20-23-fixed-issues_{context}"]
== Fixed issues

* Before this update, enabling `routingViaHost: true` and `ipForwarding: Global` on a HyperShift (HCP) KubeVirt guest cluster caused the console and ingress Cluster Operators to become degraded. Route health checks from within the cluster would fail, even though ingress routes remained accessible from external clients. This occurred due to a routing conflict when both the management cluster and the HCP guest cluster had `routingViaHost: true` enabled. With this release, the fix ensures that HCP KubeVirt guest clusters use the subnet gateway IP as the default gateway, resolving the routing conflict and allowing the console and ingress Operators to remain healthy when these networking features are enabled. (link:https://redhat.atlassian.net/browse/OCPBUGS-74395[OCPBUGS-74395])

* Before this update, creating snapshots of large Azure `UltraSSD_LRS` or `PremiumV2_LRS` persistent volumes could fail due to a hard coded 10-minute timeout in the Container Storage Interface (CSI) driver, which was insufficient for Azure's background copy process on large disks. With this release, the Azure Disk CSI driver supports instant access snapshots, configurable via the `instantAccessDurationMinutes` parameter in the `VolumeSnapshotClass`, allowing snapshot operations on large volumes to complete successfully. (link:https://redhat.atlassian.net/browse/OCPBUGS-78034[OCPBUGS-78034])

* Before this update, when multiple identity providers (IdPs) were configured and the Authentication Operator encountered an issue communicating with any subset of IdPs, the entire IdP configuration set was rolled back to the previous configuration, preventing any new IdP configuration changes from being applied. With this release, an issue communicating with any individual IdP does not block the rollout of the entire new IdP configuration, allowing IdP configuration changes to apply even if all IdPs are not responding as expected. (link:https://redhat.atlassian.net/browse/OCPBUGS-81502[OCPBUGS-81502])

* Before this release, the web console did not provide a `robots.txt` file, which meant search engine crawlers had no explicit instruction to avoid indexing console pages. With this release, the web console provides a `robots.txt` file that disallows all crawlers. As a result, search engines are instructed not to index any web console pages. (link:https://redhat.atlassian.net/browse/OCPBUGS-81575[OCPBUGS-81575])

* Before this update, the egress IP address was incorrectly assigned to the br-ex bridge when you configured additional interfaces. This issue caused egress IP address assignment conflicts and routing issues with unexpected traffic paths. With this release, the egress IP address is not assigned to the br-ex bridge. As a result, the egress IP address is correctly assigned to the secondary interface, which resolves routing conflicts. (link:https://redhat.atlassian.net/browse/OCPBUGS-82115[OCPBUGS-82115])

* Before this update, {gcp-first} Internal Load Balancers (ILBs) failed to provision when node names were configured as fully qualified domain names (FQDNs). With this update, the filtering logic is corrected. As a consequence, {gcp-first} ILBs now provision successfully when node names are FQDNs. (link:https://redhat.atlassian.net/browse/OCPBUGS-83720[OCPBUGS-83720])

* Before this update, when the cluster `Authentication` resource was configured with `type: None`, any OAuth metadata specified via `spec.oauthMetadata` field was silently ignored. The Console Operator would unconditionally delete the OAuth metadata ConfigMap object and unset the `oauthMetadataFile` field, even if a valid ConfigMap object reference was provided. This could cause the Console Operator to become degraded, as console pods could not build the OAuth cache due to missing OAuth metadata from the Kubernetes API. With this release, the `spec.oauthMetadata` field is respected when `type: None` is set. If the field references a ConfigMap object, it syncs and is served by the `kube-apiserver`. If left empty, the existing behavior is unchanged. (link:https://redhat.atlassian.net/browse/OCPBUGS-83856[OCPBUGS-83856])

* Before this update, an `ovnkube-controller` restart, typically occurring during a cluster upgrade, would trigger the removal of Dynamic Host Configuration Protocol (DHCP) options for User Defined Networks (UDN). This caused Virtual Machine (VM) IP addresses on the primary UDN interface to be lost once the DHCP lease timed out, as the controller failed to distinguish between its own leases and those of other controllers. With this release, the default network controller has been updated to ensure it does not inadvertently remove DHCP options belonging to external network controllers. (link:https://redhat.atlassian.net/browse/OCPBUGS-84054[OCPBUGS-84054])
