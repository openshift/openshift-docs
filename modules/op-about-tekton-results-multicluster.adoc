= About Tekton Results in multicluster environments

[role="_abstract"]
Tekton Results in a multicluster environment uses a hub-and-spoke architecture to centralize the storage and querying of pipeline run records and logs. This architecture separates data storage on the hub cluster from pipeline execution on spoke clusters.

[id="tekton-results-architecture_{context}"]
== Tekton Results architecture

Tekton Results components are distributed across hub and spoke clusters to support centralized querying while maintaining distributed execution:

Hub cluster::
The hub cluster runs the Tekton Results API server, which provides a unified interface for querying records and logs from all clusters. The hub cluster does not run the Results watcher or retention policy agent to prevent conflicts with pipeline runs executing on spoke clusters.

Spoke clusters::
Each spoke cluster runs a complete set of Tekton Results components:
+
* *Results watcher*: Monitors pipeline runs and task runs, sending their records to the central database
* *Results API server*: Provides local access to results while pipeline runs are executing
* *Retention policy agent*: Manages cleanup of completed pipeline runs according to retention policies

[id="records-and-logs_{context}"]
== Records and logs storage

Tekton Results provides two independent storage capabilities:

Records storage::
Pipeline run and task run metadata is stored in an external PostgreSQL database. Both hub and spoke clusters connect to the same PostgreSQL instance, enabling centralized querying of all pipeline run records regardless of which cluster executed them.

Logs storage::
Pipeline run logs are forwarded from spoke clusters to a LokiStack deployment on the hub cluster. The hub cluster provides a unified query interface for logs from all clusters through the Tekton Results API.

Both capabilities are optional and can be configured independently. For example, you can configure records storage without logs storage, or vice versa.

[id="benefits-tekton-results-multicluster_{context}"]
== Benefits

Centralized querying::
Query all pipeline run records and logs from a single API endpoint on the hub cluster, regardless of which spoke cluster executed the pipeline run.

Persistent history::
Records and logs persist in the central storage even after pipeline runs are cleaned up from spoke clusters, providing long-term audit and troubleshooting capabilities.

Simplified management::
Manage database backups, retention policies, and storage capacity from a single location instead of maintaining separate storage on each cluster.
