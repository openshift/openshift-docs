= Choosing the right IPAM method

[role="_abstract"]
Use this decision guide to select the appropriate IPAM method for your use case.

== Decision flowchart

Answer these questions in order:

. *Do you need specific, predictable IP addresses for each pod?*
+
--
* *Yes* → Use *static IP assignment*
* *No* → Continue to question 2
--

. *Do you already have a DHCP server on the SR-IOV physical network?*
+
--
* *Yes* → Use *DHCP*
* *No* → Continue to question 3
--

. *Do you need the network to provide DNS servers, default routes, or other configuration beyond IP addresses?*
+
--
* *Yes* → Set up a DHCP server and use *DHCP*
* *No* → Use *Whereabouts*
--

== Recommendation by deployment pattern

*Development and testing*

* Use static IPs for simplicity
* Few pods, easy to manage manually
* No infrastructure setup required

*Production - cloud-native applications*

* Use Whereabouts
* All configuration in {product-title}
* No external dependencies
* Good for teams familiar with Kubernetes

*Production - enterprise integration*

* Use DHCP
* Integrates with existing network infrastructure
* Familiar to network operations teams
* Good for organizations with established DHCP infrastructure

*Production - hybrid (VMs and containers)*

* Use DHCP
* Consistent IP management across VMs and containers
* Single source of truth for IP addressing
* Easier for network teams to manage

== Common decision mistakes

*Mistake:* Choosing Whereabouts when you already have DHCP infrastructure

*Why it's a mistake:* You're creating duplicate IP address management instead of using existing infrastructure

*Better approach:* Use DHCP to leverage existing infrastructure and reduce operational complexity

---

*Mistake:* Choosing DHCP when you don't have a DHCP server yet

*Why it's a mistake:* Setting up and maintaining DHCP infrastructure just for {product-title} pods adds unnecessary complexity

*Better approach:* Use Whereabouts for simpler, {product-title}-native IP management

---

*Mistake:* Using static IPs for more than 20 pods

*Why it's a mistake:* Manual IP address management does not scale and increases risk of IP conflicts

*Better approach:* Use Whereabouts or DHCP for automatic IP allocation

---

*Mistake:* Choosing Whereabouts when you need DNS or routing configuration from the network

*Why it's a mistake:* Whereabouts only provides IP addresses, not additional network configuration

*Better approach:* Use DHCP to provide DNS, routes, and other network configuration, or configure DNS/routes separately in {product-title}

== Changing your decision later

If you need to change IPAM methods after deployment:

. Create a new `SriovNetwork` resource with the new IPAM configuration
. Update pod specifications or Deployments to reference the new network
. Delete pods to force recreation with new network configuration
. Verify pods receive IP addresses from the new IPAM method
. Remove the old `SriovNetwork` resource when migration is complete

You cannot change the IPAM method on an existing `SriovNetwork` resource - you must create a new one.
