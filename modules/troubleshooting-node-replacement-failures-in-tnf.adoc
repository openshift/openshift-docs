= Troubleshooting node replacement failures in TNF

Use the following information to diagnose and resolve common errors during TNF control-plane node replacement.

MissingStaticPodControllerDegraded:: This condition indicates that static pod manifests are missing on the replacement node.

.Diagnosis

Check whether installer pods are running or have completed for the node:

[source,terminal]
----
$ oc get pods -n openshift-etcd -l app=installer
$ oc get co etcd kube-apiserver -o yaml
----

.Resolution

. Verify that you deleted stale installer pods in Phase 3. If pods remain, delete them:
+
[source,terminal]
----
$ oc get pods -n openshift-etcd -l app=installer
----

. If installer pods still do not trigger, force a static pod revision bump. See xref:tnf-phase5-node-registration_{context}[Phase 5: Wait for node registration and recovery] for instructions.

etcd does not start on the replacement node:: This condition indicates that pacemaker is unable to start the etcd container on the replacement node.

.Diagnosis

Run the following commands on the replacement node via SSH:

[source,terminal]
----
$ sudo pcs status
$ sudo podman ps -a | grep etcd
$ sudo podman logs etcd
$ sudo journalctl -u pacemaker -n 500 | grep etcd
----

.Common causes and resolution

[cols="1,2", options="header"]
|===
|Cause |Resolution

|Pacemaker not configured
|Verify that the `tnf-auth` and `tnf-update-setup` jobs completed successfully: `oc get jobs -n openshift-etcd`

|Missing etcd TLS secrets
|The CEO generates these automatically. Check whether they exist: `oc get secrets -n openshift-etcd \| grep etcd-`

|Node not `Ready`
|Verify node status and confirm that kubelet is running on the replacement node.
|===

East-west connectivity fails after replacement:: This error indicates that pods on the surviving node cannot reach pods on the replacement node.

.Diagnosis

[source,terminal]
----
$ oc get podnetworkconnectivitycheck
----

From the survivor `ovnkube-node` pod, verify the remote chassis exists:

[source,terminal]
----
$ ovn-sbctl list Chassis
$ ovn-sbctl list Encap
----

.Likely cause

This issue most commonly occurs when the node registered from a backup image before Ironic reprovisioned it.

.Resolution

Restart `ovnkube-node` on the surviving node to force a Southbound/Northbound database resync:

[source,terminal]
----
$ oc delete pod -n openshift-ovn-kubernetes \
  -l app=ovnkube-node --field-selector spec.nodeName=<survivor-name>
----

For the full recovery procedure, see xref:tnf-special-recovery-node-registers-from-backup_{context}[Recovering when a node registers from a backup image before reprovisioning].

Static pods do not appear after node becomes Ready:: This condition indicates that installer pods are not triggering for the replacement node.

.Diagnosis

Check whether old installer pods still exist for the deleted node name:

[source,terminal]
----
$ oc get pods -n openshift-etcd -l app=installer
$ oc get pods -n openshift-kube-apiserver -l app=installer
----

.Resolution

. Delete any stale installer pods. This should have been completed in Phase 3, step 8.

. If pods still do not trigger after cleanup, force a revision bump as described in xref:tnf-phase5-node-registration_{context}[Phase 5: Wait for node registration and recovery].
