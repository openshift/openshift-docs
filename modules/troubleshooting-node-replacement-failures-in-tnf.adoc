= Troubleshooting node replacement failures in TNF

[role="_abstract"]
Use the following information to diagnose and resolve common errors during control plane node replacement in a TNF cluster.

MissingStaticPodControllerDegraded:: Static pod manifests are missing on the replacement node.
+
Check whether installer pods are running or have completed for the node:
+
[source,terminal]
----
$ oc get pods -n openshift-etcd -l app=installer
$ oc get co etcd kube-apiserver -o yaml
----
+
To resolve this issue:

. Confirm that you deleted stale installer pods before provisioning the replacement node. If pods remain, delete them:
+
[source,terminal]
----
$ oc get pods -n openshift-etcd -l app=installer
----

. If installer pods do not trigger after cleanup, force a static pod revision bump.

etcd does not start on the replacement node:: Pacemaker cannot start the etcd container on the replacement node.
+
Run the following commands on the replacement node to diagnose the issue:
+
[source,terminal]
----
$ sudo pcs status
$ sudo podman ps -a | grep etcd
$ sudo podman logs etcd
$ sudo journalctl -u pacemaker -n 500 | grep etcd
----
+
The following table lists common causes and resolutions:
+
[cols="1,2", options="header"]
|===
|Cause |Resolution

|Pacemaker is not configured
|Verify that the `tnf-auth` and `tnf-update-setup` jobs completed successfully: `oc get jobs -n openshift-etcd`

|etcd TLS secrets are missing
|The CEO generates these secrets automatically. Check whether they exist: `oc get secrets -n openshift-etcd \| grep etcd-`

|Node is not `Ready`
|Check the node status and confirm that kubelet is running on the replacement node.
|===

East-west connectivity fails after replacement:: Pods on the surviving node cannot reach pods on the replacement node.
+
Check the pod network connectivity status:
+
[source,terminal]
----
$ oc get podnetworkconnectivitycheck
----
+
From the surviving `ovnkube-node` pod, verify that the remote chassis exists:
+
[source,terminal]
----
$ ovn-sbctl list Chassis
$ ovn-sbctl list Encap
----
+
This issue occurs when the node registers from a backup image before Ironic reprovisions it. To resolve this issue, restart `ovnkube-node` on the surviving node to force a Southbound and Northbound database resync:
+
[source,terminal]
----
$ oc delete pod -n openshift-ovn-kubernetes \
  -l app=ovnkube-node --field-selector spec.nodeName=<survivor-name>
----


Static pods do not appear after the node becomes Ready:: Installer pods are not triggering for the replacement node.
+
Check whether stale installer pods exist for the failed node:
+
[source,terminal]
----
$ oc get pods -n openshift-etcd -l app=installer
$ oc get pods -n openshift-kube-apiserver -l app=installer
----
+
To resolve this issue:

. Delete any stale installer pods before provisioning the replacement node.

. If pods do not trigger after cleanup, force a revision bump.
