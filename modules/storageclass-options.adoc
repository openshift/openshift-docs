= StorageClass options

[role="_abstract"]
You can configure custom `StorageClass` behaviors by defining the `storageClassOptions` field in your device class specification.

If you set an empty configuration (storageClassOptions: {}) or omit the field entirely, the Operator uses the following default settings: 

`reclaimPolicy`::
Controls what happens to the PersistentVolume (PV) and its underlying logical volume when the PersistentVolumeClaim (PVC) is deleted.
+
* Allowed values: `Delete` (default), `Retain`
+
* Immutable: Yes. You cannot change this field after creating the `LVMCluster`.
+
When set to `Retain`, deleting a PVC does not delete the PV or the underlying logical volume on disk. Data is preserved, which is useful for data protection scenarios where accidental PVC deletion must not cause data loss. However, you must perform manual cleanup before you can delete the `LVMCluster`. 
+
When set to `Delete`, both the PV and the on-disk logical volume are removed when the PVC is deleted. This is the default behavior.
+
.Example
[source,yaml]
----
storageClassOptions:
  reclaimPolicy: Retain
----

`volumeBindingMode`::
Controls when volume binding and dynamic provisioning occur.
+
* Allowed values: `WaitForFirstConsumer` (default), `Immediate`
+
* Immutable: Yes. You cannot change this field after creating the `LVMCluster`.
+
`WaitForFirstConsumer` delays PV provisioning until a pod that uses the PVC is scheduled. This enables topology-aware scheduling, where LVMS creates the PV on the node where the pod will run.
+
`Immediate` provisions and binds the PV as soon as the PVC is created, without waiting for a consumer pod. This mode is useful for pre-provisioning workflows. On multi-node clusters, PVs might be provisioned on nodes where the consuming pod cannot run. Use `Immediate` only on single-node clusters or when node affinity is managed externally.
+
.Example
[source,yaml]
----
storageClassOptions:
  volumeBindingMode: Immediate
----

`additionalParameters`::
Adds custom key-value pairs to the `StorageClass .parameters` map.
+
* Type: `map[string]string`
+
* Default: `{}` (empty)
+
* Maximum entries: 16
+
* Immutable: Yes. You cannot change this field after creating the `LVMCluster`.
+
StorageClass parameters are passed to the CSI driver (TopoLVM) during volume provisioning. TopoLVM only reads two parameters that it recognizes: `topolvm.io/device-class` and `csi.storage.k8s.io/fstype`. Any other parameter is stored on the StorageClass object but is not acted upon by TopoLVM. Use `additionalParameters` for forward-compatibility when TopoLVM or a future CSI sidecar adds support for new StorageClass parameters, or for parameters consumed by other Kubernetes components such as volume snapshot controllers.
+
The following parameter keys are managed by LVMS and are rejected at admission if you set them through `additionalParameters`:
+
* `topolvm.io/device-class` — automatically set to the device class name
* `csi.storage.k8s.io/fstype` — automatically set from the `fstype` field on the device class
+
[IMPORTANT]
====
To change the filesystem type, use the `fstype` field on the device class directly. Do not use `additionalParameters`.
====
+
.Example
[source,yaml]
----
storageClassOptions:
  additionalParameters:
    custom-param-key: custom-param-value
----

`additionalLabels`::
Adds custom labels to the StorageClass metadata.
+
* Type: `map[string]string`
+
* Default: none
+
* Maximum entries: 16
+
* Immutable: No. This is the only mutable field. You can add, change, or remove labels after creation.
+
Use additional labels for organizational tagging, integration with cluster policies, monitoring, or other labeling use cases. When you remove a label from `additionalLabels`, the operator automatically removes it from the StorageClass during the next reconciliation. Labels added directly to the StorageClass by other tools are not affected by label pruning; only labels managed through `additionalLabels` are pruned.
+
Labels must follow Kubernetes label naming rules. Keys must be valid qualified names, and values must be valid label values.
+
The following label keys are reserved by the operator and cannot be set through `additionalLabels`:


** `app.kubernetes.io/managed-by`
** `app.kubernetes.io/part-of`
** `app.kubernetes.io/name`
** `app.kubernetes.io/component`
** Any key with the prefix `owned-by.topolvm.io/`
+
.Example
[source,yaml]
----
storageClassOptions:
  additionalLabels:
    environment: production
    team: storage
----
+
If you set the `storageClassOptions: {}`, it produces the same `StorageClass` as omitting `storageClassOptions` entirely: `reclaimPolicy: Delete`, `volumeBindingMode: WaitForFirstConsumer`, no additional parameters, and no additional labels.
