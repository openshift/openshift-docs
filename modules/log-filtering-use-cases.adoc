= Understanding log filtering use cases

[role="_abstract"]
Log filtering reduces operational costs and improves system performance. It controls which log records flow through the pipeline and what data each record contains.

Filtering serves three primary operational goals:

Performance optimization::
Reducing the volume of logs that collectors process, transport, and store improves query performance and reduces resource consumption.

Cost management::
Smaller log payloads and fewer stored records directly reduce storage costs and network bandwidth consumption.

Multitenant routing::
Different teams often need different subsets of logs with team-specific filtering applied before delivery to their preferred destinations.

== Filter types and when to use them

[cols="1,2,2,1",options="header"]
|===
|Filter type |Purpose |Common use cases |Performance impact

|`drop`
|Discard entire log records based on content patterns
a|* Remove debug-level logs from production
* Drop noisy health check requests
* Exclude logs from test namespaces
* Filter out duplicate events
|High - Reduces records at source

|`prune`
|Remove specific fields from log records while keeping the record
a|* Strip verbose Kubernetes metadata
* Remove high-cardinality labels
* Exclude sensitive fields before external forwarding
* Reduce payload size for cost savings
|Medium - Reduces payload size

|`detectMultilineException`
|Combine stack traces into single records
a|* Consolidate Java exception stack traces
* Group Python traceback messages
* Merge multi-line error messages
|Low - Improves readability

|Rate limiting (output-level)
|Throttle log delivery to prevent overwhelming destinations
a|* Protect downstream systems from spike traffic
* Comply with vendor ingestion rate limits
* Prevent runaway log producers from consuming quota
|Medium - Smooths delivery rate

|===

== Decision workflow

Follow this workflow to choose the appropriate filtering approach:

. *Identify the problem:*
** Too many logs → Use `drop` filters to discard unwanted records
** Log payloads too large → Use `prune` filters to remove fields
** Downstream system overloaded → Use output rate limiting
** Many teams need different log views → Use multi-pipeline filtering

. *Determine where to filter:*
** At the source (collector) → Drop or prune before forwarding saves network and storage costs
** At the destination (output) → Rate limiting prevents overwhelming external systems

. *Verify the impact:*
** Check collector metrics for dropped record counts
** Compare payload sizes before and after pruning
** Monitor destination ingestion rates

== Filter placement and pipeline design

Filters apply at two points in the log pipeline:

Pipeline-level filters::
Defined in `spec.filters` and referenced by name in `spec.pipelines[].filterRefs`. These filters process records before the pipeline routes them to outputs. Use pipeline-level filters when many pipelines need the same filtering logic.

Output-level tuning::
Configure rate limiting in `spec.outputs[].limit.maxRecordsPerSecond`. This controls delivery rate to specific destinations. Use output-level limits to protect individual downstream systems.

.Example filter placement
[source,yaml]
----
apiVersion: observability.openshift.io/v1
kind: ClusterLogForwarder
metadata:
  name: instance
  namespace: openshift-logging
spec:
  filters:
  - name: drop-debug-logs
    type: drop
    drop:
    - test:
      - field: .level
        matches: debug

  pipelines:
  - name: application-logs
    inputRefs:
    - application
    filterRefs:
    - drop-debug-logs
    outputRefs:
    - default-lokistack

  outputs:
  - name: external-splunk
    type: splunk
    limit:
      maxRecordsPerSecond: 1000
----
+
Define filters in the `filters` section. Reference them by name in the `filterRefs` field of pipelines. Apply rate limits at the output level.

== Performance and cost trade-offs

[cols="1,2,2",options="header"]
|===
|Approach |Benefits |Considerations

|Drop at source
a|* Saves network bandwidth
* Reduces storage costs
* Improves collector performance
|You cannot recover dropped logs later

|Prune at source
a|* Reduces payload size
* Lowers storage costs
* Maintains record count for analytics
|Pruned fields are not available for queries

|Rate limit at output
a|* Protects downstream systems
* Prevents quota overruns
* Smooths traffic spikes
|Rate limiting might create backpressure during sustained high volume

|Multi-pipeline with per-team filters
a|* Each team sees relevant logs only
* Reduces noise in team-specific destinations
* Supports different retention policies
|More complex configuration to maintain

|===
