== New features and enhancements

Incidents UI now displays data refresh policy information::
The *Incidents* view in the monitoring UI plugin now includes a tooltip with the following text:
+
_Incident data is updated every few minutes. What you see may be up to 5 minutes old. Refresh the page to view updated information_.
+
This enhancement ensures clearer understanding of the displayed data, indicating that the *Incidents* UI operates differently from the real-time *Alerting* tab.
+
link:https://redhat.atlassian.net/browse/OU-1039[OU-1039]

[id="cluster-observability-operator-1-4-technology-preview-features_{context}"]
== Technology Preview features

Red Hat build of Perses dashboarding tool (Technology Preview)::
The {coo-full} 1.4 introduces a technology preview of the Red{nbsp}Hat build of Perses, a downstream version of the open source link:https://perses.dev/[Perses project], a dashboarding tool for displaying observability data with native Kubernetes RBAC support. This feature includes a dashboard management UI with create, edit, and delete capabilities directly in the {ocp-product-title} web console, a Grafana import tool for migrating existing dashboards, configurable datasources for metrics, logging, and tracing, and built-in RBAC roles for secure access control. Users can now manage observability dashboards without switching to external platforms and easily migrate from Grafana using the integrated import functionality.

[id="cluster-observability-operator-1-4-deprecated-features_{context}"]
== Deprecated features

The incidents attribute is deprecated::
Incident detection in the {coo-full} is now configured by using the `clusterHealthAnalyzer.enabled` flag instead of the `incidents.enabled` flag. The existing `incidents` attribute is now marked as deprecated and will continue to work during the deprecation period to ensure backward compatibility.
+
You can migrate existing configurations from `spec.monitoring.incidents.enabled: true` to `spec.monitoring.clusterHealthAnalyzer.enabled: true`.

[id="cluster-observability-operator-1-4-bug-fixes_{context}"]
== Bug fixes

Troubleshooting panel no longer causes console errors on the alerting rules page::
Before this update, the troubleshooting panel console plugin triggered `ResizeObserver` error notifications when you navigated to the alerting rules page in the {product-title} web console. As a consequence, the console displayed runtime error overlays that disrupted the alerting rules view.
+
With this release, the troubleshooting panel suppresses the non-critical `ResizeObserver` error notification. As a result, the alerting rules page displays correctly without runtime error interruptions.
+
link:https://issues.redhat.com/browse/COO-1668[COO-1668]


Troubleshooting panel responds to internal URL changes on the metrics page::
Before this update, the troubleshooting panel did not detect URL changes made internally by the {ocp-product-title} web console, such as when you clicked the *Insert example query* button on the metrics page. The panel remained empty and did not update to reflect the selected query. As a consequence, you could not use the troubleshooting panel to correlate signals from queries inserted through the metrics page interface.
+
With this release, the troubleshooting panel detects internal URL changes made by the web console. As a result, the panel updates correctly when you insert an example query or edit a query on the metrics page.
+
link:https://issues.redhat.com/browse/COO-1650[COO-1650]

Korrel8r displays logging alerts from Loki alerting rules::
Before this update, korrel8r could not display alerts generated by Loki alerting rules, such as `AlertingRule` resources created for log-based alerts. When you queried korrel8r for a logging alert, the response returned empty results with no nodes or edges. As a consequence, you could not correlate logging alerts with other observability signals in the troubleshooting panel.
+
With this release, korrel8r correctly retrieves and displays logging alerts from Loki. As a result, you can correlate log-based alerts with related resources in the troubleshooting panel.
+
link:https://issues.redhat.com/browse/COO-1307[COO-1307]

Troubleshooting panel graph persists when the panel is reopened::
Before this update, the troubleshooting panel correlation graph content disappeared when you closed and reopened the panel or when the browser window was resized. The graph failed to render correctly after these actions. As a consequence, you had to refresh the page to restore the correlation graph.
+
With this release, the troubleshooting panel correctly re-renders the correlation graph when the panel is reopened or the browser window is resized. The panel also includes a reset view control to restore and relayout the topology graph. As a result, you can close and reopen the troubleshooting panel without losing the correlation graph content.
+
link:https://issues.redhat.com/browse/COO-1383[COO-1383]

Korrel8r includes the user-workload monitoring Alertmanager::
Before this update, when user-workload monitoring was enabled with a separate Alertmanager instance, the {coo-full} generated a `korrel8r.yaml` configuration that only included the platform Alertmanager at `alertmanager-main.openshift-monitoring.svc:9094`. The configuration did not include the user-workload Alertmanager at `alertmanager-user-workload.openshift-user-workload-monitoring.svc:9095`. As a consequence, korrel8r could not correlate alerts from user-workload monitoring, and the troubleshooting panel did not display user-workload alerts.
+
With this release, the Operator generates a `korrel8r.yaml` configuration that includes an additional alert domain for the user-workload monitoring Alertmanager when a separate Alertmanager instance is enabled. As a result, korrel8r can correlate user-workload alerts and display them in the troubleshooting panel.
+
link:https://issues.redhat.com/browse/COO-1312[COO-1312]

Troubleshooting panel layout remains intact on small browser windows::
Before this update, when you reduced the browser window height, the troubleshooting panel title separated from the panel body and displayed in a separate column. The main console pane could also be pushed off screen with no scrollbar available. As a consequence, you could not view the full troubleshooting panel or the main console content on vertically constrained browser windows.
+
With this release, the troubleshooting panel layout remains intact regardless of browser window size. The title stays attached to the panel body, and scroll bars appear when the panel content exceeds the available space. As a result, you can use the troubleshooting panel on smaller browser windows without layout issues.
+
link:https://issues.redhat.com/browse/COO-1611[COO-1611]

Monitoring stack no longer generates transient pod security errors on creation::
Before this update, when you created a new `MonitoringStack` resource, a race condition could occur where the Prometheus pod was created before the required `ClusterRoleBinding` and `RoleBinding` resources were applied. The Prometheus pod temporarily failed to start with errors. Although the issue resolved itself within seconds, the transient errors generated log events. As a consequence, you could receive false notifications when creating a monitoring stack.
+
With this release, the Operator creates role bindings before the Prometheus pod, which makes the race condition less likely to happen. As a result, creating a new monitoring stack no longer generates transient pod errors or false events.
+
link:https://issues.redhat.com/browse/COO-1511[COO-1511]

Korrel8r detects Network Observability deployed in a custom namespace::
Before this update, when the Network Observability Operator and its Loki stack were deployed in a custom namespace instead of the default `netobserv` namespace, the {coo-full} generated a `korrel8r.yaml` configuration that still referenced the default namespace. For example, the netflow domain URL was set to `https://loki-gateway-http.netobserv.svc:8080` regardless of the actual deployment namespace. As a consequence, korrel8r could not retrieve network flow data when Network Observability was deployed in a custom namespace.
+
With this release, the Operator detects the namespace where the Network Observability Loki stack is deployed and generates the correct URL in the `korrel8r.yaml` configuration. As a result, korrel8r can retrieve and correlate network flow data regardless of the namespace used for Network Observability.
+
link:https://issues.redhat.com/browse/COO-1173[COO-1173]

Troubleshooting panel graph persists when the panel is reopened::
Before this update, the troubleshooting panel correlation graph content disappeared when you closed and reopened the panel or when the browser window was resized. The graph failed to render correctly after these actions. As a consequence, you had to refresh the page to restore the correlation graph.
+
With this release, the troubleshooting panel correctly re-renders the correlation graph when the panel is reopened or the browser window is resized. The panel also includes a reset view control to restore and relayout the topology graph. As a result, you can close and reopen the troubleshooting panel without losing the correlation graph content.
+
link:https://issues.redhat.com/browse/COO-1383[COO-1383]

Operator no longer generates excessive reconciliation log messages::
Before this update, the {coo-full} pod continuously logged repetitive `INFO` messages such as "Reconciling observability UI plugin" for each configured `UIPlugin` resource. This occurred because a loop in the plugin status update logic caused repeated reconciliation cycles. As a consequence, the Operator pod generated excessive log volume, which made it difficult to identify relevant log entries for troubleshooting.
+
With this release, the Operator avoids unnecessary reconciliation loops when setting plugin status. As a result, the Operator logs only meaningful reconciliation events and no longer spams the console log with repetitive messages.
+
link:https://issues.redhat.com/browse/COO-1282[COO-1282]

Troubleshooting panel renders alert correlation graphs on clusters with {rh-rhacm}::
Before this update, when {rh-rhacm} was installed, the troubleshooting panel included the `managed_cluster` external label in alert queries sent to korrel8r. Korrel8r could not resolve alerts with this label, so the correlation graph was not rendered. As a consequence, you could not view the correlation graph for alerts in the troubleshooting panel on clusters with {rh-rhacm} enabled.
+
With this release, the troubleshooting panel removes the `managed_cluster` label from alert queries before sending them to korrel8r. As a result, the correlation graph renders correctly for alerts on clusters with {rh-rhacm}.
+
link:https://issues.redhat.com/browse/COO-1391[COO-1391]

Incidents with many components no longer cause API request failures::         
Before this update, incidents with many different components caused the Alerts API request to fail because the queries became too complex and lengthy. As a consequence, users could not retrieve alert information for incidents involving numerous components.
+
With this release, the system handles complex incidents by breaking down large queries into smaller, manageable requests. As a result, users can now successfully access alert data for incidents regardless of the number of components involved.
+
link:https://redhat.atlassian.net/browse/OU-632[OU-632]

Incident end times in tooltips now round to 5-minute boundaries::               
Before this update, consecutive intervals in incident tooltip timestamps were not properly rounded to 5 minutes. As a consequence, tooltip end dates were inconsistent, causing issues in the user timeline view. 
+
With this release, tooltip end dates are now rounded to 5-minute boundaries for consecutive intervals. As a result, the visual clarity of incident data intervals is improved.
+
link:https://redhat.atlassian.net/browse/OU-1205[OU-1205]

Incidents page now displays error message for users with insufficient permissions::      
Before this update, users with insufficient permissions who accessed the *Incidents* view would only see a loading spinner with no indication of the underlying permission issue. 
+
With this release, the page displays a proper forbidden error message when users lack the necessary permissions. As a result, users now receive clear feedback about permission issues instead of seeing an endless loading state.
+
link:https://redhat.atlassian.net/browse/OU-1213[OU-1213]

Incident charts no longer show incorrect spacing with many alerts::           
Before this update, incidents containing 100 or more alerts displayed an unwanted gap above the alert chart bars, with the gap size increasing as the number of alerts grew. As a consequence, users experienced distorted chart visualizations that made it difficult to properly view and analyze incidents with large numbers of alerts. 
+
With this release, the chart padding and spacing is corrected to handle high alert volumes properly. As a result, incident charts now display correctly regardless of the number of alerts.
+
link:https://redhat.atlassian.net/browse/OU-1123[OU-1123]
