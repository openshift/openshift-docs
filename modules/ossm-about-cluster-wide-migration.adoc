= About cluster-wide migration

During the migration process, two cluster-wide control planes run in the same cluster while the data plane namespaces are gradually migrated to the {SMProductName} 3.0 installation. One control plane is associated with the {SMProductName} {SMv2Version} installation and the other one is associated with the {SMProduct} 3.0 installation. You must carefully plan out the migration steps in order to avoid possible conflicts between the two control planes. The following topics describe the required conditions for successful migration.

== Control planes and root certificate

During the migration, the control plane associated with the 2.6 installation and the control plane associated with the 3.0 installation must share the same root certificate. This can be achieved by installing the 3.0 control plane to the same namespace as 2.6 control plane. The migration procedures show how to verify the root certificate is shared.

== Control planes, discovery selectors, and namespace access

Both control planes must have access to all namespaces in the mesh. During the migration, some proxies are controlled by the 3.0 control plane while other proxies remain controlled by the 2.6 control plane. To ensure that mesh communication works during the migration both control planes must be aware of the same set of services. Service discovery is provided by `istiod` component, which runs in the control plane namespace.

In the {SMProduct} 3.0 installation, you can control how {istio} discovers services using discovery selectors. When using discovery selectors, ensure that the `discoverySelectors` expression defined in the {SMProduct} 3.0 {istio} resource match the namespaces that comprise the {SMProduct} 2.6 mesh. You might have to add additional labels to the {SMProduct} 2.6 application namespaces to ensure that they are captured in the {SMProduct} 3.0 installation. See "Scoping the service mesh with DiscoverySelectors" for more information.

[NOTE]
====
In {SMProduct} 2.6 installations, the `maistra.io/member-of` label is automatically created. This label automatically gets removed during the migration process so it cannot be used.
====

== Control planes and network policies

During the migration, for both control planes, ensure that no network policies block traffic between the control plane and the data plane namespaces, between the data plane namespaces and the control plane, or between the data plane namespaces themselves. By default, {SMProduct} 2.6 manages network policies that block traffic to the 3.0 control plane. 

In the pre-migration checklist, you are instructed to disable network policies. However, you can manually re-create them. Manually created network policies must allow traffic for both control planes. When a data plane namespace is migrated to 3.0, the `maistra.io/member-of` label automatically gets removed. Do not use this label in network policies. See "Migration of Network Policies" for more information.

[NOTE]
====
If network policies are configured incorrectly they can disrupt mesh traffic. When performing the migration, exercise caution when creating network policies to prevent traffic disruption. See "Network Policy migration" for more information.
====

== Control planes and sidecar injection

During the migration, only one control plane must perform sidecar injection. This is achieved by correctly using injection labels. See "Installing the Sidecar" for more information.

[NOTE]
====
During the migration, you must disable the 2.6 injector. Use the `maistra.io/ignore-namespace: "true"` label to prevent the 2.6 control plane from injecting a proxy in the namespace.
====

If both control planes try to perform sidecar injection the proxy will not start and the migration cannot be completed.

== Control planes and label selection

For {SMProduct} 3.0, you must decide if you want to use the `istio.io/rev` or `istio-injection` label to configure sidecar injection. See "Installing the Sidecar" for more information about how to use the labels.

In the {SMProduct} 2.6 installation, the way member selection was configured in the `ServiceMeshMemberRoll` resource can impact how injection labels are used in the {SMProduct} 3.0 installation.

By default, in a 2.6 installation, the `spec.memberSelectors` field in the `ServiceMeshMemberRoll` resource is configured to match the `istio-injection=enabled` label and all of the data plane namespaces in a 2.6 installation have the `istio-injection=enabled` label applied. If you are using the default 2.6 installation settings, you can either keep using that label or switch to the `istio.io/rev` label for the 3.0 installation.

If the `spec.memberSelectors` field in the `ServiceMeshMemberRoll` resource is not configured to match the `istio-injection=enabled` label and the 2.6 data plane namespace uses a custom label you must add the `istio.io/rev` or `istio-injection` label during the migration. The custom labels defined in the `spec.memberSelectors` parameter of the `ServiceMeshMemberRoll` resource have no effect on sidecar injection in the {SMProduct} 3 installation and cannot be used. 

In the 2.6 installation, if projects were added to the mesh by manually creating the `ServiceMeshMember` resource you must add the `istio.io/rev` or `istio-injection` label to the project namespaces during the migration.

== Cluster-Wide migration methods

Migrating from {SMProductName} {SMv2Version} to {SMProduct} 3.0 can be performed using one of the following methods:

* Applying the `istio.io/rev` label
* Applying the `istio-injection=enabled` label
* Performing a simple migration using the `istio-injection=enabled` label; however, this method might introduce traffic disruptions.

It is important to Understand the difference between the `istio.io/rev` or `istio-injection` labels in order to select the best migration method for your {SMProduct} installation. Read through all the migration methods before choosing the one that best fits your needs.
