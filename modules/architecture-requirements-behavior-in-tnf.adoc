= Architecture requirements and behavior in TNF

[role="_abstract"]
You need to consider some requirements and behaviors when planning control-plane node replacement in a Two-Node with Fencing (TNF) cluster.

* A dedicated fencing device is required. Only BMC via the Redfish protocol is supported.
* The fencing device must be reachable from both control-plane nodes.
* Node replacement reuses the same node name, for example `master-0`, with a new node UID.
* Pacemaker health is visible through the `PacemakerCluster` CR and the `pcs status` command.

You can check the Pacemaker cluster health by running the following command:
[source,terminal]
----
$ oc get pacemakercluster cluster -o yaml
----

[IMPORTANT]
====
Because etcd runs as a podman container managed by pacemaker rather than as a Kubernetes static pod, standard diagnostic commands such as `oc -n openshift-etcd get pods` do not apply in a TNF cluster. Use `sudo pcs status` and `sudo podman logs etcd` instead.
====
