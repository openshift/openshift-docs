= Architecture requirements and behavior in two-node OpenShift cluster with fencing

[role="_abstract"]
Before replacing a control plane node in your two-node OpenShift cluster with fencing (TNF), review these essential requirements and system behaviors to ensure a smooth, disruption-free recovery process.

The essential requirements are as follows:

Dedicated Fencing Device:: 
You must use a dedicated fencing device. The cluster explicitly requires a Baseboard Management Controller (BMC) configured with the *Redfish* protocol.

Network Reachability:: 
Ensure that both control-plane nodes have an uninterrupted network path to the BMC fencing device.

Reusing Node Names:: 
When replacing a node, you must reuse the exact same node name (for example, `master-0`). The cluster will automatically assign a new unique identifier (UID) to the replacement hardware.

Verifying Cluster Status:: 
You can easily monitor and verify Pacemaker health at any time by using either the `PacemakerCluster` custom resource (CR) in OpenShift or by running the `sudo pcs status` command directly on a node.

You can check the Pacemaker cluster health by running the following command:

[source,terminal]
----
$ oc get pacemakercluster cluster -o yaml
----

[NOTE]
====
Pacemaker manages etcd as a Podman container rather than a standard Kubernetes static pod. Because of this architecture, standard diagnostic commands such as `oc -n openshift-etcd get pods` do not show the status in a TNF cluster. Use `sudo pcs status` and `sudo podman logs etcd` instead.
====
