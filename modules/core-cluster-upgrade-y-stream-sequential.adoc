= Upgrade through sequential y-stream releases

[role="_abstract"]
You can upgrade bare metal Core clusters through multiple sequential y-stream releases when you need to move from an older version to a newer version that is more than one minor release away.
You must upgrade through each intermediate y-stream version because skipping versions is not supported.

For example, to upgrade from 4.20 to 4.22, you must first upgrade from 4.20 to 4.21, and then upgrade from 4.21 to 4.22.

.Prerequisites

* Pre-upgrade health check completed successfully.
* {cgu-operator} is installed and configured on the {rh-rhacm} hub cluster.
* All intermediate y-stream releases are available.
* Maintenance windows planned for each y-stream upgrade.

.Procedure

. Determine the complete upgrade path:
+
Identify your current version and target version, then map out the required intermediate versions.
+
.Example upgrade paths
[cols="2,3",options="header"]
|===
|Current version
|Upgrade path to 4.22

|4.20.20
|4.20.20 → 4.21.latest → 4.22.0

|4.19.10
|4.19.10 → 4.20.latest → 4.21.latest → 4.22.0
|===

. Upgrade to the first intermediate y-stream version:
+
Follow the procedure in "Upgrade bare metal Core clusters through y-stream releases" to upgrade to the first intermediate version.
+
For example, upgrade from 4.20.20 to 4.21.latest.

. Verify the first y-stream upgrade completed successfully:
+
[source,terminal]
----
$ oc --context=spoke1 get clusterversion
----
+
[source,terminal]
----
$ oc --context=spoke1 get co | grep -v "True.*False.*False"
----
+
All cluster operators must show healthy status before proceeding.

. Wait at least 24 hours, and up to 48 hours, between y-stream upgrades:
+
Allow time to monitor cluster stability after the first upgrade.
Extend this period for large production deployments or if any anomalies are observed.
+
During this wait period:
+
--
* Monitor workload health and performance
* Check for unexpected behavior or errors
* Run smoke tests for critical CNF workloads
* Verify all custom resources specific to your deployment are functioning correctly
--

. Review release notes for the next y-stream version:
+
Check for additional API deprecations or configuration changes introduced in the next version.

. Update OLM operator subscriptions if needed:
+
Some operators might require intermediate channel updates when upgrading through multiple y-streams.
+
[source,terminal]
----
$ oc --context=spoke1 get csv -A
----
+
Verify all operators are at versions compatible with your current cluster version before proceeding.

. Create GitOps policies for the second y-stream upgrade:
+
Update the cluster version policy to target the next y-stream:
+
[source,yaml]
----
apiVersion: policy.open-cluster-management.io/v1
kind: Policy
metadata:
  name: core-upgrade-4.20-policy
  namespace: openshift-upgrade-policies
spec:
  remediationAction: inform
  disabled: false
  policy-templates:
  - objectDefinition:
      apiVersion: policy.open-cluster-management.io/v1
      kind: ConfigurationPolicy
      metadata:
        name: upgrade-cluster-version
      spec:
        remediationAction: inform
        severity: high
        object-templates:
        - complianceType: musthave
          objectDefinition:
            apiVersion: config.openshift.io/v1
            kind: ClusterVersion
            metadata:
              name: version
            spec:
              channel: stable-4.20
              desiredUpdate:
                version: 4.20.0
----

. Apply the updated policy:
+
[source,terminal]
----
$ oc apply -f core-upgrade-4.20-policy.yaml
----

. Create a new ClusterGroupUpgrade CR for the second y-stream:
+
[source,yaml]
----
apiVersion: ran.openshift.io/v1alpha1
kind: ClusterGroupUpgrade
metadata:
  name: core-ystream-upgrade-4.20
  namespace: openshift-upgrade-policies
spec:
  clusters:
  - spoke1
  managedPolicies:
  - core-pause-worker-nodes
  - core-upgrade-4.20-policy
  enable: false
  remediationStrategy:
    maxConcurrency: 1
    timeout: 7200
----

. Enable the CGU to start the second y-stream upgrade:
+
[source,terminal]
----
$ oc apply -f core-ystream-upgrade-4.20-cgu.yaml
$ oc patch cgu core-ystream-upgrade-4.20 \
  -n openshift-upgrade-policies \
  --type merge \
  -p '{"spec":{"enable":true}}'
----

. Monitor the second y-stream upgrade:
+
[source,terminal]
----
$ oc get cgu core-ystream-upgrade-4.20 -n openshift-upgrade-policies -w
----

. After the second y-stream control plane upgrade completes, verify health:
+
[source,terminal]
----
$ oc --context=spoke1 get clusterversion
$ oc --context=spoke1 get co
----

. Update operators for the second y-stream:
+
[source,terminal]
----
$ oc --context=spoke1 get csv -A | grep -v Succeeded
----
+
Update operator channels as needed for the new cluster version.

. Unpause worker nodes and complete the second y-stream upgrade:
+
Follow the worker node unpause procedure from "Manage worker nodes during upgrades".

. Repeat steps 2-14 for any additional intermediate y-stream versions:
+
If upgrading through more than two y-streams, repeat the process for each intermediate version.

.Verification

. Verify the final cluster version:
+
[source,terminal]
----
$ oc --context=spoke1 get clusterversion -o jsonpath='{.items[0].status.desired.version}'
----

. Run comprehensive health checks:
+
Follow the procedure in "Perform health checks before bare metal Core cluster upgrades".

. Verify workload functionality after all upgrades:
+
Run smoke tests and validate that all CNF workloads operate correctly at the final version.

.Troubleshooting

* If the cluster becomes degraded after the first y-stream upgrade, resolve all issues before proceeding to the second y-stream.
* If operator versions are incompatible with an intermediate y-stream, update the operator subscription before upgrading the cluster.
* If configuration changes are needed between y-streams, apply those changes during the wait period before proceeding.
