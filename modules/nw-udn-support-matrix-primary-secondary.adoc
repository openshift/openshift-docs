= Support matrix for primary and secondary networks

The `UserDefinedNetwork` and `NetworkAttachmentDefinition` custom resources (CRs) provide cluster administrators and users the ability to create customizable network configurations and define their own network topologies, ensure network isolation, manage IP addressing for workloads, and leverage advanced network features. A third CR, `ClusterUserDefinedNetwork`, is also available, which allows administrators the ability to create and define additional networks spanning multiple namespaces at the cluster level. Both user-defined networks and network attachment definitions can serve as both the primary and secondary network interface, and each support `layer2` and `layer3` topologies; a third network topology, Localnet, is also supported with network attachment definitions with secondary networks. 

[NOTE]
====
As of {product-title} 4.18, the Localnet topology is unavailable for use with the `UserDefinedNetwork` leveraging primary networks and `ClusterUserDefinedNetwork` APIs. It is available for use with network attachment definitions that leverage secondary networks only.
====

== Network attachment definition support matrix

Network attachment definitions (NADs) offer {product-title} *administrators* the ability to define additional network interfaces to pods, which enables the pods to connect multiple networks with distinct configurations and plugins. This network type is useful for complex networking cases such as network isolation, performance tuning, and dedicated access to external resources.
