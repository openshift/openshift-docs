= Configuring /dev/fuse on unprivileged pods

As an alternative to the virtual filesystem, you can configure the `/dev/fuse` device to the `io.kubernetes.cri-o.Devices` annotation to access faster builds within unprivileged pods. Using `/dev/fuse` is secure, efficient, and scalable, and allows unprivileged users to mount an overlay filesystem as if the unprivileged pod was privileged. 

.Procedure

. Create the pod.

[source,terminal]
----
oc apply -f podman-pod.yaml
----

[IMPORTANT]
====
Ensure that the pod is created and running commands before continuing this procedure. Here is an example configuration similar to what you should see.

[source,yaml]
----
apiVersion: v1
kind: Pod
Metadata:
name: podman-pod
Annotations:
io.kubernetes.cri-o.Devices: "/dev/fuse"
Spec:
containers:
- name: podman-container
image: quay.io/podman/stable
       Args:
- sleep
- "1000000"
securityContext:
runAsUser: 1000
----
====


. Access the shell.
[source,terminal]
----
oc exec -ti podman-pod -- /bin/bash
----

[NOTE]
====
If you receive a `Permission denied` error, you might be in the wrong working directory. If this happens, specify a directory.

[source,terminal]
----
oc exec -ti podman-pod -- /bin/bash -c "cd /desired/directory && bash"
----
====

. Create a Dockerfile and build the image.

[source,terminal]
----
cat >> Dockerfile <<EOF
FROM registry.access.redhat.com/ubi9
EOF
podman build .
----

. Ensure the `/dev/fuse` device is available and properly configured:

[source,terminal]
----
ls /dev/fuse
----
