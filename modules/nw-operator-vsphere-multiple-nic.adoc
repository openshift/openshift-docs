= Specifying multiple network interface controllers (NICs)

Before you install an {product-title} cluster on a {vmw-short} host, you can specify multiple network interface controllers (NICs) for a networking implementation so that the {vmw-short} cloud controller manager (CCM) can select the appropriate NIC for a given networking situation. The {vmw-full} can use the NIC for managing pods and services on your cluster. For this configuration, you must specify internal and external Classless Inter-Domain Routing (CIDR) implementations in the {vmw-short} cloud controller manager (CCM) configuration. Each CIDR implementation lists an IP address range that the CCM uses to decide what NICs interact with traffic from internal and external networks.

If you do not configure internal and external CIDR implementations in the {vmw-short} CCM configuration, the {vmw-short} CCM selects the first NIC that the host operating system reports as the primary network interface. This configuration can cause new nodes that associate with a `MachineSet` object with a single NIC to become unusable as each new node receives the `node.cloudprovider.kubernetes.io/uninitialized` taint. In certain situations, the reported network interface might not be the expected network interface. These situations can cause communication issues with the Kubernetes API server that can cause installation of the cluster to fail.

[IMPORTANT]
====
To prevent any IP address selection issues for each IP protocol, only configure the amount of IP addresses and IP protocols that you will use. If you configured multiple NICs for a network implementation but you intend to only use the IP protocols on a single NIC, ensure that you deploy your cluster nodes on the single NIC.
====

.Example error message because of the {vmw-short} CCM selecting an incompatible NIC
[source, text]
----
ERROR Bootstrap failed to complete: timed out waiting for the condition
ERROR Failed to wait for bootstrapping to complete. This error usually happens when there is a problem with control plane hosts that prevents the control plane operators from creating the control plane.
----

.Prerequisites

* You created Kubernetes manifest files for your {product-title} cluster.

.Procedure

. From the directory where you store your {product-title} cluster manifest files, open the `manifests/cluster-infrastructure-02-config.yml` manifest file.

. Add a `nodeNetworking` object to the file and specify internal and external network subnet CIDR implementations for the object.
+
[NOTE]
====
For most networking situations, consider setting the standard multiple-NIC configuration. This configuration requires that you set the same IP address ranges in the `nodeNetworking.internal.networkSubnetCidr` and `nodeNetworking.external.networkSubnetCidr` parameters.
====
+
.Example of a configured `cluster-infrastructure-02-config.yml` manifest file
[source,yaml]
----
apiVersion: config.openshift.io/v1
kind: Infrastructure
metadata:
  name: cluster
spec:
  cloudConfig:
    key: config
    name: cloud-provider-config
  platformSpec:
    type: VSphere
    vsphere:
      failureDomains:
      - name: generated-failure-domain
      ...
       nodeNetworking:
         external:
           networkSubnetCidr:
           - <machine_network_cidr_ipv4>
           - <machine_network_cidr_ipv6>
         internal:
           networkSubnetCidr:
           - <machine_network_cidr_ipv4>
           - <machine_network_cidr_ipv6>
# ...
----
