= Verify SR-IOV network connectivity

[role="_abstract"]
Test that your SR-IOV network interface is working correctly.

.Procedure

. Verify the SR-IOV interface is up:
+
[source,terminal]
----
$ oc exec -it sriov-test-pod -n default -- ip link show net1
----
+
.Example output
[source,terminal]
----
3: net1: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc mq state UP mode DEFAULT group default qlen 1000
    link/ether 3c:fd:fe:b8:7f:f8 brd ff:ff:ff:ff:ff:ff
----
+
Verify the interface shows `UP` state.

. Check the IP address configuration:
+
[source,terminal]
----
$ oc exec -it sriov-test-pod -n default -- ip addr show net1
----
+
Verify your static IP address (`192.168.1.100/24`) is configured on the interface.

. If you have another device on the same network segment, test connectivity by pinging it:
+
[source,terminal]
----
$ oc exec -it sriov-test-pod -n default -- ping -c 4 -I net1 192.168.1.1
----
+
Replace `192.168.1.1` with an IP address of a device on your SR-IOV network.

. View network statistics for the SR-IOV interface:
+
[source,terminal]
----
$ oc exec -it sriov-test-pod -n default -- ip -s link show net1
----
+
This shows packet statistics (RX/TX packets and bytes) for the interface.

.Troubleshooting

If the SR-IOV interface is not working:

* Verify the virtual function was allocated: Check the pod's resource requests and limits
* Verify the network attachment definition exists: `oc get network-attachment-definitions -n default`
* Check the SR-IOV network configuration: `oc get sriovnetwork -n openshift-sriov-network-operator`
* Review pod events: `oc describe pod sriov-test-pod -n default`
* Check SR-IOV operator logs: `oc logs -n openshift-sriov-network-operator deployment/sriov-network-operator`

[NOTE]
====
If you don't have other devices on the SR-IOV network to test with, you can create a second pod with a different IP address and test connectivity between the two pods.
====
