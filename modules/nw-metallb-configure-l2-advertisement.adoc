= Configuring MetalLB with an L2 advertisement

[role="_abstract"]
You can configure MetalLB so that the `IPAddressPool` is advertised with the L2 protocol.

.Prerequisites

* Install the {oc-first}.
* Log in as a user with `cluster-admin` privileges.
* Configure an `IPAddressPool`.

.Procedure

. Create an IP address pool, if you have not already done so.
+
.. Create a file, such as `ipaddresspool.yaml`, with content like the following example:
+
[source,yaml]
----
apiVersion: metallb.io/v1beta1
kind: IPAddressPool
metadata:
  namespace: metallb-system
  name: doc-example-l2
spec:
  addresses:
    - 4.4.4.0/24
  autoAssign: false
# ...
----
+
.. Apply the configuration for the IP address pool:
+
[source,terminal]
----
$ oc apply -f ipaddresspool.yaml
----

. Create an L2 advertisement.
+
.. Create a file, such as `l2advertisement.yaml`, with content like the following example:
+
[source,yaml]
----
apiVersion: metallb.io/v1beta1
kind: L2Advertisement
metadata:
  name: l2advertisement
  namespace: metallb-system
spec:
  ipAddressPools:
   - doc-example-l2
  # ...
----
+
.L2Advertisement parameters
[cols="2,1,3",options="header"]
|===
| Parameter | Format | Description

| `spec.ipAddressPools`
| List of strings
| Names of `IPAddressPool` resources to advertise. If empty, all pools are advertised.

| `spec.ipAddressPoolSelectors`
| Label selectors
| Optional. Select pools by label instead of by name.

| `spec.nodeSelectors`
| Label selectors
| Optional. Limit which nodes announce the IP addresses. Technology Preview.

| `spec.interfaces`
| List of strings
| Optional. Restrict which network interfaces advertise the IP addresses.
|===
+
.. Apply the configuration:
+
[source,terminal]
----
$ oc apply -f l2advertisement.yaml
----

.Verification

. Verify the `L2Advertisement` resource is created:
+
[source,terminal]
----
$ oc get l2advertisement -n metallb-system
----
+
.Example output
[source,terminal]
----
NAME              AGE
l2advertisement   10s
----

. Create a test `LoadBalancer` service to verify that MetalLB assigns an external IP from the pool:
+
[source,terminal]
----
$ oc create deployment test-web --image=registry.access.redhat.com/ubi9/httpd-24
$ oc expose deployment test-web --type=LoadBalancer --port=8080
----

. Verify an external IP is assigned:
+
[source,terminal]
----
$ oc get svc test-web
----
+
The `EXTERNAL-IP` column must show an IP address from the configured pool.

. Clean up the test resources:
+
[source,terminal]
----
$ oc delete svc test-web
$ oc delete deployment test-web
----

.Troubleshooting
[cols="2,2,2",options="header"]
|===
| Symptom | Likely cause | Resolution

| `EXTERNAL-IP` stuck at `<pending>`
| `L2Advertisement` does not reference the correct pool name, or no `IPAddressPool` is configured
| Verify pool names match: `oc get ipaddresspool -n metallb-system` and check the `ipAddressPools` field in the `L2Advertisement`.

| Service gets an IP but is not reachable from outside the cluster
| `IPAddressPool` addresses are on a different subnet than the cluster node interfaces. L2 mode uses ARP, which only works within the same broadcast domain.
| Verify the pool addresses are on the same subnet as the node IPs: `oc get nodes -o jsonpath='{.items[*].status.addresses[?(@.type=="InternalIP")].address}'`. The pool range must share the same network prefix as the node addresses.

| Slow failover when a node goes down
| L2 mode limitation: clients must update their ARP caches
| This is expected behavior for L2 mode. Failover time depends on client ARP cache timeout. For faster failover, consider BGP mode.
|===
