[role="_abstract"]
The Compliance Operator can evaluate Common Expression Language (CEL) checks that are shipped inside a compliance content image, in addition to Extensible Configuration Checklist Description Format (XCCDF) checks that OpenSCAP runs.
To support that workflow, the `Rule` custom resource definition (CRD) includes optional CEL fields, and the `ProfileBundle` object can reference a bundled CEL content file next to the existing DataStream file.
The profile parser creates `Rule` and `Profile` objects from both sources so that you can run scans through `ScanSetting` and `ScanSettingBinding` objects without authoring a separate `CustomRule` object for every shipped CEL check.

[NOTE]
====
The CEL scanner is available as a link:https://access.redhat.com/support/offerings/techpreview/[Technology Preview] feature.
Technology Preview features are not supported with Red Hat production service level agreements (SLAs) and might not be functionally complete.
For the latest support statement, see the xref:../../security/compliance_operator/compliance-operator-release-notes.adoc#compliance-operator-release-notes[Compliance Operator release notes].
====

[id="compliance-operator-cel-profilebundle-about_{context}"]
= Understanding CEL rules and ProfileBundles

OpenShift compliance content is distributed in an image that the `ProfileBundle` object references.
Traditionally, that image contains an XCCDF DataStream (for example, an `ssg-ocp4-ds.xml` file) that the Compliance Operator turns into `Profile` and `Rule` objects for OpenSCAP scans.

When CEL content is included, the image can also provide a single YAML bundle file that lists CEL `Rule` definitions and the `Profile` objects that reference them.
The Compliance Operator profile parser reads that file when you set the optional `celContentFile` field on the `ProfileBundle` object.
The parser validates CEL expressions when it creates `Rule` objects and records validation errors in the `ProfileBundle` object status if content is invalid.

Each `Rule` object stores an explicit `scannerType` value (`OpenSCAP` or `CEL`) so that you can see which scanner evaluates the rule.
A given `Profile` contains only one kind of rule: CEL rules or OpenSCAP rules, not both.

User-authored CEL checks remain available through the `CustomRule` object and `TailoredProfile` objects, as described in xref:../../security/compliance_operator/co-scans/compliance-operator-tailor.adoc#compliance-operator-tailor[Tailoring the Compliance Operator].
Shipped CEL checks use the same `Rule` CRD as OpenSCAP rules so that content authors can deliver CEL and XCCDF profiles through one `ProfileBundle`.

[id="compliance-operator-cel-profilebundle-example_{context}"]
= Example ProfileBundle object with CEL content

The following example shows a `ProfileBundle` object that references both the XCCDF DataStream file and a CEL bundle file inside the content image:

[source,yaml]
----
apiVersion: compliance.openshift.io/v1alpha1
kind: ProfileBundle
metadata:
  name: ocp4
spec:
  contentImage: registry.example.com/compliance/ocp4-content:latest
  contentFile: ssg-ocp4-ds.xml <1>
  celContentFile: cel-content.yaml <2>
----
<1> The XCCDF DataStream for OpenSCAP profiles.
<2> The bundled YAML file that contains CEL `Rule` and `Profile` definitions for the CEL scanner.

After the `ProfileBundle` object reaches a valid state, the Compliance Operator creates `Profile` and `Rule` objects that you can bind to scans.
For general steps to run scans, see xref:../../security/compliance_operator/co-scans/compliance-scans.adoc#compliance-operator-scans[Compliance Operator scans].

[IMPORTANT]
====
Do not mix OpenSCAP rules and CEL rules in the same `Profile`.
Select a profile that matches the scanner you intend to use, and use `TailoredProfile` objects only to extend profiles of the same scanner type.
====
