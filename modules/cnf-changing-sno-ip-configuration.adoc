= Perform a {sno} network reconfiguration

[role="_abstract"]
You can change the network configuration of a {sno} cluster by editing the `IPConfig` custom resource (CR) and transitioning through the configuration stages.

.Prerequisites

* You have a {sno} cluster.
* You have installed the {lcao}.
* You have the new network configuration details, including IP addresses, gateways, and DNS servers.
* You have cluster administrator privileges.
* You have installed the OpenShift CLI (`oc`).

.Procedure

. Verify that the `IPConfig` CR exists and check its current state:
+
[source,terminal]
----
$ oc get ipc ipconfig -o yaml
----
+
If the CR does not exist, verify that you installed the {lcao}.

. Verify that the `spec.stage` field is set to `Idle`, the `Idle` status condition is set to `true`, and review the current network configuration in the `status` fields.
+
[NOTE]
====
You can only modify spec fields when the CR is in the `Idle` stage.
====

. Edit the `IPConfig` CR to specify the new network configuration:
+
[source,terminal]
----
$ oc edit ipc ipconfig
----

. Update the spec fields with your new network configuration. The following example shows a dual-stack configuration with VLAN and DNS settings:
+
[source,yaml]
----
apiVersion: lca.openshift.io/v1
kind: IPConfig
metadata:
  name: ipconfig
spec:
  stage: Idle <1>
  ipv4:
    address: 192.0.2.10 <2>
    machineNetwork: 192.0.2.0/24 <3>
    gateway: 192.0.2.1 <4>
  ipv6:
    address: 2001:db8::10
    machineNetwork: 2001:db8::/64
    gateway: 2001:db8::1
  dnsServers: <5>
  - 192.0.2.53
  - 2001:4860:4860::8888
  vlanID: 100 <6>
  dnsFilterOutFamily: none <7>
  autoRollbackOnFailure:
    initMonitorTimeoutSeconds: 1800 <8>
----
<1> Set the `stage` field to `Config` when you are ready to apply the new network settings.
<2> The target IPv4 address.
<3> The target machine network CIDR.
<4> The target default gateway.
<5> Ordered list of DNS servers.
<6> Optional VLAN ID. Only specify if the cluster already has VLAN configuration.
<7> Optional DNS filtering for dual-stack clusters. Set to `ipv4` or `ipv6` to filter out A or AAAA records respectively.
<8> Timeout in seconds for automatic rollback if the configuration does not complete. Default is 1800 seconds (30 minutes).

. After saving the configuration, change the stage to `Config` to start the network reconfiguration:
+
[source,terminal]
----
$ oc patch ipc ipconfig --type merge -p '{"spec":{"stage":"Config"}}'
----
+
[NOTE]
====
After triggering the network reconfiguration, update your external DNS servers to resolve the cluster's new API and ingress endpoints.
====

. Monitor the progress of the configuration by running the following command:
+
[source,terminal]
----
$ oc get ipc ipconfig -o yaml
----
+
Watch for the following progression:
+
--
* The controller sets the `ConfigInProgress` condition
* The pre-pivot phase runs and triggers a reboot
* After reboot, the post-pivot phase applies the network changes
* The controller waits for cluster stabilization
* The `ConfigCompleted` condition is set when successful
--

. After the configuration completes successfully, verify that the `status.validNextStages` field includes `Idle` and `Rollback`.

. Verify the new network configuration by running the following command:
+
[source,terminal]
----
$ oc get nodes -o wide
----

. Verify cluster health by running the following command:
+
[source,terminal]
----
$ oc get clusteroperators
----

. When you are satisfied with the new configuration, finalize the change by setting the stage to `Idle`. Run the following command:
+
[source,terminal]
----
$ oc patch ipc ipconfig --type merge -p '{"spec":{"stage":"Idle"}}'
----
+
[IMPORTANT]
====
After you finalize the configuration by transitioning to `Idle`, you cannot roll back to the previous network configuration. The old stateroot is removed during Idle cleanup.
====
