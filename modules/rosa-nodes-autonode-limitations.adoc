[id="rosa-nodes-autonode-limitations_{context}"]
= {autonode} limitations and known issues

[role="_abstract"]
Current limitations, constraints, and known issues for {autonode} on {product-title}.

[id="rosa-nodes-autonode-limits-private-preview_{context}"]
== Private preview limitations

The following limitations apply during the {autonode} private preview period and might be removed in future releases.

Region restrictions::
{autonode} private preview is only available in the `us-east-1` Amazon Web Services (AWS) region. Clusters in other regions cannot enable {autonode} during the private preview.

Cluster shard requirement::
Clusters must be created in the private preview shard (shard ID: `9f11dd2b-98c1-11f0-8fe5-0a580a830a08`). Existing clusters not in this shard cannot enable {autonode} during the private preview.

OpenShift version requirement::
{autonode} requires {product-title} version 4.19 or later. Earlier versions do not support {autonode}.

Hosted control plane requirement::
{autonode} is only supported on {product-title} clusters with hosted control planes (HCP). Clusters with classic control plane architecture cannot use {autonode}.

Manual resource tagging::
During the private preview, you must manually tag subnets and security groups with the Karpenter discovery tag (`karpenter.sh/discovery=<cluster-id>`). Future releases might automate this tagging.

[NOTE]
====
These private preview limitations are expected to be removed when {autonode} reaches general availability.
====

[id="rosa-nodes-autonode-limits-nodepool_{context}"]
== NodePool constraints

NodePools have the following constraints and limitations.

Resource limits are soft caps::
Resource limits in NodePools (`spec.limits.cpu` and `spec.limits.memory`) are soft caps. {autonode} provisions nodes to satisfy pending pods until adding another node would exceed the limit. Actual resource usage might slightly exceed configured limits.

Maximum NodePools per cluster::
While there is no hard limit on the number of NodePools, creating many NodePools can impact cluster performance. As a best practice, limit the number of NodePools to 10-15 per cluster.

Instance type availability::
{autonode} can only provision instance types that are available in the selected AWS availability zones. If the required instance types are unavailable, pods remain pending. Configure NodePools with multiple instance type options to improve availability.

Minimum node size::
{autonode} provisions nodes based on pending pod requirements. Very small pods might result in over-provisioned nodes if no appropriately sized instances exist in the NodePool requirements.

[id="rosa-nodes-autonode-limits-spot_{context}"]
== Spot instance limitations

Spot instances provide cost savings but have important limitations.

Interruptions::
Spot instances can be interrupted by AWS with short notice (typically 2 minutes). Applications running on spot instances must tolerate interruptions and implement graceful shutdown handling.

Availability::
Spot instance availability varies by region, availability zone, and instance type. During periods of high demand, AWS might not have spot capacity available, causing provisioning delays or failures.

No service level agreement::
Spot instances have no service level agreement (SLA) for availability. Do not use spot instances for workloads that require guaranteed uptime.

Interruption handling::
While {autonode} receives interruption warnings and attempts to gracefully drain nodes, applications must implement appropriate shutdown logic to handle interruptions within the 2-minute warning window.

[id="rosa-nodes-autonode-limits-integration_{context}"]
== Integration limitations

{autonode} has the following integration limitations with other {product-title} and AWS features.

Cluster Autoscaler::
{autonode} and the Cluster Autoscaler cannot be used together. Disable the Cluster Autoscaler before enabling {autonode}. {autonode} replaces Cluster Autoscaler functionality for scaling worker nodes.

Machine pools::
{autonode} and machine pools can coexist in the same cluster, but they operate independently. Machine pools do not benefit from {autonode}'s automatic scaling capabilities.

Local persistent storage::
{autonode}-provisioned nodes are ephemeral. Workloads requiring persistent local storage (data stored on instance local disks) should use machine pools or persistent volumes instead.

Custom machine configurations::
Unlike machine pools, you cannot directly specify custom MachineConfig resources for {autonode}-provisioned nodes. Use the `userData` field in OpenShiftEC2NodeClass for custom initialization scripts.

Node labels from machine pools::
Labels automatically applied to machine pool nodes are not automatically applied to {autonode}-provisioned nodes. Configure required labels in the NodePool template.

[id="rosa-nodes-autonode-limits-networking_{context}"]
== Networking limitations

Subnet requirements::
{autonode} requires private subnets tagged with `karpenter.sh/discovery=<cluster-id>`. Nodes provisioned in public subnets are not supported.

Security group requirements::
Security groups must allow traffic required for {product-title} node operation, including kubelet communication on port 10250 and container network traffic.

IPv6::
{autonode} currently supports only IPv4 networking. IPv6-only or dual-stack network configurations are not supported during the private preview.

Multiple VPCs::
{autonode} can only provision nodes in the same Virtual Private Cloud (VPC) as the cluster. Multi-VPC configurations are not supported.

[id="rosa-nodes-autonode-limits-storage_{context}"]
== Storage limitations

Instance store volumes::
While you can configure instance store volumes through block device mappings, data on instance store volumes is lost when nodes are terminated. Use persistent volumes for data that must survive node replacement.

EBS volume encryption::
{autonode} supports Amazon Elastic Block Store (Amazon EBS) volume encryption through block device mappings. However, you cannot specify custom AWS Key Management Service (AWS KMS) keys during the private preview. Encrypted volumes use the default EBS encryption key.

Volume attachment limits::
Each Amazon Elastic Compute Cloud (Amazon EC2) instance type has a maximum number of Amazon EBS volumes that can be attached. {autonode} does not validate that pod volume requirements are within the instance type's attachment limit.

[id="rosa-nodes-autonode-limits-iam_{context}"]
== IAM and security limitations

IAM role modification::
The {autonode} IAM role and policy are created during setup and should not be modified. Modifying the IAM role or policy can cause node provisioning failures.

Custom instance profiles::
While you can specify custom instance profiles in OpenShiftEC2NodeClass, the instance profile must include all permissions required for {product-title} worker nodes. Missing permissions prevent nodes from joining the cluster.

IAM Roles for Service Accounts (IRSA)::
{autonode} uses IRSA to grant the Karpenter controller permissions. The OIDC provider must be configured correctly, and the trust policy must allow the Karpenter service account to assume the role.

[id="rosa-nodes-autonode-limits-monitoring_{context}"]
== Monitoring and observability limitations

CloudWatch integration::
{autonode} does not automatically send metrics to AWS CloudWatch. You must configure monitoring separately to track Amazon EC2 usage and costs.

Prometheus metrics::
While the Karpenter controller exposes Prometheus metrics, these metrics are not integrated with {product-title} monitoring during the private preview. You cannot view {autonode} metrics in the OpenShift console.

Node lifecycle events::
Standard Kubernetes events are generated for node lifecycle operations, but {autonode}-specific events (such as consolidation decisions) might not appear in the {product-title} event stream.

[id="rosa-nodes-autonode-limits-known-issues_{context}"]
== Known issues

The following known issues exist in the current {autonode} implementation.

NodeClass READY=False on first creation::
+
*Symptom*: When you create an OpenShiftEC2NodeClass, it might initially show `READY=False` for 10-30 seconds.
+
*Cause*: The controller needs time to validate AWS resources and create the corresponding EC2NodeClass.
+
*Workaround*: Wait 30 seconds and check the status again. If the status remains `False`, check the resource status conditions for errors.

Pods stuck pending with insufficient capacity::
+
*Symptom*: Pods remain in `Pending` state with events indicating insufficient capacity, even though NodePool limits have not been reached.
+
*Cause*: AWS does not have available capacity for the requested instance types in the selected availability zones.
+
*Workaround*: Broaden the instance type requirements in your NodePool to include more instance families, generations, or availability zones.

Consolidation causes excessive pod rescheduling::
+
*Symptom*: Nodes are repeatedly created and deleted in short cycles, causing frequent pod rescheduling.
+
*Cause*: Aggressive consolidation settings combined with workload characteristics that cause nodes to become underutilized shortly after creation.
+
*Workaround*: Increase the `consolidateAfter` duration or change the consolidation policy to `WhenEmpty`. Configure appropriate pod disruption budgets.

Manual OIDC provider cleanup required::
+
*Symptom*: After deleting a cluster with {autonode} enabled, the OIDC provider and associated IAM resources remain.
+
*Cause*: {autonode} resources are not automatically cleaned up when the cluster is deleted during the private preview.
+
*Workaround*: Manually delete the {autonode} IAM role, policy, and OIDC configuration after cluster deletion.

Node fails to join cluster with custom AMI::
+
*Symptom*: Nodes are provisioned but never reach `Ready` state when using a custom Amazon Machine Image (AMI).
+
*Cause*: The custom AMI is not compatible with {product-title} or is missing required software.
+
*Workaround*: Verify that the custom AMI is based on Red Hat Enterprise Linux CoreOS (RHCOS) and includes all required {product-title} components. Test the AMI with a machine pool before using it with {autonode}.

[id="rosa-nodes-autonode-limits-differences_{context}"]
== Differences from upstream Karpenter

{product-title} {autonode} is based on the upstream Karpenter project but includes ROSA-specific modifications.

OpenShiftEC2NodeClass wrapper::
ROSA introduces the OpenShiftEC2NodeClass custom resource that wraps the upstream EC2NodeClass. This ensures that node classes have appropriate configuration for {product-title} clusters. Always create and modify OpenShiftEC2NodeClass resources, not EC2NodeClass resources directly.

AMI selection::
The default AMI selection differs from upstream Karpenter. {autonode} uses RHCOS AMIs by default, while upstream Karpenter uses Amazon Linux AMIs.

IAM role configuration::
The IAM role and trust policy setup for {autonode} is specific to ROSA and differs from upstream Karpenter documentation. Follow the ROSA-specific IAM setup procedures.

OIDC configuration::
ROSA uses a managed OIDC provider for IRSA. The OIDC provider URL and configuration differ from self-managed Kubernetes clusters.

Feature availability::
Some upstream Karpenter features might not be available in the ROSA {autonode} implementation during the private preview. Refer to this documentation for supported features.

[id="rosa-nodes-autonode-limits-unsupported_{context}"]
== Unsupported features

The following features are not supported in the current {autonode} implementation.

Drift detection::
Automatic drift detection and node replacement based on configuration changes is not supported. Changes to OpenShiftEC2NodeClass do not trigger automatic node replacement.

Custom instance metadata tags::
You cannot add custom tags to instance metadata during the private preview. Tags specified in OpenShiftEC2NodeClass are applied to Amazon EC2 instances and volumes but not to instance metadata.

Multi-architecture clusters::
While you can specify `arm64` architecture in NodePool requirements, multi-architecture clusters with mixed `amd64` and `arm64` nodes are not fully tested during the private preview.

GPU instance scheduling optimization::
Advanced GPU scheduling optimizations, such as time-slicing or multi-instance GPU (MIG) support, are not integrated with {autonode} during the private preview.

Custom node initialization::
Complex node initialization workflows that require multiple stages or conditional logic are not supported. Use the `userData` field for simple initialization scripts only.

[id="rosa-nodes-autonode-limits-planning_{context}"]
== Planning for general availability

When {autonode} reaches general availability, the following changes are expected:

* Removal of region restrictions (all ROSA regions supported)
* Removal of cluster shard requirement
* Automated resource tagging
* Enhanced monitoring integration
* Additional instance type support
* Improved documentation and examples

Plan your {autonode} adoption with the understanding that some private preview limitations and manual steps might be removed or automated in future releases.
