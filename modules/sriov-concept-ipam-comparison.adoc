= Comparing DHCP and Whereabouts

[role="_abstract"]
Both DHCP and Whereabouts provide dynamic IP address assignment - you do not specify IP addresses in pod specifications. However, they work very differently.

== Key difference

*DHCP:* External server on the physical network assigns IP addresses using the DHCP protocol

*Whereabouts:* CNI plugin running inside {product-title} assigns IP addresses from a configured pool

== Common confusion: "Aren't they the same?"

Many users confuse DHCP and Whereabouts because both are "dynamic" - pods get IP addresses automatically without you specifying them in pod specifications.

The key difference is *where* the IP address comes from:

* *DHCP* - Asking the physical network for an address (like a VM or physical server)
* *Whereabouts* - {product-title} assigning from its own pool (cloud-native approach)

Think of it this way:

* *DHCP* is like checking into a hotel - you get a room number from the hotel's reservation system
* *Whereabouts* is like your company assigning you a desk - the assignment comes from internal tracking, not an external system

== Side-by-side comparison

[cols="1,2,2"]
|===
| Aspect | DHCP | Whereabouts

| IP address source
| External DHCP server on physical network
| {product-title} CNI plugin

| Infrastructure required
| DHCP server must exist on physical network
| Nothing - built into {product-title}

| IP pool definition
| DHCP server configuration (outside Kubernetes)
| `SriovNetwork` resource IPAM field (in Kubernetes)

| Additional configuration
| Can provide DNS servers, default routes, NTP servers, boot parameters
| Only IP address, subnet, and gateway

| State storage
| DHCP server database (external)
| Kubernetes CustomResources (etcd)

| Troubleshooting location
| DHCP server logs and lease database
| Kubernetes resources and CNI plugin logs

| Familiar to
| Network administrators
| Kubernetes/platform administrators

| Management team
| Network operations team
| Platform/cluster operations team

| Best for
| Integrating with existing network infrastructure
| Cloud-native, {product-title}-managed deployments

| IP address lifecycle
| Leases with expiration and renewal
| Allocated when pod created, released when pod deleted

| Failure modes
| DHCP server unavailable, pool exhausted, lease expiration
| IP pool exhausted, etcd issues, CNI plugin failures

| Works with existing VMs/servers
| Yes - same DHCP infrastructure
| No - {product-title}-only solution
|===

== Protocol-level differences

*DHCP:*

* Uses DHCP protocol (DISCOVER, OFFER, REQUEST, ACK)
* Broadcasts requests on the physical network
* DHCP server responds with offered IP address
* Client must renew leases periodically

*Whereabouts:*

* No network protocol - internal {product-title} function
* CNI plugin reads pool configuration from network resource
* Allocates next available IP from pool
* No lease renewal required - allocation persists until pod deletion

== Can you use both?

You cannot use both on the same network attachment. Each `SriovNetwork` resource uses one IPAM method:

* Static IPs (specified per pod)
* DHCP (external server)
* Whereabouts ({product-title} pool)

However, you can create multiple `SriovNetwork` resources, each using different IPAM methods for different use cases:

[source,yaml]
----
# Network 1 uses DHCP
apiVersion: sriovnetwork.openshift.io/v1
kind: SriovNetwork
metadata:
  name: sriov-dhcp-network
spec:
  ipam: '{"type": "dhcp"}'
---
# Network 2 uses Whereabouts
apiVersion: sriovnetwork.openshift.io/v1
kind: SriovNetwork
metadata:
  name: sriov-whereabouts-network
spec:
  ipam: '{"type": "whereabouts", "range": "10.10.0.0/16"}'
----

== Migration between IPAM methods

If you start with one IPAM method and need to switch:

* You cannot change IPAM method on an existing network (must create new network)
* Pods using the old network must be recreated to use the new network
* Plan for a transition period where both networks exist
* Update pod specifications or Deployments to reference the new network
