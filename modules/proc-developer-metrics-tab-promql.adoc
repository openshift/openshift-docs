= Running custom PromQL queries in the Metrics tab

[role="_abstract"]
You can create custom Prometheus Query Language (PromQL) queries to monitor specific application metrics in the Developer perspective. The Metrics tab provides PromQL autocomplete and supports both predefined and custom queries.

.Prerequisites

* You have access to the cluster as a developer or as a user with view permissions for the project.
* User-defined monitoring is enabled in the cluster.
* ServiceMonitor or PodMonitor custom resources exist in your project to expose application metrics.
* Your application exposes Prometheus-compatible metrics on an HTTP endpoint.
* You have the `monitoring-rules-view` or `monitoring-rules-edit` role for the project.

.Procedure

To run a predefined query:

. In the Developer perspective of the {product-title} web console, click *Observe* and go to the *Metrics* tab.

. In the *Select Query* dropdown list, select a predefined query.
+
The Metrics tab displays the query results in a graph. Available predefined queries include:
+
--
* CPU usage
* Memory usage
* Receive bandwidth
* Transmit bandwidth
* Rate of received packets
* Rate of transmitted packets
* Rate of received packets dropped
* Rate of transmitted packets dropped
--

. Optional: Click the colored legend boxes next to pod names to toggle their visibility in the graph.
+
This filters the query results to show only the pods you want to monitor.

To create a custom PromQL query:

. In the *Select Query* dropdown list, select *Custom Query*.

. In the *Expression* field, enter your custom PromQL query.
+
The Expression field provides autocomplete support:
+
--
* Metric names appear as you type
* Label names appear after you type an opening brace `{`
* Label values appear after you select a label name
* Functions and operators appear based on query context

Use arrow keys to navigate suggestions, press *Tab* or *Enter* to accept a suggestion, or press *Esc* to dismiss the autocomplete menu.
--
+
[NOTE]
====
All queries are automatically scoped to the selected project namespace.
For example, if you enter `up`, the system automatically rewrites it to `up{namespace="my-project"}`.
Attempting to manually specify a different namespace (for example, `up{namespace="other-project"}`) returns no results because the namespace filter overrides your selection.
You cannot query metrics from other projects in the Developer perspective.
====

. Optional: Click *Show PromQL* to see the complete Prometheus query, including the automatically added namespace filter.

. Click *Run queries* or press *Enter* to execute the query.
+
The Metrics tab displays the query results in a graph.

To view query results in different formats:

. After running a query, the default view shows results in a graph format.

. Optional: Click the *Table* tab to view query results in tabular format.
+
The table view displays metric labels and values, which is useful for comparing specific data points.

To set a time range for query results:

. In the *Time range* dropdown list, select a predefined time range:
+
--
* 5 minutes
* 15 minutes
* 30 minutes
* 1 hour
* 2 hours
* 6 hours
* 12 hours
* 1 day
* 2 days
* 1 week
* 2 weeks
--

. Optional: Select *Custom time range* to specify exact start and end times:
.. Click *Custom time range*.
.. Set the *From* date and time.
.. Set the *To* date and time.
.. Click *Save*.

. Optional: Click *Reset zoom* to return to the default time range after zooming in on a specific time period in the graph.

To set a refresh interval:

. In the *Refresh interval* dropdown list, select how often the query results refresh:
+
--
* Off (no automatic refresh)
* 15 seconds
* 30 seconds
* 1 minute
* 5 minutes
* 15 minutes
* 30 minutes
* 1 hour
* 2 hours
* 1 day
--

.Verification

* The Metrics tab displays query results in graph or table format.
* Custom PromQL queries return metrics scoped to the selected project namespace.

.Example custom queries

The following examples demonstrate custom PromQL queries you can run in the Metrics tab:

* Query all up metrics for the namespace:
+
[source,terminal]
----
up
----
+
Expected output: A list of all monitored targets in your project with their up/down status (1 = up, 0 = down).

* Query HTTP request rate for a specific service:
+
[source,terminal]
----
rate(http_requests_total{service="my-app"}[5m])
----
+
Expected output: The per-second rate of HTTP requests for the `my-app` service over the last 5 minutes.

* Query average CPU usage across all pods:
+
[source,terminal]
----
avg(rate(container_cpu_usage_seconds_total[5m]))
----
+
Expected output: The average CPU usage (in cores) across all containers in your project over the last 5 minutes.

* Query pod memory usage sorted by highest to lowest:
+
[source,terminal]
----
sort_desc(container_memory_usage_bytes)
----
+
Expected output: A list of container memory usage values (in bytes) sorted from highest to lowest.
