= Performance tuning guide for Multicloud Object Gateway

The default configuration of {rh-storage} for Multicloud Object Gateway is optimized for low resource consumption and not performance.


* Large files: In the case of large files, the metadata-to-data ratio is low. Increasing the resources for the endpoints (memory and CPU) and the number of endpoints would be the first thing to do. In the case of namespace buckets, only memory would be sufficient. 
+
The CPU is important mainly for data buckets, where the endpoints use the CPU for encryption and deduplication.

* Small objects: In the case of small objects, the metadata-to-data ratio is high. For data buckets, this means a high level of resource involvement from the core and database. Increasing those pods resources would be the first step.
+
Endpoints memory will probably not be under pressure if the core and database respond quickly. If they do not respond quickly enough, then pressure will build, and endpoints will eventually be under pressure. In this case, you should increase both core and database, with more emphasis on the database itself.

* A high number of configuration entities, such as a large number of buckets or accounts: This would also point to the database and core, with more emphasis on the core than the database.

When using namespace buckets, increasing the endpoint's memory and the database's memory and CPU would be the first step.

*  MCG database resources: You need to increase CPU and memory based on the workload characteristics.

* MCG auto-scales the minimum and maximum size: This would improve the response to peaks, but it also has a delay until it reacts. Therefore, it is important to set both the minimum and maximum size.

* MCG Core resources: You need to increase CPU and memory based upon the workload characteristics.

* Ensure you connect to the NooBaa endpoint by using its service address, link:https://s3.openshift-storage.svc[https://s3.openshift-storage.svc] or link:http://s3.openshift-storage.svc[http://s3.openshift-storage.svc] since this connects directly to the NooBaa endpoints.


[id="mcg-auto-scaling_{context}"]
== Adjust the auto-scaling

* Adjust the auto-scaling with a command like this:

+
[source,yaml]
----
oc patch -n openshift-storage storagecluster ocs-storagecluster \
    --type merge \
    --patch '{"spec": {"multiCloudGateway": {"endpoints": {"minCount": 3,"maxCount": 10}}}}'
----

+
+
This would set the NooBaa Endpoint Horizontal Pod Autoscaling, with the aim of automatically scaling the workload to match demand, to deploy at least 3 pods and scale up to 10 pods when needed. The default is to deploy at least 1 pod and at most 2 pods.

== Tuning MCG core and database resources

* Tuning MCG core and database resources by using the storage cluster Custom Resource (CR).

+
[source,yaml]
----
apiVersion: ocs.openshift.io/v1
kind: StorageCluster
metadata:
  creationTimestamp: "2022-01-15T13:40:42Z"
  generation: 4
  name: ocs-storagecluster
  namespace: openshift-storage
  resourceVersion: "29406833"
  selfLink: /apis/ocs.openshift.io/v1/namespaces/openshift-storage/storageclusters/ocs-storagecluster
  uid: 9f970119-379c-11ea-949c-02bb7e7e425c
spec:
  manageNodes: false
  resources:
    noobaa-core:
      limits:
        cpu: "3"                    <-----
        memory: "4Gi"       <-----
      requests:
        cpu: "3"                    <-----
        memory: "4Gi"       <-----
    noobaa-db:
      limits:
        cpu: "3"                    <-----
        memory: "4Gi"       <-----
      requests:
        cpu: "3"                    <-----
        memory: "4Gi"       <-----
    noobaa-endpoint:
      limits:
        cpu: "3"                    <-----
        memory: 4Gi          <-----
      requests:
        cpu: "3"                    <-----
        memory: 4Gi          <-----
 storageDeviceSets:
.

----

* Apply the above values by issuing the following command:

+
[source,yaml]
----
oc patch -n openshift-storage storagecluster ocs-storagecluster \
    --type merge \
    --patch '{"spec": {"resources": {"noobaa-core": {"limits": {"cpu": "3","memory": "4Gi"},"requests": {"cpu": "3","memory": "4Gi"}},"noobaa-db": {"limits": {"cpu": "3","memory": "4Gi"},"requests": {"cpu": "3","memory": "4Gi"}},"noobaa-endpoint": {"limits": {"cpu": "3","memory": "4Gi"},"requests": {"cpu": "3","memory": "4Gi"}}}}}'
----

[id="mcg-pv-backing-store_{context}"]
== PV pool backing store

When you use the PV pool backing store, you might not have the expected performance due to low default values. To change it:

* Open the OpenShift console.

* Go to *Storage* -> *Object Storage* -> *Backing Store*

* Select the relevant backing store and click *YAML*.

* Go to *spec* -> *pvPool* and update the requests with CPU and memory.

* Add a new property of limits and add CPU and memory as well. For example:

+
[source,yaml]
----
spec:
  pvPool:
    resources:
      limits:
        cpu: 1000m
        memory: 4000Mi
      requests:
        cpu: 800m
        memory: 800Mi
        storage: 50Gi
----

To retrieve the NooBaa logs, you could use the following `NooBaa log collector` script:

[source,bash]
----
#!/bin/bash# Define the function to get logs for StatefulSets
function get_logs_statefulset() {
    local statefulsets=$(oc get statefulset | grep noobaa | awk '{print $1}')
    for ss in $statefulsets; do
        local pods=$(oc get pods -l statefulset.kubernetes.io/pod-name -o name | grep $ss)
        for pod in $pods; do
            echo "Logs for $pod"
            oc logs $pod
        done
    done
}# Define the function to get logs for Deployments
function get_logs_deployment() {
    local deployments=$(oc get deployment | grep noobaa | awk '{print $1}')
    for deploy in $deployments; do
        local pods=$(oc get pods -l app.kubernetes.io/name -o name | grep $deploy)
        for pod in $pods; do
            echo "Logs for $pod"
            oc logs $pod
        done
    done
}# Execute the log retrieval functions
get_logs_statefulset
get_logs_deployment
----
