= Static IP assignment

[role="_abstract"]
With static IP assignment, you specify the exact IP address in the pod specification or network attachment annotation.

== When to use static IPs

Use static IP assignment when:

* You have a small number of pods (fewer than 10-20)
* You need specific, predictable IP addresses
* You are connecting to external systems that require specific source IPs
* You do not have a DHCP server on the SR-IOV network
* You want complete control over addressing

== How it works

You specify the IP address directly in the pod's network annotation:

[source,yaml]
----
apiVersion: v1
kind: Pod
metadata:
  annotations:
    k8s.v1.cni.cncf.io/networks: |
      [{
        "name": "sriov-network",
        "ips": ["192.168.1.10/24"]
      }]
spec:
  containers:
  - name: app
    image: myapp:latest
----

Each pod specification includes the exact IP address the SR-IOV interface should use.

== Advantages

* *Simple* - Easy to understand and configure
* *Predictable* - You know exactly which pod has which IP address
* *No dependencies* - Does not require external infrastructure
* *Complete control* - You manage every address assignment
* *Easy troubleshooting* - IP addresses are visible in pod specifications

== Limitations

* *Does not scale* - Manually managing IP addresses for many pods becomes impractical
* *Error-prone* - Risk of IP address conflicts if you assign the same address to multiple pods
* *Inflexible* - Must update pod specifications to change IP addresses
* *No automation* - You manage every address manually
* *Not suitable for dynamic workloads* - Difficult to use with Deployments that scale up and down

== Best practices

If you use static IP assignment:

* Document your IP address assignments to avoid conflicts
* Use IP addresses from a reserved range that will not conflict with DHCP or other systems
* Consider using ConfigMaps or environment variables to make IP addresses easier to manage
* Plan to migrate to dynamic IPAM (DHCP or Whereabouts) if you scale beyond 10-20 pods
