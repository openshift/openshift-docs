= Comparison between TNF and standard etcd member replacement

[role="_abstract"]
The Two-Node with Fencing (TNF) control-plane node replacement differs from the standard etcd member replacement procedure. The following diagnostic commands and concepts from the standard procedure are also applicable to TNF clusters:

Checking etcd member health::
[source,terminal]
----
$ oc get co etcd -o jsonpath='{.status.conditions[?(@.type=="EtcdMembersAvailable")].message}'
----

Listing etcd members:: In a TNF cluster, run this command from inside the etcd container rather than from a Kubernetes pod:
+
[source,terminal]
----
$ sudo podman exec etcd etcdctl member list -w table
----

Verifying node status::
[source,terminal]
----
$ oc get nodes -l node-role.kubernetes.io/master
----

[NOTE]
====
Both procedures require cluster-admin access and a current etcd backup before you begin.
====

Identifying an unhealthy member:: Check the `EtcdMembersAvailable` condition in both procedures.

The following table shows key differences between standard and TNF replacement.

[cols="1,2,2", options="header"]
|===
|Area |Standard procedure |TNF procedure

|Quorum restoration
|Run `etcdctl member remove` command from a healthy member.
|Cluster etcd Operator (CEO) automatically detects the failed member and runs the quorum restore job. Pacemaker brings etcd online in single-member mode.

|Member removal
|Run `etcdctl member remove <member-id>` command from the etcd pod.
|Delete the Node, BareMetalHost, and Machine resources. CEO handles member removal automatically during quorum restore.

|Member addition
|Re-create the Machine and Node. The etcd cluster operator adds the member and generates TLS certificates.
|CEO runs `tnf-auth-<nodename>` and `tnf-update-setup-<nodename>` jobs after the node becomes Ready. Pacemaker manages the etcd join process.

|Installer pod cleanup
|Not required. Different node names or no same-name replacement scenario.
|Required. Delete stale `app=installer` pods in static-pod operator namespaces after the Node is deleted to prevent blocking.

|Open Virtual Network(OVN) considerations
|Not applicable. Multi-node clusters use a different OVN topology.
|Stale chassis cleanup is required only if the node registers from a backup image before Ironic reprovisions it. The `ovnkube-node` pod on the surviving node might need restarting to resync remote chassis.

|etcd runtime
|etcd runs as a Kubernetes static pod (`/etc/kubernetes/manifests/etcd-pod.yaml`). Use `oc -n openshift-etcd get pods` and `oc logs` for diagnostics.
|etcd runs as a Podman container on the host, managed by the Pacemaker `podman-etcd` resource agent. Use `pcs status` and `podman logs etcd` for diagnostics.

|Pacemaker integration
|No Pacemaker component.
|Pacemaker cleanup (`pcs resource cleanup`, `pcs stonith cleanup`) is required after replacement.

|TLS secret management
|Delete old secrets and recreate them manually.
|Secrets are auto-generated by the CEO. No manual re-creation is needed.

|Static pod trigger
|Installer pods for kube-apiserver, KCM, and scheduler run automatically when the Node joins.
|Installer pods trigger automatically after stale installer pods are deleted. A manual operator log level bump is available as a fallback.
|===
