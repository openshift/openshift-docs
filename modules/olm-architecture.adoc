[role="_abstract"]
Operator Lifecycle Manager (OLM) is composed of two Operators: the OLM Operator and the Catalog Operator. Understanding the division of responsibilities between these Operators helps you troubleshoot issues with custom resource definitions (CRDs).

.CRDs managed by OLM and Catalog Operators
[cols="2a,1a,1a,8a",options="header"]
|===
|Resource |Short name |Owner |Description

|`ClusterServiceVersion` (CSV)
|`csv`
|OLM
|Defines application metadata, such as name, version, icon, and required resources.

|`InstallPlan`
|`ip`
|Catalog
|Lists the calculated resources that are required to automatically install or upgrade a CSV.

|`CatalogSource`
|`catsrc`
|Catalog
|Defines a repository of CSVs, CRDs, and packages that define an application.

|`Subscription`
|`sub`
|Catalog
|Identifies a package and channel to track for updates.

|`OperatorGroup`
|`og`
|OLM
|Configures Operators to watch for custom resources (CRs) in specific namespaces or on the cluster.
|===

The OLM and Catalog Operators create the resources listed in the following table:

.Resources created by OLM and Catalog Operators

|===

|`Deployment`

|`ServiceAccount`
|`Role` or `ClusterRole`
|`RoleBinding` or `ClusterRoleBinding`

|`CustomResourceDefinition` (CRD)
.2+.^|Catalog
|`ClusterServiceVersion`
|===
