= About network observability alerts
////
:FeatureName: Network observability alerts
include::snippets/technology-preview.adoc[]

[role="_abstract"]
Network observability includes predefined alerts so you can make the most of network observability's built-in metrics. The predefined alerts provide a quick health indication of your cluster's network in the *Network Health* dashboard. You can also customize alerts using Prometheus Query Language (PromQL) queries.

By default, network observability creates alerts that are contextual to the features you enable.

For example, packet drop-related alerts are only created if the `PacketDrop` agent feature is enabled in the `FlowCollector` custom resource (CR). Because alerts are built on metrics, you might also see configuration warnings if some enabled alerts are missing their required metrics.

You can configure these metrics in the `spec.processor.metrics.includeList` object of the `FlowCollector` CR.

[id="network-observability-default-alerts_{context}"]
== List of default alerts

These alert templates are installed by default:

`PacketDropsByDevice`:: Triggered on a high percentage of packet drops from devices (`/proc/net/dev`). This alert is grouped by node, with a *Warning* severity above 5%.
`PacketDropsByKernel`:: Triggered on a high percentage of packet drops by the kernel. This alert requires the `PacketDrop` agent feature. Two variants are installed by default:
    * Grouped by node, with *Info* severity above 5% and *Warning* above 10%.
    * Grouped by namespace, with *Info* severity above 10% and *Warning* above 20%.
`IPsecErrors`:: Triggered when the Network Observability Operator detects IPsec encryption errors. This alert requires the `IPSec` agent feature.
`NetpolDenied`:: Triggered when the Network Observability Operator detects traffic that is denied by network policies. This alert requires the `NetworkEvents` agent feature.
`LatencyHighTrend`:: Triggered when the Network Observability Operator detects an increase in TCP latency. This alert requires the `FlowRTT` agent feature.
`DNSErrors`:: Triggered when the Network Observability Operator detects DNS errors. This alert requires the `DNSTracking` agent feature.

These are operational alerts that relate to the self-health of network observability:

`NetObservNoFlows`:: Triggered when no flows are being observed for a certain period.
`NetObservLokiError`:: Triggered when flows are being dropped due to Loki errors.

You can configure, extend, or disable alerts for network observability. The configured alerts generate a `PrometheusRule` resource, which is used to feed the `Prometheus AlertManager` in the `FlowCollector` CR.

[id="network-health-dashboard_{context}"]
== Network Health dashboard

When you enable alerts in the Network Observability Operator, two things happen:

* New alerts appear in *Observe* → *Alerting* → *Alerting rules* tab in the {product-title} web console.
* A new *Network Health* dashboard appears in {product-title} web console → *Observe*.

The *Network Health* dashboard provides a summary of triggered alerts, distinguishing between critical, warning, and minor issues, and also shows pending alerts. The alerts for rule violations are displayed in the following tabs:

* *Global* tab: Shows alerts that are global to the cluster.
* *Nodes* tab: Shows alerts for rule violations per node.
* *Namespaces* tab: Shows alerts for rule violations per namespace.

You can click on a card to see more information. Next to each alert, a three dot menu appears, and you can navigate to *Network Traffic* → *Traffic flows* to see more detailed information for the selected resource.
