= Creating a ClusterLogging resource by using the web console

You can use the {product-title} web console to create a `ClusterLogging` custom resource (CR), as an alternative to applying YAML files by using the {oc-first}.

.Prerequisites

* You have administrator permissions.
* You have installed the Red Hat Openshift Logging Operator.
// TODO: add an attribute for the Operator name and replace in all our docs.
* You have access to the {product-title} web console.

.Procedure

. In the *Administrator* perspective of the web console, navigate to *Operators* -> *Installed Operators*.
. Click *Cluster Logging* in the list of *Provided APIs*.
. In the *ClusterLoggings* page, click *Create ClusterLogging*. You can use the *Form view* or the *YAML view* to complete your CR setup.
+
[NOTE]
====
The *Form view* of the web console does not include all available options. The *YAML view* is recommended for completing your setup.
====
. Verify that the *Name* is `instance`.
. Click *collection*, then select either *fluentd* or *vector* in the *Collector Implementation* drop-down menu.
+
--
include::snippets/logging-fluentd-dep-snip.adoc[]
--
. Click *logStore*, then select either *elasticsearch* or *lokistack* in the *type* drop-down menu.
+
--
include::snippets/logging-elastic-dep-snip.adoc[]
--
. Click *visualization*, then select either *kibana* or *ocp-console* in the *type* drop-down menu.
+
--
include::snippets/logging-kibana-dep-snip.adoc[]
--
. Select *Managed* or *Unmanaged* in the *managementState* drop-down menu. For most deployments, *Managed* is recommended.
+
[IMPORTANT]
====
In some cases, if you change the {logging} default settings, you must set this to *Unmanaged*. However, unmanaged deployments do not receive updates.
====

. Click *Create*.
