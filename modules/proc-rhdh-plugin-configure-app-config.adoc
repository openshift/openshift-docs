= Configure the application settings

[role="_abstract"]
Configure the {rhdh} application settings to enable authentication, permissions, and access for the {prodname} {rhdh} plugin.

.Prerequisites

* You enabled API catalog integration in {rhdh}.
* You provisioned your custom {rhdh} configuration.
* You are logged into an {ocp} cluster with administrator privileges.
* You installed the {oc-first}.

.Procedure

. Append your `app-config.yaml` file with the following base configuration:
+
[source,yaml,subs="+quotes"]
----
app:
  title: Red Hat Developer Hub
  baseUrl: https://frontend.$CLUSTER_HOSTNAME
----

. Enable the permissions framework by adding the following configuration:
+
[source,yaml]
----
permission:
  enabled: true
----

. Configure your authentication by adding your organization's identity provider to the `auth` section by adding the content:
+
[source,yaml]
----
auth:
  providers: {}
----

. Create a role-based access (RBAC) file, such as `rbac-policy.csv`, to define user roles and permissions by using the following example:

----
+
`api consumer`: Browses apis, requests access.
`api owner`: Publishes apis they own, approves requests for their APIs.
`api admin`: People who manage all API products.
`g, group:default`: Assign groups to roles.
`g, user:default/guest, role:default/api-admin`: Only for development and testing. Assigns a guest user an admin role. For production environments, remove this line and assign roles to your actual users or groups

. Create a `ConfigMap` object from the RBAC policy file by running the following command:
+
[source,terminal]
----
$ oc create configmap rbac-policies --from-file=rbac-policy.csv --namespace=$RHDH_NS
----

. Add the RBAC policy reference to the `app-config.yaml` file:
+
[source,yaml]
----
permission:
  enabled: true
  rbac:
    policies-csv-file: /opt/app-root/etc/rbac-policy.csv
    policyFileReload: true
----
+
Make sure that the mounting path `/opt/app-root/etc` configured in the `Backstage` CR matches the path set here.

. Configure Kubernetes access using in-cluster mode by adding one of the following configurations:
+
.. Option 1. Omit the `kubernetes` section entirely by adding the following content:
+
[source,yaml]
----
kubernetes: null
----
+
.. Option 2. Configure the multitenant service locator without a service account token by adding the following content:
+
[source,yaml]
----
kubernetes:
  serviceLocatorMethod:
    type: multiTenant
  clusterLocatorMethods:
    - type: config
      clusters:
        - name: local-cluster
          url: ignored
          authProvider: serviceAccount
----

.. Option 3. Configure the multitenant service by using the `config` cluster locator method with explicit cluster URLs and access tokens by adding the following content:
+
[source,yaml,subs="+quotes"]
----
kubernetes:
  serviceLocatorMethod:
    type: multiTenant
  clusterLocatorMethods:
    - type: config
      clusters:
        - name: remote-cluster
          url: https://example.com
          authProvider: serviceAccount
          serviceAccountToken: ${K8S_ACCESS_TOKEN}
----

. Add the `APIProduct` entity type to the catalog rules by adding the following content:
+
[source,yaml]
----
catalog:
  rules:
    - allow: [Component, System, API, APIProduct, Resource, Location]
----

. Optional. Add catalog locations as required by adding the following content:
+
[source,yaml]
----
catalog:
  locations:
    - type: file
      target: /opt/app-root/src/catalog-entities/all.yaml
----

. Create a `ConfigMap` from the `app-config.yaml` file by running the following command:
+
[source,terminal]
----
$ oc create configmap rhdh-app-config --from-file=app-config.yaml --namespace=$RHDH_NS
----

.Verification

. Verify that the `ConfigMap` resources exist by running the following command:
+
[source,terminal]
----
$ oc get configmap -n $RHDH_NS
----

. Confirm that both `rhdh-app-config` and `rbac-policies` appear in the output.
