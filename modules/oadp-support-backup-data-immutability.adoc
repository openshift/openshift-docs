= OADP does not support backup data immutability

Starting with {oadp-short} 1.3, backups may not function as expected when the target object storage has an immutability option configured. These immutability options are referred to by different names. 

The primary reason for the absence of support lies in the fact that {oadp-short} initially saves the state of a backup as _finalizing_ and then scrutinizes whether any async operations are in progress. If there are async operations in progress, {oadp-short} needs to wait for all of them to be finished before moving the backup state to _complete_. If there are no async operations, and the state is moved to _complete_ immediately. Regardless, {oadp-short} must modify the metadata in object storage, which will not be possible if some kind of immutability is configured on the object storage.

With versions before {oadp-short} 1.3, there was no explicit support in {oadp-short} to work with object storage with an 'immutability' configuration. You may see some problems even though backups seem to work, for example, version objects not being deleted when backup is deleted.

[NOTE]
====
Depending on the specific provider and configuration, backups may still work in some cases.
====

* AWS S3 service supports backups because S3's object lock only applies to versioned buckets, and the object data can still be updated as the new version. However, when backups are deleted, old versions of the objects will not be deleted.

* Azure Storage Blob is capable of supporting both versioned-level immutability and container-level immutability. In a versioned-level situation, data immutability can still work in {oadp-short}, but not at the container level.

* GCP Cloud storage policy only supports bucket-level immutability. Therefore, it is not feasible to implement it in the GCP environment.
