= Upgrading a hosting cluster for virtualized control planes

[role="_abstract"]
Upgrade the hosting cluster and {VirtProductName} while maintaining the availability of the target clusters.

During a hosting cluster upgrade, the Machine Config Operator drains and reboots nodes one at a time.
If pod disruption budgets and anti-affinity rules are configured correctly, the upgrade proceeds without losing `etcd` quorum in any target cluster.

.Prerequisites

* Control plane VMs are distributed across hosting cluster nodes by using anti-affinity rules.
* A `PodDisruptionBudget` with `minAvailable: 2` is configured for each target cluster's control plane VMs.
* The VM eviction strategy is set to `None` for all control plane VMs.

.Procedure

. Verify that all target cluster control plane VMs are healthy before starting the upgrade by running the following command:
+
[source,terminal]
----
$ oc get vm -A -l redfish-enabled=true
----
+
All control plane VMs must show a `Running` status.

. Verify `etcd` health on each target cluster by running the following command:
+
[source,terminal]
----
$ oc --kubeconfig=<target_cluster_kubeconfig> get etcd -o=jsonpath='{range .items[0].status.conditions[?(@.type=="EtcdMembersAvailable")]}{.message}{end}'
----

. Start the hosting cluster upgrade by following the standard {product-title} update process.
+
The Machine Config Operator drains and reboots each node in sequence.
The `PodDisruptionBudget` prevents more than one control plane VM for each target cluster from being evicted at a time.

. Monitor the upgrade progress by running the following commands:
+
[source,terminal]
----
$ oc get clusterversion
$ oc get nodes
----

. After the hosting cluster upgrade completes, upgrade {VirtProductName} if a new version is available.

. Verify that all control plane VMs are running after the upgrade by running the following command:
+
[source,terminal]
----
$ oc get vm -A -l redfish-enabled=true
----

.Verification

* Verify `etcd` health on each target cluster after the upgrade:
+
[source,terminal]
----
$ oc --kubeconfig=<target_cluster_kubeconfig> get etcd -o=jsonpath='{range .items[0].status.conditions[?(@.type=="EtcdMembersAvailable")]}{.message}{end}'
----

* Verify that the target cluster API is responsive:
+
[source,terminal]
----
$ oc --kubeconfig=<target_cluster_kubeconfig> get nodes
----
