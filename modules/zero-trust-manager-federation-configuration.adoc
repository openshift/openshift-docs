= Configuring SPIRE federation

[role="_abstract"]
The {zero-trust-full} includes SPIRE Federation support, allowing multiple independent SPIRE deployments to establish trust relationships. By sharing trust bundles through secure federation endpoints, workloads across different trust domains and clusters can authenticate and communicate with each other.

.Prerequisites

* You have installed the OpenShift CLI (`oc`).

* You have a running SPIRE deployment managed by the {zero-trust-full}.

* You have network connectivity between the clusters you intend to federate.

.Procedure

. Edit your `SpireServer` custom resource to add the `federation` section.
+
.Example YAML
+
[source,yaml]
----
apiVersion: spire.spiffe.io/v1alpha1
kind: SpireServer
metadata:
  name: spire-server
  namespace: spire
spec:
  trustDomain: example.org

  federation:
    bundleEndpoint:
      address: "0.0.0.0"
      port: 8443
      profile: "https_spiffe"
      manageRoute: true
    federatesWith:
      - trustDomain: "remote-domain.org
        bundleEndpointURL: "https://federation.remote-domain.org:8443"
        bundleEndpointProfile: "https_spiffe"
        endpointSpiffeID: "spiffe://remote-domain.org/spire/server"
----

. Configure the 'bundleEnpoint' parameter:
+
[source,yaml]
----
apiVersion: spire.spiffe.io/v1alpha1
kind: SpireServer
metadata:
  name: spire-server
spec:
  # ... other configuration ...
  federation:
    bundleEndpoint:
      address: "0.0.0.0"
      port: 8443
      profile: "https_spiffe"
      manageRoute: true
----

profile:: Set to `https_spiffe` or `https_web`. This profile cannot be changed once set.
manageRoute:: When set to `true`, the Operator automatically creates the OpenShift Route.

. Configure the 'federatesWith' parameter:
+
[source,yaml]
----
federatesWith:
      - trustDomain: "example.org"
        bundleEndpointURL: "https://federation.example.org:8443"
        bundleEndpointProfile: "https_spiffe"
        endpointSpiffeID: "spiffe://example.org/spire/server"
----

endpointSpiffeID:: Required for the `https_spiffe` profile to validate the remote server's identity.

. Apply the configuration changes by running the following command:
+
[source,terminal]
----
$ oc apply -f spire-server-config.yaml
----

. Bootstrap the initial trust bundle:

.. Fetch the trust bundle from the remote federation endpoint using 'curl':
+
[source,terminal]
----
$ curl -k https://<federation.trust-domain>:8443
----

.. Create the `ClusterFederatedTrustDomain` resource, populating the `trustDomainBundle` field with the data fetched in the previous step.
+
[source,yaml]
----
apiVersion: spire.spiffe.io/v1alpha1
kind: ClusterFederatedTrustDomain
metadata:
  name: example.org
spec:
  trustDomain: example.org
  bundleEndpointURL: "https://federation.example.org:8443"
  trustDomainBundle: |
    -----BEGIN CERTIFICATE-----
    MIID... (Content fetched via curl) ...
    -----END CERTIFICATE-----
----

.Verification

. Verify that the OpenShift Route has been created by running the following command:
+
[source,terminal]
----
$ oc get route spire-server-federation -n <namespace>
----

. Check the SPIRE Server logs to ensure it is successfully watching the federated trust domains and updating bundles.
