= Managing high availability (HA) for the NUMA-aware scheduler

The NUMA Resources Operator manages the high availability (HA) of the topology-aware secondary scheduler based on how the `spec.replicas` field is configured in the NUMAResourcesScheduler Custom Resource (CR). As an OpenShift Administrator, you have three options for controlling the scheduler's behavior.

[discrete]
[id="default-ha-mode_{context}"]
== Default high-availability mode

If the `spec.replicas` field is not specified in the NUMAResourcesScheduler custom resource (CR), the NUMA Resources Operator automatically enables a high-availability (HA) mode. In this mode, the operator creates one scheduler replica for each control-plane node in the cluster to ensure redundancy.

The following manifest demonstrates this default behavior. The replicas field is omitted, which enables automatic replica detection.

[source,yaml,subs="attributes+"]
----
apiVersion: nodetopology.openshift.io/v1
kind: NUMAResourcesScheduler
metadata:
  name: example-auto-ha
spec:
  imageSpec: 'registry.redhat.io/openshift4/noderesourcetopology-scheduler-rhel9:v{product-version}'
  # The 'replicas' field is not included, enabling auto-detection.
----

[discrete]
[id="customizing-replicas_{context}"]
== Customizing replicas

You can explicitly set a specific number of scheduler replicas by defining the `spec.replicas` field. This action overrides the default HA behavior. The controller then ensures the number of running scheduler pods matches the specified value.

[source,yaml,subs="attributes+"]
----
apiVersion: nodetopology.openshift.io/v1
kind: NUMAResourcesScheduler
metadata:
  name: example-custom
spec:
  imageSpec: 'registry.redhat.io/openshift4/noderesourcetopology-scheduler-rhel9:v{product-version}'
  replicas: 2
----

[discrete]
[id="disabling-numa-aware-scheduling_{context}"]
== Disabling NUMA-aware scheduling

To disable the scheduler, set the `spec.replicas` field to `0`. The NUMA Resources Operator then terminates all running scheduler pods and prevents new ones from starting.

The following manifest disables the scheduler:

[source,yaml,subs="attributes+"]
----
apiVersion: nodetopology.openshift.io/v1
kind: NUMAResourcesScheduler
metadata:
  name: example-custom
spec:
  imageSpec: 'registry.redhat.io/openshift4/noderesourcetopology-scheduler-rhel9:v{product-version}'
  replicas: 0
----

.Verification 

After applying a `NUMAResourcesScheduler` manifest, an OpenShift Administrator should verify that the changes have taken effect.


Check the number of running scheduler pods to confirm it matches your configuration. Use the `oc get pods` command with a label selector to list only the scheduler pods.

* If you used the **default HA mode**, the number of pods should equal the number of control-plane nodes in your cluster.
* If you **customized the replicas**, the number of pods should match the value you set.
* If you **disabled the scheduler**, there should be no running pods with this label.

[source,terminal]
----
oc get pods -n openshift-numaresources-operator -l app=numaresources-scheduler
----

*Expected Output:* The output will show a list of scheduler pods, and you can count them to verify the replica count. For example, if you set `replicas: 2`, you should see two pods listed.

For a more detailed verification, you can inspect the status of the `NUMAResourcesScheduler` custom resource itself. This shows the controller's view of the configuration and its current state.

Use the `oc describe` command to get a full report of the CR.

[source,terminal]
----
oc describe numaresourcesschedulers -n openshift-numaresources-operator example-custom
----

*Expected Output:* In the output, look for the `Status` section. It should provide information about the number of `replicas` and the `readyReplicas` to confirm the desired state has been reached.

Example `Status` section:

[source,yaml]
----
...
Status:
  Conditions:
    ...
  Observed Generation: 1
  Ready Replicas: 2
  Replicas: 2
...
----
