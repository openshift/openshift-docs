= Managing high availability (HA) for the NUMA-aware scheduler

The NUMA Resources Operator manages the high availability (HA) of the topology-aware secondary scheduler based on how the `spec.replicas` field is configured in the `NUMAResourcesScheduler` Custom Resource (CR). Control scheduler behavior by using one of three available options.

--
:FeatureName: Managing high availability
include::snippets/technology-preview.adoc[]
--

[discrete]
[id="default-ha-mode_{context}"]
== Default high-availability mode

If the `spec.replicas` field is not specified in the `NUMAResourcesScheduler` custom resource (CR), the NUMA Resources Operator automatically enables HA mode. In this mode, the Operator creates one scheduler replica for each control-plane node in the cluster to ensure redundancy. A limit of 3 replicas is enforced for the NUMA-aware scheduler.

The following manifest demonstrates this default behavior. Omit the replicas field to enable automatic replica detection. 

[source,yaml,subs="attributes+"]
----
apiVersion: nodetopology.openshift.io/v1
kind: NUMAResourcesScheduler
metadata:
  name: example-auto-ha
spec:
  imageSpec: 'registry.redhat.io/openshift4/noderesourcetopology-scheduler-rhel9:v{product-version}'
  # The 'replicas' field is not included, enabling auto-detection.
----

[discrete]
[id="customizing-replicas_{context}"]
== Customizing replicas

You can explicitly set a specific number of scheduler replicas by defining the `spec.replicas` field. This action overrides the default HA behavior. The controller then ensures the number of running scheduler pods matches the specified value.

[source,yaml,subs="attributes+"]
----
apiVersion: nodetopology.openshift.io/v1
kind: NUMAResourcesScheduler
metadata:
  name: example-custom
spec:
  imageSpec: 'registry.redhat.io/openshift4/noderesourcetopology-scheduler-rhel9:v{product-version}'
  replicas: 2
----

[discrete]
[id="disabling-numa-aware-scheduling_{context}"]
== Disabling NUMA-aware scheduling

To disable the scheduler, set the `spec.replicas` field to `0`. The NUMA Resources Operator then terminates all running scheduler pods and prevents new ones from starting.

The following manifest disables the scheduler:

[source,yaml,subs="attributes+"]
----
apiVersion: nodetopology.openshift.io/v1
kind: NUMAResourcesScheduler
metadata:
  name: example-custom
spec:
  imageSpec: 'registry.redhat.io/openshift4/noderesourcetopology-scheduler-rhel9:v{product-version}'
  replicas: 0
----

.Verification 

. Run the following command with a label selector to list only the scheduler pods:
+
[source,terminal]
----
$ oc get pods -n openshift-numaresources -l app=secondary-scheduler
----
+
.Expected Output

If you used the **default HA mode**, you should see that the number of pods is equal to the number of control-plane nodes. For a standard HA {product-title} cluster, which typically has three control plane nodes, you would expect to see three pods, for example:
+
[source,terminal]
----
NAME                                   READY   STATUS    RESTARTS   AGE
secondary-scheduler-5b8c9d479d-2r4p5   1/1     Running   0          5m
secondary-scheduler-5b8c9d479d-k2f3p   1/1     Running   0          5m
secondary-scheduler-5b8c9d479d-q8c7b   1/1     Running   0          5m
----
+
* If you **customized the replicas**, the number of pods should match the value you set.
* If you **disabled the scheduler**, there should be no running pods with this label.
+
[NOTE]
====
A limit of 3 replicas is enforced for the NUMA-aware scheduler. On a on hosted control planes cluster, the scheduler pods are running on the worker nodes of the hosted-cluster.
====

. Run the following command to verify the number of replicas and their status:
+
[source,terminal]
----
$ oc get deployment secondary-scheduler -n openshift-numaresources
----
+
.Expected Output
+
[source,terminal]
----
NAME                  READY   UP-TO-DATE   AVAILABLE   AGE
secondary-scheduler   3/3     3            3           5m
----
+
In this output, 3/3 means 3 replicas are ready out of an expected 3 replicas.


. Run the following command for a more detailed verification:
+
[source,terminal]
----
$ oc describe deployment secondary-scheduler -n openshift-numaresources
----
+
.Expected output

The replicas line shows a deployment configured for 3 replicas, with all 3 updated and available.
+
[source,yaml]
----
Replicas:        3 desired | 3 updated | 3 total | 3 available | 0 unavailable
----
+
This indicates that the deployment is healthy, with all three pods running as intended.
