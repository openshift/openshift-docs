= Deploy application workloads in each mesh
:context: ossm-deploy-application-workloads-in-each-mesh

This procedure demonstrates how to deploy application workloads in separate namespaces.

.Procedure

. Create an application namespace called `app-ns-1`:
+
[source,terminal]
----
$ oc create namespace app-ns-1
----

. Add the `istio-discovery=mesh-1` label to ensure that the namespace is discovered by the first control plane.
+
[source,terminal]
----
$ oc label namespace app-ns-1 istio-discovery=mesh-1
----

. Add the `istio.io/rev=mesh-1` label to the namespace to enable sidecar injection into all the pods by default while ensuring that pods in this namespace are mapped to the first control plane.
+
[source,terminal]
----
$ oc label namespace app-ns-1 istio.io/rev=mesh-1
----
+
[NOTE]
====
You can verify the `mesh-1` revision name by running the following command:
[source,terminal]
----
$ oc get istiorevisions
----
====

. Deploy the `sleep` and `httpbin` applications:
+
[source,terminal]
----
$ oc apply -n app-ns-1 \
   -f https://raw.githubusercontent.com/openshift-service-mesh/istio/release-1.23/samples/sleep/sleep.yaml
   -f https://raw.githubusercontent.com/openshift-service-mesh/istio/release-1.23/samples/httpbin/httpbin.yaml
----

. Wait for the `httpbin` and `sleep` pods to run with sidecars injected:
+
[source,terminal]
----
$ oc get pods -n app-ns-1
----
+
You should see output similar to the following example:
+
.Example output
[source,terminal]
----
NAME                       READY   STATUS    RESTARTS   AGE
httpbin-7f56dc944b-kpw2x   2/2     Running   0          2m26s
sleep-5577c64d7c-b5wd2     2/2     Running   0          91m
----

. Create two more namespaces called `app-ns-2` and `app-ns-3`:
+
[source,terminal]
----
$ oc create namespace app-ns-2
----
+
[source,terminal]
----
$ oc create namespace app-ns-3
----

. Add labels to both namespaces with `istio-discovery=mesh-2` and the revision label `mesh-2` to match the discovery selector of the second control plane:
+
[source,terminal]
----
$ oc label namespace app-ns-2 app-ns-3 istio-discovery=mesh-2 istio.io/rev=mesh-2
----

. Deploy the `sleep` and `httbin` applications to both namespaces:
+
[source,terminal]
----
$ oc apply -n app-ns-2 \
   -f https://raw.githubusercontent.com/openshift-service-mesh/istio/release-1.23/samples/sleep/sleep.yaml \
   -f https://raw.githubusercontent.com/openshift-service-mesh/istio/release-1.23/samples/httpbin/httpbin.yaml
----
+
[source,terminal]
----
$ oc apply -n app-ns-3 \
   -f https://raw.githubusercontent.com/openshift-service-mesh/istio/release-1.23/samples/sleep/sleep.yaml \
   -f https://raw.githubusercontent.com/openshift-service-mesh/istio/release-1.23/samples/httpbin/httpbin.yaml
----
+
[NOTE]
====
You can use the following command to wait for deployment to be ready:

[source,terminal]
----
$ oc wait deployments -n app-ns-2 --for condition=Available
----
====

.Verification

After deploying the applications, use the `istioctl ps` command to verify that each application workload is managed by its assigned control plane.

.. Run the following procedure to verify workloads assigned to the control plane in `istio-system-1`:
[source,terminal]
----
$ istioctl ps -i istio-system-1
----
You should see output similar to the following example:

.Example output
[source,terminal]
----

Use the following procedure to verify that application connectivity is restricted to workloads within their respective mesh:

. Send a request from the `sleep` pod in `app-ns-1` to the `httpbin` service in `app-ns-2`. The communication should fail:
+
[source,terminal]
----
$ oc -n app-ns-1 exec deploy/sleep -c sleep -- curl -sIL http://httpbin.app-ns-2.svc.cluster.local:8000
----
+
You should see output similar to the following example:
+
.Example output
[source,terminal]
----
HTTP/1.1 503 Service Unavailable
content-length: 95
content-type: text/plain
date: Wed, 16 Oct 2024 12:05:37 GMT
server: envoy
----

. Send a request from the `sleep` pod in `app-ns-2` managed by `mesh-2` to the `httpbin` service in `app-ns-3` also managed by `mesh-2`. The communication should work:
+
[source,terminal]
----
$ oc -n app-ns-2 exec deploy/sleep -c sleep -- curl -sIL http://httpbin.app-ns-3.svc.cluster.local:8000
----
+
You should see output similar to the following example:
+
.Example output
[source,terminal]
----
HTTP/1.1 200 OK
access-control-allow-credentials: true
access-control-allow-origin: *
content-security-policy: default-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' camo.githubusercontent.com
content-type: text/html; charset=utf-8
date: Wed, 16 Oct 2024 12:06:30 GMT
x-envoy-upstream-service-time: 8
server: envoy
transfer-encoding: chunked
----
