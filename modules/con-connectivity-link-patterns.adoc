= Using {prodname} technologies and patterns

[role="_abstract"]
You can use the following technologies and patterns with {prodname}:

Gateway API::
You can use Gateway API to set up ingress policies on each {ocp} cluster to be identical, consistent, and implemented simultaneously.

In Kubernetes-based environments, link:https://gateway-api.sigs.k8s.io/[Gateway API] is the standard for deploying ingress gateways and managing application networking. Gateway API provides standardized APIs for ingress traffic management and support for many protocols.
+
.Gateway API user-persona-based design
image::connectivity-link-gateway-api.png[{prodname} Gateway API user-persona-based design]

Policy-based configuration::
You can use the {prodname} _policy attachment_ pattern to add behavior to a Kubernetes object by using configuration that cannot be described in the object `spec` field. Policy attachments also provide the concept of defaults and overrides. These defaults and overrides mean that you can configure different roles to operate with policy APIs at different levels of the object hierarchy. These policies are then merged with specific rules and strategies to form an effective policy.
+
For example, the following rate-limiting policy configures a specified limit of 5 requests per 10 seconds for every listener defined in the target gateway that does not have its own rate limiting policy defined:
+
.Rate-limiting policy example
[source,yaml]
----
apiVersion: kuadrant.io/v1
kind: RateLimitPolicy
metadata:
  name: gw-rlp
spec:
  targetRef: # Specifies Gateway API policy attachment
    group: gateway.networking.k8s.io
    kind: Gateway
    name: external
  defaults: # Means it can be overridden
    limits: # Limitador component configuration
      "global":
        rates:
        - limit: 5
          window: 10s

----

WebAssembly plugin::

As a is a link:https://webassembly.org/[WebAssembly (WASM)] plugin developed for the Envoy proxy, {prodname} is lightweight, hardware independent, non-intrusive, and secure. This means that clusters that are using {ocp} Service Mesh, Istio, or Envoy for ingress do not require major changes to their existing ingress objects and policies to begin using {prodname}.

Multicluster configuration mirroring::

You can configure and deploy your policies across different cloud service providers in a consistent way with {prodname}. {prodname} uses multicluster configuration mirroring across multicloud and hybrid cloud environments. You can deploy your routing, configuration, and policies wherever you need them with one set of CRDs.
