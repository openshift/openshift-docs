= Scanner V4 sources

[role="_abstract"]
Scanner V4 uses many vulnerability data sources including Red{nbsp}Hat VEX, OSV, NVD, and various Linux distribution security databases.

Scanner V4 uses the following vulnerability sources:

Red{nbsp}Hat VEX:: Use this source with release 4.6 and later. This source offers vulnerability data in Vulnerability Exploitability eXchange (VEX) format. {product-title-short} takes advantage of VEX benefits to significantly decrease the time needed for the initial loading of vulnerability data, and the space needed to store vulnerability data. VEX also offers improved accuracy over OVAL.
+
{product-title-short} might list a different number of vulnerabilities when you are scanning with a {product-title-short} version that uses OVAL, such as {product-title-short} version 4.5, and a version that uses VEX, such as version 4.6. For example, {product-title-short} no longer displays vulnerabilities with a status of "under investigation," while earlier versions that used OVAL data included these vulnerabilities.
+
For containers from or built on top of containers from the Red{nbsp}Hat Ecosystem Catalog, {product-title-short} offers an option to show only vulnerabilities from Red{nbsp}Hat's VEX data. VEX data for Red{nbsp}Hat images is the most exact because the Red{nbsp}Hat security team vets the vulnerabilities in those images and reports the results in VEX. Other vulnerability sources such as OSV can report vulnerabilities that Red{nbsp}Hat has determined are not applicable to the image. This can cause false positives in vulnerability results. Enabling the setting to use only VEX data for Red{nbsp}Hat images minimizes these false positives.
+
The option to use VEX data for Red{nbsp}Hat containers defaults to disabled. To enable this option, in Scanner V4 Matcher, set the environment variable `ROX_SCANNER_V4_RED_HAT_LAYERS_RED_HAT_VULNS_ONLY` to `true`. Be aware that in rare instances, using this option can cause valid vulnerabilities to not appear in scan results, or false negatives. For example, Red{nbsp}Hat does not track vulnerabilities for products that have reached end of life. Also be aware that Red{nbsp}Hat's VEX data is missing exact security data for many Middleware products.
+
For more information about Red{nbsp}Hat security data, including information about the use of OVAL, Common Security Advisory Framework Version 2.0 (CSAF), and VEX, see "The future of Red{nbsp}Hat security data".
OSV:: Use this for language-related vulnerabilities, such as Go, Java, JavaScript, Python, and Ruby. This source might offer vulnerability IDs other than CVE IDs for vulnerabilities, such as a GitHub Security Advisory (GHSA) ID.
+
[NOTE]
====
{product-title-short} uses the OSV database available at OSV.dev under the Apache License 2.0.
====
NVD:: Use this for various purposes such as filling in information gaps when vendors do not offer information. For example, Alpine does not offer a description, CVSS score, severity, or published date.
+
[NOTE]
====
This product uses the NVD API but is not endorsed or certified by the NVD.
====
Additional vulnerability sources::
* Alpine Security Database
* Data tracked in Amazon Linux Security Center
* Debian Security Tracker
* Oracle OVAL
* Photon OVAL
* SUSE OVAL
* Ubuntu OVAL
* StackRox: The upstream StackRox project maintains a set of vulnerabilities that you might not discover due to data formatting from other sources or absence of data.

Scanner V4 Indexer sources:: Scanner V4 indexer uses the following files to index Red{nbsp}Hat containers:

* repository-to-cpe.json: Maps RPM repositories to their related CPEs, which you need for matching vulnerabilities for RHEL-based images.
* container-name-repos-map.json: This matches container names to their corresponding repositories.
