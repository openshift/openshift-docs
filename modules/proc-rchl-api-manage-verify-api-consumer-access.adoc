= Verifying the API consumer access role

[role="_abstract"]
You can verify that your role-based-access controls are configured the way you intend by using the CLI.

[NOTE]
====
You can replace `--as-group` with `--as=_<username>_` or `--as=system:serviceaccount:_<namespace>_:_<name>_` as needed.
====

.Prerequisites

* You enabled the {prodname} {ocp} web console plugin.
* You have either the `api-owner` role bound in your namespace, or the `api-admin` role bound cluster-wide.
* You configured other role bindings as required for your use case.

.Procedure

. Verify API consumer catalog browsing permissions by running the following command:
+
[source,terminal,subs="+quotes"]
----
$ oc auth can-i list apiproducts \
  --as-group=_<consumer_group>_ \
  --all-namespaces
----
+
* Replace `_<consumer_group>_` with the name of the API consumer group.
+
.Example output
[source,terminal]
----
yes
----

. Verify API consumer `APIKey` management permissions in their namespace by running the following command:
+
[source,terminal,subs="+quotes"]
----
$ oc auth can-i create apikeys \
  --as-group=_<consumer_group>_ \
  -n _<consumer_namespace>_
----
+
* Replace `_<consumer_group>_` with the name of the API consumer group.
* Replace `_<consumer_namespace>_` with a the API consumer namespace.
+
.Example output
[source,terminal]
----
yes
----

. Verify that API consumers cannot create `APIKey` resources in other namespaces by running the following command:
+
[source,terminal,subs="+quotes"]
----
$ oc auth can-i create apikeys \
  --as-group=_<consumer_group>_ \
  -n _<other_namespace>_
----
+
* Replace `_<consumer_group>_` with the name of the API consumer group.
* Replace `_<other_namespace>_` with a namespace that is not theirs.
+
.Example output
[source,terminal]
----
no
----
