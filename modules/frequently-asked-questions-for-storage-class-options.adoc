= Frequently asked questions for storage class options

[role="_abstract"]
You can find answers to frequently asked questions about configuring and managing storage class options in Logical Volume Manager Storage (LVMS). Learn how to handle file system types, custom parameters, labels, empty configurations, and immutable fields.

*Can I change the filesystem type through `additionalParameters`?*

No. The `csi.storage.k8s.io/fstype` parameter is managed by LVMS and is rejected at admission if set through `additionalParameters`. To use `ext4` instead of the default `xfs`, use the `fstype` field on the device class:

[source,yaml]
----
deviceClasses:
- name: vg1
  fstype: ext4
----

Note that `fstype` is also immutable after creation.

*What happens if I set a custom parameter that TopoLVM does not recognize?*

Nothing. The parameter is stored on the StorageClass object and passed to the CSI driver during provisioning, but TopoLVM ignores any parameters it does not recognize. PVC provisioning, pod mounting, and volume operations continue to work normally.

*Can I add labels directly to the StorageClass outside of the LVMCluster CR?*

Yes. Labels added directly to the StorageClass, for example with `oc label storageclass lvms-vg1 my-label=value`, are preserved by the operator. Only labels managed through `additionalLabels` in the `LVMCluster` CR are subject to pruning when removed from the CR.

*What happens if I set `storageClassOptions: {}` (empty)?*

This produces the same StorageClass as omitting `storageClassOptions` entirely: `reclaimPolicy: Delete`, `volumeBindingMode: WaitForFirstConsumer`, no additional parameters, and no additional labels.

*How do I change an immutable field such as `reclaimPolicy` or `volumeBindingMode`?*

Delete the `LVMCluster` and recreate it with the new values. You cannot update immutable fields in place. See xref:deleting-an-lvmcluster[Deleting an LVMCluster] and xref:changing-an-immutable-field[Changing an immutable field] for the full procedure.
