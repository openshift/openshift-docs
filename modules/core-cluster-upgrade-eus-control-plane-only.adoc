= Updating with EUS-to-EUS control-plane-only updates

[role="_abstract"]
You can update bare-metal core clusters from one Extended Update Support (EUS) release to another using control-plane-only updates.
This approach updates the control plane while leaving worker nodes at the previous EUS version, enabling staged rollouts that minimize workload disruption.

EUS releases provide 18 months of support and include even-numbered minor versions such as 4.18, 4.20, and 4.22.
Control plane and worker nodes can be up to two minor versions apart during EUS-to-EUS updates.

WARNING: Control plane and worker version skew is limited to N-2 (two minor versions).
For example, if the control plane is at 4.20, workers must be updated before the control plane reaches 4.22, or they will be in an unsupported configuration.
Plan worker updates accordingly to avoid exceeding the version skew policy.
For more information about version skew policies, see the Kubernetes version and version skew support policy.

.Prerequisites

* Pre-update health check completed successfully.
* {cgu-operator} is installed and configured on the {rh-rhacm-first} hub cluster.
* Current cluster is at the latest z-stream of a source EUS version (for example, 4.20.latest).
* Target EUS version is available (for example, 4.22).
* EUS-to-EUS update path is supported (verify in release notes).

.Procedure

. Verify the current cluster version is at the latest z-stream of the source EUS by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get clusterversion
----
+
.Example output
[source,terminal]
----
NAME      VERSION   AVAILABLE   PROGRESSING   SINCE   STATUS
version   4.20.35   True        False         30d     Cluster version is 4.20.35
----
+
If not at the latest z-stream, update to the latest patch release before proceeding.

. Review the EUS-to-EUS release notes:
+
Check the {product-title} documentation for:
+
--
* Supported EUS-to-EUS update paths
* API deprecations between EUS versions
* Breaking changes affecting your workloads
* Control plane and worker version skew policies
--

. Validate that cloud-native network function (CNF) workloads tolerate control plane and worker version skew:
+
Verify that your network functions can operate with the control plane at 4.22 while workers remain at 4.20.
+
Consult with application teams if needed.

. Plan the worker node update schedule:
+
Determine when to update worker nodes after the control plane update completes.
+
Worker updates can be scheduled days or weeks after the control plane, depending on your maintenance windows.

. Create a GitOps policy to pause worker nodes:
+
[source,yaml]
----
apiVersion: policy.open-cluster-management.io/v1
kind: Policy
metadata:
  name: core-pause-worker-nodes
  namespace: openshift-upgrade-policies
spec:
  remediationAction: inform
  disabled: false
  policy-templates:
  - objectDefinition:
      apiVersion: policy.open-cluster-management.io/v1
      kind: ConfigurationPolicy
      metadata:
        name: pause-worker-mcp
      spec:
        remediationAction: inform
        severity: high
        object-templates:
        - complianceType: musthave
          objectDefinition:
            apiVersion: machineconfiguration.openshift.io/v1
            kind: MachineConfigPool
            metadata:
              name: worker
            spec:
              paused: true
----

. Verify the current cluster channel before creating the update policy by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get clusterversion -o jsonpath='{.items[0].spec.channel}'
----
+
.Example output
[source,terminal]
----
eus-4.20
----
+
If the cluster is on a non-EUS channel (such as `stable-4.20`), changing to an EUS channel and setting the required version simultaneously can cause unexpected update behavior.

. Get the release image reference for the target version.
+
The image reference is required for the update policy. Retrieve it by running the following command:
+
[source,terminal]
----
$ oc adm release info 4.21.5 | head -9
----
+
.Example output
[source,terminal]
----
Name:           4.21.5
Digest:         sha256:93879f84b3165c5b5bd1fdf4563a11155dc61ea35cd93e67dc61c2b66e11c8bb
Created:        2025-03-13T09:17:26Z
OS/Arch:        linux/amd64
Manifests:      758
Metadata files: 2

Pull From: quay.io/openshift-release-dev/ocp-release@sha256:93879f84b3165c5b5bd1fdf4563a11155dc61ea35cd93e67dc61c2b66e11c8bb
----
+
Use the `Pull From` value as the `image` field in the update policy.

. Create a policy for the EUS update.
+
For an EUS-to-EUS update, you need a separate policy for each Y-stream step. The following example shows a policy for the first Y-stream update:
+
[source,yaml]
----
apiVersion: policy.open-cluster-management.io/v1
kind: Policy
metadata:
  name: upgrade-ocp-20to21
  namespace: upgrade
  annotations:
    policy.open-cluster-management.io/categories: CM Configuration Management
    policy.open-cluster-management.io/controls: CM-2 Baseline Configuration
    policy.open-cluster-management.io/standards: NIST SP 800-53
  labels:
    app.kubernetes.io/instance: policies
spec:
  disabled: false
  policy-templates:
    - objectDefinition:
        apiVersion: policy.open-cluster-management.io/v1
        kind: ConfigurationPolicy
        metadata:
          name: upgrade-ocp-20to21
        spec:
          object-templates:
            - complianceType: musthave
              objectDefinition:
                apiVersion: config.openshift.io/v1
                kind: ClusterVersion
                metadata:
                  name: version
                spec:
                  channel: eus-4.22
                  desiredUpdate:
                    force: true
                    image: quay.io/openshift-release-dev/ocp-release@sha256:df00d64554b9b5d860273f938e0ff3b0c4a7af8356fbd103658ebe6fa2a830c8
                    version: 4.21.29
                status:
                  history:
                    - state: Completed
                      version: 4.21.29
          remediationAction: inform
          severity: low
  remediationAction: inform
----
+
Replace the following values:
+
--
* `channel`: Set the cluster channel to the target EUS release. Refer to the link:https://access.redhat.com/labs/ocpupgradegraph/update_path/[update selection tool] when setting the channel.
* `image`: Specify the release image reference obtained from the `oc adm release info` command.
* `version`: Set the specific Y-stream and Z-stream release version for the cluster update. The `status.history` section ensures the policy remains noncompliant until the update completes.
--
+
IMPORTANT: For an EUS-to-EUS update, create a policy for each Y-stream step. For example, updating from 4.20 to 4.22 requires one policy for 4.20-to-4.21 and another for 4.21-to-4.22.

. Apply the cluster version policy to {rh-rhacm} by running the following command:
+
[source,terminal]
----
$ oc apply -f upgrade-ocp-20to21-policy.yaml
----
+
Apply the worker pause policy by running the following command:
+
[source,terminal]
----
$ oc apply -f pause-mcp-policy.yaml
----

. Create a `ClusterGroupUpgrade` CR for the EUS update:
+
[source,yaml]
----
apiVersion: ran.openshift.io/v1alpha1
kind: ClusterGroupUpgrade
metadata:
  name: core-eus-upgrade-4.20-to-4.22
  namespace: openshift-upgrade-policies
spec:
  clusters:
  - spoke1
  managedPolicies:
  - core-pause-worker-nodes
  - core-eus-upgrade-4.22-policy
  enable: false
  remediationStrategy:
    maxConcurrency: 1
    timeout: 10800
----
+
EUS updates might take longer than single y-stream updates, so the timeout is set to 10800 seconds (3 hours).

. Apply the `ClusterGroupUpgrade` CR by running the following command:
+
[source,terminal]
----
$ oc apply -f core-eus-upgrade-cgu.yaml
----
+
Enable the `ClusterGroupUpgrade` CR by running the following command:
+
[source,terminal]
----
$ oc patch cgu core-eus-upgrade-4.20-to-4.22 \
  -n openshift-upgrade-policies \
  --type merge \
  -p '{"spec":{"enable":true}}'
----

. Monitor the EUS control plane update progress by running the following command:
+
[source,terminal]
----
$ oc get cgu core-eus-upgrade-4.20-to-4.22 -n openshift-upgrade-policies -w
----

. On the target cluster, monitor cluster version by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get clusterversion -w
----
+
The cluster version progresses through intermediate versions during the EUS update.
For example, updating from 4.20 to 4.22 goes through 4.21 internally.

. Monitor cluster operator status by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get co
----
+
Cluster operators might take longer to stabilize during EUS updates because of the larger version jump.

. After the control plane update completes, verify control plane health by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get nodes -l node-role.kubernetes.io/master=
----
+
.Example output
[source,terminal]
----
NAME                   STATUS   ROLES                  AGE   VERSION
master-0.example.com   Ready    control-plane,master   60d   v1.30.1
master-1.example.com   Ready    control-plane,master   60d   v1.30.1
master-2.example.com   Ready    control-plane,master   60d   v1.30.1
----
+
Control plane nodes should show the new kubelet version (4.22).

. Verify cluster operators are healthy by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get co | grep -v "True.*False.*False"
----
+
All operators must be healthy before updating worker nodes.

. Verify worker nodes remain at the previous EUS version by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get nodes -l node-role.kubernetes.io/worker=
----
+
.Example output
[source,terminal]
----
NAME                   STATUS   ROLES    AGE   VERSION
worker-0.example.com   Ready    worker   60d   v1.27.10
worker-1.example.com   Ready    worker   60d   v1.27.10
worker-2.example.com   Ready    worker   60d   v1.27.10
----
+
Worker nodes should still show the previous kubelet version (4.20).

. Update OLM operators for the new EUS version:
+
Operators might need multiple version jumps for EUS-to-EUS updates.
+
[source,terminal]
----
$ oc --context=spoke1 get csv -A | grep -v Succeeded
----
+
Update operator subscriptions to EUS-compatible channels by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 patch subscription sriov-network-operator-subscription \
  -n openshift-sriov-network-operator \
  --type merge \
  -p '{"spec":{"channel":"stable-4.22"}}'
----

. Schedule worker node updates:
+
Determine the timing for worker node updates based on your maintenance schedule.
+
Workers can remain at the previous EUS version for an extended period if needed.

. When ready to update worker nodes, create and apply an unpause policy:
+
Follow the procedure in "Pausing and unpausing worker nodes by using {cgu-operator}" to unpause and update workers.

. Monitor worker node updates to completion by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get mcp worker -w
----

.Verification

. Verify all nodes are at the target EUS version by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get nodes
----

. Verify all cluster operators are healthy by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get co
----

. Validate Performance Profiles by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get performanceprofile
----
+
Validate PTP configurations by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get ptpconfig -n openshift-ptp
----
+
Validate SR-IOV network node policies by running the following command:
+
[source,terminal]
----
$ oc --context=spoke1 get sriovnetworknodepolicy -n openshift-sriov-network-operator
----

. Run comprehensive smoke tests for all CNF workloads.

.Troubleshooting

* If the EUS update path is not available, verify you are at the latest z-stream of the source EUS version.
* If the control plane update gets stuck, check for operator compatibility issues with the target EUS version.
* If worker nodes fail to update after unpause, check the `MachineConfigPool` resource status and node logs.
