= Automatic Certificate Management Environment not working (Configuration 1)

[role="_abstract"]
Troubleshoot Automatic Certificate Management Environment (ACME) configuration failures for SPIRE federation by auditing server logs, verifying public domain reachability, and validating the ACME parameters within the SPIRE server custom resource (CR).

Identify the root cause of the federation connectivity failure by reviewing these primary symptoms:

* SPIRE server logs show ACME errors

* Federation endpoint certificate is not valid

.Procedure

. Filter the SPIRE server logs for ACME-related errors by running the following command:
+
[source,terminal]
----
$ oc logs spire-server-0 -c spire-server -n zero-trust-workload-identity-manager | grep -i acme
----

. Verify that the federation domain is reachable from the public internet by running the following command:
+
[source,terminal]
----
$ curl -v https://federation.apps.cluster3.example.com
----

. Check the `SpireServer` custom resource for a valid email address and ACME configuration by running the following command:
+
[source,terminal]
----
$ oc get spireserver cluster -o yaml | grep -A5 acme
----

Infrastructure and configuration constraints::

The following infrastructure and configuration constraints are frequently responsible for the failure to issue federation certificates:

* *Domain accessibility*: Domain not publicly accessible.

* *Network Restrictions*: A firewall or security group is blocking Let's Encrypt from performing validation.

* *Configuration Errors*: The email format in the SpireServer CR is invalid or missing.

* *Rate Limiting*: The ACME provider has temporarily blocked requests; wait for the cooldown period or switch to a staging environment.
