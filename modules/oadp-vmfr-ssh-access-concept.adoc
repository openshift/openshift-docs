= Understanding SSH-based file access

[role="_abstract"]
SSH-based file access provides programmatic file recovery by using standard SSH tools such as `rsync`, `scp`, and `sftp`.
This method is appropriate for automation scenarios, bulk file recovery, and integration with existing backup workflows.

SSH access operates on port 2222 with key-based authentication enforced.
Files are served from a chroot jail environment with read-only FUSE mounts to prevent modification of backup data.
The SSH service accepts connections by using autogenerated or custom SSH key pairs stored in Kubernetes secrets.

== SSH access architecture

The file-serving pod runs an SSH server configured for restricted access:

* Port 2222 is used instead of the standard port 22 to avoid conflicts and clearly identify VMFR traffic.
* Chroot jail isolation limits SSH sessions to the backup file system tree.
* Read-only FUSE mounts enforce data protection at the mount level by using libguestfs `--ro` flag.
* Key-based authentication prevents password-based attacks and integrates with automated recovery scripts.

File paths follow the format `/restores/<date>/<backup_name>/<vm_name>/<path>` where `<date>` is the backup creation timestamp, `<backup_name>` is the Velero backup name, and `<vm_name>` is the virtual machine name.

== Supported tools

rsync:: Synchronizes directories between the backup and local systems with incremental transfer capabilities.
Useful for bulk recovery operations and comparing files across multiple backup versions.

scp:: Copies individual files or directories with a simple syntax.
Appropriate for recovering known file paths without interactive browsing.

sftp:: Provides an interactive file transfer session with commands similar to FTP.
Useful for browsing backup contents before transferring files.

All tools require the `-p 2222` flag to specify the non-standard SSH port.
The private key location is specified with the `-i` flag pointing to the key retrieved from the autogenerated Kubernetes secret.

== Authentication model

SSH access uses key-based authentication only.
Password authentication is disabled for security.

Auto-generated key pairs:: The VMFR controller creates an SSH key pair and stores it in a Kubernetes secret.
The private key is retrieved from the secret and used by SSH tools to authenticate.
The default username is `oadp`.

Custom key pairs:: Users can provide a custom username and public key in the `VirtualMachineFileRestore` CR.
The private key is managed by the user and not stored in the cluster.
This enables integration with existing SSH key management systems.

== When to use SSH access

SSH access is appropriate for the following scenarios:

Automation and scripting:: When file recovery operations are part of automated disaster recovery workflows or scheduled validation processes.

Bulk file recovery:: When recovering large directories or many files where web browser download would be inefficient.

Directory synchronization:: When by using `rsync` to maintain local copies of backup contents or compare file versions across multiple backups.

Integration with existing tools:: When recovery operations integrate with backup automation that already uses SSH-based tools.

Command-line preference:: When administrators prefer command-line tools and scripted workflows over graphical interfaces.

== Performance benefits

SSH-based tools provide performance advantages for large transfers:

* `rsync` transfers only changed portions of files with incremental mode enabled.
* Compression can be enabled with the `-z` flag to reduce network transfer time.
* Bandwidth limits can be enforced with the `--bwlimit` flag to avoid saturating network links.
* Progress indicators provide transfer status for long-running operations.

== Security model

SSH access provides defense-in-depth security:

* Key-based authentication prevents brute-force password attacks.
* Chroot jail limits access to the backup file system tree.
* Read-only mounts prevent modification or corruption of backup data.
* Non-standard port 2222 reduces exposure to automated SSH scanning.
* Kubernetes RBAC controls which users can create `VirtualMachineFileRestore` CRs and access credentials.
