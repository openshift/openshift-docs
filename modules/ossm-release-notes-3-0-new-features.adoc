= {SMProductName} 3.0 new features and enhancements

This release adds new features, addresses Common Vulnerabilities and Exposures (CVEs), and is supported on {ocp-product-title} 4.14 and later.

For a list of supported component versions and support features, see "Service Mesh 3.0 feature support tables".

For a complete list of changes between {SMProduct} 2 and {SMProduct} 3, see "Important information to know if you are migrating from OpenShift Service Mesh 2.6".

== Istioctl with support for select platforms & commands

This release adds support for the Istioctl command line utility, with support for select platforms and commands included. This utility offers several commands for inspecting, debugging and diagnosing issues with a service mesh.

[NOTE]
====
Istio installation using the `istioctl` utility is not supported.
====

== New Istio distribution and operator

{SMProduct} 3.0 is based on a Red Hat distribution of the link:https://istio.io/[Istio project] and is deployed with a new operator for Istio based on the Sail Operator project that is part of the istio-ecosystem organization. The Sail Operator includes a new set of custom resource definitions (CRDs) for managing Istio. For example, the `Istio` CRD replaces the `ServiceMeshControlPlane` CRD in previous releases of {SMProduct}.

== Support for multi-cluster

This release introduces support for Istio's multi-cluster deployment models, namely: multi-primary, primary-remote, and an external control plane. The federation feature introduced in {SMProduct} 2.1 is not available in {SMPRoduct} 3.0.

== Multiple control planes in a cluster

This release adds support for the Istio feature of multiple control planes in a single cluster. This replaces the `MultiTenant` deployment model (mode) in {SMProduct} 2.

== Revision based upgrades

This release introduces support for a `RevisionBased` strategy, which is based on the Istio canary control plane upgrade procedure. This enables a new Istio control plane to be created alongside the existing Istio control plane so that workloads may be migrated incrementally. This strategy is in addition to the `InPlace` upgrade strategy, which was the only supported approach for upgrading {SMProduct} 2.6 and earlier. The upgrade strategy is configured using the `spec.updateStrategy` in the `Istio` resource.

== IstioCNI CRD

This release introduces the `IstioCNI` CRD, which is used to manage the lifecycle of the Istio CNI DaemonSet. A single instance of this resource must be created per cluster to configuring traffic redirection for pods in the mesh. The Istio CNI lifecycle is independent of the Istio control plane(s).

== Dual Stack (IPv4/IPv6)
:FeatureName: Dual Stack
include::../snippets/technology-preview.adoc[]

This release includes IPv4/IPv6 Dual Stack support as a technology preview. This aligns with the 'Alpha' status of the Istio upstream project, and is feature-complete for Istio when using sidecars. Dual Stack helps organizations smoothly transition to IPv6, while still maintaining compatibility with their existing IPv4 setup.

In {SMProduct} 3.0, dual stack is disabled by default. You can enable it with specific configuration changes, such as these in the following example:
+
.Example configuration for IPv4/IPv6 Dual Stack
[source,yaml]
----
spec:
  values:
    meshConfig:
      defaultConfig:
        proxyMetadata:
          ISTIO_DUAL_STACK: "true"
    pilot:
      ipFamilyPolicy: RequireDualStack
      env:
        ISTIO_DUAL_STACK: "true"
----

== Istio Ambient mode
:FeatureName: Istio Ambient mode
include::../snippets/developer-preview.adoc[]

Istio Ambient Mode provides a sidecar-less service mesh architecture that reduces resource overhead, simplifies operations, and allows incremental adoption without application changes. It maintains security and observability through a layered security model with mTLS and authorization. OpenShift Service Mesh 3 operator includes deploying the Ambient profile as a developer preview feature using the community ztunnel image. However, the ambient profile should not be used on clusters with production workloads or for multi-control plane use cases. Additionally, the community ztunnel image does not work on IBM Power and Z Platforms.
