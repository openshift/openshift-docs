== Identifying and removing {op-system-base} worker nodes

With the introduction of {product-title} 4.19, package-based {op-system-base} worker nodes are no longer supported. Attempting to update your cluster while having these nodes up and running will prevent you from successfully completing the update.

The following procedure describes how to identify {op-system-base} nodes for cluster removal.

.Procedure

. Run the following command to identify nodes running on {op-system-base}:
+
[source,bash]
----
<short script that iterates across nodes based on ROLE (worker) and grabs NAME and SYSTEMINFO:OS IMAGE (RHEL)>

$ oc get nodes | grep -i "worker"
----

. Cordon the nodes with the following command:
+
[source,terminal]
----
$ oc adm cordon <node_name(s)>
----

. Drain the nodes using the following command:
+
[source,terminal]
----
$ oc adm drain <node_name(s)> --force --delete-emptydir-data --ignore-daemonsets
----

. Ensure there are no current jobs or cronjobs for these nodes.

. Run the following command to back up node information into a YAML file:
+
[source,terminal]
----
$ oc get node <node_names(s)> -o yaml > backupnodes.yaml
----
+
This information might be useful during provisioning of {op-system} worker nodes.

. Power off the nodes.

. Delete the nodes from the cluster with the following command:
+
[source,terminal]
----
$ oc delete node <node_name(s)>
----

. Decommission the nodes.

. Optional: Destroy all data from the worker nodes and delete all software installed on them by running the following command:
+
[source,terminal]
----
# nohup shred -n 25 -f -z /dev/<HDD>
----

For more information on this process, see the following link:https://access.redhat.com/solutions/4976801[Red{nbsp}Hat Knowledgebase solution].
