= Understanding OpenAPI callback and callback state
:context: serverless-logic-openAPI-callback
include::_attributes/common-attributes.adoc[]

You can use OpenAPI callbacks and Callback states to handle actions that require waiting for events. Both methods perform an action and wait for an event to continue the workflow, but they achieve this differently. Both are effective for `fire & wait-for-result` operations.

OpenAPI callback manages asynchronous, external responses from other services. Your service makes a request to another service. That service responds to specific events by sending a request back to your callback URL. Whereas callback state, executes an action that triggers an external service to send a CloudEvent back to the workflow. The workflow performs an action and waits for a CloudEvent to return, which then resumes the workflow.

When the workflow makes an asynchronous service call, it includes a callback URL in the request body. The external service uses this callback URL to send a CloudEvent back to the workflow when the action is done.

To make sure the workflow can identify the correct event to wait for, the external service should add the workflow instance ID in the CloudEvent header and correlate the event with the workflow instance using IDs or metadata.
